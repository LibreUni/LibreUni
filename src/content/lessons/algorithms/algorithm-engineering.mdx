# Algorithm Engineering

An $O(n \log n)$ algorithm in C++ takes 45 seconds on a 100M-element dataset. A colleague's $O(n^2)$ implementation handles the same data in 30 seconds. Asymptotic analysis predicts the opposite ordering.

The abstract RAM model counts arithmetic and memory operations uniformly. Real hardware penalises non-local access: a cache miss costs ~60 ns (main memory) vs ~1 ns (L1 cache). An $O(n \log n)$ algorithm that scatters memory access across a large working set can lose to a cache-friendly $O(n^2)$ algorithm at practical input sizes. Branch mispredictions, failure to auto-vectorize, allocation overhead, and NUMA effects compound the gap.

Algorithm engineering, formalised by Peter Sanders and colleagues in the 1990s [Sanders 2009], treats the gap between algorithmic theory and fast software as a research problem in its own right. It proposes a cycle: **design** an algorithm with asymptotic guarantees, **analyze** its resource demands beyond operation counts, **implement** with representation choices that match the hardware, **experiment** with rigorous benchmarks, and **refine** the design based on measurement.

**Prerequisites:** asymptotic analysis (big-O notation, RAM model), basic graph representations, cache hierarchy concepts.

## Specification Contracts

An algorithm is not complete when its recurrence is solved. An implementation has:

- **Input contract:** accepted types, ranges, invariants (e.g., sorted input, connected graph).
- **Output contract:** return value, side effects, ownership, stability.
- **Error behavior:** overflow, malformed input, resource exhaustion.
- **Performance contract:** expected latency, memory, allocation count.

<MathStatement id="contract-dijkstra" kind="definition" title="Implementation contract: Dijkstra with binary heap" proofId="contract-dijkstra-just">
For graph $G = (V,E)$ with nonnegative integer weights in $[0, 2^{31}-1]$ and source $s \in V$:
- **Input:** $G$ in CSR format, `weights` array of length $|E|$, `source` index.
- **Output:** array `dist[0..V-1]` where `dist[v] = \delta(s,v)$ (or $2^{31}-1$ for unreachable).
- **Error:** iff $G$ contains a negative edge, behavior is undefined.
- **Performance:** $O((V+E)\log V)$ comparisons, $O(V+E)$ memory, at most $2V$ heap operations.
</MathStatement>

The performance contract is the most frequently omitted clause. Without it, a user cannot distinguish between an implementation that is slow due to a bug and one that is slow because the contract makes different trade-offs (e.g., prioritising low memory over throughput).

## Representation Selection

| Representation | Edge existence | Iterate neighbors | Memory | Update |
|---------------|---------------|-------------------|--------|--------|
| Adjacency matrix | $O(1)$ | $O(V)$ | $O(V^2)$ | $O(1)$ |
| Adjacency list | $O(\deg(v))$ | $O(\deg(v))$ | $O(V+E)$ | $O(1)$ |
| Compressed sparse row (CSR) | $O(\log \deg(v))$ | $O(\deg(v))$ | $O(V+E)$ | $O(V+E)$ amortized |

**CSR format.** Two arrays: `offsets[0..V]` (start index in `edges` for each vertex) and `edges[0..2E-1]` (target and weight pairs). Successive neighbors are contiguous in memory, maximising cache efficiency for sequential scans.

### Worked example: CSR traversal

Consider a graph with $V = 4$ vertices and edges $(0,1), (0,3), (1,0), (1,2), (1,4), (2,1), (3,1)$. CSR stores:

- `offsets = [0, 2, 5, 6, 7]`
- `edges = [1, 3, 0, 2, 4, 1, 1]`

Iterating neighbors of vertex 1 reads `edges[offsets[1]:offsets[2]] = edges[2:5] = [0, 2, 4]`. This is a contiguous memory access of three integers — one cache line at most. An adjacency list would follow three linked-list nodes scattered across the heap, likely causing three cache misses.

## Cache-Oblivious Algorithms

An algorithm is *cache-oblivious* if it achieves asymptotically optimal cache complexity without knowing cache parameters. The *ideal-cache model* [Frigo et al. 1999] assumes a two-level memory hierarchy with automatic LRU replacement, cache line size $B$, and cache size $M$.

<MathStatement id="cache-oblivious-def" kind="definition" title="Cache-oblivious algorithm" proofId="cache-oblivious-claim">
An algorithm with work $W(n)$ and cache complexity $Q(n; M, B)$ in the ideal-cache model achieves optimal cache usage if $Q(n; M, B) = \Omega(W(n)/B)$.
</MathStatement>

<MathProof id="cache-oblivious-claim" for="cache-oblivious-def" title="Cache-oblivious complexity bound" collapsible>
The lower bound $\Omega(W(n)/B)$ arises because each cache line of size $B$ can serve at most $B$ memory accesses. If the algorithm touches $W(n)$ words, it must incur at least $W(n)/B$ cache misses. An algorithm that achieves this bound is *cache-optimal* because it makes the best possible use of each loaded cache line.

The key property: the bound holds for *any* $M$ and $B$, provided the cache uses LRU or FIFO replacement and is *tall* ($M = \Omega(B^2)$). The algorithm does not need to know $M$ or $B$ — it exploits the cache automatically through recursive divide-and-conquer that keeps working sets small at each recursion depth.
</MathProof>

**Example: Recursive matrix transpose.** Divide the matrix into quadrants and recursively transpose each. At the leaves where the submatrix fits in a cache line, the transpose is cache-efficient. The cache complexity is $O(n^2/B)$, matching the lower bound [Gustavson 1997].

### Worked example: blocked matrix multiplication

Naive $O(n^3)$ matrix multiply computes $C[i][j] = \sum_k A[i][k] \cdot B[k][j]$. The inner loop over $k$ reads $A$ row-wise (good) but $B$ column-wise (bad — each access is a cache miss).

Blocked multiplication, also known as tiled multiplication [Cormen et al. 2022, Ch. 4], partitions matrices into $b \times b$ blocks. Each block of $B$ fits in cache. The inner loop loads one block of $A$ and one block of $B$, computes a partial result, and writes one block of $C$. For $b \approx \sqrt{M/3}$, all three blocks fit in cache, and the cache complexity becomes $O(n^3/(B\sqrt{M}))$, compared to $O(n^3/B)$ for the naive version.

import time
import random

def naive_mult(A, B, n):
    C = [[0.0] * n for _ in range(n)]
    for i in range(n):
        for k in range(n):
            aik = A[i][k]
            for j in range(n):
                C[i][j] += aik * B[k][j]
    return C

def blocked_mult(A, B, n, block=32):
    C = [[0.0] * n for _ in range(n)]
    for i in range(0, n, block):
        for k in range(0, n, block):
            for j in range(0, n, block):
                for ii in range(i, min(i+block, n)):
                    for kk in range(k, min(k+block, n)):
                        aik = A[ii][kk]
                        for jj in range(j, min(j+block, n)):
                            C[ii][jj] += aik * B[kk][jj]
    return C

n = 128
A = [[random.random() for _ in range(n)] for _ in range(n)]
B = [[random.random() for _ in range(n)] for _ in range(n)]

t0 = time.time()
C1 = naive_mult(A, B, n)
t1 = time.time()
print(f"Naive: {t1-t0:.3f}s")

t0 = time.time()
C2 = blocked_mult(A, B, n, block=32)
t1 = time.time()
print(f"Blocked (b=32): {t1-t0:.3f}s")

# Verify correctness
max_err = max(abs(C1[i][j] - C2[i][j]) for i in range(n) for j in range(n))
print(f"Max error: {max_err:.2e}")

<MathStatement id="blocked-mm-complexity" kind="theorem" title="Cache complexity of blocked matrix multiplication" proofId="blocked-mm-proof">
For $n \times n$ matrices multiplied in $b \times b$ blocks on a machine with cache line size $B$ words and cache size $M$ words, the cache complexity is
$$Q(n) = O\left(\frac{n^3}{B \sqrt{M}}\right)$$
when $b = \Theta(\sqrt{M})$ and $M = \Omega(B^2)$.
</MathStatement>

<MathProof id="blocked-mm-proof" for="blocked-mm-complexity" title="Proof of cache complexity bound" collapsible>
Each block multiplication loads $3b^2$ words ($A$-block, $B$-block, $C$-block). With $b = \Theta(\sqrt{M})$, all three blocks fit in cache simultaneously ($3b^2 \leq M$). Each block is loaded $O(n^3 / b^3)$ times, and each load incurs $O(b^2 / B)$ cache misses (streaming $b^2$ contiguous words in $B$-word lines). Total misses: $O((n^3 / b^3) \cdot (b^2 / B)) = O(n^3 / (b B)) = O(n^3 / (B \sqrt{M}))$. The naive algorithm incurs $O(n^3 / B)$ misses because each column-wise access of $B$ misses once per row.
</MathProof>

## Counterexample: When Cache-Oblivious Algorithms Underperform

Cache-oblivious algorithms assume a *tall cache*: $M = \Omega(B^2)$. On real hardware, this assumption holds for L1 ($M/B \approx 32\text{KB} / 64\text{B} = 512 \gg 1$) but may fail for L3 on some embedded or GPU architectures where the cache line is large relative to total cache size.

**Example: radix sort vs cache-oblivious sorting.** A tuned radix sort processes data in sequential passes, each pass reading and writing the entire array linearly. Modern CPUs prefetch sequential accesses, hiding cache miss latency. A recursive cache-oblivious sorting algorithm (e.g., funnel sort) may incur more TLB misses and branch mispredictions than a simpler sequential algorithm, even though its cache complexity is asymptotically optimal.

<MathStatement id="failure-mode" kind="remark" title="When cache-oblivious loses">
Cache-oblivious algorithms can underperform when:
1. The cache is not tall ($M = O(B)$), so the recursion cannot exploit locality.
2. Sequential prefetching already hides miss latency, making the recursive overhead unnecessary.
3. The working set fits entirely in cache — asymptotic cache complexity is irrelevant.
4. TLB pressure from recursive subdivision exceeds TLB reach.
</MathStatement>

The practical lesson: profile before optimising. A cache-oblivious algorithm is not automatically faster than a simple cache-aware one on your specific hardware.

## Numeric Safety

A correct implementation must guard against arithmetic overflow, especially in sentinel-based algorithms like Dijkstra.

**Overflow detection.** Before computing $d[u] + w$, check if $d[u] > \text{MAX} - w$ (where MAX is the sentinel for infinity). Use a wider type (e.g., `int64` for `int32` inputs) when possible [Cormen et al. 2022, Ch. 24].

**Sentinel design.** Use $\infty = \text{MAX}/2$ to allow safe addition without overflow. Alternatively, use a separate `visited` flag and skip unreachable vertices.

**Probabilistic stability.** For algorithms multiplying probabilities, work in log-space: $\log(pq) = \log p + \log q$, avoiding underflow.

## Benchmark Design

Rigorous benchmarking is the only way to validate the engineering cycle. A protocol must exclude confounding factors.

<MathStatement id="benchmark-protocol" kind="definition" title="Rigorous benchmark protocol">
1. **Specify the question.** (e.g., "Which priority queue gives lower median latency for Dijkstra on sparse graphs with $V=10^6$, $E=10V$?")
2. **Pre-register metrics.** Median latency, p95, allocation rate, cache misses.
3. **Control variables.** Hardware, compiler flags, input generator, random seed, JVM warmup.
4. **Warm-up.** Execute the algorithm on a separate input before timed trials.
5. **Randomize order.** Interleave trials from different implementations.
6. **Report distributions.** Box plots or violin plots, not just means.
7. **Validate correctness.** Every timed run must produce correct output.
</MathStatement>

5. Allocate two copies of the CSR in each socket's local memory (doubling memory but halving cross-socket traffic). Or partition the vertex set by NUMA node and store each partition's CSR locally, with a small remote cache for cross-partition edges.

Merge $k$ sorted runs using a tournament tree. Expected passes: 1 for sorting, 1 for merging. Cache complexity of internal sort: $O(n/B \log_2(n/M))$.

7. Protocol: (a) fix hardware, compiler flags, and input generator; (b) report machine specs (CPU model, cache sizes, RAM); (c) 10 warm-up iterations before measurement; (d) randomise trial order across implementations; (e) at least 30 trials per configuration; (f) report min, median, p95, and full distribution; (g) validate correctness of every timed run. Threats: omitted warmup, single input size, only mean reported, no statistical test, compiler optimisations not stated.

8. **Hybrid representation.** Store the graph as an adjacency matrix during phase 1 and convert to CSR at the phase boundary. Conversion cost: $O(V^2)$ for matrix-to-CSR (scanning all $V^2$ entries) plus $O(V+E)$ for CSR construction. The conversion is worthwhile when the number of edge-existence checks exceeds $V^2 / (V+E)$ — for a dense matrix, that is roughly $V$ checks.

9. **Prefetcher-aware recursion.** The recursive subdivision of matrix transpose creates access patterns with stride that changes at each level. To preserve prefetcher benefits, use a cache-oblivious layout (e.g., recursive Morton-order or Z-order layout) that converts the recursive subdivision into a linear memory scan at the leaf level. The Z-order layout ensures that the leaf-level blocks are contiguous in memory, enabling sequential prefetching at the cache-line granularity [Frigo et al. 1999].

10. **Diagnosis.** The road network has low degree, so its working set for CSR is small (edges stored contiguously, good cache behaviour). The social network has high degree and a power-law distribution: a few vertices have millions of neighbours. The working set for CSR on the social network is $O(V + E)$ — the high-degree vertices scatter memory access across a large contiguous range, but the range itself is large enough to exceed L3 cache. The 8× slowdown is explained by the working set exceeding L3 capacity ($\approx$ 8 MB), causing DRAM-bound access (60 ns vs 12 ns).

## Deliberately Omitted

- **Just-in-time compilation effects on algorithm performance.** The benchmark section notes warmup but does not analyse JIT compilation, inlining, and escape analysis effects. These are language-runtime topics outside algorithm engineering — a JIT-aware engineering cycle would require controlling the JIT state (e.g., tiered compilation thresholds, deoptimisation guards) as a separate experimental variable. Omitted because the standard engineering cycle should apply regardless of execution model.
- **Energy-aware algorithm design.** Power consumption as a function of algorithm choice is a growing concern in green computing. This lesson focuses on time and memory as the primary resources. Energy adds a third dimension (joules per operation, DRAM refresh power, core sleep states) that is not yet captured by the ideal-cache or RAM models. Omitted because the measurement methodology differs substantially (power meters, RAPL counters) and belongs in a dedicated treatment.
- **Parallel algorithm engineering with OpenMP/MPI.** Shared-memory and distributed-memory parallelism introduce communication costs, load balancing, synchronisation overheads, and false-sharing effects. These are essential for modern performance but require a separate treatment of parallel complexity models (e.g., work-depth model, BSP) and cache coherence protocols (MESI, MOESI). Omitted to maintain focus on single-threaded algorithm engineering.
- **Persistent memory and NVM-aware algorithms.** Emerging non-volatile memory (Optane, CXL-attached) introduces asymmetric read/write costs, durability guarantees, and wear-leveling constraints not covered by the ideal-cache model. The engineering cycle adapts (e.g., write-combining, flushes, fences), but the hardware is still evolving. Omitted pending stabilisation of NVM programming models.
- **Hardware performance counter analysis.** Cycle-level profiling with `perf stat`, PMC interpretation, and top-down microarchitecture analysis (TMA) is a separate skill. This lesson focuses on algorithmic and representation-level engineering that is portable across microarchitectures. Omitted because PMC analysis is a debugging tool, not a design methodology.
- **Amortised analysis of dynamic data structures.** The CSR update cost ($O(V+E)$ amortised) is stated but not derived. Amortised analysis (potential method, accounting method) is a separate topic that belongs in a data structures module. Omitted because the lesson's focus is on the engineering cycle, not on formal complexity analysis techniques.
