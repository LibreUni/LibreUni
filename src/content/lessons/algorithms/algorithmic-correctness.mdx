# Algorithmic Correctness

An algorithm is correct when it halts on every input in its domain and returns a result satisfying its specification. A few successful examples are not a proof, and a complexity bound does not imply correctness. This lesson develops proof patterns that scale from loops over arrays to graph algorithms, dynamic programs, greedy choices, randomized procedures, and reductions.

## Specifications

A specification defines input precondition, output postcondition, and permitted effects. Sorting requires that output is ordered and is a permutation of input; sortedness alone permits losing records. A shortest-path algorithm must return distances or unreachable markers and, if paths are returned, each path must be valid and have the reported weight. A data-processing algorithm may promise stability, ownership, or no mutation in addition to value correctness.

Separate partial correctness from termination. Partial correctness says if the algorithm halts, the output satisfies the postcondition. Total correctness adds a proof that it halts for every valid input. A loop invariant plus a decreasing well-founded measure often establishes both. If a loop can preserve the invariant while cycling, the postcondition proof is insufficient.

## Loop invariants

An invariant P holds before the first iteration, is preserved by each iteration, and combines with loop termination to imply the postcondition. For insertion sort, after processing positions below i, the prefix is sorted and is a permutation of the original prefix; the suffix remains untouched. The next iteration inserts A[i] into the sorted prefix, preserving the invariant.

Invariant discovery starts from the desired final fact and weakens it to a statement true initially. It should be strong enough to support the next operation. A statement such as “some elements are sorted” is too weak to prove that shifting and insertion produce a sorted prefix. Include exact boundaries and ownership of elements.

**Worked binary-search invariant.** For a sorted array and half-open interval [lo,hi), if target exists, its index lies in that interval. Each comparison discards a half that cannot contain target, and the interval strictly shrinks. When empty, target is absent; when found, return its index. Midpoint arithmetic must avoid overflow, and the implementation must use the same endpoint convention as the proof.

## Induction and recursion

Recursive correctness uses induction on a size measure. Prove base cases directly. Assume the algorithm correctly solves all smaller valid instances, show recursive calls meet their preconditions, invoke the hypothesis, and prove the combine step. The induction measure must strictly decrease; “the subarray is smaller” fails if partition can return the original range.

For merge sort, induction proves child arrays sorted and the merge invariant proves their union sorted. For a graph DFS, induction over reachable subgraphs is less direct because recursion can revisit vertices; use a visited invariant instead. Choose the proof principle that matches the algorithm’s state, not the one that sounds most familiar.

## Greedy exchange and cuts

Greedy proofs show that an optimal solution can be transformed to contain the algorithm’s choice. State feasibility after exchange, objective non-worsening, and the residual problem. Interval scheduling exchanges the first selected interval; MST exchanges a cut edge; Huffman exchanges deepest sibling leaves. A local improvement argument without an exchange preserving global constraints is not enough.

## Dynamic programming proofs

DP correctness follows from a state definition and optimal substructure. Prove that every feasible solution falls into one transition case, and every transition constructs a feasible solution. Then show the optimal value is the maximum/minimum of transition values. The table order must compute dependencies first. Reconstruction needs a separate proof that stored choices correspond to the value and produce a feasible witness.

## Graph algorithm invariants

BFS maintains that dequeued vertices have minimum unweighted distance and that parent pointers form shortest discovery paths. Dijkstra maintains settled distances final under nonnegative weights. Bellman-Ford maintains correctness for paths with at most k edges after k passes. Max-flow maintains capacities and conservation; residual reachability certifies optimality. Naming the invariant prevents silently applying an algorithm outside its assumptions.

## Randomized correctness

A Las Vegas algorithm has a deterministic correctness invariant for every random outcome, with randomness only affecting cost. A Monte Carlo algorithm proves (Pr[text()error()] at most epsilon) over random bits for each valid input. If the algorithm outputs a certificate, verify the certificate deterministically to obtain one-sided error or a Las Vegas wrapper. State whether random bits are independent and how seeds are generated.

## Certificates and verification

A certificate is evidence that can be checked independently. An MST certificate combines tree shape and non-tree edge cycle inequalities; a max-flow certificate combines feasible flow and equal cut capacity; a SAT assignment evaluates clauses; a sorted output can be checked for order and permutation. Certificate checking is not necessarily as hard as finding the solution, but it must cover all postcondition clauses.

## Termination measures

A termination proof identifies a measure into a well-founded set that decreases on every iteration or recursive call. Natural-number interval length works for binary search; remaining unprocessed elements works for a scan; recursion depth alone does not, because depth increases while descending. For nested loops, prove the inner loop terminates for each fixed outer state, then show the outer measure decreases after the inner loop completes.

A measure can be lexicographic. An algorithm may first reduce the number of unresolved components and, within one component count, reduce a finite queue or interval. Prove that no transition increases an earlier coordinate while decreasing a later coordinate in a way that permits an infinite sequence. This is useful for rewriting systems, graph algorithms with phases, and algorithms that alternate cleanup and progress.

## Total correctness under errors

If inputs can violate the precondition, either reject them before the main proof or extend the specification to define behavior. Integer overflow, malformed edges, NaN comparisons, and missing dictionary keys can break mathematical assumptions. A production algorithm should validate or constrain input and state the cost of validation. An algorithm that is correct only for well-formed input is acceptable if the interface makes that contract explicit.

Exception and cancellation behavior also affects total correctness. If a caller cancels a recursive search, the algorithm may not return a result but should restore ownership and release resources according to its contract. If a comparator or allocation fails, partial mutation may be visible unless strong exception safety is proved. These are not separate from correctness when the algorithm is a library component.

## Proof debugging

When a proof fails, find the first unproved implication rather than patching the conclusion. A loop invariant may be too weak, a subproblem may not satisfy its precondition, or a complexity recurrence may omit copying. Construct a small counterexample targeted at the gap, run a reference implementation, and revise the specification or algorithm. A proof review and an adversarial test should inform each other: the invariant predicts which state is dangerous, while the counterexample shows whether the prediction is real.

## Correctness report format

For a complete algorithm report, give the problem/model, pseudocode or executable description, preconditions/postconditions, invariant or induction statement, termination measure, proof, complexity by parameter and model, edge cases, and test/certificate strategy. If randomness, approximation, or amortization is used, add the relevant quantifiers and assumptions. This format lets a reader evaluate both mathematical validity and implementation readiness.

## Exercises

1. Write a full specification for sorting.
2. Give insertion-sort loop invariant and proof.
3. Prove binary-search termination and correctness.
4. State a DP proof for LCS or knapsack.
5. Give a greedy exchange proof.
6. List BFS/Dijkstra/max-flow invariants.
7. Contrast Las Vegas and Monte Carlo correctness.
8. Design an independently checkable MST certificate.

### Diagnostic solutions


1. A sorting specification must name the input sequence and comparator, require an output ordered under that comparator, and require that the output is a permutation of the input. Mutation, stability, and error behavior are separate clauses.

2. Before each insertion-sort iteration, the prefix `[0,i)` is sorted and is a permutation of the original prefix, while the suffix is unchanged. Shifting larger prefix values and inserting `A[i]` preserves both clauses; at `i=n` the whole array is sorted and permuted.

3. For binary search, maintain that a target, if present, lies in `[lo, hi)`. The midpoint comparison retains a valid half and strictly shrinks `hi-lo`. An empty interval proves absence; equality returns a valid index. Sortedness is the precondition.

4. For dynamic programming, partition feasible solutions by their final choice or last characters. Show that each case maps to a smaller state and that the optimum is the best transition. Compute states in dependency order and prove reconstruction follows stored optimal transitions.

5. For a greedy proof, take an optimum, exchange the greedy choice into its first position while preserving feasibility and not reducing value, then apply induction to the residual instance. Identify the exact structural property that makes the exchange valid.

6. BFS finalizes distance when a vertex is dequeued; Dijkstra settles a label under nonnegative edges; Bellman-Ford reasons by edge count; max-flow preserves capacity and conservation, and a reachable cut after no residual path certifies optimality.

7. Las Vegas algorithms always return correct output and randomness changes runtime. Monte Carlo algorithms have a bounded or controlled runtime but a probability of wrong output. A certificate checker can make some Monte Carlo outcomes one-sided verifiable procedures.

8. For an MST certificate, check `n-1` edges, acyclicity, and connectivity. Then for every non-tree edge `e`, verify that its weight is at least the maximum edge weight on the tree path between its endpoints. These conditions rule out every improving cycle exchange.
