# Algorithms Capstone: Design a Reliable Route Planner

Design a route-planning engine for a transportation network with directed and undirected links, travel times, tolls, capacity constraints, temporary closures, and user preferences. The engine must answer shortest-time queries, identify unreachable destinations, produce a route certificate, and support a batch mode that builds a minimum-cost infrastructure plan. It must reject invalid input and state which query variants are exact, approximate, or unsupported.

## Requirements and models

Define a graph model with vertex IDs, edge IDs, direction, travel time, toll, capacity, and active interval. Decide whether weights are nonnegative, whether time-dependent travel is FIFO, and whether a route may repeat vertices. A shortest route under time is not necessarily shortest under toll or a combined scalar; specify the objective and units. If multiple routes tie, define deterministic tie-breaking or return any optimum.

The service supports static batch queries and online updates. Static queries may preprocess; online updates may invalidate distances. A batch mode selects links connecting all depots at minimum total infrastructure cost, which is an MST problem only if the graph is undirected and the objective is total edge weight. A directed or reliability-constrained variant needs a different model. Make this distinction explicit.

## Algorithm selection

Use BFS only for unit costs. Use Dijkstra for nonnegative static travel times; use Bellman-Ford if negative reduced costs are truly meaningful and negative cycles are handled; use DAG relaxation when the route graph is acyclic. For many sources on sparse graphs, consider Johnson reweighting; for dense small graphs, Floyd-Warshall may be appropriate. Use Kruskal or Prim for the undirected infrastructure plan and max flow for capacity feasibility if the project includes a throughput query.

For online edge updates, a correct baseline recomputes from the changed source. A sophisticated incremental algorithm may reduce work but requires a new proof and update contract. Do not reuse stale distances after a decrease/increase without a validity argument. If approximate routes are allowed, state an additive or multiplicative guarantee and how updates affect it.

## Correctness obligations

For shortest paths, provide a label invariant and predecessor reconstruction proof. For Dijkstra, prove settlement finality under nonnegative weights. For Bellman-Ford, prove the bounded-edge-path induction and negative-cycle detection. For MST, prove the cut property and forest invariant. For max flow, prove residual updates and return a cut certificate. If the planner combines constraints, prove that the reduction preserves feasible routes in both directions.

The route certificate should include the vertex/edge sequence, total cost, and enough evidence to validate it. A checker verifies consecutive endpoints, active-time and capacity constraints, no forbidden repeats, and arithmetic total. A shortest-path certificate may include final labels and edge inequalities; an MST certificate includes cycle inequalities; a flow certificate includes a cut.

## Data structures

Use adjacency lists with stable edge IDs for sparse networks, a binary heap or specialized queue for frontier labels, predecessor arrays for reconstruction, and union-find for Kruskal. If edge updates are frequent, use an update log and versioned graph snapshots. A priority queue must handle stale entries or implement decrease-key correctly. A graph index must preserve endpoint validity when edges are removed or compacted.

For multi-criteria routes, a single scalar label is insufficient unless a dominance property is proved. Maintain nondominated labels (time,toll), but the frontier can grow exponentially. Approximate discretization or a user-selected scalar objective can bound work; report the loss and memory tradeoff.

## Testing plan

Build a brute-force oracle for tiny graphs by enumerating simple paths, spanning trees, and feasible flows. Generate disconnected graphs, parallel edges, self-loops, zero and negative weights, equal ties, cycles, and sparse IDs. Verify returned routes against certificates and compare algorithms where their assumptions overlap. For negative-cycle cases, test explicit undefined status rather than expecting a finite answer.

For dynamic updates, compare the incremental result with recomputation after every update. Test edge insertion, deletion, weight changes, activation interval boundaries, and updates that disconnect the source. Retain old snapshots if the API promises versioned queries and verify they do not change after later updates.

## Performance study

Vary vertices, edges, density, degree skew, source count, update frequency, weight distribution, and query selectivity. Measure preprocessing, query latency, p50/p95/p99, heap operations, edge relaxations, memory, and certificate-checking cost. Compare adjacency list/heap with dense matrix/array approaches on regimes where each should be competitive. Keep graph instances identical across implementations.

An asymptotic claim must name its model: Dijkstra with binary heap is O((V+E)log V), array-based dense Dijkstra O(V²), Kruskal O(E log E), and BFS O(V+E). Measurements may differ because of locality and constants. Explain any crossover rather than presenting a single universal winner.

## Failure and operations

If a route uses an inactive edge, inspect snapshot/version mismatch or update propagation. If a returned path’s cost is lower than the sum of its edges, check overflow or unit conversion. If Dijkstra changes a settled label, find a negative edge or stale-queue bug. If MST has fewer than component-count-adjusted edges, inspect vertex mapping or disconnected status. If a max-flow certificate disagrees with value, inspect residual reverse updates and cut state.

Deploy with a known-good fallback algorithm and bounded resource budgets. Reject graphs that exceed memory or numeric limits. Cache query results only with a graph-version key. A cache hit under a newer edge weight is not a correct route. Log algorithm choice, graph version, assumptions, and certificate status without logging sensitive user data.

## Deliverables and rubric

Submit problem specification (15%), algorithm selection and alternatives (15%), pseudocode/implementation (20%), correctness proofs/certificates (20%), tests and failure analysis (15%), complexity/performance study (10%), and reproducibility/clarity (5%). Include one deliberately invalid instance and show a safe rejection; include one instance where a tempting algorithm is invalid and explain why.

## Oral defense prompts

Explain why Dijkstra is or is not valid, how a predecessor path is guaranteed to be optimal, why an MST is not a shortest-path tree, how an update invalidates cached labels, what a certificate independently proves, and which workload would cause your chosen data structure to lose. A strong defense names assumptions before conclusions and distinguishes theorem, measurement, and heuristic.

## Detailed project workflow

Begin with a small formal instance and solve it by exhaustive search. Use that instance to test the graph parser, objective calculation, path reconstruction, and certificate checker before implementing the optimized algorithm. Then add one feature at a time: negative or time-dependent weights, update versions, capacity constraints, and multiple criteria. At each step, update the specification and identify which prior proof no longer applies.

Create a baseline implementation that is intentionally simple, even if slow. A brute-force simple-path solver on tiny graphs, Bellman-Ford for general signed weights, and a copied graph per update provide reference behavior. The optimized implementation should be compared against these baselines on random and hand-designed cases. If the results disagree, preserve the smallest graph and classify whether the bug is in parsing, arithmetic, algorithm assumptions, or the oracle.

## Time-dependent extension

If edge travel time depends on departure time, represent an edge by an arrival function (A_e(t)=t+tau_e(t)). FIFO means departing later cannot arrive earlier on the same edge: (A_e(t_1) at most  A_e(t_2)) whenever (t_1 at most  t_2). Under FIFO and suitable waiting rules, a label-setting generalization of Dijkstra can be valid because an earliest settled arrival cannot be improved by a later departure through an unsettled vertex. Without FIFO, waiting and later departures can create nontrivial improvements, and ordinary Dijkstra’s proof fails.

State whether waiting at vertices is allowed, whether edge availability intervals are hard constraints, and whether a route’s feasibility depends on a reservation resource. A route with individually feasible edges may be globally infeasible if it arrives after a time window or exhausts capacity. The state must include the relevant time/resource context, and a single distance label may no longer summarize all futures.

## Robustness and uncertainty

If travel times are estimates, optimize expected time, worst-case time, or a risk measure explicitly. A route shortest under mean weights can be poor under variance or failure probability. Robust shortest paths may require intervals and a min-max objective; the resulting problem can be harder. Do not present a deterministic shortest path over point estimates as a reliability guarantee.

Sensitivity experiments perturb weights and remove edges. Report whether the route remains feasible, how much cost changes, and which edges are critical. For the static shortest-path algorithm, a certificate verifies optimality for the supplied graph; it does not verify that the graph’s measurements are accurate or that future failures will not occur.

## Reproducibility package

Include graph files, schemas, generator seeds, exact command lines, software versions, benchmark raw data, plots, and checker output. A result table should identify graph version, algorithm, data structure, preprocessing, query count, and units. Keep theorem assumptions next to each result. The project is assessed on the ability to make a defensible algorithmic claim, not on selecting a fashionable implementation.

## Final self-audit

Before submission, ask: Can another reader reconstruct the objective from the specification? Can they identify every precondition? Does every returned route pass an independent checker? Does the proof cover the actual code’s edge cases and update semantics? Are complexity claims parameterized and model-aware? Are measured results reproducible and interpreted with uncertainty? If any answer is no, the project is not ready for an advanced course final.
