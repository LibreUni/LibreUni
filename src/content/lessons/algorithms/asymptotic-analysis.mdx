# Asymptotic Analysis and Cost Models

Algorithm analysis predicts resource growth as input size increases. The prediction is meaningful only after defining the input representation, cost model, resource being measured, and whether the claim is worst-case, average-case, expected, amortized, or empirical. This lesson builds the mathematical vocabulary used to compare algorithms and to detect analyses that hide expensive operations behind constant-time assumptions.

## Input size and representation

Choose a size measure that captures the encoded input. For an array, n elements is natural; for a graph, use V and E; for a binary integer x, input length is (lfloorlog_2xrfloor+1), not x. An algorithm polynomial in numeric x may be exponential in its binary encoding. For strings, key length ell matters if comparisons or hashing inspect characters.

A multi-parameter bound should retain relevant parameters. BFS is O(V+E), not simply O(n), unless n is defined to include both. A matrix algorithm may be O(V^3); a sparse-list algorithm may be O(V+E). Replacing a graph with a single “input size” can hide density transitions and misrepresent which instances are expensive.

## Asymptotic notation

For eventually nonnegative functions, (f=O(g)) if there are constants c,n0 such that f(n) at most c g(n) for n at least n0. (f=Omega(g)) reverses the inequality, and (f=Theta(g)) means both. Little-o means f/g- greater than 0; little-omega means f/g- greater than infinity. Constants and lower-order terms disappear, but only after the threshold n0; asymptotics do not assert that one function is faster at every practical size.

Useful rules include (log_a n=Theta(log_b n)) for fixed bases, polynomial terms dominate logarithms, and the highest-degree polynomial term dominates lower degrees. Products and nested loops need structural analysis, not blind multiplication: an inner loop may depend on the outer index, yielding a sum such as (sum_()i=1()^n i=Theta(n^2)).

**Worked loop analysis.** For i=1 to n, set j=1 and repeatedly double j while j at most i. The inner loop runs (Theta(log i)) times, so total work is (sum_()i=1()^nTheta(log i)=Theta(nlog n)), not (Theta(nlog n)) by treating every inner bound as log n without justification. The upper bound follows from log i at most log n; the lower bound uses the last half of i values, each with u03a9(log n) iterations.

## Cost models

The unit-cost RAM model treats word arithmetic, array access, and pointer operations as O(1), usually for word-sized values. A bit-complexity model charges for arithmetic on ell-bit integers; addition is not constant when ell grows. A comparison model charges only comparisons, useful for sorting lower bounds. An external-memory model charges block transfers; a cache model distinguishes lines and capacity.

State which model supports a claim. “Hash lookup is O(1)” may mean expected probes under a load assumption while ignoring key hashing cost. “Matrix multiplication is O(n^3)” may mean scalar arithmetic in RAM, not bytes moved or big-integer bit operations. Model choice is not pedantry; it determines what the algorithm is allowed to do in one step.

## Worst-case, average, expected, amortized

Worst-case (T(n)) is the maximum cost over all size-n inputs and internal randomness if randomness is adversarially fixed. Average-case takes an input distribution and computes expected cost over that distribution. Expected randomized analysis averages over algorithmic random choices, often for every fixed input. Amortized analysis bounds total cost over every operation sequence, distributing occasional expensive work.

Quicksort has expected O(n log n) comparisons under random pivots, but O(n^2) for an unlucky pivot sequence. Dynamic-array append has O(1) amortized cost but an O(n) resize operation. A hash table has expected O(1) lookup under a distribution or universal hash assumption but can have linear worst-case lookup. Do not collapse these claims into “the algorithm is O(1)” without qualifiers.

## Correctness and complexity together

An algorithm description should include a termination argument, correctness proof, and cost analysis. A loop invariant can support both: it describes progress and bounds the remaining work. For binary search, the invariant is that any target lies within the active interval if present; interval length halves each iteration, giving O(log n) comparisons. If midpoint arithmetic overflows or the interval fails to shrink, both correctness and termination fail despite the textbook recurrence.

For recursive algorithms, define the subproblem size and prove the recurrence matches actual calls and nonrecursive work. An algorithm may call a subroutine whose cost is not constant; hiding a sort or a tree traversal inside a “helper” changes the total. Sum costs over recursion paths and account for output size, copying, and allocation.

## Lower and upper bounds

An upper bound gives an algorithm. A lower bound says every algorithm in a model needs at least a resource on some inputs or in the worst case. The comparison sorting lower bound uses decision trees; an input-specific lower bound may use adversarial arguments. Matching upper and lower bounds establish optimality in that model, but a stronger model such as word operations or randomization may evade a comparison lower bound.

Output size itself is a lower bound: enumerating k objects costs u03a9(k) time to write them. Range reporting should be O(search+k), not claimed O(log n) independent of output. Reading the entire input costs u03a9(input length), so an algorithm with a smaller stated bound has omitted work or uses an oracle.

## Recurrences and recursion trees

A recurrence describes the cost of recursive calls only when the subproblem overlap and nonrecursive work are specified. Merge sort has two subproblems of size n/2 and linear merge work: (T(n)=2T(lfloor n/2rfloor)+Theta(n)). Binary search has one subproblem of size n/2 and constant work: (T(n)=T(lfloor n/2rfloor)+Theta(1)). A recurrence with floors, ceilings, or unequal sizes can usually be bounded by monotonic upper/lower recurrences, but the inequalities must be directionally correct.

The recursion-tree method expands each level. In merge sort, level j has (2^j) calls of size n/2^j, so the total merge work is Theta(n) per level and there are Theta(log n) levels. The leaves contribute Theta(n), yielding Theta(n log n). For (T(n)=2T(n/2)+Theta(n^2)), level costs shrink geometrically upward from the root’s n^2 term, so the total is Theta(n^2). Looking at the tree reveals which level dominates rather than relying on a memorized case.

## Loop and recursion invariants

A loop invariant must hold before each iteration, survive the body, and combine with termination to imply the postcondition. A termination measure is a well-founded quantity that strictly decreases; a bound on iterations is separate from logical correctness. For nested loops, prove the inner postcondition before using it in the outer invariant. For recursion, the induction hypothesis applies only to strictly smaller subproblems satisfying the precondition.

For divide-and-conquer, the proof should match the partition invariant and the recurrence. Quicksort’s correctness comes from the partition ensuring every left item is no greater than every right item, then induction on subarrays; its cost comes from pivot-dependent subproblem sizes. An implementation can be correct for every pivot while having worst-case quadratic cost, so correctness and efficiency are related but distinct claims.

## Constant factors and engineering

Asymptotics are deliberately coarse. A hash lookup, a pointer chase, and a vectorized scan may all be O(1) per item but have different cache and branch costs. For small n, a quadratic algorithm with contiguous memory can beat a logarithmic or n log n algorithm with expensive allocation. Hybrid algorithms choose a threshold empirically while retaining a large-input bound. Document the threshold and test that the fallback preserves correctness.

If an algorithm is intended for a service, include memory, allocation count, recursion depth, and tail latency in the resource contract. An O(n log n) algorithm that allocates O(n) temporary objects per request may exhaust memory under concurrency. “Time complexity” is one projection of a multidimensional cost, not the entire performance specification.

## Exercises

1. Choose size measures for arrays, graphs, strings, and binary integers.
2. Prove a loop with inner bound log i costs Theta(n log n).
3. Distinguish expected, average-case, and amortized claims.
4. Analyze binary search with a loop invariant and termination measure.
5. Give a model where integer sorting avoids comparison lower bounds.
6. Explain why output size lower bounds range reporting.
7. Find a hidden nonconstant operation in a proposed complexity argument.

### Diagnostic solutions


1. Array n; graph V,E; string length ell; binary integer bit length u230alog2 xu230b+1. Preserve multiple parameters when density or key length affects the work.

2. The inner count is Theta(log i). Upper bound by n log n. For i at least n/2, log i=u03a9(log n), and there are Theta(n) such i, giving u03a9(n log n).

3. Expected averages random algorithm choices for a fixed input; average-case averages over an input distribution; amortized bounds total cost of every valid operation sequence. State the probability or sequence quantifier.

4. Invariant: if target exists, it lies in [lo,hi]. Choose midpoint, compare, and retain the half that can contain target. Use interval length as a decreasing nonnegative measure; after O(log n) iterations it is empty or one element.

5. Counting sort or radix sort inspect bounded integer digits and use O(n+k) or O(d(n+b)) time under domain parameters. The comparison decision-tree lower bound does not apply because the algorithm obtains information from representations without pairwise comparisons.

6. Returning k records requires writing or visiting k outputs, so any explicit-output algorithm needs u03a9(k) time. A range query with O(log n+k) is output-sensitive; omitting k is impossible for large output.

7. A loop may call a helper that scans a list, compare strings of length ell, resize an array, or allocate/copy a result. Replace the constant claim with its parameterized cost and sum it over calls.
