# Divide and Conquer

Divide-and-conquer algorithms solve a problem by dividing it into smaller subproblems, solving those subproblems recursively, and combining their answers. The paradigm is powerful when subproblems are substantially smaller and their solutions contain enough information to build the original answer. The design work is in choosing the subproblem boundary and combine summary; a recurrence alone does not guarantee correctness or efficiency.

## Three design questions

First, define the subproblem precisely. “Sort the left half” is clear for an array, but a graph or geometric set may require a boundary summary. Second, ensure progress: every recursive call must satisfy a smaller measure and eventually reach a base case. Third, specify the combine invariant: what facts about child answers are sufficient to prove the parent answer?

If subproblems overlap heavily, divide and conquer may recompute work and dynamic programming may be preferable. If the combine step requires scanning the entire input at every level, the recurrence may be n log n or worse. If a subproblem can be empty, the base contract must cover it.

## Merge sort

Merge sort divides an array into two halves, recursively sorts each, and merges them. The correctness proof uses induction on n. Base arrays of size zero or one are sorted. For n greater than 1, induction gives sorted left and right halves; merge returns their stable sorted union. Therefore the complete result is sorted and is a permutation of the input.

The merge invariant is: after writing t output elements, the output prefix is the t smallest remaining elements of the two input runs, in sorted order; the unconsumed suffixes remain sorted. At each step choose the smaller front, choosing the left on equality for stability. When one run empties, append the other; the invariant and final permutation follow.

The recurrence `T(n)=2T(n/2)+Theta(n)` gives `Theta(n log n)`. Auxiliary space is O(n) for a standard array merge, plus O(log n) recursion stack. Bottom-up mergesort avoids recursive stack and can exploit runs already present. Linked lists can merge by relinking nodes, while arrays favor contiguous scans.

## Divide-and-conquer selection

Finding the kth smallest can use partitioning. Choose a pivot, partition into less/equal/greater, and recurse only into the region containing rank k. Unlike quicksort, only one side is processed, giving expected O(n) with randomized pivots. Correctness follows from partition regions: discarded elements are provably all below or above the desired rank.

Deterministic median-of-medians selection groups elements, finds group medians, recursively selects their median, and partitions. The pivot guarantees a constant fraction discarded, yielding `T(n)  at most  T(n/5)+T(7n/10)+O(n)=O(n)`. The proof counts groups whose medians lie on each side and accounts for groups near the boundaries.

## Counting inversions

An inversion is a pair with `i  less than  j` and `A[i]  greater than  A[j]`. Merge sort can count inversions while merging. If a right-run element is selected before remaining left elements, it forms one inversion with each remaining left element. The merge invariant remains sortedness; an additional counter maintains the number of cross inversions. Total time O(n log n), versus O(n^2) pair enumeration.

The algorithm demonstrates a general technique: enrich a combine step with a summary that composes. Child recursion counts inversions internal to each half; merge counts cross inversions. The sum is exactly the full count because every pair is either internal to one half or crosses the split, never both.

## Closest pair geometry

For points in the plane, brute-force closest pair is O(n^2). Divide by median x-coordinate, recursively find the closest distance delta in each half, then inspect a vertical strip of width 2delta. Sort strip points by y; each point needs only a constant number of comparisons under a packing argument, giving O(n log n).

The combine proof depends on geometry, not simply on “merge the answers.” If two points across the split are closer than delta, both lie in the strip. Partition the strip into cells whose side is related to delta; a packing argument limits how many points can occupy neighboring cells without creating a closer pair within one half. State boundary/tie handling and duplicate coordinates.

## Convex hull intuition

Divide points by x, recursively compute hulls, and merge by finding upper and lower tangents. A hull representation must be cyclically ordered and have no interior point incorrectly included. Tangent search uses orientation tests; a sign convention and collinearity policy are required. The combine step can be logarithmic for suitable hull representations, yielding O(n log n), but implementation is delicate.

## Parallelism and memory

Independent recursive subproblems can run in parallel, but combine work and synchronization limit speedup. The span recurrence measures critical-path time; total work measures operations. A balanced merge sort has O(n log n) work and O(log^2 n) span under simple parallel merge, or better with advanced merge. Work-efficient parallelism requires avoiding duplicated copies and contention.

Memory layout affects divide-and-conquer. Recursion may create temporary arrays at each level; allocate one shared buffer or alternate source/destination arrays to reduce allocation. In-place algorithms save space but make combine proofs and stability harder. A cache-oblivious recursive layout can improve locality across cache levels without knowing line size.

## Boundary conditions and data movement

The mathematical split must match the memory partition. If an array length is odd, one half has one extra element; if a range is represented by inclusive endpoints, the midpoint and recursive calls must strictly shrink. Prefer half-open intervals [lo,hi) because length is hi-lo and adjacent subproblems do not overlap. Prove that every input position belongs to exactly one child or the combine boundary and that no position is skipped.

Copying is part of combine cost. A recursive procedure that creates a new array for every child may allocate O(n log n) total bytes even when the comparison work is O(n log n), while an in-place partition may use O(log n) stack space but write records many times. Include writes and allocations when the data type is large or the memory limit is tight.

## Parallel scheduling

Divide-and-conquer exposes a task tree. Running both children sequentially gives the ordinary recurrence; running them concurrently can reduce elapsed time only if enough processors and memory exist. A work-stealing scheduler may execute tasks in a different order, so shared output buffers need disjoint ranges or synchronization. The correctness proof should rely on disjoint ownership, not on an accidental sequential order.

The ideal parallel speedup is bounded by work/span. If total work is W and span is S, no execution on P processors can be faster than max(W/P,S). A recursive algorithm with cheap independent leaves but a serial combine may have large W and small-looking depth yet limited speedup at the root. Report parallelism separately from sequential asymptotic time.

## When not to divide

Use dynamic programming when subproblems overlap and a finite state summary can be reused. Use greedy or incremental methods when a local decision has a proof and recursion adds overhead. Use a scan when the input is small or the combine bookkeeping dominates. Paradigms are tools, not obligations; the correct design is the one whose subproblem structure matches the problem’s dependencies.

## Proof and review checklist

Before accepting a divide-and-conquer algorithm, check the exact subproblem domain, base cases, partition coverage, no-overlap condition, combine postcondition, termination measure, recurrence, auxiliary memory, and output ownership. Test odd sizes, empty ranges, duplicates, equal coordinates, already sorted inputs, and all-equal values. These cases expose assumptions that a balanced random input hides.

### Diagnostic solutions

1. Induct on n. Base size at most 1 is sorted. Child halves are sorted by induction; merge maintains a prefix of smallest remaining elements and appends the remainder, so output is sorted and contains exactly both inputs.

2. `T(n)=2T(n/2)+Theta(n)=Theta(n log n)`. Standard arrays use O(n) auxiliary merge storage plus O(log n) stack; bottom-up versions use O(n) buffer and no recursion.

3. During merge, whenever a right element precedes remaining left elements, add the number of remaining left entries. This counts exactly cross-half inversions; recursive counts handle within-half pairs.

4. Every partition places items less than pivot before, equal in middle, greater after. If k lies in a side, recursively selecting that side preserves the kth rank; if in equal region, return. Pivot randomness affects cost, not the order invariant.

5. At least half of group medians are at least the pivot; each such group contributes at least three elements at least as large, except a constant number of boundary groups. Symmetrically for smaller elements, leaving at most about 7n/10 for recursion.

6. Any cross-half pair closer than delta lies in the 2delta strip. Sorting strip points by y and applying a packing argument limits candidate comparisons per point to a constant, so combine work is linear.

7. Work sums all recursive operations and combine work; span follows the longest dependency path. Independent calls reduce span, but merge synchronization remains. A parallel algorithm can have the same work as sequential while reducing critical-path time.
