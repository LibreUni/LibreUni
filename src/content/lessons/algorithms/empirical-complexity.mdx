# Empirical Complexity

**Prerequisites:** Asymptotic notation ($\Theta$, $O$, $\Omega$), basic probability, familiarity with Python's `time` module.

An $O(n \log n)$ sorting algorithm and an $O(n^2)$ algorithm — which is faster on your laptop?

Implement both: quicksort (expected $O(n \log n)$) and insertion sort ($O(n^2)$). Test them on small arrays. Insertion sort beats quicksort for every $n < 1000$, and sometimes well beyond. Two things are happening. First, the constants: quicksort's partition step involves pointer arithmetic, swaps, and recursive calls; insertion sort is a tight inner loop with virtually no overhead. Second, the hardware: insertion sort scans memory sequentially, hitting every cache line perfectly; quicksort's partition scatters accesses across the array, causing cache misses that dwarf the instruction count at moderate sizes.

The asymptotic analysis is not wrong — $n \log n$ is eventually smaller than $n^2$ — but it is incomplete. Big-O tells you the limit as $n \to \infty$; it says nothing about $n = 1000$ on your specific CPU with your compiler's optimization flags. The crossover point where the asymptotically superior algorithm actually becomes faster may lie beyond the input sizes you care about, or may be delayed indefinitely by memory hierarchy effects.

This gap is the domain of *empirical complexity*: the experimental measurement, statistical modeling, and rigorous comparison of algorithm performance on real machines. It is not a rejection of asymptotic analysis but its complement. Knuth, in a 1977 memo on the TEX typesetting system, gave a famously cautious disclaimer: "Beware of bugs in the above code; I have only proved it correct, not tried it." The same principle applies to algorithm analysis — every bound must be confronted with measurement.

A methodology for experimental algorithmics involves: formulating testable hypotheses (e.g., $T(n) \approx c n \log n$), designing benchmarks that isolate the algorithm from system noise, collecting hardware-counter data, selecting and validating statistical models, and reporting results reproducibly. The goal is to answer the engineer's question — "How fast will this algorithm be on my data?" — with the same rigor that asymptotic analysis brings to the theorist's question "What is the growth rate as $n \to \infty$?"

## The Experimental Framework

**Hypothesis:** $T(n) = c \cdot g(n) + o(g(n))$ for known $g$, unknown $c$.

**Null:** The data is consistent with $g$.

**Alternative:** A different growth rate fits better or the model fails at certain scales.

<MathStatement id="doubling-ratio" kind="definition" title="Doubling ratio test">
For size-doubling measurements, $T(2n)/T(n)$ tends toward $2^k$ for $T(n) = \Theta(n^k)$ as $n$ grows, assuming lower-order terms become negligible.
</MathStatement>

| Growth | Doubling ratio |
|--------|---------------|
| $O(1)$ | 1 |
| $O(\log n)$ | $1 + \epsilon \to 1$ |
| $O(n)$ | 2 |
| $O(n \log n)$ | $2 + \epsilon$ |
| $O(n^2)$ | 4 |
| $O(n^3)$ | 8 |
| $O(2^n)$ | $\to \infty$ |

The doubling ratio gives a quick visual check, but transitions between cache levels, memory allocation strategies, and garbage collection can create plateaus or jumps that mimic asymptotic shifts.

<MathStatement id="model-selection" kind="definition" title="Model selection for complexity">
Choose candidate functions $g_i(n)$ from the algorithm's proof. Fit $T(n) = c_i \cdot g_i(n)$ by least squares on the largest sizes. Compare adjusted $R^2$, residual plots, and prediction error on held-out sizes. A model is consistent if residuals are patternless and prediction error is within measurement noise.
</MathStatement>

## Worked Example 1: Insertion Sort Model Selection

Consider insertion sort on input sizes $n \in \{2^{10}, 2^{11}, \ldots, 2^{20}\}$. The algorithm is proven to make $\frac{n(n-1)}{2} = O(n^2)$ comparisons in the worst case. The candidate models are therefore $g_1(n) = n$, $g_2(n) = n\log_2 n$, $g_3(n) = n^2$.

| $n$ | Time (ms) | $T(n)/n$ | $T(n)/(n\log_2 n)$ | $T(n)/n^2$ |
|-----|-----------|----------|-------------------|-----------|
| 1024 | 0.12 | $1.17\times10^{-4}$ | $1.17\times10^{-5}$ | $1.14\times10^{-7}$ |
| 2048 | 0.49 | $2.39\times10^{-4}$ | $2.15\times10^{-5}$ | $1.17\times10^{-7}$ |
| 4096 | 1.95 | $4.76\times10^{-4}$ | $3.87\times10^{-5}$ | $1.16\times10^{-7}$ |
| 8192 | 7.82 | $9.54\times10^{-4}$ | $7.08\times10^{-5}$ | $1.17\times10^{-7}$ |
| 16384 | 31.28 | $1.91\times10^{-3}$ | $1.30\times10^{-4}$ | $1.17\times10^{-7}$ |

The ratio $T(n)/n^2$ stabilizes at $1.17 \times 10^{-7}$, while $T(n)/n$ and $T(n)/(n\log_2 n)$ increase monotonically. This confirms the quadratic model. The stable ratio estimates $c \approx 1.17 \times 10^{-7}$ seconds per comparison — the constant factor hidden by the $\Theta$ notation.

        key = A[i]
        j = i - 1

sizes = [2**k for k in range(10, 16)]
for n in sizes:

An algorithm engineer comparing two sorting methods — the $O(n \log n)$ mergesort and the $O(n^2)$ insertion sort — can locate the crossover point empirically.

The insertionsort $O(n^2)$ curve is flat below $n=200$ then rises sharply. The mergesort $O(n \log n)$ curve has higher constant overhead but shallower slope. The crossover near $n=900$ is where theory and practice intersect: below this threshold, the overhead of divide-and-conquer dominates; above it, asymptotic growth takes over. The exact crossover depends on hardware, language, compiler flags, and input distribution — it should be measured, not assumed.

## Hardware Counters

Modern CPUs provide performance counters for:
- **Instructions retired:** operation count, independent of microarchitecture.
- **Cache misses (L1/L2/LLC):** memory access pattern efficiency.
- **Branch mispredictions:** indicates unpredictability in data-dependent branches.
- **TLB misses:** page-level access pattern.

<MathStatement id="cache-miss-impact" kind="theorem" title="Cache-miss dominated complexity">
A function with $O(n \log n)$ instructions but $O(n)$ cache misses may be dominated by memory latency at large $n$ if the miss penalty $M$ satisfies $M \cdot \text{misses} \gg c \cdot \text{instructions}$.
</MathStatement>

<MathProof id="cache-proof" for="cache-miss-impact" collapsible>
Let $C$ be the instruction count for a problem of size $n$, with per-instruction cost $c$, and $Q(n)$ the number of cache misses with per-miss penalty $M \approx 60\text{ns}$. The total time is $T(n) = c \cdot C + M \cdot Q(n)$. If $C = O(n \log n)$ and $Q(n) = \Theta(n)$, then as $n$ grows, the term $M \cdot Q(n)$ eventually dominates because $M / c \approx 60$ while the per-element instruction ratio is roughly $\log_2 n$, and $\log_2 n$ grows slower than 60 for any $n < 2^{60}$. On real hardware, L3 cache misses dominating performance is common at $n > 10^6$ for algorithms with poor locality.
</MathProof>

To collect counters on Linux, use `perf stat`:
```
perf stat -e instructions,cache-misses,branch-misses ./my_algorithm
```

## Counterexample: When Empirical Complexity Misleads

Consider a binary search in a sorted array: $\Theta(\log n)$ comparisons, $\Theta(1)$ space. Empirically, measuring it on $n = 2^{10}$ to $2^{20}$ reveals a steady $\log n$ curve — exactly as predicted. Now consider *linked-list* binary search (if such a thing existed): still $\Theta(\log n)$ comparisons, but each "access" requires traversing $\Theta(n)$ pointer hops. An empirical measurement that counts only operations rather than wall-clock time would fail to detect the real cost.

**The failure mode:** Empirical measurement that observes the *wrong quantity* — operation counts instead of runtime, or runtime without controlling for cache effects — can produce results consistent with a false model.

**Demonstration:** A program that touches memory sequentially at size $n$ has $T(n) \approx c_1 n$. A program that accesses a linked list randomly has $T(n) \approx c_2 n$, but $c_2 \gg c_1$ due to cache misses. A naive doubling-ratio test shows "linear" for *both*, hiding the $100\times$ constant difference. Only hardware-counter data (cache misses) reveals the true cause.

## Reproducibility and Threats to Validity

| Threat | Mitigation |
|--------|-----------|
| Compiler optimizations | State flags, inspect assembly if needed |
| Input generation | Record seed, generate algorithmically |
| Measurement overhead | Subtract empty-loop baseline |
| Thermal throttling | Randomize run order, report temperature if available |
| JIT compilation | Warmup runs before measurement |
| GC pauses | Use separate processes, measure multiple trials |

## Statistical Testing

Report **median** and **interquartile range** rather than mean and standard deviation for timing data (which is typically right-skewed). Use bootstrap confidence intervals for comparing two implementations.

<CodeExercise client:load
  title="Compute bootstrap confidence interval"
  code={`import random, statistics

def bootstrap_ci(data, stat=statistics.median,
                 trials=10000, alpha=0.05):
    \"\"\"Return (lower, upper) bootstrap CI for stat.\"\"\"
    n = len(data)
    boot_stats = []
    for _ in range(trials):
        sample = [random.choice(data) for _ in range(n)]
        boot_stats.append([!blank!](sample))
    lo = int(trials * alpha / 2)
    hi = int(trials * (1 - alpha / 2))
    boot_stats.sort()
    return (boot_stats[lo], boot_stats[hi])`}
  answers={["stat"]}
  explanation="Each bootstrap trial draws a resample with replacement from the original data and computes the statistic on that resample. The CI is the alpha/2 and (1-alpha/2) quantiles of the bootstrap distribution."
/>

## Deliberately Omitted

- **Statistical hypothesis testing for algorithm comparison (Wilcoxon, t-tests)** — Bootstrap confidence intervals are developed here; formal hypothesis testing with null-hypothesis significance testing is covered in the statistics module.
- **Microarchitecture-specific optimization (SIMD, prefetching)** — Compiler intrinsics and architecture-specific tuning (AVX, cache-line alignment) are engineering details beyond the experimental methodology focus.
- **Automated performance regression testing in CI pipelines** — Incorporating empirical complexity checks into continuous integration is a software engineering practice rather than an algorithmic methodology.
- **Statistical design of experiments (factorial designs, blocking)** — Experimental design for full factorial combinations of input type, size, and machine parameters is covered in the research methods track.
- **Energy measurement (RAPL, power meters)** — Energy-aware algorithm assessment uses different hardware counters (RAPL) and is conceptually distinct; the focus here is latency and throughput.
