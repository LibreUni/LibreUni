# Graph Algorithm Laboratory

This laboratory asks you to implement, verify, and compare graph algorithms on a shared graph interface. The goal is to connect mathematical preconditions with representation choices, invariants, certificates, and measured behavior. A program that returns plausible paths on one graph is not enough: the report must show why the algorithm is correct for its model and what happens when the model’s assumptions fail.

## Graph model and interface

Define vertices, directed or undirected edges, weights, parallel-edge policy, self-loop policy, and representation. Decide whether vertex IDs are dense integers or arbitrary objects, whether weights may be negative, and whether floating comparison uses exact or tolerance semantics. Expose neighbors, edge iteration, and endpoint validation through a contract; algorithms should not reach into representation-specific fields.

Provide at least adjacency-list and edge-list input paths, or explain why one is sufficient. Validate that every endpoint exists, weights are within range, and undirected edges are reciprocal if the representation requires it. Preserve edge IDs when parallel edges are allowed. A malformed graph should be rejected before an algorithm relies on its invariants.

## Experiment 1: traversal

Implement BFS and DFS. BFS should return reachability, parent, and distance in an unweighted graph. DFS should return discovery/finish times and parent forest. Mark vertices when discovered, not when removed from a queue, to avoid duplicates. State whether neighbor order affects parent tree and timestamps; it affects the tree but not reachability.

The BFS invariant is that when a vertex is dequeued, its distance is minimum among all undiscovered paths, and every discovered vertex has a parent path of the recorded distance. The DFS proof uses color states and the fact that a recursive call finishes after all reachable white descendants have been explored. A stack-based DFS must store continuation state if it is to reproduce recursive edge classification.

Test disconnected graphs, self-loops, parallel edges, duplicate adjacency entries, and repeated runs with fresh traversal metadata. Compare reachable sets with a reference implementation. For BFS, verify every returned parent edge exists and distance[parent]+1=distance[child]. For DFS, verify timestamps are unique and parent intervals nest according to the chosen semantics.

## Experiment 2: shortest paths

Implement BFS shortest paths for unit weights, Dijkstra for nonnegative weights, Bellman-Ford for possible negative edges, and DAG relaxation for a topologically ordered graph. Each routine must state unsupported inputs and return explicit statuses for unreachable vertices and negative cycles. Do not silently run Dijkstra on negative input.

For each returned distance, verify path reconstruction and edge inequalities. In a graph without a reachable negative cycle, check that no edge can relax the final labels. For Bellman-Ford, if a further relaxation occurs after V-1 passes, propagate affected vertices if the API reports undefined distances. Test zero-weight ties, parallel edges, disconnected components, and large weights near numeric limits.

Compare dense and sparse representations. Dijkstra with an array priority queue may be competitive on dense graphs; a binary heap and adjacency lists suit sparse graphs. Measure heap operations, stale entries, edge scans, cache effects, and total time. Do not attribute a difference to asymptotic complexity without matching graph density and input size.

## Experiment 3: minimum spanning forest

Implement Kruskal and Prim, and return a minimum spanning forest for disconnected graphs or an explicit disconnected status according to the interface. Kruskal uses union-find and edge sorting; Prim uses a priority queue and a component start. Both must handle negative and equal weights, parallel edges, and self-loops.

Validate candidate output with edge count, acyclicity, component coverage, and cycle certificates. For every non-tree edge, compare its weight with the maximum edge on the tree path. On small graphs, enumerate spanning trees or use a trusted solver to verify total weight. Compare Kruskal and Prim on the same graph; different edge sets can both be optimal under ties.

## Experiment 4: maximum flow

Implement Edmonds-Karp or another stated max-flow algorithm using explicit residual edge pairs. Return the flow and the residual-reachable cut. Validate capacities, conservation, nonnegative residuals, reverse-edge symmetry, and flow value equal to cut capacity. Include antiparallel and parallel original edges with distinct IDs.

Model a bipartite matching instance as flow and map an integral solution back to pairs. Test zero capacities, no path, multiple bottlenecks, and capacities that expose integer overflow. Record augmentations, BFS scans, and final certificate. If using floating capacities, document termination tolerance and test near-zero residual behavior.

## Failure investigations

Case A: BFS returns a distance smaller than a parent path length. Inspect whether a vertex was enqueued twice or distance was overwritten without a shorter path. Case B: Dijkstra settles a vertex and later finds a smaller path. Look for negative edges, an invalid priority queue stale-entry check, or premature settlement. Case C: Kruskal returns too few edges. Check disconnectedness, vertex mapping, or union-find root equality. Case D: flow value exceeds a cut. Check conservation, reverse updates, and whether the cut was taken from the final residual graph.

## Deliverables

Submit interface specification, graph generators, implementations, invariant validators, proof sketches, complexity analysis, benchmark tables, failure traces, and a comparison of at least two representations. Include exact seeds and commands. For each algorithm, provide one graph where its precondition is satisfied and one where an assumption fails; explain whether the implementation rejects, reports, or produces a defined alternative result.

## Evaluation rubric

* 20% graph model and representation correctness.
* 25% algorithms and invariant proofs.
* 20% certificates and adversarial tests.
* 20% complexity/measurement quality.
* 15% failure analysis and reproducibility.

## Experimental design

Keep graph topology and algorithm choice separate. Generate or load the graph once, clone it for each implementation, and verify that vertex/edge counts and weights match before timing. Warm caches and cold caches answer different questions. Measure preprocessing such as vertex compression and sorting separately from the core algorithm, then report end-to-end cost when preprocessing is unavoidable in the application.

Use size families that vary V, E, density, weight distribution, component count, and degree skew independently. A graph with the same V and E can have a long path, a star, a dense core, or many isolated vertices; these shapes affect queue size, heap behavior, and cache locality. Preserve seeds and graph serialization so a performance outlier can be reproduced.

## Algorithm selection matrix

For unit weights, BFS gives linear-time shortest paths. For arbitrary weights on a DAG, topological relaxation is linear in the graph. For nonnegative general weights, choose Dijkstra with a priority structure suited to density. For negative edges, use Bellman-Ford or reweight with a verified potential. For an MST, use Kruskal when sorting an edge list is natural, Prim when adjacency and frontier updates are natural, and Boruvka when phases or parallel scans are useful. For capacities, use a residual-flow algorithm and return a cut certificate.

The matrix is a decision aid, not an excuse to ignore output requirements. If only reachability is needed, a shortest-path routine overcomputes. If paths or certificates are needed, store predecessors or residual state. If the graph changes, static recomputation may be wrong for latency or wasteful for throughput; state whether the lab’s results apply only to a batch snapshot.

## Reproducibility

The report must distinguish proof from measurement. A proof establishes correctness under assumptions; a benchmark estimates resource behavior on sampled instances. Include failed cases, not only successful timings. The grader should be able to rerun a small deterministic graph, inspect the certificate, and reproduce at least one performance plot from the recorded seed and environment.

## Exercises

1. Design a shared graph interface that supports all four experiments.
2. State BFS and Dijkstra invariants.
3. Design a shortest-path certificate checker.
4. Design an MST certificate checker.
5. Define a max-flow residual validator.
6. Construct a graph that distinguishes adjacency-list and matrix performance.
7. Write a failure report for one invalid precondition.

### Diagnostic solutions

1. Expose vertex enumeration, neighbor iteration with edge IDs and weights, endpoint validation, and edge iteration. Keep algorithm state separate and define directed/undirected semantics and errors.

2. BFS discovered distances are shortest and parent edges increase distance by one. Dijkstra’s settled labels are final because all edge weights are nonnegative; tentative labels remain path upper bounds.

3. Verify each predecessor edge exists, the source reaches every claimed target through predecessors, summed weights equal labels, and every edge inequality `d[v]  at most  d[u] + w` holds when no negative cycle is reported.

4. Check edge count, acyclicity, and component spanning. For each non-tree edge verify its weight is at least the maximum tree-path edge; this proves no cycle exchange improves the tree.

5. Check `0  at most  flow  at most  capacity`, conservation at all nonterminals, paired residual capacities, and source value. BFS residual reachability defines S; compare cut capacity with value.

6. Use a sparse graph with E=O(V): lists scan O(V+E), while matrix traversal scans O(V squared). For a dense graph, both scan Theta(V squared), and matrix membership/locality may help.

7. State symptom, input, precondition, invariant failure, minimal reproducer, evidence, repair, and regression test. Do not label the algorithm wrong until the graph model and implementation assumptions are checked.
