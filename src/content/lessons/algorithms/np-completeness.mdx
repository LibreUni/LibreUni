# NP-Completeness and Algorithmic Limits

Algorithm design includes recognizing when an exact polynomial-time method is unlikely. NP-completeness provides a formal language for that recognition. It does not say a problem is impossible, that every instance is hard, or that no useful algorithm exists. It says that a polynomial algorithm for one complete problem would solve every problem in NP. The practical response is to exploit special cases, parameterize, approximate, search, or accept exponential behavior on small instances.

## Decision and optimization versions

Complexity classes are usually defined for decision languages. An optimization problem asks for the best feasible object; its decision version asks whether a feasible object reaches threshold K. To use a reduction, define encodings and show how a decision solver could answer the source problem. For many problems, a polynomial decision algorithm can recover an optimization solution by binary search or self-reduction, but this conversion requires a proof and may need a polynomially bounded objective.

SAT asks whether a Boolean formula has a satisfying assignment. A certificate is an assignment whose clauses can be evaluated in polynomial time, so SAT is in NP. The Cook-Levin theorem proves SAT is NP-complete by encoding a polynomial-time nondeterministic computation as a tableau of time and tape cells with local consistency constraints. The theorem is foundational: it shows every NP computation can be represented as a satisfiability instance.

## Polynomial reductions

A polynomial many-one reduction from A to B computes an instance f(x) in polynomial time with x yes iff f(x) yes. If B is in P, A is in P; if A is NP-hard and A reduces to B, B is NP-hard. Direction is the central discipline. To prove a new problem B hard, start from a known complete problem A and transform A instances into B instances.

Completeness requires both membership and hardness. A proof that only gives a reduction establishes hardness, not that the target lies in NP. For membership, specify a polynomial-size certificate and a polynomial verifier, including encoding length. For hardness, prove construction size/time and both yes and no directions.

**Worked reduction pattern.** To reduce 3SAT to CLIQUE, create a vertex for each literal occurrence, connect vertices from different clauses when they are not complementary, and ask for a clique of size equal to the number of clauses. A satisfying assignment selects one true occurrence per clause and creates a clique; a clique selects compatible occurrences and yields a satisfying assignment. No edge within a clause is needed because a clique of target size cannot select two from one clause when there are only one-per-clause compatibility edges.

## Classic complete problems

Common NP-complete problems include CLIQUE, INDEPENDENT SET, VERTEX COVER, HAMILTONIAN CYCLE, 3SAT, SUBSET SUM, PARTITION, and decision versions of scheduling and packing. Reductions exploit complement relationships: a set is independent iff it is a clique in the complement graph; a vertex cover of size k exists iff an independent set of size n-k exists.

Use known reductions carefully. A reduction from CLIQUE to a target may require complementing or changing threshold; a reduction from a problem to its optimization version needs a decision encoding. The transformation must preserve the exact parameter and feasibility constraints, not just produce a related-looking graph.

## Self-reducibility and search

SAT is self-reducible: if a polynomial SAT decision oracle exists, construct a satisfying assignment by asking whether the formula remains satisfiable after setting each variable to false; if not, set it true. Each query simplifies the formula and there are polynomially many variables. This demonstrates why efficient decision can imply efficient search for many NP problems.

When no exact polynomial algorithm is known, branch-and-bound uses upper/lower bounds to prune a search tree. A branch is safe to prune only when the bound proves no completion can beat the incumbent or satisfy the threshold. Memoization can merge repeated subproblems; parameterization can isolate a small parameter k and permit (f(k)n^()O(1)()) time.

## Weak and strong hardness

A pseudo-polynomial algorithm may solve a problem efficiently when numeric values are small but be exponential in binary input length. Subset sum is weakly NP-complete in this sense and has an FPTAS. Problems with hardness independent of numeric magnitude are strongly NP-complete; pseudo-polynomial dynamic programming is unlikely to solve them generally unless P=NP.

The distinction affects modeling and practical algorithms. A capacity of one billion stored in binary has only about 30 bits, so O(nW) DP is not polynomial in input length. If W is small in the domain, the DP may still be excellent. State numeric encoding and parameter ranges before classifying an algorithm.

## Approximation and hardness

NP-hardness does not forbid approximation. Vertex cover has a 2-approximation; metric TSP has 3/2 or 2 approximations depending on the algorithm; set cover has logarithmic approximation. Gap reductions can show that a ratio better than a threshold would imply P=NP. Distinguish optimization hardness from inapproximability and from practical difficulty on a distribution.

## Reduction hygiene

A reduction must preserve malformed-instance behavior or restrict its domain to valid encodings with a polynomial-time validity check. It must not solve the source instance while constructing the target. “Create a target object that is yes exactly when the source is yes” is incomplete until the construction is explicit and finite. Test the mapping on a source yes-instance, a source no-instance, duplicate fields, empty structures, and parameter boundary cases.

Thresholds and complements are common polarity traps. If a source asks whether an optimum is at least K, a target construction that asks whether a cost is at most K may need a complement or a transformed objective. Write the iff statement before writing the gadget. For graph reductions, map vertices, edges, weights, and parameter values explicitly; an omitted edge can create an unintended solution.

## Parameters and fixed-parameter algorithms

A parameterized problem has input size n and parameter k. An FPT algorithm runs (f(k)n^c), where the exponent c is independent of k. A brute-force algorithm (n^k) is generally XP rather than FPT because k appears in the exponent. Kernelization preprocesses an instance to an equivalent one of size g(k) in polynomial time; then an exact exponential solver may be practical for small k.

Vertex cover has an FPT branching algorithm: choose an edge (u,v), and every cover contains u or v, so branch by including each endpoint and decrease k. The recurrence T(k) at most 2T(k-1)+poly(n) gives (2^k poly(n)). The branching proof is complete because the edge must be covered in every feasible solution. A kernel can also remove vertices or edges using safe reduction rules.

## Practical responses to hardness

When a problem is NP-hard, choose among exact search with pruning, pseudo-polynomial DP, approximation, parameterization, restricted instances, heuristics, or a solver. State the guarantee: exact but exponential worst-case, approximate ratio, fixed-parameter, or empirical only. A heuristic may be excellent on a workload and still have no worst-case bound; that is acceptable when labeled and monitored.

Decision and optimization outputs may have different operational needs. A feasibility checker can be fast even when optimization is hard; an approximate solution with a certificate may be preferable to an exact answer that misses a deadline. Time budgets, incumbent solutions, and lower/upper bounds can provide anytime behavior: return the best feasible result found with a bound on the remaining gap when available.

## Hardness evidence versus proof

Empirical exponential growth on small instances is not a proof of NP-hardness. A citation to a known-complete problem is not a reduction for a new variant unless the constraints match. Conversely, a polynomial algorithm for a restricted family does not refute hardness of the general problem. Keep theorem scope, implementation scope, and benchmark scope separate.
