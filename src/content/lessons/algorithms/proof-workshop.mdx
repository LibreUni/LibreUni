# Algorithm Proof Workshop

An algorithms course is a course in justified construction. A procedure is not an algorithmic result until its inputs, outputs, assumptions, termination, correctness, and resource bounds are explicit. This workshop is organized around proof repair: each pattern identifies a common gap, supplies a stronger statement, and shows how to turn it into a checkable argument. The methods apply across sorting, graphs, optimization, dynamic programming, randomized algorithms, and approximation.

## Specification first

A specification should make incorrect outputs observable. Sorting needs sortedness and permutation; shortest paths need valid path/distance correspondence and unreachable behavior; scheduling needs feasibility and objective; approximation needs a ratio relative to OPT. If the postcondition is weaker than the problem, an algorithm can satisfy the proof while failing the user’s actual need.

State preconditions such as sorted input, nonnegative weights, acyclic graph, metric distances, or valid comparator. If the algorithm should reject invalid inputs, rejection is part of the postcondition. If inputs are assumed valid, the interface must enforce or document that boundary. A proof that silently assumes a property not stated in the problem is proving a different theorem.

## Invariant construction

A loop invariant must be true initially, preserved, and strong enough at termination. Start from the desired result, weaken it until it holds at initialization, then strengthen with information needed by the next iteration. For partition algorithms, describe regions and their contents; for BFS, describe distance and discovery; for heap construction, describe which subtrees are already heaps.

Use a table with columns: state before, operation, state after, invariant clause preserved, progress measure. Boundary branches often require separate rows. An invariant that says only “the processed part is correct” may not specify what happens to unprocessed elements or whether every input item is still represented.

**Repair example.** The claim “after each selection, the prefix is sorted” is too weak for selection sort if it does not say the prefix contains the smallest original elements. Add the permutation/minimum clause: prefix is the k smallest input elements in sorted order, and suffix contains the remainder. The next minimum scan and final postcondition then follow.

## Induction and structural recursion

For recursive algorithms, identify a well-founded size and prove child preconditions. In merge sort, children are smaller arrays and merge combines sorted permutations. In tree recursion, the child may be a subtree with inherited bounds; the induction hypothesis must include those bounds. In graph recursion with visited state, ordinary size induction may fail because edges can revisit vertices; use a reachable-set invariant.

Strengthen induction hypotheses when the naive one cannot absorb combine cost or boundary data. A claim (T(n) at most  cnlog n) may need a lower-order negative slack; a correctness claim may need to quantify over all valid subarray endpoints rather than only full arrays. The extra generality is often what makes the induction step close.

## Exchange arguments

For a greedy algorithm, take an arbitrary optimal solution and exchange the greedy choice into it. Prove feasibility, non-worsening objective, and that the residual instance has the same form. If the exchange changes multiple decisions, describe all changes and show no hidden constraint is violated. A local “greedy seems best” explanation is not an exchange argument.

For MST, the exchange follows a cycle and a cut; for interval scheduling, earliest finish leaves a compatible suffix; for Huffman, least frequencies can be deepest siblings. Different problems have different exchange structures. If no exchange can be found, search for a counterexample or change paradigms.

## Certificates

A certificate converts a global claim into local checks. For an MST, a tree plus cycle inequalities is a certificate; for max flow, feasible flow plus equal cut capacity; for shortest paths, labels plus predecessor paths and edge inequalities. The checker must validate all assumptions, endpoints, arithmetic, and output format. “The algorithm ran without error” is not a certificate.

Certificates are especially useful in testing: compare an optimized solver’s output with an independent checker, not only another implementation with the same bug. For approximate solutions, a lower/upper bound certificate can establish an objective gap even when OPT is unknown.

## Counterexamples

To disprove a universal algorithm claim, find one valid input that violates correctness or the bound. Start with the smallest structures: two or three items, tied weights, one exceptional edge, or one boundary index. For greedy rules, enumerate small instances and compare with brute force. For invariant claims, construct a state satisfying the claimed invariant but not the intended one; this exposes weakness.

Counterexamples should identify the exact failed implication. “This output seems wrong” is weaker than “the proof assumed the selected interval’s start was earliest, but the algorithm only selected shortest duration.” A repaired algorithm needs either a new rule/proof or a changed specification.

## Complexity proof review

Count all calls, loops, comparisons, copying, key length, allocation, recursion depth, and output. If randomness appears, state the probability space. If amortization appears, define a potential or aggregate sequence bound. If approximation appears, compare to a valid lower/upper bound on OPT. If the result is pseudo-polynomial, state numeric encoding.

Cross-check recurrence with code and proof. A helper that sorts a list is not O(1); a graph neighbor iteration can cost degree; a hash lookup has distribution assumptions; a heap operation may include a stale-entry skip. Complexity is a property of the full implementation under a stated model.

## Hoare-style reasoning

A useful notation is a triple (()P() C ()Q()): if precondition P holds and command C terminates, postcondition Q holds. Assignment substitutes an expression into Q; sequencing composes triples when the first postcondition implies the second precondition; conditionals require proving both branches; loops require an invariant that is established before entry and preserved by the body. The loop rule yields partial correctness, while a decreasing variant adds termination.

For an algorithm, use the triple to separate helper contracts. A partition helper can promise that returned index p separates values into ( at most ) and ( at least ) regions and preserves permutation. Quicksort then invokes the helper, recursively proves each subrange, and uses the partition postcondition to prove the full result. If the helper’s contract omits permutation, the top-level proof cannot establish that records were not lost.

## Refinement from specification to implementation

A high-level algorithm can be refined into data structures and loops while preserving a simulation relation between abstract state and concrete state. For a queue, the abstract sequence is represented by two stacks; the relation concatenates output stack order with the reverse of input stack order. Enqueue and dequeue are proved to preserve that relation. This is why a complicated implementation can still satisfy a simple ADT.

Refinement also exposes where costs enter. A mathematical set operation may become a hash lookup with expected cost or a balanced-tree lookup with worst-case cost. An abstract immutable update may become path copying with allocation. Keep the functional simulation proof separate from the resource refinement, then state both in the final theorem.

## Randomness, invariants, and probability

A randomized algorithm’s state invariant should hold for every possible random outcome if correctness is Las Vegas, or with an explicitly bounded bad event for Monte Carlo. Condition on the current history when analyzing adaptive choices; an unconditional expectation can hide an invalid step if the algorithm’s state distribution is not what the proof assumes. Use indicator variables for expected counts and concentration bounds for tails, with hypotheses stated.

## Final proof checklist

Name the input domain, output, precondition, invariant or induction statement, base case, preservation step, termination measure, complexity model, and exceptional behavior. Test the proof’s boundary cases and produce a counterexample to any tempting but unproved variant. The finished argument should let a reader identify exactly which assumption would need to change if the problem specification changes.

## Exercises

1. Repair a weak sorting invariant.
2. Write a structural induction proof for merge sort.
3. Outline an exchange proof for interval scheduling.
4. Design a shortest-path certificate.
5. Construct a counterexample to a greedy rule.
6. Review a complexity proof for hidden costs.
7. State assumptions for a randomized/approximate claim.

### Diagnostic solutions


1. Add that the processed prefix is a sorted permutation of exactly the corresponding smallest input elements and the suffix contains all remaining elements. Initialization, insertion, and termination then imply full sorted permutation.

2. Base n at most 1. Assume recursive halves return sorted permutations; merge’s prefix invariant outputs the smallest remaining element and appends the rest, yielding sorted union. Recurrence gives Theta(n log n).

3. Let g finish earliest and O be any optimum. Replace O’s first interval with g; all later intervals remain compatible because g finishes no later. Recurse on intervals beginning after g.

4. Return distances, predecessor edges, and a path. Check every path edge exists and sums to distance; check every graph edge satisfies d[v] at most d[u]+w for a graph without negative cycles. A negative-cycle status needs a separate witness.

5. Use two short compatible intervals versus one rule-selected interval, or density-first 0/1 knapsack with one high-density blocker and two lower-density items. Verify all weights/values and compute the true optimum by enumeration.

6. Inspect helpers, nested loops, copying, comparisons, and output. Replace every hidden operation with its parameterized cost and sum over calls. State whether bounds are worst-case, expected, or amortized.

7. State fixed input versus random bits, independence/adversary, error probability, and runtime. For approximation, define min/max ratio, feasibility, epsilon dependence, and the bound compared with OPT.
