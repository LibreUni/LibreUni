# Complexity Review

Classify a problem along three axes: is it decidable, what resource bounds are known, and what does one implementation measure? Use certificates and reductions where appropriate; do not infer asymptotic class from a single benchmark.

## Three distinct questions

Computability asks whether some terminating algorithm decides the problem for every valid input. Complexity asks how much resource a chosen computational model needs as a function of input length. Empirical performance asks how one implementation behaves on selected hardware and instances. A problem may be decidable but require enormous time, belong to a class with an unknown separation, or have a fast implementation on practical inputs despite a bad worst case.

Input length must be defined. An integer n represented in binary has length logarithmic in n, while a unary representation has length n. An algorithm that loops n times is polynomial in the numeric value but may be exponential in the binary input length. Complexity claims without an encoding convention are incomplete.

## Time and space classes

DTIME(f(n)) contains languages decided by a deterministic machine within O(f(n)) steps. P is the union over polynomial k of DTIME(n^k). Nondeterministic time classes describe machines with accepting branches; NP can equivalently be defined by polynomial-time verifiers for polynomial-length certificates. PSPACE contains languages decidable with polynomial workspace, allowing potentially exponential time because a machine can revisit configurations.

Containments such as P subset NP subset PSPACE follow from simulation and certificate checking. Whether P equals NP, NP equals PSPACE, or other containments are strict remains unresolved in major cases. Do not present an inclusion as an equality or use a hard instance as evidence for separation.

## Certificates and verification

For a decision problem in NP, a yes instance has a certificate whose length is polynomial in the input and a verifier that checks it in polynomial time. SAT uses a satisfying assignment; Hamiltonian Cycle uses an ordered list of vertices; graph reachability uses a path. The certificate does not need to explain why a no instance is no. NP membership therefore requires a verifier and a length bound, not merely a belief that solutions are easy to check.

Complement changes the certificate direction. coNP contains complements of NP languages. A theorem that a problem is in NP does not automatically put its complement in NP unless a further collapse is proved. Search, optimization, and decision versions are related but not identical; a decision oracle can sometimes be used through self-reduction, but that construction needs proof.

## Reductions and completeness

A polynomial-time many-one reduction from A to B computes f such that x is in A exactly when f(x) is in B. If A is in a class and A reduces to B, this can transfer hardness: a fast B solver would give a fast A solver. To prove B is NP-complete, show B is in NP and reduce one known NP-complete problem to B. The direction matters.

Cook-Levin establishes SAT as NP-complete by encoding a polynomial-time nondeterministic computation as a Boolean formula. Later reductions often use gadgets that enforce local constraints. A gadget proof must establish completeness in both directions: every source solution maps to a target solution, and every target solution decodes to a source solution. Polynomial size and polynomial construction time are separate obligations.

## Space and configuration arguments

For a deterministic machine using s(n) space, the number of possible configurations is at most exponential in s(n) times the input length. If a computation exceeds this number without accepting or rejecting, a configuration repeats and the machine loops. This observation connects space bounds to reachability in an implicit configuration graph, but it does not by itself produce a practical algorithm.

Savitch’s theorem shows that nondeterministic space s can be simulated deterministically in roughly s squared space by recursively asking whether a midpoint configuration connects two endpoints. The recursion trades time for space. Understanding this proof is useful beyond the theorem: it illustrates divide-and-conquer over paths in an exponentially large graph and the importance of encoding configurations compactly.

## Lower bounds and model dependence

Lower bounds depend on the model and resource. A comparison sorting lower bound of Omega(n log n) does not apply to integer sorting with word operations and bounded keys. A finite-state lower bound on the number of DFA states does not say a Turing machine needs the same memory. State the allowed operations, input access, uniformity, and whether randomness is permitted.

Diagonalization constructs an object that differs from every item in an enumerated list, proving limitations of a model. The halting problem uses a program that contradicts the assumption of a universal decider. These arguments establish undecidability, not merely poor performance. A very slow algorithm still terminates; an undecidable problem has no total decider in the model.

## Reading complexity claims

When a paper says an algorithm is polynomial, ask in which parameters. A graph method may be polynomial in vertices and edges but exponential in a requested accuracy parameter. A fixed-parameter algorithm may be f(k) times a polynomial in n, which is tractable for small k but not polynomial uniformly in every parameter. Approximation may provide a tunable error-time tradeoff rather than one class label.

Benchmark evidence cannot establish P versus NP. It can reveal constants, scaling regimes, and useful instance structure. Likewise, a reduction can establish conditional hardness but does not predict runtime on a particular dataset. Keep theorem, assumption, and measurement in separate paragraphs.

## Exercises

1. Explain why binary and unary encodings change a complexity statement.
2. Give an NP verifier for Hamiltonian Cycle and its certificate length.
3. State the two obligations for an NP-hardness reduction.
4. Distinguish decidability from membership in P.
5. Explain why a configuration repetition argument is relevant to space-bounded machines.
6. Critique the claim that a ten-minute runtime proves a problem is outside P.

### Diagnostic solutions


1. The binary representation of n has length about log n, so a loop of n iterations is exponential in input length; unary length is n, so the same loop is linear.
2. The certificate is a list of all vertices in proposed cyclic order. Check each vertex appears once and every consecutive pair, including the closing pair, is an edge; the certificate has polynomial length.
3. Map every source yes instance to a target yes instance and every source no instance to a target no instance, while computing a polynomial-size output in polynomial time. Also explain how target solutions decode when the reduction is gadget-based.
4. Decidability only requires eventual termination. P requires a polynomial time bound under a specified encoding and machine model.
5. A bounded-space machine has finitely many configurations on an input. Repeating one means deterministic future behavior repeats, so the run loops rather than eventually producing a new result.
6. One observed runtime is one input and implementation. It says neither what happens as input length grows nor whether a different polynomial algorithm exists.



## Proof-writing template

A complete complexity classification should have the form: “Under encoding E and model M, problem X is in class C because [algorithm/verifier], and it is C-hard because [reduction with direction and resource bound].” If the lower bound is conditional, name the assumption. If only an empirical claim is available, label it empirical. This template keeps decidability, worst-case complexity, and observed performance from being conflated.

**Exercise.** Critique a claim that “the problem is NP because the program took a long time.” State the missing formal evidence.

## Advanced synthesis: class claims and evidence

A class-membership proof has a standard shape. Define the encoding and input length, describe an algorithm or verifier, bound its resource usage in that length, and prove correctness. For NP, a verifier must receive a certificate of polynomial length and check it in polynomial time. For PSPACE, count work-tape bits rather than the number of abstract variables. For a randomized class, state the probability space and error convention.

Hardness is a different direction. To show B is hard for a class, take an arbitrary known-hard problem A and reduce A to B. The target solver is then used as a subroutine for A. Prove membership separately if claiming completeness. A reduction that merely maps examples, or maps yes instances without excluding false target solutions, is not a proof.

Input representation is part of the theorem. A loop to n is polynomial in unary n but exponential in binary n. A graph algorithm may be polynomial in V+E, while a dense adjacency matrix has V squared entries. A statement such as “the algorithm is efficient” is incomplete until the measured resource, encoding, and model are named.

Do not infer a class from a benchmark. A ten-minute run is neither an undecidability proof nor an NP-hardness proof; it is one observation under one implementation and instance. Conversely, a worst-case hardness theorem does not imply that every real instance is difficult. A mature analysis states the theorem, its assumptions, and the empirical evidence in separate paragraphs, then explains what each can and cannot establish.
