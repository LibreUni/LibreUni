# Parsing and Ambiguity

Parsing determines whether a string belongs to a grammar and constructs a derivation. Top-down parsers expand predictions; bottom-up parsers reduce recognized handles. Ambiguous grammars require a disambiguation policy or a grammar rewrite.

Operator precedence is a semantic choice encoded syntactically: multiplication productions must bind more tightly than addition if the parse tree is to represent arithmetic meaning.

## Derivations and parse trees

A context-free grammar consists of variables, terminals, productions, and a start variable. A sentential form replaces one variable at a time; a parse tree records the entire derivation independently of the choice of expansion order. A leftmost derivation always expands the leftmost variable, while a rightmost derivation expands the rightmost. For an unambiguous grammar, a word has one parse tree even though it may have several derivation orders.

The distinction matters in proofs and parser design. A parser may commit to a leftmost derivation, but ambiguity is a property of trees, not of whether one implementation chooses one result. To prove ambiguity, give one word and two distinct trees or leftmost derivations. To prove unambiguity, show that every word has at most one possible structural decomposition, often by induction on length or tree height.

## Expression grammars

The grammar E - greater than  E+E | E*E | (E) | n is ambiguous. The word n+n*n has one tree with plus at the root and one with multiplication at the root. A precedence grammar separates nonterminals: E - greater than  E+T | T, T - greater than  T*F | F, F - greater than  (E) | n. Addition now combines terms, multiplication combines factors, and parentheses recurse to E.

Associativity is also structural. Left-associative subtraction uses E - greater than  E-E | n, while right-associative exponentiation uses a production whose recursive occurrence is on the right. A grammar that encodes precedence but not associativity can still yield multiple trees. State both policies in the language specification.

## Top-down parsing

A recursive-descent parser expands a predicted production and consumes terminals. Left recursion such as E - greater than  E+T causes immediate infinite recursion, so rewrite to a tail form E - greater than  T E-prime and E-prime - greater than  + T E-prime | epsilon. The rewrite preserves the language but changes the parse-tree shape; semantic actions must be adjusted to construct the same abstract syntax tree.

LL(1) parsing chooses a production using one lookahead symbol. FIRST(A) contains terminals that can begin strings derived from A. FOLLOW(A) contains terminals that can appear immediately after A in a sentential form. For an epsilon production A - greater than  epsilon, its table entries use FOLLOW(A). An LL(1) grammar requires no conflicting table entries; otherwise one symbol of lookahead cannot choose uniquely.

Nullable variables make FIRST computation iterative. Start with direct terminal beginnings, repeatedly add FIRST of right sides, and propagate through nullable prefixes. FOLLOW computation begins with the end marker after the start variable, then propagates constraints from every production until no set changes. The fixed point is a dataflow invariant: each iteration adds only justified terminals and termination follows from finite sets.

## Bottom-up parsing

Shift-reduce parsing reads input onto a stack and reduces a handle matching the right side of a production. An LR parser constructs a rightmost derivation in reverse. Shift/reduce conflicts occur when the table cannot decide whether to consume another symbol or reduce; reduce/reduce conflicts occur when two productions could apply. Precedence declarations resolve some arithmetic conflicts, but a hidden conflict may indicate a genuinely ambiguous language or an inadequate grammar.

The CYK algorithm parses grammars in Chomsky normal form using dynamic programming. For a substring from position i to j, record variables that derive it. A binary production A - greater than  BC is added when some split k has B deriving i..k and C deriving k..j. The base case handles terminal productions. The cubic time bound comes from O(n squared) intervals and O(n) splits, multiplied by grammar factors. The table is also a proof object: a true entry has a derivation assembled from its witnesses.

## Ambiguity and semantic actions

Ambiguity is dangerous when parse trees drive evaluation. The classic dangling-else grammar allows an else to attach to either of two if statements. A language specification should resolve this by requiring the nearest unmatched if, by adding explicit delimiters, or by using a grammar whose structure enforces the policy. A parser generator warning should not be silenced without explaining the intended semantics.

Separate concrete syntax from abstract syntax. Parentheses and punctuation may be essential for parsing but disappear from the abstract syntax tree. Source locations should be retained for diagnostics. Semantic actions must not mutate shared state before a parse choice is known, or backtracking can leave inconsistent effects.

## Error recovery and validation

A parser should report the unexpected token, location, and expected categories where practical. Panic-mode recovery skips input until a synchronizing token such as semicolon or closing brace; phrase-level recovery inserts or deletes a small token sequence. Recovery is a user-interface policy, not part of the grammar’s acceptance theorem, and it must not accidentally report a recovered tree as valid without marking errors.

Validate parser implementations against an independently specified grammar. Test empty and singleton forms, nested expressions, precedence boundaries, malformed delimiters, long chains, and ambiguous-looking inputs. Differential testing against a reference parser is useful, but compare abstract trees rather than formatting.

### Diagnostic solutions


1. Use E - greater than  T E-prime and E-prime - greater than  + T E-prime | epsilon. Parse the sequence of T values and fold additions from the left in the semantic action.
2. FIRST(F) contains (, n; FIRST(T) and FIRST(E) are the same. FOLLOW(E) contains ), end, and +; FOLLOW(T) contains *, +, ), and end; FOLLOW(F) contains *, +, ), and end.
3. In “if a then if b then c else d,” one tree attaches else to the inner if and another to the outer if. The grammar must encode a policy to remove the ambiguity.
4. For each interval and split, combine a variable deriving the left interval with one deriving the right interval under a binary production. There are O(n squared) intervals and O(n) splits, hence O(n cubed) time.
5. Use P - greater than  F ^ P | F, with F for atoms and parentheses. The recursive P on the right makes a^b^c parse as a^(b^c).
6. A grammar is ambiguous if a word has multiple parse trees. A deterministic parser may still use internal alternatives or lookahead while producing one tree; operational branching alone does not prove grammar ambiguity.

## Final audit

The implementation should reject an incomplete parse table rather than silently choosing a production. Test the empty input, the shortest derivation, nested delimiters, a precedence boundary, and an input with one missing terminal. For each accepted string retain a derivation or abstract syntax tree, so recognition and construction can be checked independently.

A parser’s complexity claim must name the grammar representation and the input length. A table-driven deterministic parser can be linear in input length after table construction; a general chart parser may be cubic in the worst case. Practical performance on an unambiguous grammar does not erase the worst-case bound of the chosen algorithm.

## Advanced synthesis: parser correctness

A parser proof has two directions. If the parser returns a tree, that tree must be a valid derivation whose leaves equal the input. If the input belongs to the grammar, the parser must eventually construct a tree rather than reject it. For a deterministic predictive parser, the table must have no conflicting entries; for a chart parser, every table entry should carry a witness production and split so that the proof can be reconstructed.

FIRST and FOLLOW calculations are fixed-point computations. Initialize the sets from terminal beginnings and the end marker, repeatedly apply production constraints, and stop only when no set changes. Nullable prefixes are the subtle case: FIRST of a right side includes the first terminal after every nullable variable. FOLLOW symbols propagate through a suffix when the suffix is nullable. A single missed nullable variable creates a plausible-looking but incorrect parse table.

Ambiguity must be demonstrated with two distinct parse trees for one word. A parser choosing one tree does not remove grammar ambiguity; it only implements a policy. Encode precedence and associativity in the grammar or state the disambiguation rule explicitly. Keep semantic actions separate from speculative parsing state so backtracking or error recovery cannot commit side effects from a discarded derivation.

For testing, generate trees from the grammar and serialize their yields, then mutate delimiters, operators, and nesting depth to produce negative cases. Compare abstract syntax trees, not whitespace or concrete-parenthesis formatting. Report error recovery separately from recognition: a recovered tree with an error marker is not the same as a valid parse.
