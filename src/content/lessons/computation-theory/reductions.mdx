# Computable Reductions

Reduction is the theory of translating one problem into another while preserving the answer. It formalizes the intuition that if problem B were easy, problem A would be easy too because an A-instance could be converted into a B-instance. Reductions are the main way to transfer undecidability and later complexity lower bounds. Their power comes from precise direction and quantifiers, not from saying that two problems “seem related.”

## Many-one reductions

For languages A and B, a computable function f is a mapping reduction (Ale_m B) when, for every string x, (xin A) iff (f(x)in B). The function must halt on every input and produce one finite B-instance. It may change the representation substantially, but it cannot ask an oracle for B while computing f.

If B is decidable and A≤mB, then A is decidable: compute f(x), run the B decider, and return the same answer. The contrapositive is the standard hardness transfer: if A is undecidable, then B is undecidable. To prove B hard, reduce a known-hard A to B. The reverse reduction proves a different fact and may be useless for the desired conclusion.

**Worked direction check.** Suppose (A_()TM()le_m E_()TM()), where E_TM contains encodings of machines with empty language. A transformation must arrange (M) accepts w iff the constructed machine’s language is empty, or arrange the complement equivalence and then target the complement problem explicitly. If the constructed machine accepts all inputs when M accepts w, the direct equivalence is (M) accepts w iff the new machine is nonempty, not empty. A correct proof must account for this negation rather than silently call it a reduction to E_TM.

## Building a reduction

Use a five-part template:

1. State source A, target B, and the known status of A.
2. On input x, describe the construction of f(x) using finite effective steps.
3. Prove f is computable and produces a valid encoding.
4. Prove the forward implication (xin ARightarrow f(x)in B).
5. Prove the reverse implication (xnotin ARightarrow f(x)notin B).

The constructed target machine may contain x and a source machine as literal data. It can simulate the source on a fixed input, perform a bounded transformation, or use a wrapper that accepts only under a condition. The wrapper itself must be a finite machine that can be generated mechanically from the input encoding.

## Classic wrappers

To reduce acceptance to a language property, construct a machine N whose behavior depends on the source computation. For nonemptiness, N ignores its input and simulates M on w, accepting every input if M accepts w. For finiteness, N can accept a finite fixed set if M accepts and no strings otherwise, depending on the target property. For “accepts epsilon,” N can simulate M on w on its empty input and accept epsilon exactly when M accepts.

Wrappers can be combined with complement-sensitive target properties. If the target asks whether a language is empty but the natural wrapper produces a nonempty language on a yes-instance, use a source complement if its status is known, target the complement language, or design a wrapper that accepts nothing on the desired yes case. The proof’s iff line must be visible enough to catch this choice.

## Turing reductions

A Turing reduction allows an algorithm for A to query an oracle for B multiple times, adaptively, and use the answers to decide A. It is weaker as a hardness-transfer tool than a many-one reduction because the reduction may make several target queries and can exploit both yes and no answers. If B is decidable, a Turing reduction still makes A decidable by replacing oracle calls with the B decider.

Many-one reductions preserve a single instance and compose directly. If A≤mB and B≤mC, then A≤mC via composition. Turing reductions also compose, but the simulation must manage oracle calls. For introductory undecidability proofs, mapping reductions are usually preferable because the construction and iff proof are easier to inspect.

## Reduction and complements

If A≤mB, then (overline()A()le_moverline()B()) using the same function. This does not imply A≤m(overline()B()) or (overline()A()le_m B). When a target is an emptiness or universality property, carefully track which side corresponds to source acceptance. Closure of decidability under complement lets one transfer undecidability between a language and its complement only when the logical relation has been established.

For example, (A_()TM()) reduces to the nonemptiness problem by the wrapper above. Since A_TM is undecidable, nonemptiness is undecidable. If emptiness were decidable, nonemptiness would be decidable by complementing its answer, contradiction. This second step uses decidability closure, not an unproved direct reduction with the wrong polarity.

## Recognizability transfer

If A≤mB and B is recognizable, A is recognizable: compute f(x), then run the B recognizer. If x is in A, f(x) is in B and eventually accepts; if x is not, no acceptance is required. For co-recognizability, the same statement applies to complements. A reduction from a language known not recognizable to B proves B is not recognizable, but a reduction from an undecidable language only proves B is not decidable if the target were decidable.

This hierarchy prevents overclaiming. A target may be recognizable but undecidable. To prove non-recognizability, often reduce from the complement of A_TM or show that recognizing both target and its complement would decide a known undecidable language. State whether the conclusion is “undecidable,” “not recognizable,” or “not co-recognizable.”

## Complexity intuition

In complexity theory, reductions must also respect resource bounds. A polynomial-time many-one reduction should compute f in polynomial time and produce an output of polynomial length. A polynomial reduction can transfer membership in P, NP-hardness, and completeness. A computable reduction can be arbitrarily slow and therefore says nothing about practical efficiency. The structural template remains the same, but the cost of the construction becomes a proof obligation.

## Reductions as abstraction boundaries

A reduction is an interface contract between problem definitions. The source algorithm is allowed to know only the target answer, not the target implementation. This is why the target instance must be finite and effectively produced before the target solver is called. A construction that says “ask whether M accepts w and then build N” has smuggled in the very oracle the proof is supposed to rule out; replace that question with a literal simulator embedded in N.

The wrapper-machine technique is especially powerful because a finite description can contain an arbitrary input string and source-machine description. The wrapper does not need to predict the source computation. It simply waits for the source behavior and makes its own language reflect that behavior. If the source accepts after a long time, the target machine may also wait a long time; computability concerns eventual behavior, while complexity concerns the cost of that wait.

## Sanity checks for a written proof

Test the reduction on four conceptual cases: a source yes-instance, a source no-instance that halts, a source computation that loops, and a malformed encoding. The target’s behavior must match the stated language in each case. Then check that f halts even when the source machine loops; constructing a wrapper is finite, while simulating the source belongs inside the target machine and is not performed by f. Finally check that the target encoding is valid and that any fixed input used by the wrapper is represented unambiguously.

## Exercises

1. Prove transitivity of mapping reductions.
2. Construct a reduction from A_TM to the problem “does M accept epsilon?”
3. Design a wrapper proving nonemptiness of recognized language is undecidable.
4. Identify the polarity error in a proposed reduction from acceptance to emptiness.
5. Contrast many-one and Turing reductions with one example of extra oracle power.
6. Explain exactly what conclusion follows if an undecidable A reduces to recognizable B.
7. Give the additional cost condition for a polynomial-time reduction.

### Diagnostic solutions


1. If f witnesses A≤mB and g witnesses B≤mC, then g∘f is computable and x∈A iff f(x)∈B iff g(f(x))∈C. Thus the composition witnesses A≤mC.
2. Given (langle M,wrangle), build N that on input epsilon simulates M on w and accepts exactly when M accepts. On any nonempty input, N can reject immediately. Then M accepts w iff N accepts epsilon. The construction is effective.
3. Build N that ignores its input and simulates M on w; if M accepts, N accepts, and otherwise N never accepts. If M accepts, N recognizes every string; otherwise its language is empty. Thus A_TM reduces to nonemptiness.
4. If the wrapper’s language is empty when M fails to accept and nonempty when M accepts, the equivalence is acceptance iff nonemptiness, not acceptance iff emptiness. The proposed proof has reduced to the complement target or reversed the source answer.

5. A mapping reduction creates one target instance before receiving any answer. A Turing reduction can ask “is x in B?” and then choose a second query based on the result, or query many related instances. Replacing oracle calls with a B decider preserves decidability, but the structures are not interchangeable for every hardness claim.
6. If B were decidable, the reduction would decide A, contradiction; therefore B is undecidable. It does not follow that B is not recognizable. A recognizable undecidable target is possible.
7. The reduction function must be computable within polynomial time and output a string whose length is polynomial in the input length. Correct iff preservation remains required.
