# Space and Randomness

## Space as configuration memory

A Turing machine using s(n) work cells has a finite configuration on each input: state, input-head position, work-tape contents, and work-head positions. If the machine is deterministic and repeats a configuration, its future repeats exactly. Therefore a halting computation cannot contain more configurations than the configuration bound. This converts a space restriction into a bound on possible histories.

The input tape may be read-only and its cells may not count as work space. Clarify whether the input head is one-way or two-way, whether the work alphabet is fixed, and whether the output tape counts. A logarithmic-space algorithm can use polynomially many configurations and therefore may take polynomial or exponential time depending on the model and halting argument.

## Savitch-style reachability

To determine whether configuration u reaches v within at most t steps, recursively guess a midpoint m and ask whether u reaches m within t/2 and m reaches v within t/2. The midpoint description uses the space of one configuration; recursion depth is logarithmic in t, so the total space is roughly configuration space squared after accounting for counters and recursion bookkeeping. The time may be large because many midpoints are tried.

This is a proof of simulation, not necessarily an efficient implementation. Its significance is that nondeterministic path guessing can be replaced by systematic reachability with a space tradeoff. When writing the recurrence, include the base case for one step and the bound on the midpoint counter.

## Randomized algorithms

A randomized machine has access to random bits. One-sided error means every no instance is rejected with certainty while yes instances may be rejected with small probability, or the dual convention. Two-sided error permits both mistakes with probability bounded below a constant. Amplification repeats independent trials and takes a majority or an OR/AND appropriate to the error type.

Independence and the adversary model matter. If an adversary sees random choices and adapts inputs, a proof for a fixed input sequence may fail. A probabilistic method proof may show that a random object exists without giving a sampling algorithm; an algorithmic result needs an efficient generation and verification procedure.

## Polynomial identity testing example

Given two arithmetic expressions, evaluate them at a random point over a finite field. If the represented polynomials differ and have degree at most d, their difference has at most d roots in the field. A uniformly random evaluation therefore detects inequality with probability at least 1-d divided by field size. Repeating reduces failure probability. The proof uses a root bound, not an assumption that random inputs “usually work.”

Hashing uses the same pattern. A universal family bounds collision probability for any fixed pair. If many pairs or queries are involved, apply a union bound or a stronger concentration theorem and account for the resulting seed length. Randomness reduces expected collisions but does not make collisions impossible.

## Lower bounds and derandomization

Randomness may help an algorithm but does not automatically enlarge computability: a machine with random bits can be viewed as a family of deterministic computations, and the meaning of success must quantify over outcomes. Las Vegas algorithms always return a correct answer but have random runtime; Monte Carlo algorithms have bounded runtime but may err. Do not interchange the labels.

Derandomization replaces random choices with a pseudorandom object, limited independence, an expander walk, or a conditional-expectation construction. The replacement must preserve the analysis’s needed moments or events. A deterministic seed is not automatically pseudorandom for the property under study.

## Exercises

1. Derive the repeated-configuration argument for a deterministic space-bounded machine.
2. Distinguish one-sided, two-sided, Las Vegas, and Monte Carlo error.
3. Prove the polynomial root bound used in identity testing.
4. Explain why a union bound is needed when many random tests are performed.
5. State the space-time tradeoff in midpoint reachability recursion.

### Diagnostic solutions


1. Count state, head positions, and tape contents. A halting run cannot repeat a configuration because deterministic continuation would loop; hence its length is bounded by the number of configurations.
2. One-sided algorithms forbid one error direction; two-sided permit both. Las Vegas is always correct with random runtime; Monte Carlo has bounded runtime with possible error.
3. A nonzero univariate polynomial of degree d over a field has at most d roots, by induction using division by x-r after one root is found.
4. Each test may fail with probability delta. The probability that any of m tests fails is at most m delta, so choose per-test error small enough for the desired total confidence.
5. Store one midpoint and recurse on two subproblems at half the path length. Recursion depth is logarithmic in the time bound, while repeated subproblem exploration increases time.



Space complexity counts work tape cells, separating problems that need broad memory from those that can stream. Randomized algorithms use random bits; a bounded-error algorithm may fail with small probability, while repetition can reduce error when trials are independent.

Randomness changes guarantees, not the need for a model. A Monte Carlo algorithm has bounded resources and probabilistic correctness; a Las Vegas algorithm always returns a correct answer but has random running time.

**Exercise.** If one independent trial fails with probability $1/4$, compute the failure probability after three trials when the algorithm returns the majority result, and state the independence assumption.

## Synthesis problems

Compare a deterministic log-space machine, a polynomial-space machine, and a randomized machine on reachability. The first two differ in workspace and may revisit configurations; the randomized machine must state error probability and whether it always halts. A correct comparison names the model before discussing the bound.

When proving a randomized guarantee, define the probability space, fix the input before drawing randomness unless an adaptive adversary is handled, and identify the event being bounded. Amplification changes failure probability but also multiplies work. Limited independence may suffice for a moment calculation, but do not claim full independence without providing it.

For space proofs, count bits rather than abstract variables. A counter up to n uses logarithmic space, while an array of n values does not. For configuration arguments, include input-head and work-head positions. These bookkeeping details are frequently the difference between a valid and invalid class-membership proof.

## Integrated problem set

Consider an implicit directed graph whose vertices are configurations of a machine and whose edges are legal transitions. A deterministic machine follows one successor; a nondeterministic machine asks whether a path exists. A space-bounded simulation stores only a pair of configurations, a recursion-depth counter, and the current midpoint candidate. This explains why reachability can be solved with less space than the full graph would require, while also explaining why the running time can be large.

For a randomized algorithm, write the claim in quantified form: for every fixed input x, the probability over random seed r that the output is incorrect is at most delta. If queries are adaptive, the input is no longer fixed independently of r; condition on the history or use a construction designed for adaptive adversaries. If m separate events must all hold, choose per-event failure at most delta divided by m and apply a union bound. If a majority vote is used, independence or a suitable concentration hypothesis must be stated.

Compare the following four statements: a machine uses O(log n) space; a machine runs in polynomial time; a randomized procedure is correct with probability at least 2/3; and a program eventually halts on this one input. None implies all the others. The first is a resource claim, the second a time claim, the third a probabilistic correctness claim, and the fourth a single-instance observation.

## Diagnostic exercises

1. Count configurations for a machine with s work bits, q states, and an input of length n.
2. State the invariant in a midpoint-reachability recursion.
3. Amplify a two-sided error probability below 1/3 to below 1/100 and state the assumption.
4. Explain why a finite timeout cannot establish nonhalting.
5. Give a situation where limited independence is enough and full independence is unnecessary.

### Diagnostic solutions


1. There are at most q times (n+1) choices for state and input head, multiplied by 2^s work contents and any work-head-position factors. The exact exponent depends on the tape model, but it is finite and exponential in s.
2. The recursive predicate is true exactly when the first configuration reaches the second within the allowed number of steps; each midpoint subcall preserves that statement for a shorter bound.
3. Repeat independently O(log 100) times and take the majority; a Chernoff bound gives exponentially decreasing error. Independence is required for the standard bound.
4. A computation may halt after the timeout. The observation proves only that it has not halted within the tested prefix.
5. A second-moment collision estimate may require only pairwise independence, while a tail bound for many simultaneous events may require stronger independence or a different concentration argument.
