# Theory Capstone

## Project question

Choose one language or decision problem and study it across the hierarchy of models. A strong project does not merely collect facts. It defines the problem, constructs the strongest useful machine, proves its correctness, identifies a weaker model that cannot recognize it if appropriate, and explains the resource cost of the construction. Possible topics include modular string validation, balanced syntax, arithmetic expression parsing, reachability, a halting-style program property, or a small NP-complete problem.

## Required artifact

Submit a six-to-ten-page mathematical report and an executable checker or simulator. The report must contain: formal alphabet and language; positive and negative examples; machine or grammar definition; invariant; correctness proof; complexity analysis; impossibility or hardness proof when claimed; experiments; and references. The executable artifact must include a README, deterministic seeds for randomized tests, input validation, and at least one independently generated oracle for bounded instances.

## Suggested project paths

For a finite-state project, derive a DFA from a prefix-information invariant, minimize it, and use a product search to produce distinguishing words. Compare the minimized state count with a Myhill-Nerode lower bound. Discuss how the state count changes when a modulus, forbidden pattern, or required suffix grows.

For a context-free project, give both a grammar and a PDA. Explain the stack invariant, trace representative words, and implement a recognizer or chart parser. If ambiguity exists, exhibit two parse trees and define a disambiguation policy. If claiming non-context-freeness, use pumping or a closure reduction with all quantifiers.

For a computability project, define a semantic program property and reduce from halting. The transformation must be syntactic and effective without simulating the unknown outcome. Prove the target property holds exactly in the intended case. Discuss why a recognizer, semi-decider, or bounded tester does not solve the total decision problem.

For a complexity project, define a decision version, prove membership with a certificate, and reduce a known NP-complete problem. Include the construction size and a decoding argument. Then discuss parameterized or approximation alternatives without presenting practical performance as a complexity theorem.

## Experimental protocol

Experiments should test claims rather than decorate the report. Generate exhaustive words up to length 10 for a finite alphabet when feasible, and compare the implementation with a direct specification. For parsers, generate valid trees and corrupt them at one location. For reductions, enumerate small source instances, map them, and compare source and target answers with independent brute-force solvers. Record false positives, false negatives, runtime, and memory.

If the project involves minimization, log state counts before and after each transformation. If it involves a reduction, log input and output sizes and verify the claimed polynomial relationship. If it involves a randomized algorithm, report repeated trials and confidence rather than one successful seed. Store raw data and code so another student can reproduce the result.

## Proof standard

Every theorem needs hypotheses. A DFA proof needs total deterministic transitions; an NFA proof needs existential-run semantics; a PDA proof needs a fixed stack convention; a pumping proof needs a witness longer than the unknown pumping length; a reduction needs both directions and an effective construction; an NP claim needs a polynomial certificate bound. Explicitly list which hypothesis is used at each key step.

## Oral defense questions

1. What exactly does each state or stack symbol remember?
2. Which input is the shortest counterexample to an incorrect transition?
3. Where does your induction start, and what are all inductive cases?
4. Which quantifier is adversarial in your non-regularity proof?
5. If your reduction target had a solver, how would you solve the source?
6. What changes under a different encoding or machine model?
7. Which experimental result is evidence and which statement is proved?

## Assessment rubric

Definitions and examples are 15 percent; construction and invariant are 25 percent; correctness or impossibility proof is 30 percent; complexity and model assumptions are 15 percent; implementation, tests, and reproducibility are 10 percent; exposition is 5 percent. A polished implementation cannot compensate for a missing proof, and an elegant proof cannot excuse an artifact that cannot be run or inspected.

## Final self-audit

Read the report as a skeptical examiner. Try to find an accepted counterexample, a rejected positive instance, an unhandled transition, a reversed reduction arrow, a hidden exponential certificate, or a use of “obvious” where a quantifier is doing the work. Add tests for every discovered boundary. The capstone is complete only when the mathematical statement, construction, proof, and experiment all refer to the same formal problem.

Choose a language or decision problem from software practice: configuration validation, program termination, scheduling, or protocol trace acceptance. Define its alphabet, input encoding, and required answer. Then classify it using an automaton, grammar, decider, reduction, or complexity argument.

The deliverable must distinguish “I built a recognizer,” “I proved decidability,” and “I measured a fast implementation.” Each is a different claim with a different proof obligation.

**Deliverable.** Submit a formal definition, one constructive algorithm or impossibility proof, a worked instance, and a critique of a tempting but invalid argument.

## Reproducibility appendix

Include a notation table, a machine-readable grammar or transition table, and commands for regenerating every figure. Store generated examples with their seeds. If the implementation rejects malformed syntax, document whether that behavior is part of the language or merely an error-reporting extension. If a theorem applies only to a restricted alphabet, parameter range, or encoding, put that restriction beside the theorem statement.

Separate negative results from engineering limitations. “No DFA exists” is a theorem proved by pumping or distinguishability. “Our simulator timed out” is an experiment. “The parser uses too much memory at n=1000” is a measurement. A capstone earns trust by refusing to turn one kind of statement into another.

The final oral defense should be reproducible on a whiteboard: define the object, state the invariant, execute one representative case, and explain the failure mode of the closest incorrect construction. If the project concerns hardness, draw the reduction arrow and decode a target witness. If it concerns undecidability, explain where a hypothetical decider is composed with the constructed program.

## Technical appendix requirements

The appendix should include the exact grammar, transition table, reduction pseudocode, or simulator interface used in the project. Give one trace with every configuration or parse-table entry needed to reproduce the result. If the project reports state growth or runtime, record the input generator, seed, machine environment, and raw observations. If the project claims a lower bound, include the family of distinguishers, pumping cases, or reduction map rather than only naming a theorem.

A useful capstone compares two constructions. For example, compare a direct DFA with a product DFA; a grammar recognizer with a chart parser; a brute-force solver with a verifier; or a recognizer with a bounded simulator. Explain which guarantees are mathematical and which are engineering. A comparison makes hidden assumptions visible: determinism, completeness, encoding, memory, and termination.

## Final defense checklist

Define the object without relying on an example. State what the machine remembers. Prove the invariant at initialization and after every transition. Name the exact counterexample strategy for an impossibility claim. Draw the reduction direction and decode a target solution. Explain whether a loop is permitted. Finally, state what your experiment can support and what it cannot.

## Minimum quality gate

Before submission, delete any paragraph that merely repeats a definition and replace it with a worked derivation, counterexample, or proof obligation. The report should contain at least one construction that a reader can implement, one theorem proved from an invariant or reduction, one failure case for a tempting alternative, and one reproducible experiment. Every reference should support a theorem, model, or algorithm actually used in the report.

The examiner should be able to ask a counterfactual—what if the input encoding changes, nondeterminism is removed, the stack is bounded, or the error probability is amplified—and find the answer in the assumptions and analysis. This is the difference between a catalog of constructions and a resource from which a student can reason independently.

The final artifact should therefore be judged by transfer: can a student use the definitions to analyze a new language, use the invariant to debug a new machine, and use the reduction template to classify a new problem? Include at least one unseen practice prompt with a solution outline so this transfer is explicit.

That unseen prompt should be graded with the same rubric: formal statement, construction, invariant, proof, resource bound, counterexample, and reproducibility.

Keep the unseen prompt formal and bounded enough to solve in an examination setting.
