# Undecidability

## Deciders, recognizers, and partial computation

A decider is a machine that halts on every input and accepts exactly the words in a language. A recognizer accepts every yes instance but may loop forever on no instances. The distinction is central: a recognizer for a language and a recognizer for its complement together can be combined into a decider by dovetailing, but one recognizer alone need not decide anything.

The halting problem asks whether a machine M halts on input w. It is recognizable: simulate M(w), accepting if the simulation halts. Its complement is not recognizable, because if both were recognizable, dovetailing would decide halting, contradicting undecidability.

## Diagonal proof

Assume a decider H(P,x) that accepts exactly when program P halts on x. Construct D that on input source code P runs H(P,P), loops if H says halt, and halts if H says loop. Because H is assumed total, D is a valid program. Run D on its own source. If H(D,D) says halt, D loops; if H(D,D) says loop, D halts. Each case contradicts H, so no such decider exists.

The self-reference is not a coding trick that depends on a particular programming language. Any effective model capable of representing its own descriptions supports the argument through a compiler or universal interpreter. The contradiction targets total correctness, not efficiency.

## Mapping reductions

To show a problem B undecidable, start with a known undecidable A and compute a transformation f such that x is in A exactly when f(x) is in B. If B had a decider, compute f(x), run the B decider, and decide A. The construction must halt for every x. The target instance may be a program, grammar, automaton, or machine description.

For example, given M and w, construct a program N that ignores its own input, simulates M(w), and accepts a designated property P only after the simulation halts. A decider for P would decide halting. To prove the converse implication, note that if M halts, N eventually exhibits P; if M does not halt, N never reaches the property. Do not rely on observing a timeout as evidence of non-halting.

## Rice’s theorem intuition

Rice’s theorem states that every nontrivial semantic property of the partial function computed by a program is undecidable. A semantic property depends only on behavior, so two syntactically different programs computing the same partial function agree on it. Nontrivial means some computable partial functions have the property and some do not. Properties such as “accepts no input,” “accepts at least one input,” and “computes the zero function” fit the pattern.

Syntactic properties are different: “the source contains an unused variable” may be decidable by parsing. The theorem does not say every interesting program question is undecidable; check semantic dependence and nontriviality first. A direct reduction is often clearer than citing Rice without matching hypotheses.

## Closure and decidability consequences

Decidable languages are closed under complement, union, intersection, and difference because their deciders can be composed and all halt. Recognizable languages are closed under union and intersection by dovetailing or parallel simulation, but not generally under complement. If a language and its complement are both recognizable, run both recognizers in parallel and accept when the appropriate one accepts; one must eventually do so.

These closure facts yield useful contradictions. If a proposed decider for an undecidable language existed, composing it with complement or intersection would decide a known impossible language. Conversely, a recognizer can sometimes be shown impossible by proving its complement is also impossible to recognize and invoking the two-recognizer theorem.

## Exercises

1. Prove the halting problem is undecidable by diagonalization.
2. Explain why halting is recognizable but its complement is not.
3. Design a mapping reduction to the property “program accepts some input.”
4. State the hypotheses of Rice’s theorem and give one syntactic property outside them.
5. Prove that decidable languages are closed under complement and intersection.
6. Explain why a one-hour timeout cannot prove non-halting.

### Diagnostic solutions


1. Assume H, construct D that does the opposite of H on self-input, and run D on D. Both possible H answers contradict D’s resulting behavior.
2. Simulate M(w) and accept if it halts. If the complement were recognizable too, dovetail both machines and decide which answer occurs first, contradicting undecidability.
3. Given M,w, construct N that ignores its input, simulates M(w), and accepts an input after the simulation halts. N accepts some input exactly when M halts on w.
4. The property must be semantic and nontrivial over computable partial functions. A source-text formatting property is syntactic and can be decided by parsing.
5. Run the complement decider and swap its answer; for intersection run both deciders and accept exactly when both accept. Since both halt, the compositions halt.
6. A non-halting computation can run longer than any fixed timeout. Observing no halt by time T establishes only that it did not halt within T.



Diagonalization constructs an object that differs from every purported complete list at a chosen coordinate. For programs, self-reference turns a universal analyzer into a contradiction: build a program that does the opposite of the analyzer’s prediction about itself.

Undecidability does not mean no instances are solvable; it means no single always-correct terminating algorithm handles all instances.

**Exercise.** Distinguish three claims: a program is slow, a program may not halt, and no algorithm can decide halting for all programs. Only one is a theorem about the problem class.

## Exam synthesis

The fastest way to classify a program question is to ask whether it depends only on computed behavior or on source syntax. For a semantic nontrivial property, try a halting reduction or Rice’s theorem. For a bounded simulation, count configurations and state the input model. For a recognizability question, ask whether positive evidence can be found by finite simulation and whether the complement has the same property.

A reduction proof should define what happens when the source computation does not halt. Nonhalting is often the no case, and the target recognizer may legitimately run forever there. If the target question asks for a decider, prove the constructed instance is always syntactically valid and that the hypothetical target decider halts on it.

Use dovetailing when combining recognizers: simulate one step of each, then two, and so on. This prevents a recognizer that loops on its input from starving the other. The same schedule works for enumerating two recursively enumerable sets and is a concrete example of how computability proofs become algorithms.

## Integrated exercises

1. Construct a recognizer for a language whose members have a finite witness but whose complement is not recognizable under a standard reduction.
2. Given a semantic property P that holds for one partial function and fails for another, outline a Rice-style reduction.
3. Prove that if both a language and its complement are recognizable, then the language is decidable.
4. Explain the difference between a program that loops on one input and a language that is undecidable.
5. Design a program transformation that delays a target property until a supplied machine halts.

### Diagnostic solutions


1. HALT is recognizable by simulating the supplied program and accepting after it halts; its complement is not recognizable because simultaneous recognizers would decide HALT.
2. Given M,w, construct a program that simulates M(w) and then behaves like a fixed program known to have P; on nonhalting inputs it behaves like a fixed program without P, or use a standard wrapper appropriate to the property. Verify both cases and the effective construction.
3. Dovetail the two recognizers, one step at a time. Exactly one must eventually accept for every input, so the combined machine halts and decides the language.
4. One looping computation is an instance-level fact. Undecidability says no single total algorithm correctly decides the property on all encoded instances.
5. Embed the supplied machine and input as constants in a wrapper; simulate them first, then execute the fixed behavior witnessing P. The wrapper is finite to construct even if the embedded simulation never halts.



## Proof-quality gate

Do not use “the program runs forever” as an observed conclusion unless a proof establishes it. A reduction may arrange that nonhalting corresponds to a no instance, but the reduction itself does not observe that no case. Likewise, Rice’s theorem applies to semantic nontrivial properties of computed functions, not arbitrary source-code patterns. Checking these hypotheses is part of the proof.

A complete answer distinguishes a proof of impossibility from a practical warning. The former quantifies over every candidate algorithm and uses a contradiction or reduction; the latter reports behavior of one program on one input. Both can matter, but they answer different questions.

When presenting the diagonal argument, write both branches of the contradiction rather than saying “paradox.” The mathematical content is that the assumed decider’s answer forces the opposite behavior on the same input.

This explicit case analysis is required in both written and oral solutions.

Do not replace this case analysis with an appeal to intuition.

This is part of the formal proof standard.

## Synthesis checkpoint

When classifying a new language property, first specify the machine model and exact yes/no language, then separate recognition from decision. A recognizer that eventually accepts positive instances is not evidence that negative instances can be certified. Write the reduction direction explicitly, identify the computable transformation, and state what a hypothetical solver would do on every branch. This discipline prevents reversing a complement or treating an unbounded simulation as a decider.
