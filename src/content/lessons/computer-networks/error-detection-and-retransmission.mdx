# Error Detection and Retransmission

Reliable communication is not the same thing as communication without errors. A channel may corrupt bits, lose frames, duplicate messages, reorder packets, or delay them long enough that a sender cannot tell whether the receiver has acted. A reliable protocol is an agreement about which observable histories count as successful delivery and how each participant responds when evidence is incomplete. This lesson develops that agreement from first principles.

## What must be detected?

Suppose a sender transmits a bit string (x) and a receiver obtains (y). The receiver needs an efficiently computable predicate that says whether (y) is consistent with a valid encoding of (x). No redundancy can detect every possible corruption: if every bit string is a valid codeword, a corrupted word remains valid. The design question is therefore which error patterns are guaranteed to be caught and what probability remains for patterns outside the guarantee.

A parity bit detects every odd number of flipped bits but misses every even number. A two-dimensional parity scheme adds row and column parity; it detects many more patterns and can locate some single-bit errors, but carefully aligned multiple errors may still preserve all parities. A checksum treats words as integers and adds them with a specified carry rule. It is inexpensive and useful against many random errors, but its algebra has structured collisions.

The detection guarantee is expressed using Hamming distance. If a code (C) has minimum distance (d), then it detects every corruption of at most (d-1) bit positions. To correct (t) errors, the minimum distance must be at least (2t+1), because two radius-(t) neighborhoods must not overlap. This distinction matters: a protocol that detects corruption still needs a recovery action; detection alone does not reconstruct the original packet.

## CRC as polynomial division

A cyclic redundancy check views a bit string as a polynomial over (GF(2)). Addition and subtraction are both XOR, so long division uses XOR rather than carries. Let the generator polynomial be (G(x)) of degree (r). The sender appends (r) zero bits to the message polynomial (M(x)), divides (M(x)x^r) by (G(x)), and appends the remainder (R(x)). The transmitted codeword is (T(x)=M(x)x^r+R(x)), which is divisible by (G(x)).

If noise adds an error polynomial (E(x)), the receiver obtains (T(x)+E(x)). Division by (G) produces zero exactly when (G) divides (E). Thus a CRC detects every error pattern whose polynomial is not a multiple of the generator. A generator with a factor (x+1) detects every odd number of bit flips; a generator of degree (r) detects every burst of length at most (r); additional properties follow from its factors.

**Worked CRC calculation.** Encode message `1101011011` with generator `10011`, whose degree is 4. Append four zeros: `11010110110000`. XOR the generator under each leading one during binary long division. The final four-bit remainder is the CRC; appending that remainder makes the dividend divide evenly. At reception, the same division is performed on all transmitted bits. A nonzero remainder rejects the frame. The worked arithmetic is less important than the invariant: the sender constructs a multiple of (G), and the receiver checks divisibility.

CRC is not encryption, authentication, or a proof that a frame originated from a particular sender. An attacker who can alter a frame and recompute its CRC is not stopped. Even against accidental noise, a CRC’s undetected-error probability depends on the error model and generator; “one in (2^r)” is a useful random-error approximation, not a universal theorem.

## From detection to reliable delivery

Let a data link deliver frames with possible corruption and loss. The simplest stop-and-wait protocol sends one frame, waits for an acknowledgment, and retransmits after a timeout. The acknowledgment must identify what it acknowledges. If an ACK says only “yes,” a delayed ACK from an earlier exchange can be mistaken for the response to a later frame. Sequence numbers make message instances distinguishable.

The alternating-bit protocol uses sequence numbers 0 and 1. The sender transmits data with bit (b); the receiver delivers it only if it is the next expected bit, then acknowledges (b). If a duplicate arrives, the receiver does not deliver it again but repeats its ACK. If the sender times out, it retransmits the same numbered frame. This gives at-most-once delivery at the receiver and, assuming eventual non-corrupt delivery and a functioning timeout, eventual delivery.

The key safety invariant is: the receiver delivers a sequence number only once, and the sender never treats an acknowledgment for another sequence number as confirmation. The liveness argument is conditional. If the channel can lose every retransmission forever, no protocol can guarantee delivery. If the channel eventually allows a frame and its acknowledgment through, repeated retransmission eventually succeeds, although an overly short timer can create duplicates and an overly long timer increases delay.

## Timer design and ambiguity

The sender does not observe the one-way channel delay directly. It measures a round-trip sample from transmission to acknowledgment. Let (SRTT) be a smoothed RTT estimate and (RTTVAR) a smoothed mean deviation. A common estimator uses


\[
SRTT \leftarrow (1-\alpha)SRTT+\alpha R,
\qquad
RTTVAR \leftarrow (1-\beta)RTTVAR+\beta |SRTT-R|,
\]

and sets the retransmission timeout to (SRTT+K\cdot RTTVAR), with a lower bound. The constants are engineering choices, not magic correctness values. The timer should cover normal variation while still reacting to persistent delay.

Retransmitted segments create an ambiguity: an ACK may acknowledge the original transmission or a retransmission. Measuring a sample from an ambiguous acknowledgment can make the estimator believe the network is faster or slower than it is. Karn’s algorithm avoids using RTT samples for retransmitted data, and exponential timeout backoff prevents a sender from increasing load during congestion. Timer behavior is part of congestion control, not merely an implementation detail.

## Sliding windows and cumulative acknowledgments

Stop-and-wait permits only one outstanding packet, so bandwidth-delay product can make utilization poor. A sliding-window protocol permits up to (W) unacknowledged sequence numbers. The sender maintains `base`, the oldest unacknowledged sequence, and `next`, the next sequence to transmit. The receiver maintains the next in-order sequence it expects. Cumulative acknowledgment (k) can mean “all sequence numbers through (k) arrived in order.”

For a link rate (R), packet size (L), and RTT (T), the number of packets needed to keep the link busy is approximately (RT/L). If (W) is smaller, the sender may run out of work before an ACK returns. If (W) is large, it can fill queues and create loss. A window is therefore both a reliability bound and a flow/congestion-control parameter.

Go-Back-N has a receiver that accepts only the next expected packet and a sender that retransmits the outstanding suffix after a timeout. Selective Repeat buffers out-of-order packets and retransmits only missing ones. Selective Repeat uses more receiver memory and more state, and its sequence-number space must be large enough that old packets cannot be confused with new packets. If the window size is (W), a standard safe design uses a sequence space at least (2W) so that the sender and receiver windows do not become indistinguishable after wraparound.

**Worked window trace.** With window (W=4), packets 0–3 are sent and packet 1 is lost. A receiver accepting only in-order data acknowledges 0 and repeats that cumulative acknowledgment when 2 and 3 arrive. Go-Back-N retransmits 1, 2, and 3. Selective Repeat records 2 and 3, acknowledges them individually or with selective blocks, and retransmits only 1; after 1 arrives, buffered data can be delivered in order. The better choice depends on loss rate, memory, acknowledgment format, and implementation complexity.

## Link reliability versus end-to-end reliability

A local link can retransmit a damaged frame, reducing the rate of errors exposed to the transport layer. But local repair cannot prove end-to-end delivery: a later router may drop the packet, the destination process may reject it, or the return acknowledgment may be lost. End-to-end protocols therefore retain sequence, acknowledgment, timeout, and duplicate-suppression logic even when links are highly reliable.

Layering creates a tradeoff. Link retransmission can hide transient wireless errors and avoid expensive end-to-end recovery. It can also increase delay variance, causing transport timers to fire while the link is still retrying. A transport protocol should reason about the delivered observation stream—loss, delay, duplication, and reordering—not assume that an intermediate layer has eliminated them.
