# Networks Comprehensive Assessment

This assessment is designed as an oral-and-written final for a full networks course. A strong answer states assumptions, names the layer and state being analyzed, performs the relevant calculation, and explains what evidence would distinguish competing causes. Answers that merely recite protocol names are incomplete. For calculations, show units and identify approximations. For designs, state safety and liveness properties, failure behavior, and operational limits.

## Part I: foundational reasoning

1. A message crosses a link of rate (R), has (L) transmitted bits, and traverses propagation distance (d) at speed (v). Derive the serialization and propagation components of delay. Explain why increasing (R) does not reduce propagation delay.
2. Contrast circuit reservation, packet switching, and statistical multiplexing. Give a workload where statistical multiplexing is beneficial and a workload where queueing creates unacceptable tail latency.
3. Explain the difference between an address, an interface identifier, a port, a socket, and an application identity. Why can none of these alone prove that a remote peer is trustworthy?
4. A receiver reports a checksum failure. What has been established, and what remains unknown about the cause and security consequences?

## Part II: link and network layer

5. A switch learns A on port 1 and B on port 3. Trace the forwarding of an unknown unicast, a known unicast, and a broadcast. Explain aging and stale mappings.
6. Design a VLAN/trunk configuration for three isolated departments and one shared server segment. State where routing occurs and how a mis-tagged frame behaves.
7. A forwarding table contains `0/0 → X`, `10/8 → Y`, `10.4/16 → Z`, and `10.4.7/24 → W`. Determine the next hop for `10.4.7.20`, `10.4.9.20`, `10.8.1.1`, and `192.0.2.1`, and justify each with prefix length.
8. Run Dijkstra on a graph with edges A-B=4, A-C=2, C-B=1, B-D=5, C-D=8, and D-E=2. Give distances and predecessor paths from A. Explain the nonnegative-edge assumption.
9. Compare link-state, distance-vector, and path-vector protocols. For each, state what is exchanged, how loops are limited, and how policy enters the decision.
10. Explain how an inconsistent FIB can cause a forwarding loop even when every router’s control-plane route looks locally valid.

## Part III: transport and performance

11. Specify an alternating-bit reliable-delivery protocol and prove no data is delivered twice under arbitrary duplication and delay, assuming sequence-space ambiguity is avoided.
12. Compare Go-Back-N and Selective Repeat after one loss in a window of 16 packets. Discuss bandwidth, buffering, sequence space, and implementation complexity.
13. A path is 200 Mb/s with RTT 50 ms and 1,500-byte packets. Estimate the window required to fill the path. What additional factors make the estimate imperfect?
14. Explain the difference between receiver flow control and congestion control. Give a trace signature for each limiting a sender.
15. A TCP sender sees increasing RTT, packet loss, and a full queue. Explain a plausible congestion response and why immediate fixed-rate retransmission would worsen the condition.
16. Design a UDP request/response protocol for a state-changing operation. Include IDs, retries, duplicate handling, authentication, response loss, and overload behavior.

## Part IV: names, applications, and security

17. Reconstruct a web startup path from DNS query through TCP, TLS, and HTTP. Identify the measurements needed to localize 900 ms of delay.
18. Explain why packet boundaries cannot be used to parse HTTP/1.1 and why HTTP/2 multiplexing changes trace interpretation.
19. Describe certificate-chain and hostname validation. Give an active attack enabled by skipping either one.
20. A service’s clients retry every timeout immediately. Analyze retry amplification and design a safer policy for idempotent and non-idempotent operations.
21. Give a threat model for a public API and state one confidentiality, integrity, authentication, freshness, authorization, and availability requirement.
22. Explain why a checksum does not provide message authentication and what an AEAD record adds.

## Part V: integrated design case

You must design a multi-region service. Clients use DNS to discover an address, connect over an authenticated encrypted channel, issue idempotent reads and non-idempotent writes, and receive responses through a routed provider network. Regions have independent links but share an identity authority and a database service.

23. Draw the dependency graph and identify at least three true and three false forms of redundancy.
24. Specify the client request state machine, including timeout, retry, replay, duplicate, and unknown-outcome states.
25. Choose measurements that distinguish DNS failure, route blackhole, TLS failure, server overload, and database latency.
26. Define routing and application invariants that can be checked during a deployment.
27. Explain how a more-specific route, a certificate rotation, or a resolver cache can produce a partial outage.
28. Propose a staged change and rollback plan with blast-radius controls.

## Diagnostic solutions and grading guidance

1. Serialization is (L/R); propagation is (d/v). Higher rate reduces serialization but not the physical travel time. Queueing, processing, and transmission overhead may add further terms.
2. Circuit reservation gives predictable reserved capacity but can waste idle allocation. Packet switching shares statistical bursts efficiently but queues under offered load. Bursty web traffic benefits from sharing; interactive control traffic suffers when a shared queue has a long tail.
3. An address locates an interface in a forwarding context; a port identifies a transport endpoint; a socket is local state; an application identity is a semantic principal. Authentication requires a cryptographic binding and policy, not an observed label.
4. The receiver found the frame inconsistent with its check. The cause may be noise, truncation, implementation error, or attack. A checksum generally detects accidental corruption but does not authenticate an origin or stop recomputation by an attacker.
5. Unknown unicast is flooded within the domain; known unicast is sent to the learned port; broadcast is flooded. Sources update learning, and aging removes entries so topology changes do not leave permanent forwarding to an old port.
6. Use one access VLAN per department, tagged trunks with explicit allowed lists, and a routed gateway for inter-VLAN policy. A wrong tag places a frame in the wrong domain or causes it to be discarded, depending on ingress validation.
7. `10.4.7.20` uses W (/24); `10.4.9.20` uses Z (/16); `10.8.1.1` uses Y (/8); `192.0.2.1` uses X (/0). Longest matching prefix wins.
8. Distances are A=0, C=2, B=3 via C, D=8 via B, E=10 via D. Dijkstra settles the smallest tentative label and relaxes edges; negative edges invalidate the finality argument.
9. Link state exchanges topology and computes locally; distance vector exchanges estimates and can suffer count-to-infinity; path vector exchanges prefixes with policy attributes and AS paths. Loop prevention and policy differ accordingly.
10. A can choose B while B chooses A based on stale or independently valid state. The combined next-hop graph contains a cycle until updates converge; TTL limits packet lifetime but does not repair the control plane.
11. Tag each frame with an alternating bit, deliver only the expected bit, toggle after delivery, and repeat the acknowledgment for duplicates. Induction over receives proves each bit is delivered once; liveness additionally requires eventual successful exchanges.
12. Go-Back-N retransmits the suffix and buffers little; Selective Repeat retransmits missing packets and buffers out-of-order data. Selective Repeat uses more state and needs a sequence space large enough to distinguish old from new windows.
13. (200,000,000\times0.05/(1500\times8)\approx833.3), so about 834 packets. Headers, ACKs, variable RTT, congestion control, receive window, and queue dynamics alter usable throughput.
14. Flow control reflects receiver capacity and advertises a small window; congestion control responds to network signals such as loss, ECN, or RTT growth. Inspect advertised window, inferred in-flight bytes, loss, and delay together.
15. The sender should reduce its sending window/rate, retransmit conservatively, and back off timers. Fixed-rate retransmission adds traffic to the full queue, increasing loss and delay in a positive feedback loop.
16. Include authenticated operation IDs, bounded exponential retries with jitter, durable server deduplication, and an explicit unknown outcome. Reads can often retry safely; writes require idempotency keys or a status query. Enforce per-client and global budgets.
17. Measure resolver response, address selection, SYN/SYN-ACK, TLS messages, request-to-first-byte, and body transfer. A total alone cannot localize delay; compare packet timing and endpoint logs.
18. TCP is a byte stream, so segments can split or combine messages. HTTP/2 multiplexes streams in one encrypted connection, so a stalled object may share transport with other objects and packet-to-request mapping needs stream metadata.
19. Chain validation authenticates authority signatures and constraints; hostname validation binds the leaf to the intended service. Skipping chain validation enables an untrusted issuer; skipping hostname validation permits a valid certificate for another service.
20. Immediate retries multiply load during failure. Use classification, budgets, exponential backoff, and jitter; require operation IDs and durable deduplication for state changes.
21. A public API may face passive readers and active injectors. Requirements must separately cover plaintext secrecy, tamper detection, peer identity, replay rejection, permission checks, and bounded resource use.
22. A checksum detects many accidental errors and is recomputable by an attacker. AEAD authenticates ciphertext, associated context, and nonce discipline while providing confidentiality under its assumptions.
23. Independent power and links can be real redundancy; two IPs on one device or two services using one database are false redundancy. Shared identity authority is another common dependency.
24. Include idle, outstanding, completed, and unknown-outcome states; bind responses to authenticated IDs; retry only within a budget; query status after ambiguous writes.
25. Use resolver logs, route/FIB and path probes, handshake traces, service timing, queue/load metrics, and database spans. Take observations from multiple vantage points and preserve timestamps.
26. No unauthorized prefix origin, all next hops resolve, no critical forwarding cycle, certificate identity matches service, each write ID commits once, and tail latency/error rates remain below deployment thresholds.
27. More-specific routes can attract only part of traffic and blackhole it; certificate rotation can fail one region’s trust path; stale DNS can direct clients to a withdrawn address. Partial scope explains why some users succeed.
28. Validate offline, deploy to one region or neighbor, monitor control and application invariants, keep a timed rollback, then expand. Preserve the previous configuration and define abort thresholds before starting.
