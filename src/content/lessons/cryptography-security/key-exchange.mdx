# Key Exchange

## Goal and adversary model

Key exchange lets two parties derive a shared secret over a public network. The security target is usually authenticated key exchange: a passive observer should not learn the session key, and an active attacker should not make one party accept a key shared with the wrong peer. Forward secrecy means compromise of a long-term authentication key does not reveal completed sessions. Key confirmation means a party obtains evidence that the peer derived the same secret.

Diffie–Hellman alone provides secrecy against passive observation but is vulnerable to man-in-the-middle substitution. Each side sends a public group element; an attacker establishes one secret with each endpoint and relays traffic. Authentication binds the ephemeral exchange to a certificate, signature, pre-shared key, or trusted channel. The signature must cover the transcript and roles, not only the public key, or messages can be reflected or transplanted between sessions.

## Finite-field Diffie–Hellman

In a cyclic group G generated by g, Alice chooses secret a and sends A=g^a; Bob chooses b and sends B=g^b. Alice computes B^a=g^(ab) and Bob computes A^b=g^(ab). The discrete-log or computational Diffie–Hellman assumptions state that deriving the shared value from public values is infeasible for selected groups. The protocol must specify group order, generator, encoding, and public-element validation.

Small-subgroup attacks arise when an attacker sends an element of low order. The victim’s response can reveal the secret exponent modulo that order. Defenses include validating that the element lies in the intended subgroup, multiplying by a cofactor appropriately, using a prime-order abstraction, or selecting a protocol whose proof handles invalid inputs. Rejecting an all-zero or identity shared secret is often necessary but not always sufficient.

## Elliptic-curve exchange

Elliptic-curve Diffie–Hellman replaces modular exponentiation with scalar multiplication on a curve. X25519 specifies an encoded function with rules designed to simplify safe implementation, but callers still need to handle an all-zero shared secret and derive application keys with a KDF. The shared group element is not directly an encryption key: use a transcript-bound KDF with labels, roles, and context.

Encoding matters. A public key may be malformed, noncanonical, or from the wrong domain. A protocol must specify whether validation is performed by the primitive or by the caller. Different curve or group parameters must not be silently accepted under the same algorithm identifier.

## Transcript binding and key schedule

A robust handshake derives traffic secrets from the ephemeral shared secret and a transcript hash. The transcript includes version negotiation, algorithm selection, public shares, identities or certificate evidence, and application context. Separate labels derive client-to-server and server-to-client keys, handshake and application keys, exporter values, and resumption secrets. Key separation prevents a key used as a MAC key in one context from being interpreted as an encryption key in another.

TLS 1.3 uses HKDF-Extract and HKDF-Expand-Label in a staged schedule. A pre-shared key may contribute an early secret, an ephemeral exchange contributes the handshake secret, and later derivations produce application traffic secrets. The Finished message authenticates the transcript with a key derived from the handshake secret. This is why a seemingly harmless transcript omission can enable downgrade or identity-confusion attacks.

## Downgrade, replay, and resumption

Negotiation must prevent an attacker from forcing a weaker version or algorithm. Include the negotiated parameters in authenticated transcript material and use downgrade signals where specified. Resumption tickets are bearer credentials; protect their confidentiality and integrity, limit lifetime, and bind them to the intended server and policy. 0-RTT data can be replayed and lacks full forward secrecy for that early data, so restrict it to idempotent operations or add an application replay defense.

Key exchange does not itself solve endpoint compromise. A stolen endpoint traffic key can decrypt current records; a stolen long-term signing key may enable impersonation but should not reveal past ephemeral sessions if forward secrecy holds. State the compromise timeline in the security claim.

## Worked handshake analysis

Consider a client and server that exchange ephemeral shares and a server certificate. The client sends a ClientHello containing a fresh random value, supported versions, cipher suites, and a key share. The server responds with the selected parameters, its share, certificate, signature over the transcript, and a Finished value. The client validates the certificate name and chain, checks the signature against the transcript and server identity, derives the same traffic secret, and verifies Finished before sending application data.

For each message ask four questions: is it confidential yet, is it integrity protected, what secret authenticates it, and what replay or downgrade state is checked? Early negotiation is visible and must be authenticated later; handshake messages after the key schedule are encrypted but still require transcript authentication. The Finished value is not a user signature; it proves possession of the handshake traffic secret and transcript agreement.

Nonce and key derivation must be deterministic for both endpoints but independent across directions. Use labels such as client handshake traffic and server handshake traffic rather than deriving one key and assigning roles by convention. Include the transcript hash in derivation so two handshakes with the same ephemeral secret but different identities or parameters do not share application keys.

## Failure and interoperability tests

Test mismatched shares, invalid points, wrong certificate names, altered transcript messages, unsupported versions, duplicate extensions, downgrade attempts, all-zero shared secrets, expired tickets, replayed early data, and simultaneous key updates. Assert that the endpoint terminates or falls back according to the specification without exposing whether a secret or identity check failed. Interoperability tests must not weaken validation merely to accept a peer’s malformed message.

Measure key agreement only with public inputs and constant-time library primitives. Do not log ephemeral private keys, raw shared secrets, session tickets, or transcript material containing personal data. If debugging requires derived values, use a test-only exporter and synthetic credentials, never production secrets.

The central exam habit is to trace every secret and every authenticated transcript field. If a key is derived without a role label, or a signature omits a negotiated parameter, identify the exact substitution or reflection an active attacker can attempt. A key exchange is secure only as a composed protocol, not as an isolated algebraic equation.

Also record the compromise boundary: forward secrecy protects completed sessions from later long-term-key theft, but it cannot repair endpoint memory compromise during the session.

This distinction belongs in the protocol’s explicit security claim.

State it before implementation.

Diffie–Hellman lets two parties derive a shared value from public group elements and private exponents. Without authentication, a man-in-the-middle can establish separate keys with each endpoint. Authenticated key exchange binds the ephemeral values to identities.

Forward secrecy uses ephemeral secrets so later compromise of a long-term key does not decrypt recorded past sessions, assuming ephemeral secrets were erased.
