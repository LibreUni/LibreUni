# Password Security

## Passwords as low-entropy secrets

A password is usually chosen from a human distribution, not uniformly from all strings of a given length. An attacker therefore guesses common passwords, breached-password variants, names, dates, keyboard patterns, and transformations before attempting exhaustive search. The relevant quantity is guesswork: how many guesses an optimal attacker needs under the population distribution. Length helps, but predictable composition rules may add little entropy.

The server should not store a password or a fast general-purpose digest. It stores a verifier produced by a password hashing or KDF function with a unique salt and explicit cost parameters. When a login arrives, parse the algorithm and parameters from the stored record, recompute the verifier, compare in a timing-safe way, and update parameters after a successful login when the record is old. Do not silently fall back to a weaker algorithm when parsing fails.

## Salts, peppers, and work factors

A salt is public, random, and unique per credential record. It prevents identical passwords from producing identical verifiers and defeats a single precomputed table across users. It does not need to be backed up separately if it is stored with the verifier. A pepper is a separately protected secret, often held in a secrets manager or hardware boundary; it can increase resistance after database-only compromise but introduces availability, rotation, and recovery dependencies.

The cost parameter should be measured against the threat and hardware. A time-only cost can be parallelized cheaply by attackers with specialized hardware; a memory-hard function raises the cost of massive parallel guessing. Argon2id is a common modern choice when available, but the selected library, version, parameters, and deployment constraints must be documented. Work factors must be high enough for attackers and tolerable for legitimate login traffic, with rate limiting and monitoring providing additional protection.

## Online and offline attacks

Online attacks are constrained by the verifier endpoint and can be mitigated by throttling, exponential backoff, risk-based challenges, IP and account signals, and phishing-resistant authentication. Do not use permanent lockout as the only defense because an attacker can weaponize it for denial of service. Error messages should not reveal whether an account exists, but logging and internal metrics need enough detail to detect attacks.

Offline attacks begin when an adversary obtains the verifier database, backups, logs, memory snapshots, or a compromised KDF service. The attacker can test guesses without network rate limits. The defense is a strong password policy informed by breached-password checks, a high-cost KDF, unique salts, protected peppers where justified, and minimizing the sensitivity and retention of stored records. Encryption of the database helps against some exposures but does not help after the decryption key is stolen.

## Authentication lifecycle

Registration must validate the password policy, generate the salt with a cryptographic random generator, derive the verifier, and store the algorithm identifier, version, salt, and parameters atomically. Password changes should require current authentication or a high-assurance recovery path, invalidate or rotate sessions, and prevent reuse of a recently compromised password. Reset tokens must be high entropy, single use, short lived, bound to the account, and excluded from URLs or logs where possible.

Recovery is often weaker than login. Security questions, support overrides, email-only reset, and reused backup codes can bypass a carefully designed KDF. Treat recovery as an authentication mechanism with its own assurance level, threat model, audit trail, and abuse detection. Notify users of changes without including secrets or reset links in insecure channels.

## Multi-factor and phishing resistance

Password authentication proves knowledge of a secret, not possession of a device or origin of a website. A second factor can reduce risk, but SMS and email codes have different interception and recovery risks from hardware-backed or cryptographic authenticators. Phishing-resistant authenticators bind the origin or relying party into the authentication ceremony. MFA enrollment, replacement, and recovery must be protected at least as strongly as ordinary login, or an attacker will target the lifecycle edges.

## Timing, enumeration, and privacy

An authentication response should avoid revealing whether the username exists, whether the password was close, or whether a particular recovery state is active. The server should perform comparable work for absent and present accounts where feasible, while avoiding unnecessary expensive KDF work that can enable denial of service. Rate limits must be designed per account, source, device, and credential type rather than relying on one IP address.

Credential logs should contain event identifiers and safe metadata, not passwords, reset tokens, session cookies, or full verifier records. Breach response should revoke sessions, rotate peppers or signing keys when relevant, force resets based on evidence, and preserve an investigation trail. Privacy requirements constrain how long authentication telemetry and identity data are retained.

## Parameter calibration and migration

Calibrate a KDF on the actual authentication tier, not a developer laptop. Measure median and tail latency under expected concurrency and memory pressure, then choose a policy that leaves capacity for ordinary traffic and abuse controls. Record the parameter set in every verifier. On successful authentication, transparently rehash with the current policy; for dormant accounts, require a reset or a separate migration campaign. Never reduce cost merely because old hardware is inconvenient without documenting the changed threat model.

A verifier record should be self-describing: algorithm, version, salt, memory or iteration cost, parallelism, and derived output length. Parse it defensively and reject unknown or excessive parameters before allocating. An attacker who controls a registration or login field should not be able to force the server to perform an unbounded KDF. Set policy maxima and return a safe failure for malformed records.

## Worked compromise response

If an attacker obtains only salted verifiers, assume offline guessing has begun. Preserve forensic evidence, identify algorithm and parameter versions, invalidate sessions if account takeover is plausible, notify affected users, force resets according to risk, and screen new passwords against known compromised values. If a pepper or KDF service key may be compromised, rotate it in a way that does not make all legitimate logins impossible; a staged migration may require reauthentication. Review backups and replicas because deleting one primary copy does not remove the exposure.

Password security is a system property. Strong KDF parameters cannot repair phishing, session theft, weak recovery, logging of reset tokens, or an authorization bug. Conversely, MFA and rate limits reduce risk but do not justify storing plaintext or fast unsalted hashes.

Recovery is an authentication protocol with its own threat model. A reset link that grants access must be single-use, time-limited, and protected from account-enumeration leaks.
