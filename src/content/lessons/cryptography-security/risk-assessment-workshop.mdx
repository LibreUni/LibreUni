# Risk Assessment Workshop

Build a small risk register for a service: asset, threat, precondition, consequence, likelihood rationale, control, residual risk, owner, and evidence. Avoid treating a framework category as a control by itself.

## Risk vocabulary

Risk is not the same as vulnerability, threat, incident, or control. A threat is a potential cause of harm; a vulnerability is a condition that enables it; an incident is an occurrence; a control changes likelihood, impact, detection, or recovery; risk is the possibility and consequence of the harm under stated assumptions. A risk register should make these relationships visible rather than assigning one unexplained number.

Start from assets and outcomes. “SQL injection” is an incomplete risk statement. A complete statement might be: an unauthenticated or low-privilege attacker can cause a query to interpret supplied input as SQL, bypass tenant predicates, and read or modify confidential records. The affected assets, preconditions, consequence, and control are now testable. The same vulnerability may produce different risk in a public catalog and a medical-record service.

## Likelihood reasoning

Likelihood should cite evidence: exposure, attacker skill, exploit availability, required access, control coverage, frequency of attempts, and detection time. Avoid false precision when the data supports only qualitative reasoning. State confidence and uncertainty. A likely event with low impact may be prioritized below a rare event that destroys a root signing key, but the decision should include blast radius, reversibility, legal consequence, and time to detect.

Use scenarios rather than averages. An attacker may try one account takeover, many low-cost credential guesses, or a targeted supply-chain compromise. Controls can change the attacker’s cost and the number of opportunities. Rate limits reduce online guesses but do not reduce offline guesses after database theft. Encryption reduces storage-observer exposure but may not reduce exposure to an application process that has decryption authority.

## Control mapping

For each risk, name the control’s mechanism and enforcement location. A secure cookie prevents some client-side access to a session token; it does not prevent server-side object authorization failure. AEAD detects ciphertext modification; it does not decide whether the caller may decrypt. A key rotation schedule limits future use; it does not recover data already exposed. Controls should be mapped to preventive, detective, corrective, and compensating categories and tested separately.

Evidence should be proportionate to the claim. A source review can show that a policy predicate is present in one endpoint, while an integration test can show a cross-tenant request fails through a deployed route. A KMS access policy review does not show that an API never logs plaintext keys. A restore exercise demonstrates availability and key recovery under one scenario, not all disaster modes. Record scope, date, environment, test data, and limitations.

## Prioritization exercise

Suppose risk A is a common low-value account enumeration issue and risk B is a rare compromise of an offline root signing key that could invalidate trust across every tenant. Rank using impact, likelihood, exploitability, detection, blast radius, remediation cost, and whether a quick containment exists. It may be rational to patch A immediately while also initiating an emergency root-key protection project; prioritization is not necessarily a single total ordering.

Define service-level objectives for remediation and reassess after controls. Residual risk can rise when a feature expands exposure or a dependency becomes vulnerable. Risk acceptance requires an accountable owner, documented rationale, scope, expiry, and review. It does not turn a vulnerability into a control and should not be permanent by default.

## Risk register example

Asset: tenant note contents. Threat: compromised API service or IDOR. Preconditions: service has decryption authority or object lookup omits tenant predicate. Consequence: cross-tenant disclosure and regulatory exposure. Controls: object-level authorization in policy and database predicates, per-tenant keys, short-lived grants, negative integration tests, KMS audit, and anomaly detection. Residual risk: an attacker controlling the API process may still request plaintext within its authority. Owner: service team. Evidence: code review, test results, deployment configuration, KMS policy, and incident drill. Retest: after every new endpoint or policy change.

Asset: root signing key. Threat: operator or HSM-policy compromise. Preconditions: quorum bypass or arbitrary signing API. Consequence: forged tokens, certificates, or audit statements. Controls: nonexportable key, quorum approval, typed signing policy, offline backup, short cryptoperiod, transparent audit, and emergency trust-anchor rotation. Residual risk: authorized signing service compromise during the active interval. Evidence: ceremony record, HSM policy review, simulated revocation, and independent verification.

## Workshop procedure

1. Draw assets and data flows.
2. Enumerate attacker capabilities and assumptions.
3. Write scenario-based risks with consequences.
4. Identify existing controls and evidence.
5. Estimate likelihood and impact with uncertainty.
6. Choose remediation, containment, acceptance, or transfer.
7. Assign owner, deadline, retest, and review date.
8. Reassess after implementation and operational change.

## Exercises

1. Write a complete risk statement for a stolen refresh token.
2. Rank a frequent low-impact issue against rare catastrophic key compromise.
3. Give evidence for and limitations of an access-control test.
4. Distinguish residual risk from risk acceptance.
5. Create a control map for a backup-encryption threat.
6. Design a tabletop exercise for KMS unavailability.

## Quantitative thinking without false precision

When data supports it, estimate annualized loss as event frequency times impact, then include uncertainty ranges and dependency between events. A key compromise may have a low annual probability but a large correlated impact across every tenant; treating tenants as independent can understate systemic risk. A control may reduce probability, impact, or time to detect, and those effects should be described separately. Avoid multiplying invented numbers and calling the result objective.

A useful prioritization table records exploit prerequisites, exposure, attacker population, affected assets, maximum credible consequence, detection latency, containment options, and remediation effort. Add a confidence column and list the evidence supporting it. If confidence is low and impact high, commission a test or architectural review rather than assigning a reassuring middle score.

The register is a living decision record: close an item only after the retest evidence passes and residual risk is accepted by the named owner.

Keep superseded assumptions and decisions for audit, but mark them clearly so stale evidence cannot be mistaken for current coverage.

## Tabletop scenario

The KMS begins returning errors while the API has pending uploads, queued downloads, and an active key rotation. Operators must decide whether to reject new writes, serve cached ciphertext, pause workers, or use a recovery key. The correct answer follows the threat model: never bypass authorization or encryption silently, preserve idempotency, alert on the outage, protect queued plaintext, and restore from an approved procedure. Document which evidence proves no unauthorized fallback occurred.
