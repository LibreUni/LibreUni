# Security Assessment

Given a service design, identify trust boundaries, attacker capabilities, authentication and authorization rules, cryptographic assumptions, abuse cases, and operational response. Tie every finding to an asset and consequence.

## Assessment method

An assessment is an evidence-based argument about risk and control effectiveness. Begin with scope: systems, environments, data classes, identities, dependencies, and excluded components. Define the assessment date and threat horizon because vulnerabilities, algorithms, and deployments change. Do not make claims about an uninspected component merely because its name suggests a security feature.

Create an asset register. For each asset record owner, confidentiality, integrity, availability, privacy, safety, and regulatory consequence. Include keys, tokens, backups, logs, metadata, and recovery channels, not only the primary database. Identify the trust boundary at every flow and note whether the receiver authenticates the source, integrity, freshness, and authorization context.

## Threat enumeration

For each asset and flow, ask what an external attacker, authenticated user, compromised service, malicious administrator, supply-chain component, or insider can observe or change. Consider theft, tampering, replay, deletion, denial of service, privilege escalation, repudiation, data poisoning, downgrade, cross-tenant access, and recovery abuse. Record preconditions and assumptions so a threat is testable rather than a vague fear.

Use attack trees or abuse cases to decompose a consequence. “Read another tenant’s file” may require obtaining a session, selecting an identifier, bypassing object authorization, and receiving a response. A control at each step changes likelihood or impact. A single missing check in a bulk endpoint can make a strong single-object policy irrelevant.

## Control analysis

Classify controls as preventive, detective, corrective, or compensating. For prevention, identify enforcement point and fail-open or fail-closed behavior. For detection, define event source, signal quality, alert threshold, owner, and time to response. For correction, define credential revocation, key rotation, data restoration, patching, communication, and evidence preservation. A policy document without an enforcement mechanism is not preventive control evidence.

Cryptographic controls require primitive, parameter, key, nonce, encoding, and verification-order evidence. Identity controls require enrollment, authentication, authorization, session, recovery, and offboarding evidence. Operational controls require configuration, access logs, backups, monitoring, and incident exercises. Evaluate the composition: encryption can fail through key access; MFA can fail through recovery; an audit log can fail through unprotected write access.

## Evidence hierarchy

Evidence may include source and configuration review, unit and integration tests, independent penetration tests, formal proofs, key inventory records, logs, alert simulations, restore exercises, and operator interviews. Evidence must match the claim. A passing unit test does not establish production certificate validation; a framework badge does not prove object authorization; a code comment does not prove nonce uniqueness across restarts.

Record evidence provenance, test inputs, environment, date, result, and limitations. Findings should include reproduction steps, affected assets, preconditions, consequence, severity rationale, recommended remediation, owner, deadline, and retest criteria. Avoid reporting only a vulnerability label; explain the violated invariant and the smallest safe repair.

## Severity and residual risk

Severity combines impact and exploitability but should reflect the actual threat model. A low-probability key compromise may be catastrophic if it exposes all historical records; a frequent reflected input issue may be low impact if strong isolation prevents sensitive data access. Document uncertainty rather than hiding it in a precise score. Residual risk is the risk after controls, not the absence of a finding.

Prioritize controls by expected loss, reversibility, blast radius, remediation cost, and evidence quality. A quick isolation control may be more valuable than a long-term rewrite. Accepting residual risk requires an accountable owner and expiration or review date. Risk acceptance is not a technical mitigation and should not be represented as one.

## Worked service assessment

Assess a note-sharing API with browser clients, an identity provider, an API service, a database, object storage, a queue, and an administrator console. Assets include note contents, access grants, sessions, signing keys, encryption keys, backups, and audit records. Threats include IDOR across tenants, stolen refresh tokens, malicious file uploads, SSRF from URL previews, KMS misuse, queue replay, and administrator account takeover.

Controls include server-side object authorization and tenant predicates, secure session and refresh-token rotation, constrained parsing and storage for uploads, egress restrictions for previews, KMS operation policy, idempotent queue messages with authenticated sequence context, phishing-resistant administrator authentication, and tamper-evident audit storage. Evidence includes cross-tenant negative tests, replay tests after worker restart, KMS audit review, restore exercises, and an incident drill rotating a signing key.

Residual risks include compromise of the API process, endpoint malware on a user device, metadata leakage, and availability loss during key-service outage. The report must state what each compromise reveals and which recovery action is possible. “Encrypted at rest” does not imply that a compromised API cannot read notes; the encryption boundary and key access must be explicit.

## Assessment exercises

1. Write an evidence-backed finding for an IDOR vulnerability.
2. Distinguish preventive, detective, corrective, and compensating controls.
3. Design a retest for refresh-token reuse detection.
4. Prioritize a likely low-impact issue versus rare total key compromise.
5. State what evidence would support a nonce-uniqueness claim.
6. Explain why risk acceptance is not remediation.

5. Provide an invariant and evidence across key scope, sender identities, restarts, cloning, concurrency, backup restore, and rotation; include instrumentation that detects duplicate pairs without logging plaintext secrets.
6. Acceptance records a decision to tolerate risk; it does not remove the vulnerability, protect assets, or prove a control. It needs an owner, scope, expiry, and review.

## Incident response as an assessment target

A control is incomplete if the organization cannot respond when it fails. For each high-risk finding, define detection, triage, containment, eradication, recovery, and lessons learned. A suspected signing-key compromise may require disabling issuance, validating historical signatures, publishing revocation, rotating trust anchors, and identifying forged or replayed artifacts. A database exposure may require preserving access logs, rotating credentials, forcing resets, assessing downloaded data, and notifying affected parties.

Run tabletop exercises with realistic constraints: the primary KMS is unavailable, logs are delayed, one administrator account is compromised, a backup is restored into an old environment, or a vendor cannot answer immediately. Measure time to detect, time to contain, decisions requiring quorum, and whether evidence is preserved. Recovery plans should be tested against the same trust boundaries as normal operation; a backup that can be restored only by bypassing authorization is not a safe recovery design.

## Assessment report structure

The final report should contain scope and assumptions, asset and data-flow diagrams, threat model, prioritized findings, violated properties, evidence and limitations, remediation plan, residual risk register, and retest schedule. Include a clear separation between verified facts, analyst inference, and open questions. A student should be able to reproduce the highest-severity finding from the report without relying on private verbal context.
