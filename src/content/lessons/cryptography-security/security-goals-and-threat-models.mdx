# Security Goals and Threat Models

Security analysis starts with assets, actors, capabilities, trust boundaries, and consequences. Confidentiality limits disclosure; integrity limits unauthorized change; availability limits disruption. Authentication identifies a party, while authorization decides what it may do.

A threat model that says “the attacker can do anything” cannot guide design. State whether the attacker can read traffic, alter packets, execute code, or control an account.

## Security properties as games

A security goal should describe an experiment, not a slogan. Confidentiality can be modeled by giving an adversary two messages of equal length and asking it to distinguish an encryption of one from the other. Integrity can be modeled by allowing chosen-message access and asking whether the adversary can produce a new valid authenticated message. Authentication can be modeled by asking whether an adversary can cause a verifier to accept a statement not bound to the intended principal or session.

The game fixes what the adversary can query, what secret state is hidden, and what constitutes a win. Changing any of those rules changes the claim. “Encrypted” may mean resistant to passive observation but not to chosen-ciphertext manipulation. “Authenticated” may mean the key holder produced a tag but not that the key holder’s human identity was verified. Write the interface and winning condition before selecting a primitive.

## Assets, principals, and trust boundaries

An asset is something whose loss has a consequence: account control, payment authorization, private notes, availability capacity, audit records, or cryptographic keys. A principal is an actor or service with an identity and authority. A trust boundary is a place where assumptions change, such as a browser-to-server request, service-to-database query, tenant-to-tenant API, or application-to-HSM call.

Draw data flows across boundaries. For each flow, identify confidentiality, integrity, freshness, origin, and authorization requirements. A database connection protected by TLS may still accept an unauthorized SQL operation from a compromised application account. Conversely, encrypting a public asset may add complexity without reducing risk. Security controls must be attached to the threat they mitigate.

## Adversary capabilities and assumptions

An external network attacker may observe, delay, replay, reorder, drop, or modify packets without reading endpoint memory. A malicious authenticated user may submit validly formatted requests and exploit authorization gaps. A compromised dependency may execute code with the process privileges. A side-channel attacker may measure timing or cache behavior while lacking logical access to secrets. Distinguish these capabilities; “attacker” is not one uniform model.

Assumptions should be falsifiable. Examples include: private keys remain inside a protected module; the random generator produces unpredictable nonces; the browser enforces same-origin policy; certificate validation is performed; database backups are access controlled; and administrators are trusted for a stated operation but audited. If an assumption fails, state which property collapses and what detection or recovery exists.

## Risk and prioritization

Risk combines likelihood, impact, exposure, and uncertainty. A high-impact threat with low likelihood may still deserve mitigation when exploitation is catastrophic or detection is poor. A common qualitative matrix is useful only if its scales are defined. Record affected assets, attack prerequisites, control cost, residual risk, and owner. Avoid treating a numerical score as an objective probability when evidence is weak.

Threat modeling is iterative. Add a feature, dependency, endpoint, data class, or deployment environment and revisit trust boundaries. Security review should include abuse cases and failure states: expired credentials, clock skew, partial writes, retries, duplicate messages, revoked keys, compromised clients, and unavailable identity providers.

## Authentication versus authorization

Authentication answers “which principal is presenting this credential?” Authorization answers “may that principal perform this action on this resource under this context?” A valid session does not authorize every object. Enforce authorization server-side on every sensitive operation, using the resource identifier and current policy rather than a client-supplied role or hidden form field.

Least privilege limits the consequence of a compromise. Separate service identities, narrow database grants, short-lived tokens, scoped capabilities, and explicit administrative paths reduce blast radius. Deny-by-default is not sufficient if error handling, alternate endpoints, bulk operations, or asynchronous jobs bypass the policy.

## Security design review

For every proposed control ask what attack it blocks, where it is enforced, what evidence proves it ran, and what happens on failure. Encryption without key lifecycle is incomplete. Input validation without output encoding may not stop injection. Audit logging without tamper resistance or review is merely data collection. Multi-factor authentication without recovery protection can be bypassed through account recovery.

Use defense in depth without duplicating contradictory policies. A control should fail closed for authorization, fail safely for availability where possible, and avoid leaking secrets through errors. Define security-relevant events and retain enough context to investigate while respecting privacy and retention limits.

## Worked note-sharing threat model

Assets are note contents, account credentials, sharing permissions, and audit history. Principals are note owners, invited readers, the application, the database, and an external attacker. Boundaries include browser-to-API, API-to-database, and API-to-object-storage. Capabilities include reading and modifying network traffic, creating an account, submitting arbitrary note identifiers, and replaying requests; out of scope is compromise of the operating-system kernel.

Goals include: an unauthenticated user cannot read a private note; a reader cannot modify it; a revoked reader cannot use an old share token; and an audit record cannot be silently rewritten. Controls include authenticated encrypted transport, server-side object authorization, opaque or capability-scoped share tokens with expiry and revocation, integrity-protected audit storage, rate limits, and key rotation. Tests must attempt IDOR, replay, token substitution, stale authorization after revocation, and privilege escalation through bulk endpoints.

## Review worksheet

For each threat, record preconditions, attack steps, affected asset, control, detection signal, recovery action, and residual assumption. Then ask whether the control is preventive, detective, corrective, or merely advisory. A design review is complete only when an owner can test the control and explain what happens when it is unavailable. This worksheet turns a threat model into an engineering plan rather than a static diagram.

Repeat the review after deployment changes, because a new data flow or dependency can invalidate an old trust boundary.

Record the review date, scope, and unresolved assumptions.

Revisit them after material changes.

An unresolved assumption is a risk item, not a footnote.

Assign it an owner and a review date.

Review it.
