# Web and Application Security

## Application security model

Web security is a composition of browser behavior, transport protection, server authorization, data validation, cryptographic primitives, deployment configuration, and operational response. A vulnerability often appears where one layer assumes another layer has enforced a property. TLS can protect bytes in transit but cannot prevent an application from returning another user’s record. A signed token can prove integrity but cannot authorize an audience that the verifier fails to check.

Start with a data-flow diagram: browser, identity provider, API, database, object storage, queue, worker, third-party service, and administrator. Mark trust boundaries, origins, credentials, sensitive data, and transitions from untrusted input to side effects. Define assets and attacker capabilities, including malicious users, compromised sessions, cross-site attackers, supply-chain compromise, and insiders with limited privileges.

## Browser security boundaries

The same-origin policy separates documents and many network capabilities by scheme, host, and port. CORS selectively relaxes read access and must not be configured as “allow any origin” together with credentials. A server should validate Origin or Referer where appropriate, use CSRF defenses for state-changing browser requests, and avoid assuming that a custom header cannot be sent by a compromised origin.

Cookies carry session authority and need Secure, HttpOnly, SameSite, appropriate domain and path scope, expiration, rotation, and revocation. SameSite reduces some cross-site request risks but is not a substitute for server-side authorization or explicit CSRF design. Do not place bearer tokens in URLs, where they can leak through history, referrers, logs, screenshots, and analytics.

Content Security Policy can reduce the impact of script injection by restricting script sources, nonces, hashes, and framing behavior. It is defense in depth, not a replacement for output encoding and safe DOM APIs. Configure report-only policies during migration, then enforce and monitor violations without treating attacker-controlled reports as trusted evidence.

## Injection and output encoding

Injection occurs when data is interpreted as code or a query language. Parameterized database queries keep values separate from SQL syntax. Template autoescaping must use the correct context: HTML text, attribute, URL, JavaScript, CSS, and shell contexts have different grammars. Sanitizing once and concatenating later can reintroduce a dangerous interpretation.

Input validation should enforce type, length, range, encoding, and allowed structure, but validation is not a universal substitute for context-safe output. Canonicalize before policy checks when multiple encodings are possible, reject ambiguous encodings, and avoid double decoding. File names, redirects, hostnames, and structured documents require allowlists and parser behavior that is documented and tested.

## Authorization and object access

Every endpoint that accepts an object identifier must perform an object-level authorization check. Sequential IDs, UUIDs, and opaque capability strings do not provide authorization by themselves. Check the current principal, tenant, action, resource state, and delegation at the point where the operation occurs. Repeat checks in bulk, export, search, GraphQL, websocket, and background-job paths.

Avoid mass assignment: do not bind every request field directly to a privileged model. Define writable fields and enforce server-owned fields such as owner, role, approval status, and tenant. For multi-tenant systems, include tenant predicates in database access or policy enforcement so a missed application check cannot return another tenant’s rows.

## Session, CSRF, and request integrity

A state-changing request should carry a session and a request-integrity mechanism appropriate to its client. Synchronizer tokens store a server-generated token in the session and require it in the request; double-submit patterns bind a cookie and request value under defined assumptions. Same-site cookies, origin checks, and explicit content types can add defense in depth. APIs using authorization headers may have different CSRF exposure but still need replay and token-theft controls.

Rotate sessions after authentication and privilege changes. Invalidate old sessions after password reset or high-risk recovery. Use short-lived access tokens and protected refresh-token rotation when bearer tokens are necessary. Bind actions to audience, tenant, operation, and expiry so a valid token from one context cannot be transplanted.

## SSRF, files, and outbound requests

Server-side request forgery occurs when untrusted input controls an outbound request to internal or privileged destinations. Defenses require an allowlist of schemes, hosts, ports, and resolved addresses; protection against DNS rebinding and redirects; egress network controls; and response-size and time limits. Block loopback, link-local, metadata, private, and internal ranges according to the deployment environment, while understanding IPv4/IPv6 and parser inconsistencies.

File upload handling needs size, type, content, name, and storage controls. Do not trust a client MIME type or extension. Store uploads outside executable web roots with generated names, scan or parse in a sandbox where appropriate, and prevent path traversal, decompression bombs, archive symlink attacks, and polyglot interpretation. Download authorization must be checked per object and response headers must prevent unintended browser execution.

## Secrets and dependencies

Configuration secrets should come from an access-controlled secret manager or protected deployment channel, not source control, client bundles, logs, or error pages. Rotate credentials and scope them to the service and environment. Build pipelines should prevent secret exfiltration by untrusted pull requests and redact known patterns without relying on redaction as the primary control.

Dependencies enlarge the trusted computing base. Pin and review versions, generate a software bill of materials, verify provenance and signatures where available, monitor vulnerability advisories, and test updates. A dependency with code execution can bypass application-level cryptography; isolate high-risk parsing and minimize service privileges.

## Security headers and transport

Use HTTPS consistently, redirect safely, enable HSTS only with a deployment plan, and prevent mixed content. Validate certificates and hostnames in clients. Configure framing restrictions, content type sniffing prevention, referrer policy, and browser feature permissions according to application needs. Headers are controls with scope and compatibility consequences; test the rendered behavior rather than assuming a configuration string is honored.

## Verification workflow

Test security controls through the full request lifecycle: malformed input, authenticated but unauthorized input, cross-tenant identifiers, expired and revoked credentials, retries, concurrent requests, alternate encodings, and background execution. Combine unit tests for policy predicates with integration tests that exercise real routing, serialization, cookies, database transactions, and deployment headers. Add an abuse-case regression whenever an incident or penetration test finds a bypass.
