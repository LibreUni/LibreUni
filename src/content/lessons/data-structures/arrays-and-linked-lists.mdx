# Arrays, Dynamic Arrays, and Linked Lists

Suppose you are building a text editor. The user can insert and delete characters anywhere in the document, navigate the cursor through the text, and perform undo and redo. The fundamental operation is manipulating a sequence of characters. Two classic representations compete: a contiguous array and a linked list.

An array stores elements in adjacent memory cells. Accessing the $i$-th element is $O(1)$ — compute $\text{base} + i \cdot \text{word\_size}$ and read. Inserting at the front, however, requires shifting every subsequent element one position to the right: $O(n)$ for a document of $n$ characters. For a text editor where the user frequently types at the cursor (often near the end), the array's $O(1)$ append and $O(1)$ index access are attractive, but inserting at the beginning of a large document becomes prohibitively slow.

A linked list stores elements as nodes, each pointing to the next. Inserting a new node at any known position is $O(1)$ — just adjust a few pointers. But accessing the $i$-th element requires traversing $i$ nodes: $O(n)$ worst case. If the user jumps to line 10,000, a linked list would have to walk 10,000 pointers to reach that position, while an array would compute the address directly.

| Operation | Dynamic array | Linked list |
| --- | ---: | ---: |
| Index access ($A[i]$) | $O(1)$ | $O(n)$ |
| Insert at beginning | $O(n)$ | $O(1)$ |
| Insert at end | $O(1)$ amortized | $O(1)$ with tail pointer |
| Insert at known position | $O(n)$ shift | $O(1)$ after node |
| Sequential traversal | cache-friendly | pointer-dependent |
| Memory overhead | low (no pointers) | high (extra pointer per element) |

This fundamental tension between access speed and modification speed appears everywhere in data structures. The sequence ADT can be implemented with either strategy — both can represent the same abstract order of elements — but the choice determines performance. A text editor might use a hybrid structure: a "gap buffer" (an array with a gap at the cursor position) or a "rope" (a tree of string chunks) that combines the locality of arrays with the flexibility of linked structures. Understanding the basic tradeoff between contiguous and pointer-based storage is the first step in designing data structures that balance competing costs.

## Sequence ADT interface

<MathStatement id="sequence-adt" kind="definition" title="Sequence abstract data type">
A sequence stores an ordered collection of elements and supports at minimum:
$$
\begin{aligned}
\text{get}(i) &\mapsto \text{value at position } i \\
\text{set}(i, x) &\mapsto \text{replace value at position } i \text{ with } x \\
\text{insert}(i, x) &\mapsto \text{shift elements } i..n-1 \text{ right, place } x \text{ at } i \\
\text{remove}(i) &\mapsto \text{remove and return element at } i, \text{ shift left} \\
\text{size}() &\mapsto n
\end{aligned}
$$
Indices are zero-based: $0 \le i < n$ for a valid sequence of length $n$.
</MathStatement>

Every implementation of a sequence must provide these operations. The cost of each operation depends on the underlying representation. Arrays and linked lists are the two canonical approaches, and their tradeoffs expose the central design tension in data structures: **what you gain in access speed you lose in modification speed, and vice versa**.

## Static arrays and logical length

<StructureDiagram kind="array" title="Contiguous sequence and indices" caption="Adjacent array cells support direct index arithmetic and a separate logical length." />

<MathStatement id="static-array" kind="definition" title="Static array">
An array of capacity $C$ provides $O(1)$ address calculation for index $i$: $\text{address} = \text{base} + i \times \text{element\_size}$. A sequence stored in its first $n$ positions has logical length $n$ and capacity $C$, with $0 \le n \le C$. The unused suffix is not part of the abstract value.
</MathStatement>

`get(i)` and `set(i,x)` are constant time after checking $0 \le i < n$. Inserting at position $i$ requires shifting elements at positions $i$ through $n-1$ one place right, then writing $x$. The number of moves is $n-i$, so insertion is $O(n)$ worst case and $O(1)$ at the end if space exists. Removal shifts $n-i-1$ elements left. A sorted sequence may require finding a position by binary search but still pays linear movement for an array insertion.

Contiguous layout improves spatial locality. Sequential traversal tends to fetch adjacent elements into cache lines, while a linked representation may follow pointers to unrelated addresses. Big-O ignores this constant-factor and memory-hierarchy effect, but real workload comparisons should measure it. Arrays also permit compact storage and vectorized operations when element representation is regular.

## Geometric dynamic arrays

<MathStatement id="dynamic-array" kind="definition" title="Dynamic array (geometric resizing)">
A dynamic array wraps a static array and grows when an append encounters $n = C$. The resize strategy: allocate a new array of capacity $\alpha C$ for some growth factor $\alpha > 1$, copy all $n$ elements, and deallocate the old storage. The amortized cost of append is $O(1)$.
</MathStatement>

<MathStatement id="amortized-append" kind="theorem" title="Amortized constant append" proofId="amortized-append-proof">
For a dynamic array that doubles capacity ($\alpha = 2$) whenever $n = C$, the total cost of $m$ appends starting from $C = 1$ is $O(m)$. Hence the amortized cost per append is $O(1)$.
</MathStatement>

<MathProof id="amortized-append-proof" for="amortized-append" collapsible open>
**Worked growth trace.** Start capacity 1 and double: capacities after full resizes are 1, 2, 4, 8, 16. Appending the 9th element copies 8 elements, but the preceding copies total $1 + 2 + 4 = 7$. Across $n$ appends, copied work is less than $2n$ for this policy.

**Potential-function proof.** Define $\Phi = 2n - C$ when $n \ge C/2$, and $\Phi = 0$ otherwise. For a non-resizing append: $\Delta\Phi = 2$, actual cost $= 1$, amortized cost $= 1 + 2 = 3$. For a resize (when $n = C$): before resize $\Phi = 2C - C = C$, after resize $\Phi = 2(C+1) - 2C = 2$. Actual cost $= C + 1$ (copy $C$ elements, write one). Amortized cost $= (C+1) + (2 - C) = 3$. The bound holds for every operation. ∎
</MathProof>

**Shrinking needs hysteresis.** Shrink whenever $n < C/4$, and halve capacity. Shrinking at $n < C/2$ can cause alternating append/remove around the threshold and repeated $O(n)$ copies. With separate grow and shrink thresholds, each expensive resize requires a linear amount of intervening updates, preserving amortized constant time.

## Linked lists

<StructureDiagram kind="list" title="Pointer-based sequence" caption="A linked sequence reaches its next element through a pointer rather than an index." />

<MathStatement id="linked-list" kind="definition" title="Singly linked list">
A singly linked list represents a sequence as a chain of nodes. Each node stores a value and a pointer to the next node (or null for the last). A list with a head pointer and cached size supports:
$$
\begin{aligned}
\text{insert\_front}(x) &: O(1) \\
\text{remove\_front}() &: O(1) \\
\text{get}(i) &: O(i) \text{ (i pointer hops from head)} \\
\text{search}(v) &: O(n)
\end{aligned}
$$
</MathStatement>

A singly linked node stores value and next pointer. A list with head and cached size can insert at the front in $O(1)$, remove the front in $O(1)$, and traverse in $O(n)$. Access at index $i$ requires $i$ pointer hops, $O(i)$, and search is $O(n)$ absent ordering or auxiliary indexing. A tail pointer makes append $O(1)$, but only if the invariant updates tail when the list was empty or the last node is removed.

A doubly linked node stores previous and next pointers. Given a node, insertion or deletion can be $O(1)$ by reconnecting a constant number of pointers. The implementation must update all reciprocal relations and endpoints. A sentinel node can eliminate special cases: an empty list has sentinel.next = sentinel.prev = sentinel, and every real node lies between valid neighbors.

<MathStatement id="doubly-linked-deletion" kind="lemma" title="Doubly linked list deletion" proofId="doubly-linked-deletion-proof">
Given node $x$ in a valid doubly linked list with sentinels, deletion is $O(1)$ and preserves the list invariant.
</MathStatement>

<MathProof id="doubly-linked-deletion-proof" for="doubly-linked-deletion">
Let $p = x.\text{prev}$ and $q = x.\text{next}$. Set $p.\text{next} = q$, $q.\text{prev} = p$, decrement size, then detach $x$'s pointers ($x.\text{prev} = x.\text{next} = \text{null}$). The remaining traversal skips exactly $x$, preserves reciprocal links, and has one fewer node. Detaching prevents accidental reuse through stale references, though memory safety also depends on the language/runtime. The sentinel guarantees $p$ and $q$ are always valid nodes; without sentinels, endpoint branches must update head or tail.
</MathProof>

    def __init__(self, value):
        self.value = value
        self.next = None

class SinglyLinkedList:
    def __init__(self):
        self.head = None
        self._size = 0

    def push_front(self, value):
        new = Node(value)
        new.next = self.head
        self.head = new
        self._size += 1

    def pop_front(self):
        if self.head is None:
            raise IndexError
        value = self.head.value
        self.head = self.head.next
        self._size -= 1
        return value

    def get(self, i):
        if i < 0 or i >= self._size:
            raise IndexError
        cur = self.head
        for _ in range(i):
            cur = cur.next
        return cur.value

    def __len__(self):
        return self._size

    def __str__(self):
        vals = []
        cur = self.head
        while cur:
            vals.append(str(cur.value))
            cur = cur.next
        return " -> ".join(vals) + " -> None"

ll = SinglyLinkedList()
for v in [3, 1, 4]:
    ll.push_front(v)
print(ll)
print(f"get(1) = {ll.get(1)}")
print(f"pop_front() = {ll.pop_front()}")

## Counterexamples and failure modes

### Array pitfalls

**Thrashing at the shrink threshold.** A dynamic array that shrinks when $n < C/2$ and grows when $n = C$ will thrash under a workload that repeatedly appends and removes a single element around $C/2$. Each append triggers a resize, each remove triggers a shrink, and every resize costs $\Theta(n)$. This is not a theoretical curiosity — it appears in production systems that use a single threshold for both directions (Cormen et al. 2022, §17.4).

**Unbounded capacity growth.** Without a maximum capacity check, geometric growth can overflow `size_t` or produce an allocation larger than the address space. Multiply before allocating: verify `new_cap <= MAX_SIZE / element_size`.

**Shallow copy traps.** Resizing an array of pointers copies the pointers, not the objects. Two arrays may then share object references, and mutation through one affects the other. This is correct only when the sequence owns exclusive references.

### Linked list pitfalls

**The "constant-time insertion" myth.** Insertion at a known position is $O(1)$, but *finding* that position by index is $O(n)$. If the workload specifies insertion by position (insert at index 5000), the linked list pays $O(n)$ traversal before the $O(1)$ pointer work. The claim "$O(1)$ insertion" is only valid when the caller already holds a node reference (Cormen et al. 2022, §10.2).

**Cache thrashing.** Linked list traversal on a large dataset can be 10--20$\times$ slower than array traversal in practice, even though both are $O(n)$. Each node access may be a cache miss, while array traversal prefetches contiguous cache lines. Big-O hides this constant factor, but real-world performance is dominated by the memory hierarchy (Drepper 2007, "What Every Programmer Should Know About Memory").

**Use-after-free and dangling pointers.** After removing a node, any external reference to it becomes a dangling pointer. In manual-memory languages (C, C++), the hazard is use-after-free; in garbage-collected languages, the retained node prevents garbage collection of the entire chain reachable from it. Deciding whether iterators are invalidated on mutation is a critical API design choice.

**Sentinel omission.** Without sentinel nodes, every insertion and deletion must check whether the target is head or tail. These branches add code complexity and are a frequent source of bugs. The sentinel eliminates them but requires careful initialization.

## Arrays versus links: operational comparison

Array insertion has movement cost but excellent indexing and locality. Linked insertion avoids movement when a node or predecessor is already known but pays allocation, pointer storage, poor locality, and traversal cost. If insertion positions are specified only by integer index, finding the linked position is still $O(n)$; "linked-list insertion is $O(1)$" is conditional on having the node reference.

## Locality and benchmarking

A fair comparison must specify element size, allocator behavior, access distribution, cache state, and whether the benchmark includes construction. Sequential array traversal may be several times faster than pointer traversal despite identical $O(n)$ bounds because a cache line fetches neighboring array elements while linked nodes may each miss. Conversely, repeatedly shifting a large array can dominate even if its individual accesses are cheap.

For sequence implementations, explicitly check:

- empty state (n = 0)
- one element (n = 1)
- full capacity (n = C)
- capacity one (C = 1)
- insertion at index 0 (front)
- insertion at index n (end)
- deletion of the only element
- resize while an iterator exists

These cases exercise every endpoint branch. A proof that assumes a predecessor and successor exists is valid only for a strict interior node; sentinel designs can remove branches but still require a sentinel invariant.

## Memory safety and ownership

<CaseStudy client:load
  title="Iterator invalidation design"

  options={[

Pointer structures introduce aliasing and lifetime issues. After removing a node, external references may point to detached data. In a manual-memory language, freeing a node while an iterator retains it creates use-after-free; in a garbage-collected language, retaining it may keep unreachable values alive. Decide whether iterators are invalidated, whether nodes are public, and who owns payload objects.

Arrays have their own hazards: copying references may create shallow copies, resizing may invalidate pointers or slices, and integer overflow in capacity calculations can allocate too little. Check (`new_cap`) against maximum representable size before multiplication. A correct high-level asymptotic argument does not excuse unsafe arithmetic.

   <ExerciseSolution hint="Design a potential that grows as the array fills and drops sharply at a resize.">
   With capacity C and size n, use $\Phi = 2n - C$ when $n \ge C/2$, adjusted with a nonnegative piecewise potential near empty. A non-resizing append changes $\Phi$ by 2 and costs 1. A doubling resize costs $n+1$ but drops the capacity term by $C$; since $n = C$ before resize, the amortized cost is bounded by a constant. State the potential convention at small sizes to avoid negative credit.
   </ExerciseSolution>

3. Give a linked-list invariant with head, tail, and size, then prove append preserves it in empty and nonempty cases.

   <ExerciseSolution>
   Empty: head=tail=null and size=0. Nonempty: head and tail are reachable, tail.next=null, traversal visits size nodes, and no node repeats. Append a new node; if empty set both endpoints, otherwise set old tail.next and new prev then tail=new. Increment size and verify each invariant clause.
   </ExerciseSolution>

4. Trace deletion of a middle node in a doubly linked list and list every pointer assignment.

   <ExerciseSolution>
   Let p=x.prev and q=x.next. Set p.next=q, q.prev=p, decrement size, then x.prev=x.next=null if detachment is specified. The sentinel cases make p and q always valid; without sentinels, endpoint branches must update head or tail.
   </ExerciseSolution>

5. Construct a workload where an array dominates a linked list and one where links can dominate.

   <ExerciseSolution>
   Repeated `get` of random indices and scans favor contiguous arrays. Repeated insertion after stored node references favors linked nodes because movement is constant; if locations are found by index, traversal can erase that advantage.
   </ExerciseSolution>

6. Explain why shrinking at half-full can thrash and propose thresholds.

   <ExerciseSolution>
   If capacity halves as soon as $n < C/2$, alternating remove and append around $n = C/2$ causes a shrink followed by a grow, each copying $\Theta(n)$. Shrink below $C/4$ and grow at full, or use a potential proof with hysteresis.
   </ExerciseSolution>

7. Identify an aliasing bug caused by returning an internal array slice.

   <ExerciseSolution>
   A client stores the returned slice and later writes element j, changing the data structure without calling set and possibly violating an invariant. Return a copy or an immutable view, or document shared ownership and enforce bounds/mutation rules.
   </ExerciseSolution>

### Transfer and synthesis

8. **Design a hybrid.** A text editor needs $O(1)$ access to any character and $O(1)$ insertion at the cursor. Propose a data structure that combines an array and a linked list to achieve both. State the invariants your structure maintains and the cost of each operation.

   <ExerciseSolution>
   A gap buffer: an array with a gap at the cursor. Characters before the cursor are stored contiguously before the gap; characters after the cursor are stored contiguously after the gap. Moving the cursor shifts characters across the gap. Access is O(1) with offset computation; insertion at the cursor writes into the gap (O(1) if gap is nonempty, O(n) worst case when gap must be resized). The invariants: (gap_start + gap_size) == capacity, logical sequence is A[0..gap_start-1] ++ A[gap_start+gap_size..capacity-1].
   </ExerciseSolution>

9. **Adversarial analysis.** A colleague claims "linked lists are always faster for insert-heavy workloads." Construct a concrete counterexample: describe a specific workload and data size where a dynamic array strictly outperforms a linked list despite more insertions than lookups, and explain why.

   <ExerciseSolution>
   Insert 5000 elements at the front of a sequence with 1,000,000 existing elements. The linked list must traverse to find the insertion point (O(n) = 1,000,000 steps), then insert in O(1). The array shifts 1,000,000 elements but does so in a tight cache-friendly loop. The array's memory bandwidth dominates the list's pointer chase. With 10,000 such insertions, the array's total moved elements is 10^10, but the list's traversals total 10^10 pointer hops across scattered addresses — the list loses due to cache misses.
   </ExerciseSolution>

10. **Prove the lower bound.** Show that any sequence implementation that stores $n$ elements in $O(n)$ memory cells and supports $O(1)$ worst-case index access cannot support $O(1)$ insertion at an arbitrary position without shifting. (Hint: consider the pigeonhole principle on the number of distinct sequences representable in $O(n)$ memory.)

    <ExerciseSolution>
    Suppose $O(1)$ insertion and $O(1)$ index access were both possible. Inserting $n$ elements at the front would produce $n!$ distinct sequences. The memory state after $n$ insertions uses $O(n)$ memory cells, which can represent at most $c^n$ distinct states for some constant $c$ — far fewer than $n!$ for large $n$ by Stirling's approximation. This contradicts the pigeonhole principle, so at least one operation must be $\Omega(\log n)$ or worse. The array's choice: $O(1)$ index access, $O(n)$ insertion.
    </ExerciseSolution>

## DELIBERATELY OMITTED

**Chunked sequences (rope, gap buffer, unrolled linked list).** These combine contiguous blocks with linked nodes for logarithmic indexing and linear-time local edits. They are essential for production text editors and databases but their analysis builds on the pure array and linked models taught here. The gap buffer is briefly mentioned in Exercise 8 as a transfer exercise; a full treatment belongs in a separate lesson on hybrid sequence structures.

**Cache-aware linked-list layouts.** Linking nodes within a memory pool or arena can improve spatial locality of linked traversals. This lesson discusses locality qualitatively but does not detail specific layout strategies such as array-linked hybrids or pre-allocation patterns. See the "Locality and Layout" module for quantitative treatment.

**Persistent arrays and functional updates.** Purely functional array-like structures (e.g., the bitmapped vector trie used in Clojure and Scala) copy paths through a tree. These build on the basic array reasoning but require an understanding of persistent-structure techniques from a separate lesson.

**Memory allocator behavior for linked structures.** Slab allocators, freelist organization, and allocation order effects on linked-node placement are important for performance but belong in a systems or locality lesson.

**Concurrent sequence structures.** Lock-free linked lists, RCU-protected arrays, and concurrent deques are the subject of an entire course on concurrent data structures and are not covered here.
