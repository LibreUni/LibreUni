# Arrays, Dynamic Arrays, and Linked Lists

The sequence ADT can be implemented with contiguous arrays or linked nodes. Both can represent the same abstract order, yet their performance differs because memory layout changes the cost of indexing, insertion, traversal, resizing, and allocation. This lesson derives those costs from representation invariants instead of memorizing a table of asymptotic labels.

## Static arrays and logical length

An array of capacity C provides O(1) address calculation for index i: base+i times element width. A sequence stored in its first n positions has logical length n and capacity C, with (0le nle C). The unused suffix is not part of the abstract value. get(i) and set(i,x) are constant time after checking (0le i less than n).

Inserting at position i requires shifting elements at positions i through n-1 one place right, then writing x. The number of moves is n-i, so insertion is O(n) worst case and O(1) at the end if space exists. Removal shifts n-i-1 elements left. A sorted sequence may require finding a position by binary search but still pays linear movement for an array insertion.

Contiguous layout improves spatial locality. Sequential traversal tends to fetch adjacent elements into cache lines, while a linked representation may follow pointers to unrelated addresses. Big-O ignores this constant-factor and memory-hierarchy effect, but real workload comparisons should measure it. Arrays also permit compact storage and vectorized operations when element representation is regular.

## Geometric dynamic arrays

A dynamic array grows when an append encounters (n=C). Allocate a larger array, copy n elements, and continue. If capacity multiplies by a factor (alpha greater than 1), total copied elements over m appends are bounded by a geometric series (C+alpha C+cdots) up to the final capacity, which is O(m). Thus total append time is O(m), and amortized append is O(1), even though one resize costs O(n) worst case.

**Worked growth trace.** Start capacity 1 and double: capacities after full resizes are 1,2,4,8,16. Appending the 9th element copies 8 elements, but the preceding copies total 1+2+4=7. Across n appends, copied work is less than 2n for this policy. The constant depends on the growth factor; larger factors reduce resize frequency but increase unused capacity.

The accounting method charges each append a constant fee: one unit pays for its immediate write and one or more credits are stored on the element for future copying. When capacity doubles, the credits associated with the old block pay for moving it. Credits never become negative, proving the amortized bound. A potential function (Phi=2n-C) when the array is at least half full gives another proof; resize cost is covered by the potential drop plus actual cost.

Shrinking needs hysteresis. Shrink whenever n is below a fraction such as C/4, and halve capacity. Shrinking at n less than C/2 can cause alternating append/remove around the threshold and repeated O(n) copies. With separate grow and shrink thresholds, each expensive resize requires a linear amount of intervening updates, preserving amortized constant time.

## Linked lists

A singly linked node stores value and next pointer. A list with head and cached size can insert at the front in O(1), remove the front in O(1), and traverse in O(n). Access at index i requires i pointer hops, O(i), and search is O(n) absent ordering or auxiliary indexing. A tail pointer makes append O(1), but only if the invariant updates tail when the list was empty or the last node is removed.

A doubly linked node stores previous and next pointers. Given a node, insertion or deletion can be O(1) by reconnecting a constant number of pointers. The implementation must update all reciprocal relations and endpoints. A sentinel node can eliminate special cases: an empty list has sentinel.next=sentinel.prev=sentinel, and every real node lies between valid neighbors.

**Deletion proof.** Given node x in a valid doubly linked list with sentinels, set x.prev.next=x.next and x.next.prev=x.prev, decrement size, and detach x’s pointers. The remaining traversal skips exactly x, preserves reciprocal links, and has one fewer node. Detaching prevents accidental reuse through stale references, though memory safety also depends on the language/runtime.

## Arrays versus links

Array insertion has movement cost but excellent indexing and locality. Linked insertion avoids movement when a node or predecessor is already known but pays allocation, pointer storage, poor locality, and traversal cost. If insertion positions are specified only by integer index, finding the linked position is still O(n); “linked-list insertion is O(1)” is conditional on having the node reference.

A workload with repeated front operations favors a deque or linked head; repeated random indexing favors an array. A workload with batched middle edits may favor a chunked sequence or balanced tree of blocks, which combines logarithmic navigation with local shifts. The correct choice follows from operation frequencies and object size, not from a universal ranking.

## Locality and benchmarking

A fair comparison must specify element size, allocator behavior, access distribution, cache state, and whether the benchmark includes construction. Sequential array traversal may be several times faster than pointer traversal despite identical O(n) bounds because a cache line fetches neighboring array elements while linked nodes may each miss. Conversely, repeatedly shifting a large array can dominate even if its individual accesses are cheap. Measure throughput and tail latency, and inspect allocation and cache-miss counters when the result matters.

Benchmarking should also test adversarial and representative workloads separately. A uniform random index distribution is different from append-heavy logs, sorted insertion, or bursty deletions. Warm-up and garbage collection can distort short runs. Use a reference model to validate outputs while collecting performance, and report variance rather than one favorable timing. Complexity predicts scaling; measurement explains constants and hardware interaction.

## Boundary-case proof checklist

For sequence implementations, explicitly check empty state, one element, full capacity, capacity one, insertion at zero, insertion at n, deletion of the only element, and resize while an iterator exists. These cases exercise every endpoint branch. A proof that assumes a predecessor and successor exists is valid only for a strict interior node; sentinel designs can remove branches but still require a sentinel invariant.

## Memory safety and ownership

Pointer structures introduce aliasing and lifetime issues. After removing a node, external references may point to detached data. In a manual-memory language, freeing a node while an iterator retains it creates use-after-free; in a garbage-collected language, retaining it may keep unreachable values alive. Decide whether iterators are invalidated, whether nodes are public, and who owns payload objects.

Arrays have their own hazards: copying references may create shallow copies, resizing may invalidate pointers or slices, and integer overflow in capacity calculations can allocate too little. Check (newCapacity) against maximum representable size before multiplication. A correct high-level asymptotic argument does not excuse unsafe arithmetic.

## Exercises

1. Derive the move count for inserting and removing at every position in an array sequence.
2. Prove geometric dynamic-array append is O(1) amortized using a potential function.
3. Give a linked-list invariant with head, tail, and size, then prove append preserves it in empty and nonempty cases.
4. Trace deletion of a middle node in a doubly linked list and list every pointer assignment.
5. Construct a workload where an array dominates a linked list and one where links can dominate.
6. Explain why shrinking at half-full can thrash and propose thresholds.
7. Identify an aliasing bug caused by returning an internal array slice.

### Diagnostic solutions


1. Inserting at i shifts n-i elements; removing at i shifts n-i-1. Thus front insertion/removal is linear, end append/removal is constant when capacity permits, and average position under uniform i has linear expected movement.
2. With capacity C and size n, use potential (Phi=2n-C) when n greater than C/2, adjusted with a nonnegative piecewise potential near empty. A non-resizing append changes potential by 2 and costs 1. A doubling resize costs n+1 but drops the capacity term by C; since n=C before resize, the amortized cost is bounded by a constant. State the potential convention at small sizes to avoid negative credit.
3. Empty: head=tail=null and size=0. Nonempty: head and tail are reachable, tail.next=null, traversal visits size nodes, and no node repeats. Append a new node; if empty set both endpoints, otherwise set old tail.next and new prev then tail=new. Increment size and verify each invariant clause.

4. Let p=x.prev and q=x.next. Set p.next=q, q.prev=p, decrement size, then x.prev=x.next=null if detachment is specified. The sentinel cases make p and q always valid; without sentinels, endpoint branches must update head or tail.
5. Repeated get of random indices and scans favor contiguous arrays. Repeated insertion after stored node references favors linked nodes because movement is constant; if locations are found by index, traversal can erase that advantage.
6. If capacity halves as soon as n less than C/2, alternating remove and append around n=C/2 causes a shrink followed by a grow, each copying Θ(n). Shrink below C/4 and grow at full, or use a potential proof with hysteresis.
7. A client stores the returned slice and later writes element j, changing the data structure without calling set and possibly violating an invariant. Return a copy or an immutable view, or document shared ownership and enforce bounds/mutation rules.
