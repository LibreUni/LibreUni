# B-Trees, B+ Trees, and External Memory

A single disk read takes ~10 ms — 10 million times slower than a RAM access (1 ns). If your search tree operates on disk, a single node traversal that takes 1 $\mu$s in RAM takes 10 ms on disk. A 1000-node traversal would take 10 seconds. This is not a constant-factor slowdown; it is a fundamental change in the cost model.

The B-tree (Rudolf Bayer and Edward McCreight, 1970, at Boeing) solves this by packing 1000+ keys into a single node, so the tree height is 3–4 instead of 30. The key insight: disk reads are slow, so minimize the *number* of reads by maximizing the branching factor. A B-tree of order $m$ (each node holds up to $m-1$ keys) has height $O(\log_m n)$. For $n = 10^9$ and $m = 1000$, height is 3. For a binary tree, height is 30.

What does the "B" stand for? Bayer never specified. Popular guesses include "Boeing" (where they worked), "balanced," "Bayer," "broad," or "Bush" (after Vannevar Bush's memex). The ambiguity is intentional — Bayer later said the B stands for nothing in particular. What matters is the design: a page-oriented tree where each node fits exactly one disk block, and all leaves sit at the same depth.

## The block model

| B-tree | page transfer | high fan-out |
| B+ tree | page transfer plus leaf scan | range queries |

Assume a block holds $B$ keys and child references. A node with high fanout has height $O(\log_B n)$ rather than $O(\log_2 n)$. The CPU work inside a page can use binary search or a short linear scan; the dominant cost is the number of pages fetched. A cost contract must state whether the structure is in memory, backed by a block device, or accessed through a buffer pool.

## B-tree invariants and updates

<MathStatement id="btree-def" kind="definition" title="B-tree invariants">
A B-tree of minimum degree $t \ge 2$ satisfies:
- Every node stores between $t-1$ and $2t-1$ keys, except the root which may store as few as 1 key.
- An internal node with $k$ keys has $k+1$ children whose key ranges are separated by those keys.
- All leaves have the same depth.
- Within each node, keys appear in sorted order.
</MathStatement>

The minimum degree $t$ controls the branching factor. A node with $2t-1$ keys is *full*; a node with $t-1$ keys is at minimum occupancy. **Split** before inserting into a full node: promote the median key to the parent, and divide the remaining keys into two sibling nodes. **Merge** when a deletion causes a node to fall below $t-1$ keys: combine it with an adjacent sibling and demote the separator from the parent.

### B+ tree range query

Consider a B+ tree of order 4 storing (age, name) records keyed by age. Leaf nodes contain $[(17,\text{Alice}), (19,\text{Bob}), (21,\text{Carol}), (25,\text{David})]$, $[(28,\text{Eve}), (31,\text{Frank}), (35,\text{Grace})]$, linked in order. Root separator: $[25,28]$. Query range [20, 30]:

1. Root-to-leaf: start at root, compare 20 → go left child (since 20 < 25). Reach leaf $L_1 = [17,\text{Alice} \ldots 25,\text{David}]$.
2. Scan forward in $L_1$ from first key ≥ 20: $(21,\text{Carol})$, $(25,\text{David})$. Record both.
3. Follow leaf-next pointer to $L_2 = [28,\text{Eve}, 31,\text{Frank}, 35,\text{Grace}]$.
4. Scan $L_2$ while key ≤ 30: $(28,\text{Eve})$ qualifies; $(31,\text{Frank})$ exceeds range. Stop.
5. Return $\{(21,\text{Carol}), (25,\text{David}), (28,\text{Eve})\}$.

Block transfers: 1 (root) + 1 ($L_1$) + 1 ($L_2$) = 3 block transfers plus 2 within-leaf scans. Without linked leaves, a separate root-to-leaf search per key would cost $4 \times 2 = 8$ block transfers.

The B+ tree's leaf-linked structure makes range queries cost $O(\log_B n + k/B)$ block transfers for $k$ sequential records, compared to $O(k \log_B n)$ without leaf linkage.

## B-tree height and occupancy

<MathStatement id="btree-height" kind="theorem" title="B-tree worst-case height" proofId="btree-height-proof">
In a B-tree of minimum degree $t \ge 2$ storing $n$ keys, the height $h$ satisfies:
$$ h \le 1 + \log_t \frac{n+1}{2}. $$
</MathStatement>

<MathProof id="btree-height-proof" for="btree-height" collapsible open>
At height 0 (a single root node), the minimum number of keys is 1. For $h \ge 1$, the root has at least 1 key and 2 children; every other node has at least $t-1$ keys and at least $t$ children. The minimum number of nodes at depth $d$ is $2 t^{d-1}$ for $d \ge 1$. Summing minimum keys across all nodes:

$$ n_{\min} \ge 1 + (t-1) \sum_{d=1}^h 2 t^{d-1} = 1 + 2(t-1) \frac{t^h - 1}{t - 1} = 2 t^h - 1. $$

Solving $2 t^h - 1 \le n$ gives $h \le \log_t ((n+1)/2)$. Adding 1 for the root level: $h \le 1 + \log_t ((n+1)/2)$. ∎
</MathProof>

For $t = 100$ (a typical value for disk pages holding 4 KB keys), $n = 10^8$ gives $h \le 1 + \log_{100} 5 \times 10^7 \approx 1 + 3.85 \approx 5$ levels. Any root-to-leaf path traverses at most 5 page reads, compared to $\log_2 n \approx 27$ for a binary search tree. The height bound is a worst-case guarantee — actual occupancy is often higher, producing even shorter trees.

## B+ trees and range queries

B+ trees store records in linked leaves and separators only in internal pages. Exact lookup follows one root-to-leaf path. A range query pays for the first path, then scans adjacent leaves: $O(\log_B n + k/B)$ block transfers for $k$ sequential records, ignoring buffer effects. Leaf links are an optimization only when their mutation invariant is maintained during split and merge.

## Failure modes

A B-tree is robust, but several failure modes arise from incorrect implementation or workload mismatch:

**Deletion underflow and premature merge.** When a key is deleted from a node that already has minimum occupancy, the node must either borrow a key from a sibling (rotation) or merge with a sibling through the parent. An incorrectly implemented deletion can leave a node below $t-1$ keys, violating the occupancy invariant. The tree may still function as a search tree, but the height bound degrades: a node with $t-2$ keys reduces the minimum branching factor, potentially increasing the worst-case height by a constant factor that compounds with each underflow.

**Stale leaf-link pointers after split.** In a B+ tree, a leaf split must update the `next` pointer of the old leaf, the `next` pointer of the new leaf, and the parent's separator. If any of these updates is omitted or applied out of order, a concurrent or even a single-threaded range scan can skip the new leaf entirely. The failure is silent: point lookups remain correct because each traverses a root-to-leaf path, but range queries return an incomplete result set.

**Suboptimal fill factor under random insertions.** When keys are inserted in random order rather than sequentially, the split operation leaves nodes approximately half-full on average. The actual occupancy can be as low as $t-1$ keys per node, which is the worst case for the height bound. A workload that inserts monotonically increasing keys tends to fill the rightmost leaf to capacity before splitting, producing higher average occupancy and shorter trees. A B-tree designed for a sequential workload may perform poorly under random insertions if the fill factor assumption is not verified.

**Concurrent mutation without locking.** A B-tree structure updated concurrently by multiple threads without a proper locking or latching protocol can produce a split that a concurrent reader observes in an intermediate state: a node with more than $2t-1$ keys, or a child pointer that points to an uninitialized node. Database systems use specialized B-link tree variants or page-level latches to prevent this; the sequential invariant alone is insufficient.

   <ExerciseSolution hint="Every node except the root has a lower and upper bound on keys. Separators partition children by range.">

2. Explain why a root split is the only operation that can increase height.

   <ExerciseSolution hint="When the root is full, a new level must be created above it. Non-root splits redistribute within the existing level.">

   A non-root split redistributes entries among nodes at the same depth and inserts a separator into the existing parent. A root split creates a new root above the old root, adding exactly one level; lower-level splits do not change leaf depth.
   </ExerciseSolution>

8. **Synthesis: alternate occupancy bound.** Theorem 1 derives the worst-case height using minimum occupancy $t-1$. Prove an alternate bound when the minimum occupancy fraction is $f \le 1/2$ (i.e., each node holds at least $f \cdot M$ keys where $M = 2t-1$ is the maximum).    </ExerciseSolution>

## Deliberately Omitted

**B-link trees**: concurrency-optimized B-tree variants with link pointers for lock-free sibling traversal. B-link trees add a high-key and a link pointer to each node so that a concurrent split does not block readers. The synchronization reasoning is substantial and belongs in a concurrent-data-structures treatment.

**Write-optimized B-trees (B-ϵ trees)**: B-tree variants that buffer updates in internal nodes to amortize writes. The B-ϵ tree buffering mechanism changes the split, merge, and flush invariants significantly; it is covered in the LSM-tree lesson from a different angle.

**Cache-oblivious B-trees**: search trees with optimal block-transfer complexity without knowing block size. The van Emde Boas layout and recursive partitioning differ from the page-oriented B-tree model taught here.

**B-tree recovery and ARIES logging**: write-ahead logging protocols for B-tree persistence with undo/redo. These are database-engine implementation techniques rather than fundamental B-tree invariants.
