# Binary Search Trees and Ordered Maps

A phone directory with 100,000 entries must support insert, delete, lookup by name, and range queries ("list all names starting with A–B"). A hash table gives $O(1)$ average lookup but cannot answer range queries efficiently — scanning the table for prefix "A" costs $O(n)$. A sorted array supports binary search ($O(\log n)$ lookup) and range queries via lower/upper bounds, but insertion shifts $O(n)$ elements. A linked list offers $O(1)$ insertion at a known position but $O(n)$ lookup.

The binary search tree (BST) achieves $O(\log n)$ for all three operations simultaneously — provided the tree remains balanced. The key insight: store keys in a binary tree where every node's left subtree contains only smaller keys and its right subtree contains only larger keys. This local property guarantees that an inorder traversal visits keys in sorted order — the range-query property of a sorted array — while the branching structure supports insert and delete in $O(h)$ time, where $h$ is the tree height.

The BST was discovered by Allen Newell, Cliff Shaw, and Herbert A. Simon in their 1955–1956 work on the Information Processing Language (IPL), and independently described by William F. Palmer in the 1950s [^knuth1998]. Each comparison during a search eliminates an entire subtree. At the root, the comparison discards either the left or right half of the stored keys — analogous to binary search on an array but without the shifting cost of insertion.

The BST invariant is recursive: each subtree is itself a BST. This invites recursive algorithms and correctness proofs by induction on tree height. The price: the invariant alone does not bound height. Inserting keys in sorted order produces a chain of $n$ nodes with $h=n-1$, reducing all operations to $O(n)$. Maintaining logarithmic height requires additional invariants — the subject of balanced tree variants.

## Definition and duplicate policy

<MathStatement id="bst-invariant" kind="definition" title="Binary search tree invariant">
Let $T$ be a binary tree where each node $x$ stores a key $x.key$ drawn from a totally ordered set. $T$ is a binary search tree if for every node $x$:

- Every key in the left subtree of $x$ satisfies $key < x.key$.
- Every key in the right subtree of $x$ satisfies $key > x.key$.
- Both subtrees are themselves binary search trees.

Equivalently, with inherited bounds: a subtree rooted at $x$ must contain only keys in the open interval $(\ell, u)$, where $\ell$ and $u$ are bounds inherited from ancestors. The root has bounds $(-\infty, +\infty)$.
</MathStatement>

## Search and insertion

Search starts at the root. If the tree is empty, the key is absent. Otherwise compare target $k$ with node key $x$: return on equality, descend left if $k < x$, descend right if $k > x$. The path is unique under strict ordering. An iterative implementation uses $O(1)$ auxiliary space; recursive search uses $O(h)$ call-stack space.

Insert follows the unsuccessful search path and creates a leaf at the first null child. The existing tree satisfies the invariant, and every comparison on the path establishes the new key's bounds. A local comparison at the insertion parent is sufficient only because the path was chosen according to all ancestor comparisons.

**Worked insertion.** Insert the sequence $[8, 3, 10, 1, 6, 14, 4]$ into an empty tree. The root is 8; 3 goes left and 10 right. 1 is left of 3; 6 right of 3; 14 right of 10; 4 is greater than 3 but less than 6, so it becomes 6's left child. Inorder traversal: $1, 3, 4, 6, 8, 10, 14$. Height is 3 edges; search visits at most four nodes.

<CodeRunner client:load
  title="BST search and insert simulation"
  language="python"
  code={`class BSTNode:
    def __init__(self, key):
        self.key = key
        self.left = None
        self.right = None

def search(root, key):
    if root is None:
        return False
    if key == root.key:
        return True
    if key < root.key:
        return search(root.left, key)
    return search(root.right, key)

def insert(root, key):
    if root is None:
        return BSTNode(key)
    if key < root.key:
        root.left = insert(root.left, key)
    elif key > root.key:
        root.right = insert(root.right, key)
    return root

def inorder(root, result=None):
    if result is None: result = []
    if root:
        inorder(root.left, result)
        result.append(root.key)
        inorder(root.right, result)
    return result

root = None
for k in [8, 3, 10, 1, 6, 14, 4]:
    root = insert(root, k)
print("Inorder:", inorder(root))
print("Search 6:", search(root, 6))
print("Search 7:", search(root, 7))`}

## Deletion

<StructureDiagram kind="tree" title="Successor deletion" caption="Replacing a two-child node with its inorder successor preserves global ordering bounds." variant="successor" />

Deleting a leaf removes its parent pointer. For a node with one child, splice the child into the node's position. For two children, find the successor $y$ — the minimum node in the right subtree. Replace $x$'s key and payload with $y$'s, then delete $y$, which has no left child and therefore falls into the simpler zero/one-child case.

<MathStatement id="successor-deletion" kind="proposition" title="Successor deletion preserves BST order" proofId="successor-proof">
Let $x$ be a node with two children in a BST, and let $y$ be the inorder successor of $x$ (the minimum key in $x$'s right subtree). Replacing $x$'s key with $y$'s key and deleting $y$ produces a valid BST containing the same set of keys minus $x$'s original key.
</MathStatement>

<MathProof id="successor-proof" for="successor-deletion" collapsible>
Every key in $x$'s left subtree is less than $x.key$ and hence less than $y.key$ (since $y.key > x.key$ by definition of successor). Every key in $x$'s right subtree that is not $y$ is greater than $y.key$ because $y$ is the minimum of that subtree. The replacement therefore preserves the BST invariant at every node. The physical deletion of $y$ cannot require deleting a two-child node because $y$ has no left child (it is the minimum of its subtree). The deletion of $y$ is therefore a leaf or one-child case, which trivially preserves the BST invariant of the remaining subtree.
</MathProof>

**Worked deletion.** Delete 8 from the tree built above. Node 8 has two children (3 and 10). Its successor is 10 (the minimum of the right subtree). Replace 8's key with 10, then delete 10, which has a single child 14. Splice 14 into 10's position. The resulting tree has root 10, left child 3 (with subtree 1, 4, 6), right child 14. Inorder: $1, 3, 4, 6, 10, 14$. The operation runs in $O(h)$ time.

## Height and degeneration

Let $h$ be tree height. Search, insert, and delete take $O(h)$, with $O(h)$ recursion stack if implemented recursively. A balanced tree has $h = O(\log n)$, but an insertion sequence in sorted order creates a chain of height $n-1$.

Random insertion into an initially empty BST has expected height $O(\log n)$ under a random-permutation model [^knuth1998]. Expected behavior is not a worst-case guarantee and can be invalidated by correlated input — for example, keys arriving in sorted order from a database query.

The shape also affects iteration and memory locality. A chain has pointer traversal depth $n$; a compact balanced tree has logarithmic depth but may still have scattered nodes. A BST does not become balanced merely because it stores sorted keys; balance is a separate invariant or randomized policy.

## Ordered operations

<Quiz client:load
  title="Range-reporting complexity"
  question="What is the worst-case time complexity of reporting all keys in [a,b] from a balanced BST with n nodes, returning k keys?"
  options={["O(n)", "O(log n + k)", "O(k)", "O(n log n)"]}
  correctIndex={1}
  explanation="The algorithm prunes subtrees outside the range. It visits O(log n) nodes on the boundary paths to locate the range, then visits each of the k reported nodes. The total is O(log n + k)."
/>

Minimum follows left children until null; maximum follows right. The successor of a node with a right subtree is the minimum of that subtree; without a right subtree, walk ancestors until the node is a left child of its parent. These rules support sorted iteration with a stack or parent pointers. Each node is pushed and popped at most once, so inorder traversal is $O(n)$.

Range reporting for $[a,b]$ prunes subtrees outside the range. At each node, recurse left only if $x.key \ge a$, report $x$ if in range, and recurse right only if $x.key \le b$. The time is $O(h + k)$, where $k$ is output size, in a balanced tree.

Augment a node with subtree size to support $\text{select}(i)$ (the $i$th smallest) and $\text{rank}(k)$. If left size is $\ell$, select goes left for $i < \ell$, returns $x$ for $i = \ell$, and goes right with $i - \ell - 1$ otherwise. The size invariant is $\text{size}(x) = 1 + \text{size}(x.left) + \text{size}(x.right)$. Every insertion or deletion must update sizes along the changed path.

## Failure modes

<Quiz client:load
  title="Non-transitive comparator failure"
  question="What happens when a BST uses a comparator that is not transitive (e.g., a < b and b < c but a ≮ c)?"
  options={[
    "The tree may appear valid but produce incorrect search results",
    "The tree throws an exception on insertion",
    "Inorder traversal stops working",
    "Duplicate keys are automatically rejected"
  ]}
  correctIndex={0}
  explanation="A non-transitive comparator violates the strict total order requirement. The BST can still be built and traversed, but search results become unpredictable because the ordering assumption is broken at the comparator level."
/>

Common bugs include treating equality inconsistently, losing the successor's right child during deletion, failing to update the parent when a child subtree root changes, and using a comparator inconsistent with stored equality. A tree can still produce sorted output after a failed parent-pointer update, then crash during successor traversal. Validate both local links and global bounds.

If keys have mutable comparison fields, modifying a key after insertion can place it in the wrong order. Search may fail even though inorder still reflects the old placement. Solutions: make keys immutable while stored, remove and reinsert after mutation, or use an indirection whose ordering key cannot change.

The worst-case failure is silent corruption: a comparator that is not transitive (e.g., `a < b` and `b < c` but not `a < c`) can produce a tree that appears valid but yields incorrect search results. Verify that the comparator induces a strict total order on the key domain.

1. State a bound-based BST invariant and prove insertion preserves it.

   <ExerciseSolution hint="Consider the bounds each node inherits from its ancestors during the recursive descent">

   Every subtree key lies between inherited lower and upper bounds, and strict duplicates are absent. The search path maintains bounds; the new leaf is inserted inside the final interval, so all ancestors' constraints remain true.
   </ExerciseSolution>

2. Trace search and insertion for the sequence [5, 2, 7, 1, 3, 6, 8] into an empty BST.

   <ExerciseSolution hint="Record the comparison result at each node and follow the appropriate child pointer">

   Compare at each node and record the path. A target smaller than the current key goes left; larger goes right; equality stops. Insert at the first null link reached by the same comparisons.
   </ExerciseSolution>

3. Prove successor deletion preserves sorted order.

   <ExerciseSolution>
   The successor is the smallest key in the right subtree, so replacing x by it leaves all left keys smaller and all remaining right keys larger. The successor has no left child, so its deletion is a leaf or one-child splice that preserves the subtree invariant.
   </ExerciseSolution>

4. Construct a worst-case height sequence of 7 keys and compute the resulting height.

   <ExerciseSolution>
   Insert keys in increasing order [1, 2, 3, 4, 5, 6, 7]. Each new key follows right pointers to the current maximum, creating a chain of 7 nodes and height 6. Decreasing order [7, 6, 5, 4, 3, 2, 1] is the mirror image, producing a left-leaning chain.
   </ExerciseSolution>

5. Derive range-reporting complexity for a balanced BST.

   <ExerciseSolution>
   Search contributes $O(h)$. Subtrees whose bounds lie entirely outside $[a,b]$ are pruned; each reported node is visited and its boundary paths are $O(h)$, giving $O(h+k)$ time and $O(h)$ stack space.
   </ExerciseSolution>

6. Design subtree-size rank/select and update rules.

   <ExerciseSolution>
   Compare i with left subtree size and recurse. Store size at each node; update along insertion/deletion paths and recompute after rotations. The formula proves rank/select decisions correspond to inorder positions.
   </ExerciseSolution>

7. Explain why random insertion does not give a deterministic logarithmic bound.

   <ExerciseSolution>
   Expected height assumes a uniformly random insertion permutation. An adversary can supply sorted keys, and even a random-looking workload can be correlated with key order. Randomized priorities or explicit balancing are needed for a guaranteed policy.
   </ExerciseSolution>

8. Design a data structure that supports insert, delete, and `count_in_range(a, b)` (number of keys in $[a,b]$) in $O(\log n)$ time. Augment the BST with subtree size and modify the range-reporting algorithm.

   <ExerciseSolution>

9. You are given a BST where each node stores a mutable integer payload. After a series of insertions, a user modifies the payload of an internal node. Does this break the BST invariant? Why or why not?

   <ExerciseSolution>
   Modifying the payload does not break the BST invariant — the invariant constrains only keys, not associated payloads. However, if the payload is used as part of the comparison key (e.g., a composite key), then mutation breaks the invariant. The solution is to keep keys immutable while stored, or use a separate indirection for the ordering key.
   </ExerciseSolution>

    <ExerciseSolution>
    Path-copying creates $O(h)$ new nodes along the insertion path. The unmodified subtrees are shared between old and new versions. Space per insertion is $O(h) \subseteq O(n)$ worst case, or $O(\log n)$ expected in a balanced tree. This is the basis of persistent functional maps in languages like Clojure and Scala.

## Deliberately Omitted

<Quiz client:load
  title="Scope boundaries"
  question="Why is 'optimal binary search trees (Knuth's DP)' omitted from this lesson?"

- **Tree-drawing algorithms for visualization**: automatic layout of binary trees for pedagogical diagrams (Reingold–Tilford, etc.). These are a separate computational-geometry topic.
- **Ternary search trees**: a hybrid between BST and trie that stores strings by comparing characters at each node. TSTs have different space/time trade-offs than BSTs and tries and are covered in the trie lesson.
- **Cartesian trees and treap construction from arrays**: building a treap from a static array in $O(n)$ time using a stack. This is a niche construction technique that assumes known priorities and does not generalize to dynamic insertion.
- **Optimal binary search trees (Knuth's DP)**: constructing a BST with minimum expected search cost given a known key-access distribution. This is a dynamic-programming problem that precomputes a static tree rather than supporting dynamic updates.
- **Implicit treaps (Cartesian trees on array indices)**: a BST keyed by position rather than value, supporting split, merge, and range operations. This is a distinct data structure with different invariants.
- **Weight-balanced trees**: a balance family that uses subtree weight (number of nodes) rather than height. These are covered in the balanced search trees lesson.
