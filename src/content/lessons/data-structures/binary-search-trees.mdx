# Binary Search Trees and Ordered Maps

A binary search tree stores keys so that an inorder traversal is sorted. This simple invariant turns comparisons into navigation: a search discards an entire subtree at each comparison. The benefit depends on height, which is determined by insertion history unless a balancing policy is added. Understanding the unbalanced structure first makes the obligations of AVL, red-black, treap, and external trees precise.

## Definition and duplicate policy

A BST node contains a key, payload, and left and right child references. Under the strict policy, every left-subtree key is less than the node key and every right-subtree key is greater; duplicates are rejected or stored in a separate count. Under a non-strict policy, equal keys consistently go to one side, but deletion and height behavior change. State the comparator and duplicate semantics before proving anything.

The RI can be expressed with inherited bounds: a subtree rooted at x must contain only keys in an open interval (lo,hi), or closed intervals if duplicates are allowed. This recursive formulation catches violations that local parent-child checks miss. A node’s left child may be smaller than x but still violate an ancestor’s lower bound.

## Search and insertion

Search starts at the root. If the tree is empty, the key is absent. Otherwise compare target k with node key x: return on equality, descend left if k less than x, and descend right if k greater than x. The path is unique under strict ordering. An iterative implementation uses O(1) auxiliary space; recursive search uses O(h) call-stack space.

Insert follows the unsuccessful search path and creates a leaf at the first null child. The existing tree satisfies the RI, and every key on the path establishes the new key’s bounds. A local comparison at the insertion parent is enough only because the path was chosen according to all ancestor comparisons; state that fact in the proof.

**Worked insertion.** Insert 8, 3, 10, 1, 6, 14, 4. The root is 8; 3 goes left and 10 right. 1 is left of 3; 6 right of 3; 14 right of 10; 4 is greater than 3 but less than 6, so it becomes 6’s left child. Inorder traversal is 1,3,4,6,8,10,14. The height is 3 edges, so search visits at most four nodes on this example.

## Deletion

Deleting a leaf removes its parent pointer. For a node with one child, splice the child into the node’s position. For two children, find the successor y, the minimum node in the right subtree. Replace x’s key and payload with y’s, then delete y, which has no left child and therefore falls into the simpler zero/one-child case. If payload identity matters, move a node or update references instead of copying fields without a contract.

The successor replacement preserves order: every key in x’s left subtree is less than x and hence less than y; y is the smallest key in the right subtree; keys after y remain greater. The physical deletion of y cannot require deleting a two-child node because y has no left child. Recursive deletion returns the new subtree root so parent links are updated correctly.

## Height and degeneration

Let h be tree height. Search, insert, and delete take O(h), with O(h) recursion stack if implemented recursively. A balanced tree has h=O(log n), but an insertion sequence in sorted order creates a chain of height n-1. Random insertion into an initially empty BST has expected height O(log n) under a random-permutation model, but expected behavior is not a worst-case guarantee and can be invalid under correlated input.

The shape also affects iteration and memory locality. A chain has pointer traversal depth n; a compact balanced tree has logarithmic depth but may still have scattered nodes. A BST does not become balanced merely because it stores sorted keys; balance is a separate invariant or randomized policy.

## Ordered operations

Minimum follows left children until null; maximum follows right. The successor of a node with a right subtree is the minimum there; without a right subtree, walk ancestors until the node is in a left branch. These rules support sorted iteration with a stack or parent pointers. Each node is pushed and popped at most once, so inorder traversal is O(n).

Range reporting for [a,b] prunes subtrees outside the range. At each node, recurse left only if x.key≥a, report x if in range, and recurse right only if x.key≤b. The time is O(h+k), where k is output size, in a balanced tree. Returning an iterator requires define mutation behavior and stack lifetime.

Augment a node with subtree size to support select(i), the ith smallest, and rank(k). If left size is ell, select goes left for i less than ell, returns x for i=ell, and goes right with i-ell-1 otherwise. The size RI is (size(x)=1+size(left)+size(right)). Every insertion/deletion must update it along the changed path; rotations must recompute affected nodes.

## Iterators and maps

A map associates one payload with each key. Updating an existing key should not change tree shape; inserting a new key does. An iterator may use an explicit ancestor stack and yield keys in sorted order. If a mutation occurs, options include invalidation, weak consistency, or a fully specified live-order behavior. A fail-fast modification count detects many accidental mutations but is not a concurrency guarantee.

## Correct recursive interfaces

The cleanest recursive update returns the new root of the subtree. An insert operation returns the existing root if the key is already present; otherwise it recursively assigns the returned left or right child and returns the current root. A delete operation follows the same pattern and returns the replacement root in the zero/one-child cases. This return-value convention handles deletion of the global root without a special external parent pointer.

Every recursive call should receive a subtree whose inherited bounds are known and should return a subtree satisfying the same bounds. The proof is structural induction on subtree size. The base case is null. The inductive step uses the comparator to choose a child, invokes the induction hypothesis on the corresponding interval, and reconnects the returned root. This is stronger than testing a few inorder traversals because it proves bounds and pointer structure together.

## Failure modes

Common bugs include treating equality inconsistently, losing the successor’s right child during deletion, failing to update the parent when a child subtree root changes, and using a comparator that is inconsistent with stored equality. A tree can still produce sorted output after a failed parent-pointer update, then crash during successor traversal. Validate both local links and global bounds.

If keys have mutable comparison fields, modifying a key after insertion can place it in the wrong order. Search may fail even though inorder still reflects the old placement. Either make keys immutable while stored, remove/reinsert after mutation, or use an indirection whose ordering key cannot change.

## Memory and external storage

A pointer BST incurs one or more pointers per node and unpredictable allocation. A packed implicit layout is efficient only for near-complete trees; a skewed BST wastes array positions. B-trees group many keys in a block and are preferable when cache lines or storage pages dominate. Their split/merge invariants generalize the BST order but require a different height and I/O analysis.

For persistent or snapshot maps, parent pointers and in-place rotations are unsafe under sharing. Path-copy updates preserve old roots but allocate O(h) nodes. For a mutable ordered map with stable iterators, rotations can invalidate node addresses unless nodes are individually allocated and handles are stable. The client contract must identify which identities survive structural updates.

## Testing and model comparison

Maintain a simple sorted list or reference map and execute randomized insert, update, delete, search, rank, and range operations against both implementations. After each mutation, verify inorder keys, size, bounds, parent links, and cached augmentation. Include duplicate keys, deletion of absent keys, deleting the root repeatedly, and comparator ties. Compare an iterator’s output to the reference sorted order and test mutations according to the documented invalidation rule.

### Diagnostic solutions


1. Every subtree key lies between inherited lower and upper bounds, and strict duplicates are absent. The search path maintains bounds; the new leaf is inserted inside the final interval, so all ancestors’ constraints remain true.

2. Compare at each node and record the path. A target smaller than the current key goes left; larger goes right; equality stops. Insert at the first null link reached by the same comparisons.

3. The successor is the smallest key in the right subtree, so replacing x by it leaves all left keys smaller and all remaining right keys larger. The successor has no left child, so its deletion is a leaf or one-child splice that preserves the subtree invariant.

4. Insert keys in increasing order. Each new key follows right pointers to the current maximum, creating a chain of n nodes and height n-1. Decreasing order is the mirror image.

5. Search contributes O(h). Subtrees whose bounds lie entirely outside [a,b] are pruned; each reported node is visited and its boundary paths are O(h), giving O(h+k) time and O(h) stack space.

6. Compare i with left subtree size and recurse. Store size at each node; update along insertion/deletion paths and recompute after rotations. The formula proves rank/select decisions correspond to inorder positions.

7. Expected height assumes a uniformly random insertion permutation. An adversary can supply sorted keys, and even a random-looking workload can be correlated with key order. Randomized priorities or explicit balancing are needed for a guaranteed policy.
