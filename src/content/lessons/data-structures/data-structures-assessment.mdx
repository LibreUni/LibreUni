# Data Structures Comprehensive Assessment

A production system processes 100,000 range-sum queries per second on a key-value store. The naive approach — scanning every record — costs $O(n)$ per query. A Fenwick tree or segment tree can answer each query in $O(\log n)$ but requires a specific invariant, a correct update procedure, and a proof that the representation matches the abstract array. One wrong index in the construction loop silently corrupts every later query. This assessment requires you to connect every data structure's abstract specification to its concrete representation through explicit invariants, then prove that operations preserve them.

For every answer, state the model, assumptions, duplicate and error policy, and whether a bound is worst-case, amortized, or expected. A correct complexity label without the invariant that justifies it is incomplete. A correct invariant without an abstraction function does not establish user-visible behavior.

The problems span sequences, dictionaries, priority queues, connectivity, sorting, persistence, and memory-aware design.

## Part I: abstraction and sequences
<StructureDiagram kind="array" title="Assessment: sequence representation" caption="A sequence answer must distinguish logical length from allocated capacity." />

1.  Define an abstract sequence ADT with indexed read, update, insertion, deletion, iteration, and size. Specify invalid indices and mutation during iteration.

    <ExerciseSolution hint="Think about the minimal contract a sequence must guarantee to its callers.">

    Define sequence value as ordered elements; get/set require $0\le i<n$; insertion allows $0\le i\le n$; removal requires valid i; state iterator invalidation or snapshot behavior. All operations must specify ownership and errors.
    </ExerciseSolution>
2.  Give a representation invariant and abstraction function for a dynamic array with capacity $C$ and logical size $n$. Prove append in the spare-capacity case and resize case.

    <ExerciseSolution hint="What constraints relate capacity, logical size, and the prefix of stored elements?">

    RI: $0\le n\le C$ and elements occupy prefix $[0,n)$; AF is that prefix. Spare append writes at $n$ and increments. Resize allocates $C'>n$, copies the prefix in order, publishes it, writes $x$, and sets $n+1$; old state remains valid until commit.
    </ExerciseSolution>
3.  Compare arrays, singly linked lists, and doubly linked lists for random access, front insertion, known-node insertion, memory overhead, and locality.

    <ExerciseSolution>
    Arrays give $O(1)$ indexing and locality but shifts for middle insertion; singly linked lists give $O(1)$ front and known-node splicing but $O(n)$ indexing; doubly linked lists support $O(1)$ known-node deletion with two pointers at extra space. Bounds condition on a known node versus an index.
    </ExerciseSolution>
4.  A circular queue stores head, tail, count, and capacity. State the modular invariant and explain how it distinguishes empty and full.

    <ExerciseSolution>
    Logical element $j$ is at $(head+j)\bmod C$ for $0\le j<count$; $tail=(head+count)\bmod C$; $0\le count\le C$. Empty iff $count=0$, full iff $count=C$. Enqueue writes at tail and increments; dequeue reads head and decrements.
    </ExerciseSolution>
5.  Prove that a two-stack queue has $O(1)$ amortized operations using a potential function.

    <MathStatement id="tsq-amortized" kind="theorem" title="Two-stack queue amortized cost" proofId="tsq-amortized-proof">
    A queue implemented with two stacks (input and output) supports each operation in $O(1)$ amortized time.
    </MathStatement>

    <MathProof id="tsq-amortized-proof" for="tsq-amortized" collapsible>
    Set potential $\Phi = 2\cdot|input|$. Enqueue pushes to input: actual cost $1$, $\Delta\Phi = 2$, amortized cost $3$. Dequeue: if output is nonempty, pop; actual cost $1$, $\Delta\Phi = 0$, amortized cost $1$. If output is empty, transfer $k$ elements from input to output: actual cost $k$, $\Delta\Phi = -2k$, amortized cost $k-2k = -k$ plus the final pop gives $1$. Since $\Phi\ge 0$ always, the sum of amortized costs bounds the sum of actual costs, giving $O(1)$ per operation.
    </MathProof>

6.  A client receives a mutable slice of an internal array. Give a counterexample to representation independence and a safe API alternative.

    <ExerciseSolution>
    The client can mutate an element without set or write beyond logical size, violating the RI. Return a copy, immutable view, or controlled iterator; document ownership if sharing is intentional.
    </ExerciseSolution>

## Part II: dictionaries and order

| Claim type | Must state |
| --- | --- |
| worst-case | bound for every valid input |
| amortized | sequence and accounting method |
| expected | probability space and assumptions |

<Quiz client:load question="What is missing from a bare complexity label?" options={["The model and invariant supporting it", "The font size", "The lesson title", "The number of references"]} correctIndex={0} explanation="A bound is meaningful only with the representation, input model, and proof obligation that justify it." />

7.  State a chaining hash-table invariant and explain lookup correctness under collisions.

    <ExerciseSolution>
    Every live pair is in exactly one bucket $h(k)$, equality has no duplicate ambiguity, and size counts pairs. Lookup computes the same bucket and scans it; collision changes cost, not correctness.
    </ExerciseSolution>
8.  Why are tombstones needed in open addressing? Trace a false negative caused by clearing a slot.

    <ExerciseSolution>
    A key inserted after a collision may lie beyond an earlier slot. Clearing the earlier slot makes lookup stop and falsely report absence. Tombstones mean "continue searching, but reusable for insertion."
    </ExerciseSolution>
9.  Explain load factor, resizing, and the probability assumption behind expected $O(1)$ hashing.

    <ExerciseSolution>
    $\alpha=n/m$ is average occupancy. Under uniform or universal hashing and bounded $\alpha$, expected chain/probe cost is constant. The bound is not worst-case and can fail under concentration or adversarial keys.
    </ExerciseSolution>
10. Define a BST bound invariant and prove insertion preserves it.

    <ExerciseSolution>
    Every subtree obeys inherited lower/upper key bounds. The search path narrows the interval; the new leaf is placed in the final valid interval, preserving all ancestor constraints.
    </ExerciseSolution>
11. Trace deletion of a two-child BST node and prove successor replacement preserves order.

    <ExerciseSolution>
    Replace a two-child node by the minimum of its right subtree, then delete that successor, which has no left child. All left keys remain smaller and all remaining right keys larger.
    </ExerciseSolution>
12. Compare unbalanced BST, AVL, red-black, treap, and B-tree guarantees and memory models.

    <ExerciseSolution>
    Unbalanced BST is $O(h)$, worst-case linear; AVL and red-black are worst-case logarithmic with different balance/update tradeoffs; treaps are expected logarithmic under random priorities; B-trees reduce block I/O with wide nodes.
    </ExerciseSolution>
13. Design subtree-size augmentation for rank/select and show how a rotation updates it.

    <ExerciseSolution>
    Store $size(x)=1+size(left)+size(right)$. Select compares rank with left size. After rotation recompute the old root then new root; all unaffected subtrees retain their sizes.
    </ExerciseSolution>
14. A trie stores prefixes and terminal keys. Define exact lookup, prefix enumeration, and safe pruning.

    <ExerciseSolution>
    Follow symbols and test terminal for exact lookup. For a prefix, traverse then enumerate descendants; output complexity includes results. Prune only nonterminal childless nodes while moving upward.
    </ExerciseSolution>

## Part III: priority and connectivity
<StructureDiagram kind="heap" title="Assessment: validate shape and order" caption="A heap answer must check both complete-tree shape and parent-child priority order." />

15. Define the binary-heap shape/order invariant and prove sift-down after delete-min.

    <MathStatement id="heap-invariant" kind="definition" title="Binary heap invariant">
    A binary heap is a complete binary tree stored in an array $A[0..n-1]$ such that for every index $i$, if $2i+1 < n$ then $A[i] \le A[2i+1]$, and if $2i+2 < n$ then $A[i] \le A[2i+2]$.
    </MathStatement>

    <ExerciseSolution>
    Complete-tree array layout plus $parent\le children$. After root replacement, child subtrees are heaps and only the root path can violate order; swap with the smaller child until fixed.
    </ExerciseSolution>
16. Prove bottom-up heap construction is $O(n)$, not $O(n\log n)$.

    <ExerciseSolution>
    At height $h$ there are at most $n/2^{h+1}$ nodes and each costs $O(h)$; $\sum h/2^h$ is constant, giving $O(n)$.
    </ExerciseSolution>
17. Explain indexed priority queues and the map/array bijection invariant.

    <ExerciseSolution>
    The item-to-index map and array contents are inverse bijections. Every swap updates both entries. Decrease-key locates an item, changes priority, and sifts up.
    </ExerciseSolution>
18. State union-find partition semantics, union-by-size proof, and path-compression guarantee.

    <MathStatement id="uf-amortized" kind="theorem" title="Union-find amortized complexity" proofId="uf-amortized-proof">
    Union-find with union-by-size and path compression supports any sequence of $m$ operations in $O(m\,\alpha(n))$ time, where $\alpha$ is the inverse Ackermann function.
    </MathStatement>

    <MathProof id="uf-amortized-proof" for="uf-amortized" collapsible>
    Union-by-size ensures every tree has height $O(\log n)$ even without path compression. Path compression flattens the access path on each find. The combined analysis uses a potential function based on the logarithm of subtree sizes; the inverse Ackermann bound arises from the extremely slow growth of the iterated logarithm under repeated compression.
    </MathProof>

19. Why cannot ordinary union-find support arbitrary edge deletion? Describe rollback union-find for offline dynamic connectivity.

    <ExerciseSolution>
    Parent pointers discard enough edge history to know whether a deleted edge was a bridge. Offline segment-tree intervals plus rollback union-find add active edges on recursion paths and undo them at exit.
    </ExerciseSolution>
20. Compare adjacency matrix, adjacency list, CSR, and edge list by space and primitive costs.

    <ExerciseSolution>
    Matrix uses $\Theta(V^2)$, $O(1)$ edge lookup, $O(V)$ neighbor scan; lists use $O(V+E)$, $O(degree)$ scan; CSR is compact and scan-friendly but update-heavy; edge lists use $O(E)$ and favor global scans.
    </ExerciseSolution>

## Part IV: complexity and memory
<CaseStudy client:load title="Integrated answer" scenario="A student claims expected constant hash lookup under attacker-controlled keys without a hash model." question="What should the reviewer request?" options={["A probability model plus collision and load assumptions", "A longer code listing", "A different diagram color", "No clarification"]} correctIndex={0} explanation="Expected claims require explicit assumptions; adversarial input can invalidate random-key reasoning." />

21. Distinguish worst-case, amortized, expected, and average-case analysis with one data-structure example each.

    <ExerciseSolution>
    Worst-case bounds every call; amortized bounds every sequence; expected bounds average over specified randomness; average-case may assume an input distribution. Dynamic-array append is amortized, hash lookup is conditional expected, and BST search has worst-case height.
    </ExerciseSolution>
22. Explain cache locality and why two $O(n)$ traversals can differ substantially.

    <ExerciseSolution>
    Arrays fetch adjacent values in cache lines and can be prefetched; pointer chasing may miss per node. Both perform $n$ logical steps, but transfer and latency costs differ.
    </ExerciseSolution>
23. Design a benchmark that measures a data-structure operation without timing input generation or allowing dead-code elimination.

    <ExerciseSolution>
    Pre-generate inputs, consume outputs via checksum, warm up separately, randomize repetitions, record environment, report distributions, and validate against a model outside the timed core or in a controlled phase.
    </ExerciseSolution>
24. Explain stable sorting, comparison lower bounds, quickselect, and a domain-specific non-comparison sort.

    <ExerciseSolution>
    Stable sorting preserves equal-key order; comparison sorts require $\Omega(n\log n)$ comparisons; quickselect finds the $k$th element in expected linear time; counting/radix sort exploit bounded representations and avoid the comparison lower bound.
    </ExerciseSolution>
25. A persistent tree update changes an old version. Identify the violated principle and path-copy repair.

    <ExerciseSolution>
    A shared node was mutated. Copy every node on the update path, reuse untouched subtrees, and publish a new root; old roots must never reach a write.
    </ExerciseSolution>
26. Give a test strategy that combines a reference model, invariant validator, property tests, and adversarial workloads.

    <ExerciseSolution>
    Differential reference model, per-operation invariant checks, property/metamorphic relations, boundary cases, adversarial distributions, reproducible seeds, and performance tests separated from correctness.
    </ExerciseSolution>

## Part V: integrated debugging cases
<StructureDiagram kind="graph" title="Assessment: representation choices" caption="The same abstract relationships can be represented with different concrete edges and storage costs." />
<StructureExercise client:load title="Debug the invariant" prompt="Which evidence is strongest when reviewing a broken structure?" options={["A counterexample state and the missing invariant clause", "A single happy-path trace", "The class name", "An asymptotic label without a model"]} answer={0} explanation="A concrete counterexample connects the observed failure to the exact predicate the implementation failed to preserve." kind="graph" />

27. A hash map passes uniform random tests but times out in production. What evidence and repairs do you seek?

    <ExerciseSolution>
    Measure bucket/probe distributions, key patterns, load/resizes, and hash seed. Use keyed/universal hashing, bucket trees, lower thresholds, rate limits, or a fallback; expected random tests did not model production.
    </ExerciseSolution>
28. A balanced tree returns sorted inorder output but rank is wrong after rotations. Which invariant and update rule are missing?

    <ExerciseSolution>
    Recompute subtree sizes after every rotation and update path. Inorder can remain correct while cached augmentation is stale; validate size equations recursively.
    </ExerciseSolution>
29. A queue fails only when capacity is a power of two. What modular-index and full/empty bugs are plausible?

    <ExerciseSolution>
    Overflow/masking may confuse capacity with index range; $head=tail$ can mean both empty and full if count is not used consistently; modulo updates may skip the final slot. Test empty, full, wrap, and resize transitions.
    </ExerciseSolution>
30. A persistent map retains excessive memory after old snapshots are released. Analyze roots, reclamation, and measurement.

    <ExerciseSolution>
    Old roots or retained handles still reach shared nodes; garbage collection cannot reclaim reachable data. Track root ownership, release snapshots, compact versions, and measure retained graph size rather than allocation alone.
    </ExerciseSolution>
31. A graph traversal visits a vertex twice after a repeated run. Explain stale metadata and a generation-based fix.

    <ExerciseSolution>
    Visited marks from the previous run were not cleared or scoped. Store a generation number and compare against the current traversal token, or reset all marks before starting.
    </ExerciseSolution>
32. A rollback dynamic-connectivity solver reports connections in sibling time ranges. Identify the temporal leak.

    <ExerciseSolution>
    A union made in one segment-tree branch was not rolled back before entering its sibling. Snapshot the history length on entry and pop exactly to it on exit; test nested branches against a brute-force model.
    </ExerciseSolution>

### Typical failure patterns

| Failure | Root cause | Detection |
| --- | --- | --- |
| Amortized bound fails under adversarial sequence | Assumed independence; adversary forces worst-case every operation | Sequence-level validator, not per-operation |
| Persistent update mutates past version | Shared node written in place | Snapshot comparison after every mutation |
| Hash map returns wrong value for colliding key | Equality not consistent with hash | Differential test against reference model |
| Tree iterative traversal enters infinite loop | Parent pointers not updated after rotation | Invariant checker after every mutation |
| Queue underflows (removes from empty) | Client did not check size before dequeue | Contract precondition enforcement |

### Detailed counterexample: dynamic array append

<MathStatement id="append-failure" kind="example" title="Counterexample: append that violates the invariant">
A dynamic-array append implementation that writes the new element at position $n$ but forgets to increment $n$ before the next operation.
</MathStatement>

Trace: initial state $n=3$, $C=5$, `arr = [a, b, c, _, _]`. Append $d$ writes `arr[3] = d` but does not update $n$. The RI still reports $n=3$, so the next read or write uses index $3$ as the new logical size. The client sees only $[a, b, c]$ — $d$ is permanently invisible. If the next operation is another append, it overwrites $d$ with $e$ at position $3$. The element $d$ is lost.

The missing invariant clause: "after every mutation, $n$ equals the number of elements stored." The fix: increment $n$ after every successful append.

## Transfer and synthesis exercises

These problems require combining concepts from multiple parts of the assessment. They cannot be solved by recall alone.

1.  **Synthesis: range-sum with concurrent updates.** A Fenwick tree supports point update and prefix sum in $O(\log n)$. Design a version that supports concurrent readers and writers. Specify the linearization protocol, the invariant the tree must maintain under concurrent access, and the memory-ordering guarantees (e.g., acquire/release semantics on the tree array). Prove that a reader who observes a partially updated tree either sees the old value or the new value, never a mix of both.

    <ExerciseSolution hint="Readers must see a consistent snapshot of the tree. Consider how to publish updates atomically.">

    Represent the Fenwick tree as an immutable structure per version: a writer constructs a new tree array via path copying, publishes it with a release-store to a global root pointer, and readers acquire-load the root. The invariant: every prefix sum computed from a published root reflects exactly one complete update sequence. The linearization point is the release-store. Partial updates are invisible because the new array is fully constructed before publication.
    </ExerciseSolution>

2.  **Synthesis: multi-criteria index selection.** A workload requires point lookups by key $A$, range queries by key $B$, and prefix queries by key $C$. The keys are independent (no functional dependency). You have one primary storage structure and may build up to two auxiliary indexes. The data is 100 million records, each 512 bytes. The workload is 70% point lookups by $A$, 20% range queries by $B$, 10% prefix queries by $C$. Write throughput is 500 writes/second. Design the index architecture and justify:
    - Which index structures you choose for each access pattern
    - The total memory budget for indexes
    - The write amplification per operation
    - How you handle consistency across indexes during a write

    <ExerciseSolution hint="Consider hash maps for point lookups, B-trees for range queries, and tries for prefix queries. Each index adds write amplification.">

    Use a hash map on $A$ for point lookups (expected $O(1)$), a B+ tree on $B$ for range queries ($O(\log n + k)$), and a trie on $C$ for prefix queries ($O(|prefix| + output)$). Memory: hash map ~8 bytes/entry × 100M = 800 MB; B+ tree ~20 bytes/key × 100M = 2 GB; trie ~40 bytes/node × 50M nodes = 2 GB. Total ~4.8 GB for indexes. Write amplification: each write updates all three indexes (hash insert, B+ tree insert, trie insert), so ~3× write amplification. Consistency: use a write-ahead log and apply all three index updates within a transaction; readers see a versioned root pointer.
    </ExerciseSolution>

3.  **Transfer: external-memory priority queue.** A priority queue must process 1 billion insertions and extract-min operations on a machine with 64 GB RAM. The data does not fit in memory. Adapt the binary heap to the external-memory model using B-tree nodes as the underlying storage. Define the representation invariant for a B-tree-based heap, specify the merge policy for underfull nodes, and analyze the I/O cost of insert and extract-min. Compare against a buffer tree (Arge, 1995).

    <ExerciseSolution hint="Replace the implicit array of a binary heap with an explicit tree of B-tree nodes.">

    Store each heap node as a B-tree block (page) of $B$ keys. The heap invariant: every key in a parent block is $\le$ every key in its child blocks. Insert: append to the last block; if full, split and propagate the split upward. Extract-min: remove the minimum from the root block; if the root becomes empty, refill from the smallest child. I/O cost: insert costs $O(\log_B n)$ I/Os (traverse height + potential split write); extract-min costs $O(\log_B n)$ I/Os (read root, refill). Buffer trees achieve amortized $O((1/B)\log n)$ I/Os per operation by batching inserts, making them superior for write-heavy workloads.
    </ExerciseSolution>

4.  **Transfer: proving amortized cost with a potential function under adversarial input.** The two-stack queue amortized proof assumes each element is transferred at most once. Construct an adversarial sequence of operations that forces a transfer on every dequeue, and show that the amortized analysis still holds because the potential function accounts for the cost. Then construct a different data structure (e.g., a stack with a multipop operation) and design a potential function that proves $O(1)$ amortized per operation despite an adversary.

    <ExerciseSolution hint="The adversary can force a transfer by alternating enqueue and dequeue on a single-element output stack.">

    Adversarial sequence: enqueue $x$, dequeue, enqueue $y$, dequeue, repeating. Each dequeue triggers a transfer of one element. The potential function $\Phi = 2\cdot|input|$ handles this: after enqueue, $\Phi=2$; dequeue transfers 1 element (cost 1) and pops it (cost 1), and $\Delta\Phi = -2$, giving amortized cost $2-2+1=1$. For a stack with multipop: let $\Phi$ be the number of elements in the stack. A push costs $1+\Delta\Phi=1+1=2$ amortized; a multipop of $k$ costs $k$ actual but $\Delta\Phi=-k$, amortized cost $k-k=0$. The sum of amortized costs bounds the actual cost for any sequence.
    </ExerciseSolution>

## Deliberately Omitted

**Advanced persistence and concurrency models**: the assessment tests understanding of basic persistence (path copying) and concurrency concerns (snapshot isolation). Fully persistent data structures with branching histories and concurrent linearizability proofs are beyond the assessment scope.

**External-memory data structures beyond B-trees**: cache-oblivious B-trees, buffer trees, and priority queues for the I/O model. The assessment tests B-tree invariants as a representative external-memory structure but does not cover the broader I/O-algorithms landscape.

**Probabilistic data structures**: Bloom filters, Count-Min Sketch, HyperLogLog, and skip lists. These structures rely on randomized guarantees rather than deterministic invariants and follow a different correctness and analysis model.

**Formal verification of data structures**: encoding specifications in Dafny, VeriFast, or Coq and mechanically verifying invariants. The assessment tests the ability to write and reason about invariants by hand, not the use of automated verification tools.

**Cache-oblivious and cache-adaptive algorithms**: the assessment covers cache locality and B-tree memory models but does not require analysis of the ideal-cache model or cache-oblivious matrix multiplication.

**Distributed data structures**: consistent hashing, distributed hash tables, CRDTs, and gossip-based structures. These belong to distributed systems, not core data structures.
