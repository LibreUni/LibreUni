# Graph Representations

A social network with 2 billion users must answer "are these two people connected?" and "who can introduce me to the shortest path to a target person?" Storing every possible pair would require $4 \times 10^{18}$ entries — physically impossible. Instead, each user stores roughly 200 friends on average, so a representation proportional to edges rather than vertex pairs makes feasibility possible. Graph algorithms depend not only on abstract connectivity but on how neighbors are stored. Choosing a representation *is* an algorithmic decision.

## Formal graph model

<MathStatement id="graph-def" kind="definition" title="Directed graph">
A directed graph is $G = (V, E)$ where $E \subseteq V \times V$. An edge $(u, v)$ goes from $u$ to $v$. An undirected edge is an unordered pair $\{u, v\}$, conventionally represented by two directed adjacency entries. A weighted graph assigns a weight $w: E \to \mathbb{R}$; a multigraph permits parallel edges; self-loops $(v, v)$ may be allowed or forbidden.
</MathStatement>

A graph ADT may support `add-vertex`, `add-edge`, `remove-edge`, `adjacent(u, v)`, `neighbors(u)`, `degree`, and edge iteration. Vertex identifiers can be dense integers (allowing array indexing) or arbitrary objects requiring an index map. If vertices are removed, decide whether identifiers are reused, whether edges disappear automatically, and whether external handles remain valid.

### Representation choice as an algorithmic primitive

<StructureDiagram kind="graph" title="Toy graph: 5 vertices, 7 edges" caption="All three representations encode this same connectivity. Edge list: (A,B), (A,E), (B,C), (B,D), (B,E), (C,D), (D,E)." variant="default" />

<MathStatement id="degree-hammer" kind="theorem" title="The degree hammer">
If an algorithm performs $k$ neighbor iterations per vertex and the average degree is $d$, using an adjacency matrix costs $k \cdot |V|$ per iteration vs. $k \cdot d$ for adjacency lists. When $|V| = 10^6$ and $d = 10$, the ratio is $100\,000\! :\! 1$.
</MathStatement>

The key insight: the operation set of the *algorithm* determines which representation is asymptotically faster. A representation that makes neighbor iteration fast may make edge existence checks slow, and vice versa.

## Adjacency matrix

An adjacency matrix $A$ has one row and column per vertex. For an unweighted directed graph, $A[u][v] = 1$ when edge $(u, v)$ exists, $0$ otherwise. For weighted graphs, use a sentinel for absence and define whether zero or infinity is a legal weight.

<MathStatement id="matrix-prop" kind="proposition" title="Adjacency matrix properties">
- Edge existence and weight lookup: $O(1)$
- Adding or removing an edge: $O(1)$ (dense identifiers)
- Space: $\Theta(|V|^2)$ regardless of edge count
- Neighbor iteration of $u$: scans entire row — $O(|V|)$ — even when outdegree is 1
</MathStatement>

The matrix uses $\Theta(|V|^2)$ space regardless of edge count. Iterating neighbors of $u$ scans an entire row, $O(|V|)$, even if degree is small. Matrix multiplication and Boolean algebra can support specialized dense-graph algorithms, and predictable contiguous storage can outperform pointers on moderate dense graphs. An undirected matrix should maintain symmetry; updating only one triangle violates the representation invariant.

<CodeRunner client:load

  language="python"

    def __init__(self, n):
        self.n = n
        self.A = [[0]*n for _ in range(n)]

    def add_edge(self, u, v, w=1):
        self.A[u][v] = w

    def adjacent(self, u, v):
        return self.A[u][v] != 0

    def neighbors(self, u):
        return [v for v in range(self.n) if self.A[u][v] != 0]

## Adjacency lists

An adjacency-list representation stores, for each vertex, a collection of outgoing edges.

<MathStatement id="list-prop" kind="proposition" title="Adjacency-list properties">
- Space: $\Theta(|V| + |E|)$
- Neighbor iteration of $u$: $O(\text{outdegree}(u))$
- Complete traversal over all adjacency lists: $O(V + E)$
- Edge membership in unsorted list: $O(\text{degree})$; sorted list: $O(\log \text{degree})$ with slower insertion
</MathStatement>

For an undirected graph, inserting an edge usually adds $u$ to $v$'s list and $v$ to $u$'s list. The invariant is that each undirected edge has exactly two reciprocal entries, unless parallel edges are explicitly allowed. A directed graph has one entry per directed edge. Weighted entries store destination and weight, and equality for removing an edge must include endpoint and duplicate policy.

Neighbor membership in an unsorted list is $O(\text{degree})$; sorting each list enables binary search but makes insertion more expensive. A per-vertex hash set gives expected constant membership and insertion but loses order and requires hash assumptions. A balanced tree gives ordered neighbors and worst-case logarithmic membership. The best choice depends on the algorithm's primitive operations.

<CodeRunner client:load
  title="Adjacency list in Python"
  language="python"
  code={`class AdjList:
    def __init__(self, n):
        self.n = n
        self.adj = [[] for _ in range(n)]

    def add_edge(self, u, v, w=1):
        self.adj[u].append((v, w))

    def neighbors(self, u):
        return self.adj[u]

    def all_edges(self):
        for u in range(self.n):
            for v, w in self.adj[u]:
                yield u, v, w

g = AdjList(3)
g.add_edge(0, 1)
g.add_edge(1, 2)
g.add_edge(2, 0)
print("Neighbors of 0:", g.neighbors(0))
print("All edges:", list(g.all_edges()))`}
  output="Neighbors of 0: [(1, 1)]\nAll edges: [(0, 1, 1), (1, 2, 1), (2, 0, 1)]"

## Edge lists and incidence

An edge list stores records $(u, v, w)$. It uses $O(E)$ space and supports scanning all edges in $O(E)$, useful for Kruskal, Bellman-Ford, and input/output. Neighbor queries require scanning all edges unless an index is added.

<CodeRunner client:load
  title="Edge list in Python"
  language="python"
  code={`edges = [(0, 1, 4), (0, 2, 3), (1, 2, 1), (1, 3, 2), (2, 3, 5)]

# Neighbor lookup: O(E)

An incidence matrix stores vertex-edge relationships and uses $O(VE)$, rarely ideal for general sparse graphs but useful for algebraic formulations or hypergraph-like constraints.

## Worked representation comparison

Consider $V = 10\,000$ and $E = 20\,000$. A matrix has $10^8$ possible entries, while an adjacency list stores roughly $10\,000$ vertex headers plus $20\,000$ directed entries for a directed graph. BFS on lists examines $O(V+E)$ records; on a matrix it scans $10\,000$ cells per dequeued vertex, $O(V^2)$, even though the graph is sparse. Conversely, testing whether one of many possible edges exists may favor a matrix when the memory budget permits it.

<MathStatement id="bfs-cost" kind="theorem" title="BFS cost by representation">
For BFS on a graph with $V$ vertices and $E$ edges:
- **Adjacency list:** $O(V + E)$ — each vertex dequeued once, each edge examined once.
- **Adjacency matrix:** $O(V^2)$ — each dequeued vertex scans all $V$ cells in its row, regardless of degree.
</MathStatement>

## Counterexample and failure mode

<BookOnly>

### Failure mode: stale visited metadata

</BookOnly>

A correct BFS marks vertices as visited. If a prior traversal leaves marks in a shared array, a later BFS silently declares the graph disconnected — it skips already-marked vertices and never discovers new paths.

<CodeRunner client:load
  title="Stale visited metadata corrupts BFS"
  language="python"
  code={`def bfs(adj, start, visited):
    from collections import deque
    visited[start] = True
    q = deque([start])
    while q:
        u = q.popleft()
        for v in adj[u]:
            if not visited[v]:
                visited[v] = True
                q.append(v)

# Two components: 0-1 and 2-3
adj = [[1], [0], [3], [2]]
visited = [False] * 4

print("First BFS (component 0-1):")
bfs(adj, 0, visited)
print("  visited:", visited)  # marks [T, T, F, F]

# Oops — visited is not reset
print("Second BFS (from vertex 2) with stale array:")
bfs(adj, 2, visited)
print("  visited:", visited)
print("  Vertex 0 reachable from 2?", visited[0])  # True! Wrong.
print("  Actually 0 is NOT reachable from 2.")`}
  output="First BFS (component 0-1):\n  visited: [True, True, False, False]\nSecond BFS (from vertex 2) with stale array:\n  visited: [True, True, True, True]\n  Vertex 0 reachable from 2? True\n  Actually 0 is NOT reachable from 2."
/>

The fix: reset the visited array before each traversal. Alternatively, use a traversal timestamp or generation field so marks are scoped to one run.

### Failure mode: vertex identity mutation

If vertex objects are mutable and their identity-determining fields change after insertion into the graph, the vertex may become unreachable from the internal map — exactly as a mutable dictionary key breaks a hash table.

## Traversal state

BFS and DFS add metadata to the graph: color or visited status, parent, distance, discovery and finish times, or component label. Keep traversal metadata separate from the graph representation unless the graph ADT owns algorithm state. A reused visited array must be reset or timestamped.

For a directed graph, transpose construction reverses every edge. With adjacency lists, scan each list and append the reversed entry; time $O(V+E)$. For a matrix, transpose swaps entries or accesses columns; copying takes $O(V^2)$. Representation affects the cost of a mathematically identical operation.

## Dynamic and external graphs

If edges arrive online, choose whether neighbor iteration sees a snapshot or concurrent updates. A graph with millions of vertices may store adjacency in compressed sparse row (CSR) form: one offset array and a contiguous destination array. CSR gives excellent iteration locality but expensive insertion; batch updates and rebuilds can amortize the cost. A mutable sparse graph often uses vectors, hash maps, or log-structured adjacency with periodic compaction.

External-memory graphs must count block transfers rather than RAM operations. Grouping neighbors by vertex can reduce I/O for traversals; random adjacency queries may require indexes or caching. Distributed graphs add partition boundaries, ghost vertices, and communication cost. An $O(V+E)$ sequential algorithm can be dominated by cross-partition messages.

## Identity, ordering, and mutation

If vertex objects are mutable, using their fields as hash keys can make them unreachable from a vertex map. Give each vertex a stable identity or prohibit identity-field mutation. If neighbor order is observable, use a deterministic collection and state whether insertion order, identifier order, or no order is promised. Algorithms that rely on a stable iteration order should not receive an unordered graph without an explicit tie policy.

Removing a vertex is a multi-structure mutation. Delete its outgoing edges, remove it from every incoming neighbor collection, update degree counts, and invalidate or remap its identifier. A lazy tombstone can make deletion cheap, but traversals must skip tombstones and memory usage can grow. Batch compaction is an amortized alternative.

## Validation and property tests

Validate that every endpoint exists, weights satisfy the chosen domain, undirected entries are reciprocal, edge counts match, and no forbidden duplicates occur. Generate small random graphs and compare matrix and list implementations for adjacency, degree, traversal reachability, and transpose. For an undirected graph, the number of adjacency entries should equal twice the logical edge count when parallel/self-loop conventions are accounted for. These cross-representation tests are especially effective because each implementation can act as an oracle for the other.

    # Build matrix
    mat = [[0]*V for _ in range(V)]
    # Build list
    adj = [[] for _ in range(V)]
    for u, v in edges:
        mat[u][v] = 1

## Algorithm-facing design

Before selecting a representation, list the operations the next algorithm performs. BFS needs neighbors and visited state; Kruskal needs a global edge ordering and union-find; dense shortest-path algorithms may repeatedly test all pairs; dynamic connectivity needs updates and component queries. A representation optimized for one operation can make another algorithm asymptotically or operationally unsuitable. Treat the graph ADT as a contract between storage and algorithm, not as a single universal class.

## Transfer and synthesis exercises

1. An algorithm on an undirected graph requires: (a) fast neighbor iteration and (b) checking whether any pair of vertices is adjacent, both operations interleaved. The graph has $V = 10^5$, $E = 10^6$. Design a hybrid representation. State its invariants, space usage, and operation costs.

   <ExerciseSolution hint="Could one data structure serve both needs poorly but both well through composition?">
   Store two representations: an adjacency list for iteration and a hash set of canonicalized edge keys $(\\min(u,v), \\max(u,v))$ for adjacency checks. Space: $O(V + E)$ for lists plus $O(E)$ for the hash set. Neighbor iteration is $O(\\text{degree})$; adjacency check is expected $O(1)$. The invariant: every edge appears in both the list and the hash set; removal must delete from both.
   </ExerciseSolution>

2. Reconstruct the space formula for CSR: derive the offset array size and the destination array size in terms of $V$ and $E$. Show that CSR uses exactly one integer per edge plus one integer per vertex.

   <ExerciseSolution>
   CSR stores: an offset array `row_ptr` of length $V+1$ (one per vertex plus sentinel $\\text{row_ptr}[V] = E$), and a destination array `col_ind` of length $E$. The offset entries are cumulative degree sums: $\\text{row_ptr}[i] = \\sum_{j < i} \\text{degree}(j)$. Space: $(V+1) + E$ integers.
   </ExerciseSolution>

3. Adapt adjacency lists to support a multigraph with weighted parallel edges. Define a `remove-edge` operation that deletes exactly one specified edge when duplicates exist. Does your design allow $O(1)$ removal?

   <ExerciseSolution>
   Store a list of edge records per source; each record has destination, weight, and a unique edge ID (monotonic counter). remove-edge(edge_id) uses the ID to locate and delete the record, requiring a mapping from edge ID to (source, list position). This mapping can be a hash table for expected $O(1)$ removal. Without IDs, removing one occurrence of a parallel edge requires linear scan or assumes FIFO/LIFO policy.
   </ExerciseSolution>

4. A BFS traverses a graph stored in CSR format. Write pseudocode for the BFS inner loop using only the offset and destination arrays. What is the cache behavior compared to a pointer-chasing adjacency list?

   <ExerciseSolution>
   ```python
   for u in queue:
       start = row_ptr[u]
       end = row_ptr[u + 1]
       for i in range(start, end):
           v = col_ind[i]
           if not visited[v]:
               visited[v] = True
               queue.append(v)
   ```
   CSR reads neighbors of $u$ from a contiguous segment of `col_ind`, producing a predictable sequential access pattern. A pointer-chasing list follows scattered heap allocations, causing more cache misses. CSR's locality advantage grows with vertex degree.
   </ExerciseSolution>

## Deliberately omitted

- **Hypergraphs** (edges connect more than two vertices): incidence matrix techniques exist but are not required for the core graph algorithm curriculum.
- **Graph databases** and query-language internals (Cypher, SPARQL) build on these primitives but belong in a databases or information systems course.
- **Spectral graph theory** (Laplacian, eigenvalues, Cheeger's inequality): adjacency structures here are matrices, but the algebraic treatment and its applications are deferred to advanced algorithms or numerical linear algebra.
- **Succinct graph representations** (e.g., $o(|E| \log |V|)$ bits): specialized encodings for web graphs and social networks use delta-encoding, gap compression, and variable-length codes. These are engineering extensions of the CSR concept.
- **Streaming and semi-streaming graph models** where edges arrive in a single pass and sublinear memory is required.
