# Graph Representations

A graph is a set of vertices together with edges relating pairs of vertices. Graph algorithms depend not only on abstract connectivity but on how neighbors are stored. An adjacency matrix offers constant-time edge existence and predictable dense storage; adjacency lists offer space proportional to vertices plus edges and efficient neighbor iteration for sparse graphs; edge lists are compact and useful for global edge scans. Choosing a representation is an algorithmic decision.

## Formal graph model

A directed graph is (G=(V,E)) with (Esubseteq Vtimes V). An edge (u,v) goes from u to v. An undirected edge is an unordered pair, often represented by two directed adjacency entries. A weighted graph assigns a weight w(u,v); a multigraph permits parallel edges, and a graph may include self-loops. State which model an ADT supports because degree, traversal, and duplicate behavior differ.

A graph ADT may support add-vertex, add-edge, remove-edge, adjacent(u,v), neighbors(u), degree, and edge iteration. Vertex identifiers can be dense integers or arbitrary objects requiring an index map. If vertices are removed, decide whether identifiers are reused, whether edges disappear automatically, and whether external handles remain valid.

## Adjacency matrix

An adjacency matrix A has one row and column per vertex. For an unweighted directed graph, A[u][v] is true exactly when edge (u,v) exists. For weighted graphs, use a sentinel for absence and define whether zero or infinity is a legal weight. Edge existence and weight lookup are O(1), and adding/removing an edge is O(1) for dense identifiers.

The matrix uses (Theta(|V|^2)) space regardless of edge count. Iterating neighbors of u scans an entire row, O(|V|), even if degree is small. Matrix multiplication and Boolean algebra can support specialized dense-graph algorithms, and predictable contiguous storage can outperform pointers on moderate dense graphs. An undirected matrix should maintain symmetry; updating only one triangle violates the representation invariant.

## Adjacency lists

An adjacency-list representation stores, for each vertex, a collection of outgoing edges. Total space is (Theta(|V|+|E|)). Iterating neighbors of u costs O(outdegree(u)) plus collection overhead. A complete traversal over all adjacency lists costs O(V+E), which is the natural input size for sparse graph algorithms.

For an undirected graph, inserting an edge usually adds u to v’s list and v to u’s list. The invariant is that each undirected edge has exactly two reciprocal entries, unless parallel edges are explicitly allowed. A directed graph has one entry per directed edge. Weighted entries store destination and weight, and equality for removing an edge must include endpoint and duplicate policy.

Neighbor membership in an unsorted list is O(degree); sorting each list enables binary search but makes insertion more expensive. A per-vertex hash set gives expected constant membership and insertion but loses order and requires hash assumptions. A balanced tree gives ordered neighbors and worst-case logarithmic membership. The best choice depends on the algorithm’s primitive operations.

## Edge lists and incidence

An edge list stores records (u,v,w). It uses O(E) space and supports scanning all edges in O(E), useful for Kruskal, Bellman-Ford, and input/output. Neighbor queries require scanning all edges unless an index is added. An incidence matrix stores vertex-edge relationships and uses O(VE), rarely ideal for general sparse graphs but useful for algebraic formulations or hypergraph-like constraints.

## Worked representation comparison

Consider V=10,000 and E=20,000. A matrix has 100 million possible entries, while an adjacency list stores roughly 10,000 vertex headers plus 20,000 directed entries for a directed graph. BFS on lists examines O(V+E) records; on a matrix it scans 10,000 cells per dequeued vertex, O(V^2), even though the graph is sparse. Conversely, testing whether one of many possible edges exists may favor a matrix when the memory budget permits it.

For a dense graph with E=Theta(V^2), an adjacency list still stores O(E) entries and neighbor iteration is optimal up to constants, while a matrix offers compact fixed-size booleans and O(1) membership. “Sparse” and “dense” are relative to V and the workload; state the density threshold or compare actual primitive costs.

## Traversal state

BFS and DFS add metadata to the graph: color or visited status, parent, distance, discovery and finish times, or component label. Keep traversal metadata separate from the graph representation unless the graph ADT owns algorithm state. A reused visited array must be reset or timestamped; stale marks can make a correct graph appear disconnected.

For a directed graph, transpose construction reverses every edge. With adjacency lists, scan each list and append the reversed entry; time O(V+E). For a matrix, transpose swaps entries or accesses columns; copying takes O(V^2). Representation affects the cost of a mathematically identical operation.

## Dynamic and external graphs

If edges arrive online, choose whether neighbor iteration sees a snapshot or concurrent updates. A graph with millions of vertices may store adjacency in compressed sparse row (CSR) form: one offset array and a contiguous destination array. CSR gives excellent iteration locality but expensive insertion; batch updates and rebuilds can amortize the cost. A mutable sparse graph often uses vectors, hash maps, or log-structured adjacency with periodic compaction.

External-memory graphs must count block transfers rather than RAM operations. Grouping neighbors by vertex can reduce I/O for traversals; random adjacency queries may require indexes or caching. Distributed graphs add partition boundaries, ghost vertices, and communication cost. An O(V+E) sequential algorithm can be dominated by cross-partition messages.

## Identity, ordering, and mutation

If vertex objects are mutable, using their fields as hash keys can make them unreachable from a vertex map just as a mutable dictionary key breaks a hash table. Give each vertex a stable identity or prohibit identity-field mutation. If neighbor order is observable, use a deterministic collection and state whether insertion order, identifier order, or no order is promised. Algorithms that rely on a stable iteration order should not receive an unordered graph without an explicit tie policy.

Removing a vertex is a multi-structure mutation. Delete its outgoing edges, remove it from every incoming neighbor collection, update degree counts, and invalidate or remap its identifier. A lazy tombstone can make deletion cheap, but traversals must skip tombstones and memory usage can grow. Batch compaction is an amortized alternative; its cost belongs in the representation contract.

## Validation and property tests

Validate that every endpoint exists, weights satisfy the chosen domain, undirected entries are reciprocal, edge counts match, and no forbidden duplicates occur. Generate small random graphs and compare matrix and list implementations for adjacency, degree, traversal reachability, and transpose. For an undirected graph, the number of adjacency entries should equal twice the logical edge count when parallel/self-loop conventions are accounted for. These cross-representation tests are especially effective because each implementation can act as an oracle for the other.

## Algorithm-facing design

Before selecting a representation, list the operations the next algorithm performs. BFS needs neighbors and visited state; Kruskal needs a global edge ordering and union-find; dense shortest-path algorithms may repeatedly test all pairs; dynamic connectivity needs updates and component queries. A representation optimized for one operation can make another algorithm asymptotically or operationally unsuitable. Treat the graph ADT as a contract between storage and algorithm, not as a single universal class.

## Exercises

1. State adjacency-list invariants for directed and undirected graphs.
2. Compare matrix and list costs for edge lookup and neighbor iteration.
3. Represent a weighted multigraph with parallel edges and define remove-edge.
4. Calculate storage and BFS work for a graph with V and E symbolic.
5. Explain CSR’s iteration/insertion tradeoff.
6. Design a safe transpose operation for an adjacency list.
7. Identify how stale visited metadata corrupts a repeated traversal.

### Diagnostic solutions


1. Directed: each edge appears exactly once in the source’s outgoing collection, with valid destination and weight. Undirected: each logical edge appears in two reciprocal collections, unless a canonical single-record design is explicitly used; duplicate rules must be stated.

2. Matrix membership is O(1), list membership O(degree) unsorted or expected O(1) hash-based. Matrix neighbor iteration is O(V) per source; list iteration is O(degree), so total list scan is O(V+E).

3. Store a list of edge records per source, allowing equal destinations with distinct edge IDs or weights. remove-edge must specify remove one matching record, all, or a chosen ID; silently collapsing parallel edges changes graph semantics.

4. Matrix storage is u0398(V^2); list storage u0398(V+E). BFS with lists is O(V+E), while scanning a matrix row for each reached vertex is O(V^2) in the worst case. Add the cost of initializing visited and queue state.

5. CSR stores offsets and neighbors contiguously, so scanning one vertex’s neighbors is cache- and I/O-friendly. Inserting an edge may shift a large suffix or require rebuilding; batch updates and periodic compaction trade update latency for scan performance.

6. Create an empty list for every destination, then for each u and each edge u→v append a reversed edge v→u. Preserve weights and edge IDs, and verify that the resulting edge count equals the original directed edge count and each original edge appears reversed.

7. If a prior BFS leaves vertices marked, a later BFS may skip them and fail to assign new parents or distances. Clear the array, use a traversal timestamp, or store a generation field so marks are scoped to one run.
