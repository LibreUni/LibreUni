# Hash Tables and Expected-Time Dictionaries

A phonebook with 100,000 entries needs lookup in constant time. The ideal: a giant array indexed by the phone number itself — but phone numbers are 10 digits, and a $10^{10}$-element array is impossible. The key insight: map the 10-digit key to a small array slot through a **hash function**.

Hashing: a function $h: U \to \{0, \dots, m-1\}$ maps keys from a universe $U$ to array indices of size $m \ll |U|$. Two distinct keys can map to the same index — a **collision**. Two main strategies resolve it:

- **Chaining**: each slot stores a bucket (linked list, dynamic array, or balanced tree) of all keys hashing to that slot.
- **Open addressing**: on collision, probe alternative slots until an empty one is found.

The first hash tables appeared in IBM's FORTRAN compiler (1953–1957) for symbol table lookups. W. Wesley Peterson published the first formal analysis of "scatter storage" in 1957, introducing open addressing. The name "hash" (from the culinary sense of "chop and mix") was popularized by the early 1960s. The core insight: with a good hash function and a bounded **load factor** $\alpha = n/m$, all operations run in expected $O(1)$ time — not worst-case, but expected over the randomness of the hash function.

## Dictionary abstraction

<MathStatement
  id="dict-adts"
  kind="definition"
  title="Dictionary ADT"
>
A dictionary stores a finite set of key-value pairs with unique keys. It supports $\text{put}(k, v)$, $\text{get}(k)$, $\text{contains}(k)$, and $\text{remove}(k)$. The contract requires:

- If $k$ is present, $\text{get}(k)$ returns the associated value; otherwise it signals absence.
- $\text{put}(k, v)$ inserts the pair or replaces an existing value for $k$.
- $\text{remove}(k)$ deletes the pair if present.
- Key equality and hash behavior must agree: if $k_1 = k_2$ then $\text{hash}(k_1) = \text{hash}(k_2)$. Collisions between distinct keys are expected and do not violate the contract.
</MathStatement>

The table has $m$ slots and a hash function $h: U \to \{0, \dots, m-1\}$. A collision occurs when distinct keys map to the same slot. Uniform hashing heuristically spreads keys across slots, but an implementation usually uses a deterministic function. Security-sensitive services may randomize a seed so an attacker cannot cheaply construct collisions from a known function.

## Separate chaining

<StructureDiagram kind="hash" title="Separate-chaining buckets" caption="Each table slot owns a bucket whose entries share the same hash destination." />

In separate chaining, each slot stores a bucket, often a linked list or dynamic array. To $\text{get}(k)$, compute $h(k)$, scan its bucket, and compare keys. If $n$ keys are distributed uniformly among $m$ buckets, the expected chain length is $\alpha = n/m$. Expected lookup is $O(1 + \alpha)$; keeping $\alpha = O(1)$ yields expected $O(1)$.

**Worked example 1 (uniform).** With $m = 8$ and hash values $0, 3, 3, 7, 3$, the table has chain lengths: slot 0 has length 1, slot 3 has length 3, slot 7 has length 1. The global load factor is $5/8$, but the cost of a specific operation depends on its bucket. Searching for a key in slot 3 scans at most 3 entries.

**Worked example 2 (adversarial worst-case).** A hash table with $m = 4$ and a poor hash function $h(k) = 0$ for all keys. Every insertion maps to slot 0. After $n = 1000$ insertions, the chain at slot 0 has length 1000. Every lookup scans the entire bucket: $O(n)$ time, not $O(1)$. This is not a failure of the algorithm but of the hash function or the assumption of randomness.

The expected bound is not a worst-case guarantee. A balanced tree in each bucket supplies $O(\log n)$ worst-case per bucket, while a collision-resistant or keyed hash reduces the chance of accidental or adversarial concentration. Bucket representation affects memory, locality, and constant factors.

## Open addressing

<StructureDiagram kind="hash" title="Open-addressed table" caption="Keys occupy slots found by a probe sequence; empty and deleted slots have different meanings." />

<HashTablePlayground client:load />

Open addressing stores entries directly in the table. A probe sequence $p(k, i)$ gives the slot checked on attempt $i$. Linear probing uses $(h(k) + i) \bmod m$, quadratic probing uses $(h(k) + c_1 i + c_2 i^2) \bmod m$, and double hashing uses $(h_1(k) + i \cdot h_2(k)) \bmod m$. Lookup follows exactly the same sequence until it finds the key, an empty slot, or a termination condition.

Deletion cannot simply clear a slot: a later key may have collided through it, and clearing would make lookup stop early. Use a **tombstone** marker or backward-shift deletion. Tombstones increase unsuccessful-search cost and require rebuilding. An empty slot means no future key in that probe sequence lies beyond it; a tombstone means a key may lie beyond but the slot itself is available for insertion.

Open addressing needs $\alpha < 1$, and performance worsens as $\alpha$ approaches one. Under simple uniform hashing, expected unsuccessful probes are approximately $1 / (1 - \alpha)$, while successful probes depend on the occupied cluster. Linear probing suffers **primary clustering**: contiguous runs attract more keys and grow. Quadratic probing reduces some clustering but has coverage constraints; double hashing can approximate independent sequences if the second hash is suitable.

## Expected-time theorem

<MathStatement
  id="expected-time"
  kind="theorem"
  title="Expected constant-time hashing"
  proofId="expected-time-proof"
  number="1"
>
Under the assumption of simple uniform hashing (each key is equally likely to hash to any slot, independently), and with a load factor $\alpha = n/m$ bounded above by a constant, the expected time for each dictionary operation is $O(1)$.
</MathStatement>

<MathProof id="expected-time-proof" for="expected-time" collapsible>
Let $n$ keys be distributed uniformly among $m$ slots. For a given key $k$, the expected number of other keys that collide with it is $n/m = \alpha$, by linearity of expectation over the $n-1$ other keys, each with collision probability $1/m$.

For chaining, an unsuccessful search scans the entire bucket at $h(k)$; the expected bucket length is $1 + \alpha$ (the key itself plus colliding keys). A successful search, assuming each position in the chain is equally likely, scans on average $1 + \alpha/2$ entries. Hence expected time is $O(1 + \alpha) = O(1)$ when $\alpha = O(1)$.

For open addressing under uniform probing, the expected number of probes for an unsuccessful search is $1/(1 - \alpha)$, and for a successful search is $(1/\alpha) \ln(1/(1 - \alpha))$. Both are $O(1)$ when $\alpha$ is bounded away from 1.

The proof depends on the probability model. If keys are not random or the hash function is not uniform, the bounds may not hold.
</MathProof>

The expected-time theorem depends on three assumptions:

1. **Uniform distribution**: the hash function spreads keys evenly.
2. **Independence**: the hash of one key does not predict the hash of another.
3. **Bounded load**: $\alpha$ stays within a constant range.

When any of these fails, the guarantees vanish.

**Example.** A web forum uses a hash table keyed by user IDs. An attacker registers 10,000 accounts whose IDs are chosen to collide under the table's hash function. Every lookup for a colliding ID scans the entire bucket — $O(n)$ per operation. The attack succeeds because the hash function is deterministic, public, and maps all attacker-chosen IDs to one slot.

Defenses: keyed hashing (random seed per-process), per-request work limits, and secondary bucket structures (balanced trees, cuckoo hashing).

## Hash-function design

A hash function should mix key bits so structured inputs do not collide. For strings, a polynomial rolling hash treats characters as coefficients modulo a large range:

$$h(s) = \left(\sum_{i=0}^{|s|-1} s_i \cdot b^{i}\right) \bmod M$$

where $b$ is a base (e.g., 131 or 257) and $M$ is a large prime. Overflow arithmetic must be defined. Combining fields should avoid commutative operations if field order matters. Hash equality must remain stable while a key is stored; mutating a key's fields can make it unreachable in its original bucket.

**Universal hashing** chooses a hash function randomly from a family with bounded collision probability for every distinct key pair. If a family is 2-universal, $\Pr[h(x) = h(y)] \le 1/m$. The expected number of colliding pairs can then be bounded without assuming keys are random. This is a mathematical guarantee over the seed choice, not necessarily over every fixed seed.

Cryptographic hashing and dictionary hashing have different goals. A cryptographic hash seeks preimage and collision resistance; a table hash seeks fast distribution and may use a secret seed. A cryptographic hash can be slower than needed, while a fast non-cryptographic hash may be vulnerable to chosen-key attacks.

## Resizing and amortization

<Quiz client:load

    "Reinsertion is cheaper than copying"
  ]}
  correctIndex={0}
  explanation="h(k) = k mod m changes with m, so a key's slot in the old table may differ from the new table."
/>

When load exceeds a threshold, allocate a new table and reinsert every live entry using the new modulus or seed. Copying entries to the same index blindly is incorrect: $h(k)$ generally changes with $m$. A resize costs $O(n)$ actual time but occurs only after a linear number of insertions under geometric capacity growth, so insertion is expected $O(1)$ amortized under the hashing assumption.

Resizing invalidates references to buckets or table slots. During incremental rehashing, operations may consult old and new tables and migrate a bounded number of entries per operation; the invariant must specify where every key can reside.

## Load factor is a policy variable

The load factor threshold is a tradeoff, not a theorem. Chaining can tolerate $\alpha > 1$ while preserving expected constant time if bucket lengths remain well behaved, but memory references and tail latency grow. Open addressing must leave empty slots and usually selects a lower threshold. A service should monitor not only average load but maximum bucket length, probe counts, tombstone fraction, resize duration, and memory headroom.

If a hash table is used for untrusted request keys, an attacker can send many distinct keys designed to collide or force repeated resizing. Bound per-request work, cap table growth, rate-limit creation, randomize the hash seed, and consider a balanced-tree fallback.

## Deletion and iteration edge cases

A remove operation must distinguish absent keys, live entries, and tombstones. In chaining, unlink exactly one equal key and decrement size; removing a node while an iterator points to it requires an invalidation or continuation rule. In open addressing, tombstones may be reused for insertion but must not terminate a search. Rebuilding after too many tombstones restores probe performance and can be charged to updates using the same amortization reasoning as resizing.

## Correctness and complexity

<Quiz client:load
  question="What must be explicitly stated in an expected-time analysis?"
  options={[

    "The size of the computer's RAM",

  ]}
  correctIndex={0}

For chaining, the representation invariant (RI) is that every live pair appears in exactly one bucket at the index determined by the current hash function and table state, no duplicate key exists, and size counts live pairs. For open addressing, every live key lies somewhere in its probe sequence before the first never-used slot, and tombstones are interpreted consistently. Prove $\text{put}$, $\text{get}$, $\text{remove}$, and resize preserve these properties.

Expected-time analysis must state the probability space. Is the randomness in keys, hash seed, or algorithmic choices? Worst-case guarantees require additional assumptions or structures. A benchmark on random keys does not prove expected behavior for production traffic if inputs are correlated or attacker-controlled.

1. Specify a dictionary RI for separate chaining and prove lookup correctness.

   <ExerciseSolution hint="A representation invariant has three clauses: shape, ordering, and metadata. What must hold for every live key's position?">

   RI: each live key-value pair occurs once in bucket $h(k)$, key equality has no duplicates, and size equals live-pair count. get computes the same $h(k)$ for an unchanged key and scans that bucket; it finds the pair iff it is present, by the RI.
   </ExerciseSolution>

2. Compute chain lengths and load factor for a given bucket assignment, then estimate expected comparisons.

   <ExerciseSolution hint="The load factor $\alpha = n/m$ is the expected bucket length under uniform hashing.">

   Count entries per slot; $\alpha = n/m$. Under uniform hashing, expected bucket length is $\alpha$, so an unsuccessful scan is expected $O(1+\alpha)$; a successful scan depends on where within its bucket the key is found, but remains expected $O(1+\alpha)$.
   </ExerciseSolution>

3. Explain why clearing an open-addressing slot on deletion breaks lookup.

   <ExerciseSolution>
   Suppose key B collided after A in A's probe sequence. If A's slot is cleared, lookup for B stops at the empty slot and falsely reports absence. A tombstone preserves the fact that probing must continue.
   </ExerciseSolution>

4. Compare linear probing, quadratic probing, and double hashing.

   <ExerciseSolution>
   Linear probing is simple and cache-friendly but clusters. Quadratic probing reduces primary clustering but may not visit every slot and can have secondary clustering. Double hashing gives more independent sequences when the second hash and table size satisfy coverage conditions.
   </ExerciseSolution>

5. Prove geometric resizing gives amortized $O(1)$ rehash work per insertion.

   <ExerciseSolution>
   If capacity grows by factor $\rho > 1$, rehash costs at capacities $C, \rho C, \rho^2 C, \dots$ form a geometric series. The total cost before $m$ insertions is $O(m)$, giving $O(1)$ amortized rehash cost per insertion, in addition to expected constant ordinary insertion.
   </ExerciseSolution>

6. State the equality/hash contract and give a mutable-key failure.

   <ExerciseSolution>
   Equal keys must have equal hashes. If a key is mutated after insertion so its hash changes, get computes a different bucket and misses it; remove and contains fail even though the pair remains stored.
   </ExerciseSolution>

7. Explain why keyed hashing can mitigate adversarial collisions but does not make every operation worst-case constant.

   <ExerciseSolution>
   A secret seed prevents an attacker from preparing collisions without knowing the seed, but a particular seed can still have a bad input set, and operations in one heavily colliding bucket remain linear. Use bucket trees, limits, or a different structure when worst-case bounds are required.
   </ExerciseSolution>

8. **Transfer.** A hash table stores points $(x, y)$ in the plane. The hash function is $h(p) = (p.x + p.y) \bmod m$. Describe a collision pattern that produces long chains, and propose a better hash.

   <ExerciseSolution>
   All points on a line $x + y = c$ collide. A better hash combines coordinates with a non-commutative operation, e.g., $h(p) = (p.x \cdot a + p.y \cdot b) \bmod m$ for distinct constants $a, b$.
   </ExerciseSolution>

9. **Synthesis.** Design a data structure that supports dictionary operations in $O(1)$ expected time and also supports iterating over all keys in sorted order in $O(n \log n)$ time. Explain the tradeoffs.

   <ExerciseSolution>
   Combine a hash table for expected $O(1)$ operations with a balanced BST or sorted list for iteration. Insert and delete update both structures. The hash table provides fast lookups; the BST provides sorted iteration. Memory doubles and update cost grows by a constant factor.
   </ExerciseSolution>

## Deliberately Omitted

**Cuckoo hashing and hopscotch hashing**: alternative collision-resolution schemes with worst-case constant-time lookups and more complex insertion algorithms. Cuckoo hashing uses two or more hash functions and relocates existing entries on collision; its analysis requires understanding random-graph properties that are beyond this lesson's scope.

**Perfect hashing and minimal perfect hashing**: constructing a hash function with no collisions for a known static set. Perfect hashing has applications in compiler symbol tables and static databases but does not support dynamic insertion.

**Consistent hashing and rendezvous hashing**: hash-based load distribution across a changing set of servers. These are distributed-systems primitives for request routing rather than in-memory dictionary data structures.

**Locality-sensitive hashing (LSH)**: hash functions that map similar items to the same bucket with high probability. LSH is an approximate nearest-neighbor technique that intentionally makes collisions for non-identical keys, which violates the dictionary ADT's key-uniqueness invariant.
