# Invariant Review Workshop

A priority queue backed by a binary heap passes unit tests. Yet a subtle heapify bug only manifests on arrays of size exactly $2^k - 1$ when the minimum element sits at the root. Random test sizes never hit the failure. The missing **representation invariant** would have caught it.

A **loop invariant** is a predicate $P$ such that (1) $P$ holds before the loop executes, (2) each iteration preserves $P$, and (3) $P$ together with the loop termination condition implies the desired postcondition. The invariant bridges informal implementation and rigorous proof (Hoare 1969; Dijkstra 1972).

<MathStatement id="invariant-def" kind="definition" title="Representation invariant">
Let $C$ be a concrete state space, $A$ an abstract value space, and $abs : C \to A$ an abstraction function. A *representation invariant* is a predicate $RI \subseteq C$ such that for every operation $op : C \to C$ corresponding to abstract operation $op_A : A \to A$,
$$
\forall c \in C.\; RI(c) \;\Longrightarrow\; RI(op(c)) \wedge abs(op(c)) = op_A(abs(c))
$$
whenever $op$ is called in a state satisfying its precondition.
</MathStatement>

An invariant without an abstraction function may prove pointer consistency but not user-visible correctness. A hash table can maintain valid buckets and count while returning the wrong value if equality or hash semantics are inconsistent with $abs$. A queue can maintain valid indices while returning newest rather than oldest data if $abs$ reverses the order.

<MathProof id="invariant-preservation-proof" for="invariant-def" title="Preservation reasoning" collapsible>
The proof obligation for each operation $op$ decomposes into:
1. **Precondition check**: the caller state $c$ satisfies $pre_{op}$.
2. **Initialisation**: $RI(c_{init})$ holds after the constructor.
3. **Preservation**: for every atomic step of $op$, if $RI$ holds before the step, it holds after.
4. **Postcondition**: $abs(c') = op_A(abs(c))$, i.e. the concrete change matches the abstract specification.

The preservation proof uses *local-change reasoning*: identify the fragment of the representation that the step can invalidate, prove frame conditions for every other fragment, then repair the broken fragment. This is the method of *data structure refinement* (Liskov & Guttag 2000).
</MathProof>

### Application to the heap example

For the binary heap whose heapify failed on $2^k-1$ sizes:

| Step | Concrete entry |
|---|---|
| RI clause (shape) | Array encodes a complete tree: $last \ge 0$, children of $i$ at $2i+1$, $2i+2$, all references non-null |
| RI clause (order) | $keys[parent(i)] \le keys[i]$ for every $i > 0$ |
| RI clause (metadata) | $size$ equals number of live elements, $capacity \ge size$ |
| Missing clause | *Heapify assumes the sub-heap property: the caller must supply a tree where only the root violates the order invariant.* The $2^k-1$ bug occurs when the root is already minimal: the sift-down loop terminates immediately, skipping the fix-up that later insertions depend on. |

The invariant was too weak: it described the final state but not the *intermediate assumption* of the heapify contract.

## Worked Example: Dynamic Array Append

A dynamic array stores elements in a contiguous buffer with logical length $n$ and capacity $C$.

**RI:**

1. $0 \le n \le C$.
2. The buffer $buf$ is a valid allocation of length $C$.
3. Abstract elements occupy exactly $buf[0..n-1]$.
4. No client-held alias permits mutation of $buf$.
5. Capacity policy: if $n < C$, the buffer is the original allocation; if $n = C$, the next append will trigger resize.
6. Integer arithmetic: all length and capacity computations do not overflow.

**AF:** $abs(buf, n, C) = \langle buf[0], buf[1], \dots, buf[n-1] \rangle$.

**Append preservation proof:**

The append operation branches on capacity:

- **Case spare capacity** ($n < C$):
  *Mutation*: write $x$ to $buf[n]$, increment $n$.
  *Frame*: $buf[0..n-1]$ unchanged, $C$ unchanged, allocation unchanged.
  *Preservation*: $0 \le n+1 \le C$ holds because $n < C$; the new element occupies $buf[n]$ by construction; all prior elements unchanged. RI satisfied.

- **Case full** ($n = C$):
  *Mutation*: allocate $buf'$ of size $C' > C$ (e.g. $2C$), copy $buf[0..n-1]$ to $buf'$, write $x$ to $buf'[n]$, publish $buf'$, $C \gets C'$, increment $n$.
  *Safety ordering*: allocate and copy *before* publishing $buf'$; if allocation fails, the original state is untouched (strong exception safety, Meyer 1997).
  *Frame* after publish: $0 \le n+1 \le C'$; all elements correctly positioned; old $buf$ becomes unreachable (or released). RI satisfied.

**Counterexample (what goes wrong):**

```java
// Java: no RI check for integer overflow
int newCapacity = oldCapacity * 2;  // overflow for oldCapacity > 2^30
```

The abstract sequence silently truncates or wraps. A correct RI must include $C \le \texttt{Integer.MAX\_VALUE} / 2$ or use overflow-safe arithmetic.

## Counterexample: BST Local vs Global Order

A binary search tree must satisfy an *inherited bound* invariant: every node's key lies within the interval inherited from its ancestors. Checking only local parent-child comparisons is insufficient.

<StructureDiagram kind="invalid-tree" title="Ancestor bound violation" caption="Each local parent-child relation is correct, but the node 1 in the right subtree of 8 violates the global ordering." />

The tree above satisfies `left.key < parent.key < right.key` at every node. Yet value 1 appears in the right subtree of the root (8), which requires all right-subtree keys to be $> 8$.

> **Warning.** The local-only check is necessary but not sufficient. A common failure mode is an insertion algorithm that walks down the tree comparing only the immediate parent, placing a node in the wrong subtree after a rotation or deletion corrupts the ancestor bound.

<MathStatement id="bst-invariant" kind="definition" title="BST inherited-bound invariant">
Let $T$ be a binary tree whose nodes carry keys from a totally ordered set $(K, <)$. Define for each node $v$:
$$
lo(v) = \max\{ key(a) \mid a \text{ ancestor of } v,\; v \in a.\text{right} \} \cup \{-\infty\}
$$
$$
hi(v) = \min\{ key(a) \mid a \text{ ancestor of } v,\; v \in a.\text{left} \} \cup \{+\infty\}
$$
Then $T$ is a BST iff for every node $v$, $lo(v) < key(v) < hi(v)$.
</MathStatement>

## Testing Invariants

Run debug validators after every mutation in randomised tests. Generate operations from a model and compare abstract results. Inject boundary values: empty, singleton, full, duplicate, maximal key, collision, root deletion, rotation, resize, and rollback.

<CodeRunner client:load
  title="Heap validator"
  language="python"
  code={`def check_heap(arr, n):
    if n < 0 or n > len(arr):
        raise RuntimeError(f"size {n} out of bounds for capacity {len(arr)}")
    for i in range(n):
        left, right = 2*i + 1, 2*i + 2
        if left < n and arr[left] < arr[i]:
            raise RuntimeError(f"heap violation at parent {i}")
        if right < n and arr[right] < arr[i]:
            raise RuntimeError(f"heap violation at parent {i}")
    return True

# Minimal test
check_heap([1, 3, 5, 7, 9], 5)
try:
    check_heap([1, 5, 3, 7, 9], 5)
except RuntimeError as e:
    print(e)`}
  output="heap violation at parent 1"
/>

Property-based tests should shrink failing sequences to a minimal counterexample. A failure after hundreds of operations often reveals the first invariant violation only if validators run incrementally. Record seeds and operation traces. Performance benchmarks are not correctness tests.

## Repairing a Broken Invariant

<CaseStudy client:load title="Repairing a failed mutation" scenario="A linked-list deletion leaves the predecessor pointing to the removed node." question="What should the repair inspect first?" options={["Reachability and reciprocal links", "Only the list's class name", "The CPU cache size", "The final output formatting"]} correctIndex={0} explanation="The broken representation must be diagnosed against the exact link and ownership clauses of the invariant." />

When a validator fails, stop at the first operation that made it fail. Classify the failure: lost element, duplicate element, wrong order, stale metadata, broken reachability, illegal alias, or resource-policy violation. Reconstruct the smallest concrete state and compare it with the operation's precondition. Repair the implementation and the proof together; changing code without updating the invariant can leave the same hidden gap under a different branch.

## Ownership and Error Atomicity

If a mutation can fail, specify whether it is atomic. An array resize that allocates a new buffer can construct and copy before publishing the new pointer, preserving the old state on allocation failure. A linked insertion should allocate and initialise a node before changing links. A hash-table resize should retain the old table until every entry is placed successfully. These ordering rules are part of the safety proof and prevent half-applied operations. Separating allocation from structural mutation is the *commit-after-construction* pattern (Meyer 1997).

### Recall

1. Write a complete RI and AF for a dynamic array.

   <ExerciseSolution hint="Cover length, capacity, buffer ownership, element occupancy, and the resize contract.">

   RI: $0 \le n \le C$, valid allocation of size $C$, elements exactly in slots $[0,n)$, no client alias can mutate storage, capacity policy $C_{\text{new}} > C_{\text{old}}$ with overflow guard. AF: the sequence of those $n$ values.
   </ExerciseSolution>

2. List the three clauses of a loop invariant.

   <ExerciseSolution>
   (1) $P$ holds before the loop. (2) Each iteration preserves $P$. (3) $P \wedge \neg \text{(loop condition)}$ implies the postcondition.
   </ExerciseSolution>

### Application

3. Review a doubly linked list deletion and enumerate every branch.

   <ExerciseSolution hint="Consider empty, singleton, head, tail, and interior removal.">

   Empty, singleton, head, tail, and interior cases differ without sentinels. Reconnect `pred.next` to `succ` and `succ.prev` to `pred`; update head/tail on endpoint removal; decrement size; detach node; verify reachability and reciprocal links.
   </ExerciseSolution>

4. State a hash-table invariant including tombstones.

   <ExerciseSolution>
   Each live key-value pair occurs exactly once, stored at its current hash location or reachable via probe sequence; $size$ counts live entries; empty ($\emptyset$) and tombstone ($T$) markers have distinct semantics; resizing eliminates or rebuilds tombstones.
   </ExerciseSolution>

### Transfer

5. Design an RI for a **lock-free linked stack** (Treiber stack). What clauses change because operations are concurrent without mutual exclusion?

   <ExerciseSolution hint="Consider the ABA problem and memory reclamation.">

   RI must include: (i) the top pointer points to a node or null; (ii) each node's `next` chain is acyclic and terminates at null; (iii) no node is reachable from top after it has been popped by any thread. The ABA problem means a CAS-based push/pop cannot rely on pointer identity alone---hazard pointers or RCU are needed. The RI must capture that a node is *logically removed* before its memory is reclaimed.
   </ExerciseSolution>

6. A persistent binary tree uses path copying. Give the RI clause that distinguishes shared subtrees from copied ones, and prove that a query on an old root sees exactly the old values.

   <ExerciseSolution>
   Let $V$ be the set of versions, $T_v$ the root of version $v$. RI: for every version $v$, the subgraph reachable from $T_v$ is a valid BST; if a node is reachable from two distinct roots $T_u$ and $T_w$, it must be identical in both (no divergent mutation through sharing). Preservation: any write creates a new node on the path from root to leaf; all other nodes are shared by reference, and queries read these shared nodes without mutation. Therefore $abs(T_v)$ remains the same snapshot after writes to unrelated versions.
   </ExerciseSolution>

### Synthesis

7. A red-black tree implementation passes all functional tests but occasionally produces a tree with two adjacent red nodes after a deletion. The deletion code uses the standard case analysis. Identify the most likely missing invariant clause and the rotation-colour-flip branch that must be fixed.

   <ExerciseSolution>
   Missing: the *no-consecutive-reds* invariant (property 4 of red-black trees). The most likely gap is the "double-black" propagation case: when a black node is removed and its replacement is also black, the algorithm must push the double-black status upward. If the double-black case analysis misses the sibling-with-red-nephew subcase, the colour fix-up terminates early, leaving two reds adjacent at a lower depth.
   </ExerciseSolution>

8. Design a differential test for persistent data-structure correctness. The test must detect accidental mutation through sharing.

   <ExerciseSolution>
   Generate a version DAG; keep random old roots; apply updates to arbitrary allowed bases; compare every query with a model that copies the entire abstract value for each version. After each update, query old roots: if any returns a different result than the model, a mutation-through-sharing bug is present. Also test concurrent reads: fork a reader on an old root while a writer mutates a descendant; the reader must see the pre-write snapshot.
   </ExerciseSolution>

## Deliberately Omitted

- **Separation logic and full Hoare logic for heap-manipulating programs:** formal program logics that extend the invariant method with frame rules, ownership, and modular aliasing. These are a full verification methodology rather than a manual review procedure.
- **Abstract interpretation and shape analysis:** static analysis tools (TVLA, Predator) that infer invariants automatically. Useful in practice but depend on the abstract-domain framework rather than manual invariant construction.
- **Concurrent invariants and linearizability proofs:** invariants for thread-shared data structures with linearisation points and memory ordering. The concurrency dimension changes the review procedure fundamentally and requires a separate treatment.
- **Dependent-type encoding in proof assistants:** encoding the RI as a type in Coq, Agda, or Lean so type-checking guarantees preservation. Applicable only when the data structure is implemented in the proof assistant's language.
