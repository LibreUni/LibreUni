# Persistent and Immutable Data Structures

A mutable structure has one current state; an update destroys or changes the previous representation. A persistent structure preserves access to prior versions while producing a new version. Immutability simplifies reasoning, supports undo and snapshots, and makes concurrent readers safer, but it can increase allocation and memory traffic. Structural sharing lets versions reuse unchanged substructure rather than copying the entire value.

## Versions and semantics

An immutable sequence operation does not change its input. A set operation on version v returns a new version v' such that v' differs at i while v remains observable and unchanged. A version is a root handle into a representation graph. The abstract correctness condition is functional: AF(v') is the specified result and AF(v) remains the old value for every previously returned version.

Persistence can be partial or full. In partial persistence, updates are allowed only on the newest version but all versions can be queried. In full persistence, any version can be updated to produce a new branch. Confluent persistence permits merging versions, a harder model because two histories must be combined. State which model an API offers; a version pointer alone does not define update legality.

Aliasing is intentional inside a persistent implementation but invisible at the abstract level. If two versions share a node and neither operation mutates nodes, sharing is safe. A single accidental in-place write corrupts every version that reaches that node. Use private constructors, immutable fields, copy-on-write checks, or a type system to enforce the no-mutation invariant.

## Path copying in trees

A persistent binary search tree can implement insertion by copying nodes along the root-to-insertion path and reusing untouched subtrees. If old root r has left child a and a new key belongs in the right subtree, create a new r' with the same key and left pointer a, and a newly constructed right path. The old root and all old nodes remain unchanged.

**Worked update.** Let version v contain root 8, left subtree keys 3,5, and right subtree 10. To insert 4, copy 8, then copy 3, then create a leaf 4 under 3’s right; reuse 8’s right subtree and 3’s left subtree. The new version has O(log n) new nodes if the tree is balanced, while v still yields the original sorted set. The abstraction proof follows the copied path and unchanged-subtree equality.

Deletion uses the same path-copying idea, including successor replacement. Balancing adds rotations: copy every node whose child pointer changes, and ensure no new version mutates an old node. A rotation can share subtrees that remain in the same position, but parent pointers complicate sharing because a shared child cannot have two mutable parent fields. Functional trees usually omit parent pointers or represent navigation through recursion.

## Persistent sequences

A singly linked list is naturally persistent: prepending x creates a new node pointing to the old head. Both versions share the old tail. Access remains O(i), and append requires copying the path or using a different tree representation. A persistent stack is therefore exceptionally simple; a persistent queue can use two immutable lists and lazy reversal, with care about evaluation and amortization.

For random access, a tree of fixed branching factor stores a sequence in leaves and copies one root-to-leaf path per update. A bitmapped vector trie uses chunks and path copying to provide near-constant or logarithmic access with sharing. Updating the branching factor trades depth against node size and allocation. The representation invariant must describe logical length, child occupancy, and index mapping.

## Persistent maps and hash tries

A persistent map can use a balanced tree, a trie, or a hash-array-mapped trie. A trie shares prefixes of keys; a hash trie uses hash bits at successive levels and handles collisions with a node storing equal-hash entries. Updating a key copies the nodes along the path and reuses unrelated branches. Hash stability and collision semantics remain part of correctness.

Structural sharing can reduce total memory from O(number of versions times n) to O(n+updates times depth) when updates are localized. It can also retain large subtrees longer than a mutable program would, because an old root keeps all reachable nodes alive. Garbage collection or reference counting reclaims a node only after no version reaches it. Lifetime analysis matters for snapshot-heavy workloads.

## Persistence and concurrency

Immutable versions can be read concurrently without locks because no reader observes fields changing. Publishing a new root still needs a memory-order guarantee: readers must see a fully initialized reachable graph, not a partially constructed node. Atomic root replacement provides a simple linearization point for publishing a version. Garbage reclamation remains necessary in manual memory systems if old roots can disappear while readers hold them.

Snapshots provide consistent reads if a reader holds one root for the duration of a query. A mutable structure that is copied lazily must define which updates a snapshot sees and whether iteration is isolated. Version identifiers can support optimistic concurrency: a writer creates a new version only if the expected base version still matches, otherwise it reports a conflict.

## Correctness and cost

Prove persistence with two claims: old versions are observationally unchanged, and the new version has the requested abstract value. A structural induction over shared subtrees is useful. Cost analysis counts new nodes, pointer reads, traversal comparisons, and reclamation, not just abstract operations. Path copying in a height-h tree costs O(h) time and space per update, while queries cost O(h); a balanced tree yields O(log n).

Persistence changes cache behavior and allocator load. Sharing can improve memory footprint but scattered nodes can harm locality. Batch updates may benefit from transient mutable builders that are frozen before publication; the freeze boundary must prevent later mutation through aliases. A fast implementation that violates the version contract is not a persistent structure.

## Lazy evaluation and amortization

A persistent queue built from front and rear lists can delay reversing the rear until the front is empty. If reversal is represented as a lazy computation, a single forced operation may perform linear work. An amortized proof must account for when that work is forced and ensure a computation is not forced repeatedly for many versions. Memoization can make a forced result shareable, but it adds storage and synchronization considerations.

Persistence can make amortization more subtle because old versions preserve old potential. If two branches share a delayed computation and both force it independently, the total work may differ from a mutable one-version analysis. Define whether thunks are memoized, whether forcing mutates the thunk, and which version owns the potential. A proof that ignores branch sharing can undercount work.

## Version algebra

Versions form a history graph rather than a single timeline. A query should be pure with respect to the version it receives; an update should return a new root and leave its base root valid. For full persistence, updating an old root creates a branch without changing sibling branches. Testing can generate a version DAG, retain random roots, perform updates, and compare every root against a simple copied model. This directly tests the central guarantee that historical observations remain stable.

## Failure and serialization

If versions are persisted to disk or sent between processes, pointers cannot be serialized as raw addresses. Use stable node IDs, content hashes, or a traversal that writes shared nodes once and references them by offset. Recovery must preserve sharing or at least the abstract value, and cyclic representations require cycle-aware encoding. A partially written version should not become the published root; use a commit record or atomic rename boundary.

## Exercises

1. Specify partial persistence for a stack and prove old versions remain unchanged.
2. Trace path-copying insertion in a BST and list shared nodes.
3. Explain why mutable parent pointers complicate structural sharing.
4. Analyze persistent linked-list prepend time and space.
5. Design version publication for concurrent readers.
6. Prove a balanced path-copying update uses O(log n) new nodes.
7. Identify a memory-retention problem caused by long-lived roots.

### Diagnostic solutions


1. Each push creates a new immutable node pointing to the prior root; pop returns the prior node and its next pointer. The old root is never written, so every earlier version’s traversal remains identical. New version size differs by one and operations are O(1).
2. Copy each node on the search path from root to insertion, create the new leaf, and point copied nodes to copied child paths or shared untouched children. Shared nodes are exactly subtrees not on the path; no shared node may be mutated.
3. A node shared by two versions cannot store one mutable parent pointer that is correct for both paths. Omitting parent pointers or copying every parent context avoids the conflict; otherwise the parent field leaks version identity and breaks old traversals.
4. Prepend is O(1) time and one new node; the old list is shared as the new node’s tail. Access remains O(i), and appending requires copying all nodes or a tree-based representation.
5. Fully construct an immutable graph, then atomically publish its root with release semantics. Readers acquire the root once and traverse immutable fields. Retire old roots only after a garbage collector, epoch, hazard pointer, or reference-count protocol proves no reader can reach them.
6. A balanced tree has height O(log n). Path copying creates one new node per level plus a leaf or replacement, so O(log n) nodes and pointer updates. Unchanged subtrees are shared; the old root and nodes remain valid.
7. A long-lived snapshot root keeps every subtree reachable from that version alive, including large data no current query needs. Limit snapshot lifetime, compact versions, use content-addressed sharing carefully, or release roots through a documented ownership protocol.
