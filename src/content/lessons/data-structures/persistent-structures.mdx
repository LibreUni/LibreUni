# Persistent and Immutable Data Structures

Git stores every version of every file without using $1000\times$ the space of a single checkout. A text editor supports unbounded undo. A database provides snapshot isolation under concurrent writers. These systems rely on **persistent data structures**: structures that preserve prior versions when modified.

The foundational treatment is Driscoll, Sarnak, Sleator, and Tarjan (1989), "Making Data Structures Persistent." The core mechanism—**path copying**—creates a new root-to-leaf path for each update while sharing every unchanged subtree. Each version costs $O(\log n)$ space and time under balanced trees, not $O(n)$ for a full copy. This technique underpins functional programming collections, version-control internals, and database MVCC.

Requires: understanding of binary search trees and asymptotic analysis.

<MathStatement id="persistence-def" kind="definition" title="Persistence">
A data structure is **persistent** if every update produces a new distinguishable version and all prior versions remain observable and unchanged. An **ephemeral** structure discards prior versions on update. **Partial persistence** permits queries on any version but updates only on the newest. **Full persistence** permits updates on any version, creating a branching history graph. **Confluent persistence** additionally permits merging two versions.
</MathStatement>

<StructureDiagram kind="versions" title="Persistent version roots" caption="Each update publishes a new root while old roots remain valid and readable." />

| Model | Old versions queryable | Old versions updatable | History shape |
| --- | --- | --- | --- |
| Ephemeral | no | — | single state (discarded) |
| Partial persistence | yes | no | linear timeline |
| Full persistence | yes | yes | tree (branching) |
| Confluent persistence | yes | yes (mergeable) | DAG |

## Path copying in binary search trees

<MathStatement id="path-copying" kind="proposition" title="Path copying correctness">
Let $T_v$ be a balanced BST with root $r_v$. An insertion of key $k$ into $T_v$ produces a new version $T_{v'}$ by copying every node on the root-to-insertion-path, reusing all other subtrees by reference. Then:
1. $T_v$ is observationally unchanged—every query on $r_v$ returns the original set.
2. $T_{v'}$ is a valid BST containing $\text{set}(T_v) \cup \{k\}$.
3. The update allocates $O(\log n)$ new nodes and completes in $O(\log n)$ time.
</MathStatement>

<MathProof id="path-copying-proof" for="path-copying" collapsible>
Proceed by induction on tree height. The base case (empty tree) creates one node. For the inductive step, let $r_v$ store key $x$ and have left child $L$, right child $R$. If $k < x$, recurse into $L$: construct new left child $L'$ by path copying, then allocate $r_{v'}$ with key $x$, left pointer to $L'$, and right pointer to $R$ (shared). The case $k > x$ is symmetric. Since the tree is balanced, recursion depth is $O(\log n)$. Exactly one node per level is allocated. The old root $r_v$ retains its original pointers, so queries on $r_v$ traverse the unchanged $L$ and $R$. Thus set equality holds for both versions.
</MathProof>

### Worked example: BST insertion

Let $v$ contain root **8**, left subtree $\{3,5\}$, right subtree $\{10\}$. Insert **4**:

1. Copy **8** (new node, same key).
2. $4 < 8$, so recurse left. Copy **3**.
3. $4 > 3$, so recurse right. Create leaf **4** under copied **3**.
4. Point copied **3**'s left to existing **5** (shared). Point copied **8**'s right to existing **10** (shared).

Result: $v'$ has three new nodes (8, 3, 4). $v$ still yields $\{3,5,8,10\}$; $v'$ yields $\{3,4,5,8,10\}$. Shared subtrees $\{5\}$ and $\{10\}$ occupy exactly one copy in memory.

### Worked example: Deletion with successor replacement

Delete **3** from the same tree. Copy **8**, then copy its left path. Node **3** has right child **5** (leaf), so replace **3** with **5** in the copied path. No additional allocation for **5**—it is shared. The old version $v$ retains **3** and its original linkage. Deletion copies $O(\log n)$ nodes; balancing rotations (if needed) copy every node whose child pointers change.

## Persistent sequences

A singly linked list is naturally persistent: prepending $x$ creates a new node pointing to the old head. Both versions share the tail. Access remains $O(i)$, and append requires a tree representation or path copying through a finger.

<StructureDiagram kind="list" title="Persistent sequence update" caption="Prepending copies one node; the tail is shared between both versions." />

A persistent queue can be built from two immutable lists (front and rear) with lazy reversal. When the front empties, the rear list is reversed to become the new front. The reversal is a delayed thunk to amortize cost. Under persistence, the same thunk may be forced in multiple branches, so memoization prevents duplicate work at the cost of additional storage.

<MathStatement id="persistent-queue" kind="definition" title="Persistent queue (lazy, amortized)">
A persistent queue is a pair $(F, R)$ of immutable lists where $F$ is the front (head for dequeue) and $R$ is the rear (head for enqueue). Enqueue $x$ yields $(F, x :: R)$. Dequeue on non-empty $F$ yields $(\text{tail}(F), R)$. When $F$ is empty, the representation is rebuilt as $(\text{reverse } R, [])$, where the reversal is a delayed lazy computation. Each operation runs in $O(1)$ amortized time when shared forcing is accounted with memoization.
</MathStatement>

## Counterexample: Shared-node mutation

Suppose two versions $v_1$ and $v_2$ share node $n$ via path copying. An in-place write to $n$—for example, mutating `n.left` or `n.value`—immediately corrupts every version whose representation graph reaches $n$. The abstract value of $v_1$ changes even though the client never updated $v_1$. This is not permitted by the persistence definition.

<StructureDiagram kind="tree" title="Corrupted sharing" variant="invalid-tree" />

Consider this violation in Python:

```python
class EphemeralNode:
    def __init__(self, key):
        self.key = key
        self.left = None
        self.right = None

# Pretend to be persistent, but mutate in place
root = EphemeralNode(8)
root.left = EphemeralNode(3)
snapshot = root  # "version" v0
root.key = 99    # oops — snapshot.key is now 99
```

The `snapshot` root no longer represents the original set. This is **not persistence**—it is mutable state with aliasing. Real persistent implementations enforce immutability through private constructors, frozen fields, or a type system (e.g., `@dataclass(frozen=True)` in Python, `final` fields in Java, or persistent collections in Clojure/ML).

## Correctness and cost analysis

<MathStatement id="correctness-claim" kind="theorem" title="Persistence correctness">
For any sequence of updates applied to a persistent structure via path copying, every prior version returned to the client remains observationally equivalent to its state at the moment of creation. Formally, if $v$ is a version returned by operation $op$ at logical time $t$, and $q$ is any pure query, then $q(v) = q(v)$ for all logical times > $t$.
</MathStatement>

Proof follows by structural induction on the sharing graph: new nodes are allocated without mutating existing nodes, and every pointer in an old root points to a node whose fields have not been written since the root was created.

**Cost.** In a balanced BST, path copying uses $O(\log n)$ new nodes and $O(\log n)$ time per update. Queries remain $O(\log n)$. A persistent trie (bitmapped vector trie) with branching factor $b$ copies $O(\log_b n)$ nodes per update. Unchanged subtrees are shared, reducing total memory across $m$ updates from $O(m \cdot n)$ to $O(n + m \cdot \log n)$.

**Subtlety: amortization under persistence.** Okasaki (1998) showed that amortized bounds for ephemeral structures do not automatically hold under persistence because old versions preserve old potential. A lazy thunk forced in one branch and later forced in another may perform the same work twice. Solutions include memoization (share the forced result), scheduling (real-time guarantees), or explicit accounting that charges each branch independently.

## Persistence and concurrency

<CaseStudy client:load
  title="Concurrent readers and one writer"

  question="Why can persistence simplify the concurrency design?"
  options={[
    "Readers observe a consistent snapshot without locks because old roots are immutable",
    "It eliminates all memory allocation costs",
    "Every query becomes constant time",
    "It forbids any form of sharing between versions"
  ]}
  correctIndex={0}
  explanation="Immutable old roots avoid in-place data races. Readers can traverse a snapshot without synchronization. The writer still needs atomic root publication (e.g., an atomic pointer swap) and a reclamation protocol (epoch-based reclamation, hazard pointers, or a GC)."
/>

Immutable versions can be read concurrently without locks. Publishing a new root requires an atomic store with release semantics so readers see a fully initialized graph. Reclamation must ensure no active reader holds a reference to a retired node. In garbage-collected languages (Java, Go, ML), the collector handles this automatically. In manual-memory languages (C++, Rust), epoch-based reclamation or hazard pointers are necessary.

## Exercises

### Recall and application

1. Trace path-copying insertion of key **6** into BST version with root **8**, left $\{3,5\}$, right $\{10\}$. List every copied node and every shared subtree.

<ExerciseSolution hint="The search path is 8 → 3 → 5 (right child). Copy each visited node and create the new leaf.">

Copied: **8** (new root), **3** (copied), and leaf **6**. Shared: **5** (left child of copied 3), right subtree **10** (right child of copied 8). Total: 3 new nodes.
</ExerciseSolution>

2. Explain why mutable parent pointers violate structural sharing.

<ExerciseSolution>
A node shared by two versions cannot store one parent pointer correct for both paths. If node **5** is shared by $v_1$ and $v_2$, its parent in $v_1$ is **3** but in $v_2$ could be a different copied node. A single `parent` field would be wrong for at least one version. Functional trees omit parent pointers or navigate via recursion paths in the call stack.
</ExerciseSolution>

3. Persistent linked-list prepend: analyze time and space relative to the number of versions.

<ExerciseSolution>
Prepend is $O(1)$ time and allocates one new node per call. The old list is shared as the new node's tail. Appending $x$ to the end of an $n$-element list requires copying all $n$ nodes (or using a different representation, such as a finger tree). Memory across $m$ prepend operations is $O(n + m)$ rather than $O(m \cdot n)$ for full copies.
</ExerciseSolution>

### Transfer and synthesis

4. Design a persistent hash map using a hash-array-mapped trie (HAMT) with branching factor 32. Each internal node stores a 32-element array of child pointers and a bitmap of occupied slots. Describe which nodes are copied on insertion and which are shared. How many new nodes does a single insertion allocate in expectation?

<ExerciseSolution>
A HAMT uses $h = \lceil \log_{32} N \rceil$ levels. Each insertion copies one node per level (the node whose child array changes). With a 32-way split and reasonable load, expected depth is $\log_{32} N \leq 5$ for $N \leq 33$ million. Expected new nodes per insertion: $h$ (about 2–5). All sibling branches at each level are shared. Collision nodes (for equal-hash entries) are copied when modified.
</ExerciseSolution>

5. A database uses persistent B-trees for MVCC snapshot isolation. Suppose a transaction $T_1$ holds a snapshot root and $T_2$ inserts a key, causing a node split. Which nodes must $T_2$ copy? Will the split affect $T_1$'s traversal? Explain.

<ExerciseSolution>
$T_2$ copies the split node and every node on the path from root to the split node (path copying in a B-tree). $T_1$'s root is unchanged; it traverses old nodes that still point to the pre-split children. The split is invisible to $T_1$. This is exactly the persistent B-tree semantics used in systems like CouchDB and ZFS for consistent snapshots.
</ExerciseSolution>

6. Prove that a persistent queue using two lists and lazy reversal satisfies the queue ADT under persistence with memoized thunks. What happens if the thunk is not memoized?

<ExerciseSolution hint="Consider two branches that both force the same reversal thunk. Without memoization, each branch pays $O(n)$ for the reversal. The total work becomes $O(b \cdot n)$ for $b$ branches, making the amortized analysis invalid. With memoization (the thunk caches its result), the first branch pays $O(n)$ and all subsequent branches pay $O(1)$ to read the cached reversed list. Show the potential function must account for shared thunks.">
The queue representation invariant: $F$ contains front elements in order, $R$ contains rear elements in reverse insertion order. When $F$ is empty, the thunk $\text{reverse } R$ is forced, producing a new $F'$ of length $|R|$. With memoization, this result is cached. A second branch that triggers reversal on the same $(F, R)$ state reuses the cached $F'$ without recomputation. The amortized cost per operation remains $O(1)$ even under full persistence when all thunks are memoized. The proof extends Okasaki's bank account method by charging each thunk once and distributing the cost across the operations that create it.
</ExerciseSolution>

## Deliberately omitted

- **Fat-node and node-splitting (Driscoll–Sarnak–Sleator–Tarjan general transformation).** The fat-node method stores multiple field-version pairs per node and uses an explicit version stamp to select the correct field. It makes any pointer-based structure partially persistent but is rarely competitive with path copying in practice and adds version-stamp overhead on every pointer dereference.
- **Confluent persistence and version merging.** Merging two versions produces a DAG version history. The shared-structure accounting changes significantly because the version graph is no longer a tree. Analysis is complex and specialized; the CLRS coverage is the standard reference.
- **Persistent graphs with cycles.** Path copying cannot clone a simple cycle. Techniques such as indirection nodes, hash-consing, or explicit cycle breaking are needed. These are well covered in Peyton Jones (1987) *Implementation of Functional Programming Languages*.
- **Persistent structures in lazy languages (Okasaki's scheduling framework).** A full treatment of real-time queues, implicit queues, and the scheduling approach to worst-case bounds. Okasaki (1998) is the authoritative source; this lesson assumes amortized analysis is sufficient for the conceptual introduction.
- **Transactional memory and STM.** While persistent structures are used in STM implementations, the transaction model (conflict detection, retry, rollback) is independent of the data-structure persistence model and belongs in a concurrency course.
