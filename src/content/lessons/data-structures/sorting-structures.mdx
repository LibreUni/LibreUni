# Sorting Structures and Selection

Sorting is an algorithmic task, but data representation determines how comparisons, moves, memory, and stability behave. A sorting routine receives records and a comparator, then returns an ordered sequence under a total preorder or total order. The data structure may be an array, linked list, external runs, or an index over immutable records. A complete specification must define equal-key behavior, ownership, mutation, and whether the result is stable.

## Order and stability

A comparator should be transitive and consistent with equality for the algorithm’s assumptions. If comparator(a,b) and comparator(b,a) are both false, keys may be equivalent; a stable sort preserves their original relative order. Stability matters when sorting by a secondary key after a primary-key sort, or when equal records carry timestamps or provenance.

Sorting can rearrange records, rearrange pointers, or produce an array of indices. Moving large records is expensive; sorting compact references can reduce bytes moved but adds indirection on later access. If records are immutable and shared, an index sort preserves ownership. The representation choice must say whether input is mutated and whether output aliases input storage.

## Comparison lower bound

Any comparison sort must distinguish among n! possible input permutations. A deterministic comparison algorithm can be represented by a decision tree whose leaves correspond to possible output orders. A binary comparison gives at most two branches, so a tree with at least n! leaves has height at least (lceillog_2(n!)rceil=Omega(nlog n)). This lower bound applies to worst-case comparisons, not to integer-key algorithms that inspect bits or exploit a restricted domain.

Stability does not change the asymptotic comparison lower bound, but it constrains which equal-key permutations are legal. A comparator that is not transitive can make “sorted” undefined: a cycle a less than b less than c less than a has no sequence satisfying all pairwise demands. Validate comparator contracts in tests or document that behavior is unspecified.

## Merge and linked structures

Mergesort divides a sequence, recursively sorts halves, and merges two sorted sequences. The merge invariant is that the output prefix contains the smallest remaining elements from the two input fronts, in stable order when ties choose the left input first. The recurrence (T(n)=2T(n/2)+O(n)) gives O(n log n) time. Array merges use O(n) auxiliary space; linked-list merges can relink nodes with O(1) extra pointer storage.

For linked lists, split using slow/fast pointers or a counted midpoint, recursively sort, and splice runs. Ensure the split terminates for size 0 and 1; a bad midpoint can recurse forever. During merge, update exactly one next pointer at a time and terminate the final node with null. A cycle introduced by a pointer error may make later traversal nonterminating even if order tests pass on short prefixes.

## Quicksort and partition structures

Quicksort partitions around a pivot, recursively sorts subarrays, and can use O(log n) expected stack depth with randomized pivots. A partition invariant describes regions less than, equal to, and greater than the pivot. Two-way partition performs poorly on many duplicates; three-way partition groups equal keys and can reduce work dramatically.

Worst-case time is O(n^2) for adversarial pivots. Randomized pivot selection gives expected O(n log n) comparisons under an appropriate randomness model, but an attacker who observes the seed or a deterministic pivot policy can still force bad inputs. Introsort switches to heapsort after excessive recursion depth, combining average quicksort performance with worst-case O(n log n).

## Heapsort and selection

Heapsort provides worst-case O(n log n) comparisons and O(1) auxiliary array space under ordinary in-place representation, but is not stable. A heap’s order invariant ensures the maximum or minimum is at the root; repeated extraction establishes a sorted suffix.

Selection asks for the kth smallest without fully sorting. Quickselect partitions and recurses into one side; randomized expected time is O(n), while worst case is O(n^2). The median-of-medians pivot gives deterministic linear worst-case time by guaranteeing a constant fraction discarded each round. Its proof groups elements, finds group medians, recursively selects their median, and counts elements on both sides.

**Worked selection reasoning.** Group n elements into groups of five, sort each group, and take its median. At least half the group medians are at least the median of medians; each such group contributes at least three elements at least as large, yielding a constant lower bound on discarded elements. The recurrence (T(n)le T(n/5)+T(7n/10)+O(n)) is linear because the recursive fractions sum below one.

## Integer and domain-specific sorting

If keys are integers in a bounded range, counting sort can run in O(n+k) time and space by counting occurrences, then prefix sums, and outputting stable positions. Radix sort applies a stable digit sort repeatedly; with d digits and base b, cost O(d(n+b)). These algorithms avoid the comparison lower bound by inspecting representations. They require key-domain and digit-cost assumptions, and large sparse ranges can make counting arrays impractical.

Bucket sort distributes values under a distribution assumption and can have expected linear time, but worst-case concentration makes a bucket expensive. State whether the guarantee is expected under a distribution or deterministic under bounded keys. Sorting algorithms are often chosen by key width, range, stability, memory, and input distribution rather than comparison count alone.

## External sorting

When records do not fit in memory, external mergesort creates sorted runs that fit in memory, writes them, then performs multiway merges. The main cost is block I/O, not comparisons. A k-way merge uses a priority queue of one head per run and reads sequential blocks. Buffer sizing and run generation determine I/O passes; a cache-aware design should minimize random disk operations.

Sorting a linked in-memory structure may be less attractive than extracting records to a contiguous array because sequential scans and writes improve locality. Conversely, a database may sort record IDs and fetch payloads later. Separate logical order from physical record movement when analyzing storage systems.

## Verification and engineering

For every sorting implementation, test that the output is a permutation of the input and is ordered under the comparator. A sortedness-only test can miss lost or duplicated records; a permutation-only test can miss ordering. Stable tests attach original positions to equal keys and check that those positions remain increasing. Property-based tests should include empty arrays, singleton arrays, already sorted and reverse-sorted inputs, all-equal keys, repeated values, and sizes around partition or buffer boundaries.

Instrumentation can expose the true bottleneck: comparisons, moves, allocations, recursion depth, cache misses, and I/O operations. A comparison-efficient algorithm may move large records; a stable algorithm may require auxiliary storage; an in-place algorithm may have worse locality. Report the metric that matches the deployment constraint instead of presenting one asymptotic number as a complete performance claim.

## Selection under mutation

In-place selection rearranges elements, so callers must know whether the input is consumed or copied. If records are shared, returning an index or a reference can avoid moving payloads but requires an ownership contract. A selection routine that partially partitions an array before throwing or being canceled should document whether the partial mutation is observable. Strong exception safety requires building a result separately or providing rollback, which changes space and time costs.

### Diagnostic solutions


1. There are n! input permutations requiring distinct sorted-leaf outcomes. A binary comparison tree of height h has at most 2^h leaves, so (2^hge n!), giving hu2265log2(n!)=u03a9(n log n).

2. At each merge step, output the smaller front; if equal, choose the left run first. The output prefix is sorted and contains exactly the smallest consumed elements, while each remaining run stays sorted. When one run empties, append the other unchanged.

3. Two-way partition may produce subproblems n-1 and 0 repeatedly, yielding quadratic work. Three-way partition places all equal keys in the middle, so equal inputs finish after one linear partition and recursion on empty sides.

4. Randomized quickselect has expected linear time over pivot choices but quadratic worst-case branches. Median-of-medians guarantees a constant fraction eliminated each round and gives deterministic O(n), with larger constants.

5. Sorting groups and recursively finding their median costs O(n)+T(n/5). The chosen pivot leaves at most about 7n/10 elements on the larger side, so (T(n)le T(n/5)+T(7n/10)+O(n)=O(n)).

6. Comparison sorts require only an order comparator and have u03a9(n log n) comparison lower bound. Counting requires a manageable integer range; radix requires fixed digit representation and stable passes; bucket uses distribution assumptions. Report memory and stability costs too.

7. Generate in-memory sorted runs, write them sequentially, then use a min-heap over run heads while buffering each run. Each pass reads and writes all records sequentially; the bottleneck is block transfers and number of merge passes, not the O(n log n) comparison count.
