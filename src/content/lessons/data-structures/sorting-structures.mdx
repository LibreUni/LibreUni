# Sorting Structures and Selection

**Prerequisites:** Asymptotic notation ($O$, $\Omega$, $\Theta$), recursion and recurrences, arrays and linked lists, basic sorting concepts.

---

You need the top 20 best-selling products from a catalog of 10 million. Sorting all 10 million ($O(n \log n)$) is wasteful when only 20 are needed. A min-heap of size 20 turns selection into $O(n \log 20)$ — nearly 500$\times$ faster. Sorting is not always the right answer.

This lesson examines how data representation constrains sorting: **stability** (order of equal keys), **space usage** (in-place vs. auxiliary memory), **cache behavior** (contiguous vs. pointer-chasing access), and **external memory** (when data exceeds RAM). Understanding these constraints lets you choose the right strategy rather than reaching for a generic sort.

## Order and stability

<StructureDiagram kind="sort" title="Stable merge fronts" caption="A merge chooses from two sorted fronts and preserves the left-to-right order of equal keys." />

| Requirement | Meaning | Failure symptom |
| --- | --- | --- |
| total order | comparisons are transitive | "sorted" has no answer |
| stability | equal records retain order | pagination changes |
| ownership | mutation is specified | caller data changes |

<Quiz client:load question="What does stability constrain?" options={["The order of equal-key records", "The number of input records", "The comparator's memory address", "Whether arrays have indices"]} correctIndex={0} explanation="A stable sort preserves the relative order of records whose keys compare equal." />

A comparator must be transitive and consistent with equality. If `comparator(a,b)` and `comparator(b,a)` are both false, keys are equivalent; a stable sort preserves their original relative order. Stability matters when sorting by a secondary key after a primary-key sort, or when equal records carry timestamps or provenance.

Sorting can rearrange records, rearrange pointers, or produce an array of indices. Moving large records is expensive; sorting compact references reduces bytes moved but adds indirection on access. If records are immutable and shared, an index sort preserves ownership. The representation choice must specify whether input is mutated and whether output aliases input storage.

## Comparison lower bound

<MathStatement id="comparison-lower-bound" kind="theorem" title="Comparison sorting lower bound" proofId="comparison-lb-proof">
Every deterministic comparison-based sorting algorithm performs at least $\lceil \log_2(n!) \rceil = \Omega(n \log n)$ comparisons in the worst case to sort $n$ distinct keys.
</MathStatement>

<MathProof id="comparison-lb-proof" for="comparison-lower-bound" collapsible open>
Any deterministic comparison sort can be modeled as a binary decision tree. Each internal node represents a comparison between two keys; each leaf represents a permutation of the input that is the output order. There are $n!$ possible input permutations, and a correct algorithm must distinguish all of them, so the tree must have at least $n!$ leaves.

A binary tree of height $h$ has at most $2^h$ leaves. Therefore $2^h \ge n!$, giving $h \ge \lceil \log_2(n!) \rceil$. By Stirling's approximation:

$$
n! \sim \sqrt{2\pi n}\left(\frac{n}{e}\right)^n
\quad\Longrightarrow\quad
\log_2(n!) = n\log_2 n - n\log_2 e + O(\log n) = \Omega(n \log n)
$$
</MathProof>

The lower bound applies to worst-case comparisons, not to integer-key algorithms that inspect bits or exploit a restricted domain. Stability does not change the asymptotic comparison lower bound, but constrains which equal-key permutations are legal. A non-transitive comparator can make "sorted" undefined: a cycle $a < b < c < a$ has no sequence satisfying all pairwise demands. Validate comparator contracts in tests or document that behavior is unspecified.

Mergesort divides a sequence, recursively sorts halves, and merges two sorted sequences. The merge invariant: the output prefix contains the smallest remaining elements from the two input fronts, in stable order when ties choose the left input first. The recurrence $T(n) = 2T(n/2) + O(n)$ gives $O(n \log n)$ time. Array merges use $O(n)$ auxiliary space; linked-list merges relink nodes with $O(1)$ extra pointer storage.

For linked lists, split using slow/fast pointers or a counted midpoint, recursively sort, and splice runs. Ensure the split terminates for size 0 and 1; a bad midpoint can recurse forever. During merge, update exactly one next pointer at a time and terminate the final node with null. A cycle introduced by a pointer error may make later traversal nonterminating even if order tests pass on short prefixes.

## Quicksort and partition structures

<StructureDiagram kind="array" title="Partition state" caption="A partition separates regions before recursive work; each region has a precise meaning." variant="capacity" />

Quicksort partitions around a pivot, recursively sorts subarrays, and can use $O(\log n)$ expected stack depth with randomized pivots. A partition invariant describes regions less than, equal to, and greater than the pivot. Two-way partition performs poorly on many duplicates; three-way partition groups equal keys and can reduce work dramatically.

Worst-case time is $O(n^2)$ for adversarial pivots. Randomized pivot selection gives expected $O(n \log n)$ comparisons under an appropriate randomness model, but an attacker who observes the seed or a deterministic pivot policy can still force bad inputs. Introsort switches to heapsort after excessive recursion depth, combining average quicksort performance with worst-case $O(n\log n)$.

## Heapsort and selection

<StructureDiagram kind="heap" title="Heap order property" caption="Parent priority is no greater than children; the root is a global extremum." />

Heapsort provides worst-case $O(n\log n)$ comparisons and $O(1)$ auxiliary array space under ordinary in-place representation, but is not stable. A heap's order invariant ensures the maximum or minimum is at the root; repeated extraction establishes a sorted suffix.

Selection asks for the kth smallest without fully sorting. Quickselect partitions and recurses into one side; randomized expected time is $O(n)$, while worst case is $O(n^2)$. The median-of-medians pivot gives deterministic linear worst-case time by guaranteeing a constant fraction discarded each round.

<MathStatement id="median-of-medians" kind="theorem" title="Deterministic linear-time selection" proofId="mom-proof">
There exists a deterministic algorithm that selects the $k$th smallest element from $n$ elements in $O(n)$ worst-case time.
</MathStatement>

<MathProof id="mom-proof" for="median-of-medians" collapsible open>
Group the $n$ elements into $\lceil n/5 \rceil$ groups of five (the last group may have fewer). Sort each group and take its median. Recursively select the median of these $\approx n/5$ medians. Use this median-of-medians as the pivot.

At least half the group medians are $\ge$ the median-of-medians. Each such group contributes at least three elements $\ge$ the pivot (the median plus the two larger elements in the group). So at least $3 \cdot \lceil (1/2) \cdot \lceil n/5 \rceil \rceil \ge 3n/10$ elements are $\ge$ the pivot. By symmetry, at most $7n/10$ elements can lie on the larger side after partition.

The recurrence is $T(n) \le T(n/5) + T(7n/10) + O(n)$. Since $1/5 + 7/10 = 9/10 < 1$, the recurrence solves to $T(n) = O(n)$.
</MathProof>

  language="python"

def quickselect(A, k):
    if len(A) == 1:
        return A[0]
    pivot = random.choice(A)
    left  = [x for x in A if x < pivot]
    mid   = [x for x in A if x == pivot]
    right = [x for x in A if x > pivot]
    if k < len(left):
        return quickselect(left, k)
    elif k < len(left) + len(mid):
        return pivot
    else:
        return quickselect(right, k - len(left) - len(mid))

## Integer and domain-specific sorting

<Quiz client:load
  question="Why can counting sort beat the Ω(n log n) lower bound?"
  options={["It does not use comparisons at all", "It runs on a quantum computer", "It sorts only when n is small", "It uses a hash function"]}
  correctIndex={0}
  explanation="Counting sort uses occurrences as array indices rather than pairwise comparisons, so the comparison lower bound does not apply."
/>

If keys are integers in a bounded range, counting sort runs in $O(n+k)$ time and space by counting occurrences, computing prefix sums, and outputting stable positions. Radix sort applies a stable digit sort repeatedly; with $d$ digits and base $b$, cost is $O(d(n+b))$. These algorithms avoid the comparison lower bound by inspecting representations. They require key-domain and digit-cost assumptions, and large sparse ranges can make counting arrays impractical.

Bucket sort distributes values under a distribution assumption and can have expected linear time, but worst-case concentration makes a single bucket expensive. State whether the guarantee is expected under a distribution or deterministic under bounded keys. Sorting algorithms are often chosen by key width, range, stability, memory, and input distribution rather than comparison count alone.

When records do not fit in memory, external mergesort creates sorted runs that fit in memory, writes them to disk, then performs multiway merges. The main cost is block I/O, not comparisons. A $k$-way merge uses a priority queue of one head per run and reads sequential blocks. Buffer sizing and run generation determine I/O passes; a cache-aware design should minimize random disk operations.

Sorting a linked in-memory structure may be less attractive than extracting records to a contiguous array because sequential scans and writes improve locality. Conversely, a database may sort record IDs and fetch payloads later. Separate logical order from physical record movement when analyzing storage systems.

## Verification and engineering

<Quiz client:load
  question="What two properties must a sorting correctness test verify?"
  options={["Permutation and sortedness", "Memory and speed", "Input size and key range", "Stability and recursion depth"]}
  correctIndex={0}
  explanation="A correct sort must produce both a permutation of the input (no lost/duplicated elements) and a sequence ordered under the comparator. A sortedness-only test misses lost records."
/>

For every sorting implementation, test that the output is a permutation of the input and is ordered under the comparator. A sortedness-only test can miss lost or duplicated records; a permutation-only test can miss ordering. Stable tests attach original positions to equal keys and check that those positions remain increasing. Property-based tests should include empty arrays, singleton arrays, already sorted and reverse-sorted inputs, all-equal keys, repeated values, and sizes around partition or buffer boundaries.

Instrumentation can expose the true bottleneck: comparisons, moves, allocations, recursion depth, cache misses, and I/O operations. A comparison-efficient algorithm may move large records; a stable algorithm may require auxiliary storage; an in-place algorithm may have worse locality. Report the metric that matches the deployment constraint instead of presenting one asymptotic number as a complete performance claim.

## Counterexample and failure mode

A quicksort implementation using two-way Lomuto partition on an array of 100,000 identical keys degrades to $O(n^2)$ because each partition produces one element (the pivot) and $n-1$ remaining — the recursion depth becomes $n$, and total comparisons become $n(n-1)/2$. This is not a theoretical curiosity; it occurs in production when sorting database columns with low cardinality (e.g., a boolean flag or a small enum field). Three-way partition or introsort's fallback to heapsort prevents this.

A non-transitive comparator is a second failure mode. If `compare(a,b)` and `compare(b,c)` both return `-1` but `compare(a,c)` returns `1`, no arrangement satisfies all pairwise constraints. The sort may crash, loop indefinitely, or produce garbage output. This violates the total-order prerequisite implicitly assumed by every comparison sort.

## Selection under mutation

In-place selection rearranges elements, so callers must know whether the input is consumed or copied. If records are shared, returning an index or a reference can avoid moving payloads but requires an ownership contract. A selection routine that partially partitions an array before throwing or being canceled should document whether the partial mutation is observable. Strong exception safety requires building a result separately or providing rollback, which changes space and time costs.

### Recall and application

1. Prove the comparison sorting lower bound with a decision tree.

   <ExerciseSolution hint="A binary decision tree with height h has at most 2^h leaves. How many distinct output permutations must a correct comparison sort distinguish?">

   There are $n!$ input permutations requiring distinct sorted-leaf outcomes. A binary comparison tree of height $h$ has at most $2^h$ leaves, so $2^h \ge n!$, giving $h \ge \log_2(n!) = \Omega(n\log n)$.
   </ExerciseSolution>

2. State a stable merge invariant and prove merge correctness.

   <ExerciseSolution hint="When the two fronts compare equal, which one should be emitted to preserve original input order?">

   At each merge step, output the smaller front; if equal, choose the left run first. The output prefix is sorted and contains exactly the smallest consumed elements, while each remaining run stays sorted. When one run empties, append the other unchanged.
   </ExerciseSolution>

3. Analyze quicksort with all equal keys under two-way and three-way partition.

   <ExerciseSolution>
   Two-way partition may produce subproblems n-1 and 0 repeatedly, yielding quadratic work. Three-way partition places all equal keys in the middle, so equal inputs finish after one linear partition and recursion on empty sides.
   </ExerciseSolution>

4. Explain randomized expected versus deterministic worst-case selection.

   <ExerciseSolution>
   Randomized quickselect has expected linear time over pivot choices but quadratic worst-case branches. Median-of-medians guarantees a constant fraction eliminated each round and gives deterministic $O(n)$, with larger constants.
   </ExerciseSolution>

5. Derive the median-of-medians recurrence.

   <ExerciseSolution>
   Sorting groups and recursively finding their median costs $O(n) + T(n/5)$. The chosen pivot leaves at most about $7n/10$ elements on the larger side, so $T(n) \le T(n/5) + T(7n/10) + O(n) = O(n)$.
   </ExerciseSolution>

6. Compare comparison, counting, and radix sort assumptions.

   <ExerciseSolution>

   </ExerciseSolution>

7. Design an external multiway merge and identify its I/O bottleneck.

   <ExerciseSolution>
   Generate in-memory sorted runs, write them sequentially, then use a min-heap over run heads while buffering each run. Each pass reads and writes all records sequentially; the bottleneck is block transfers and number of merge passes, not the $O(n\log n)$ comparison count.
   </ExerciseSolution>

### Transfer and synthesis

8. You are given two sorted singly linked lists of lengths $m$ and $n$ ($m \ll n$). Design a merge that runs in $O(m \log(n/m))$ time by jumping through the longer list. Prove the bound and state the space cost.

   <ExerciseSolution hint="Use exponential search (galloping) to find where each element of the short list belongs in the long list.">

   For each element in the short list, use binary search (or galloping search followed by binary search) to locate its insertion point in the long list. Each search costs $O(\log n)$ in isolation, but because the short list elements are already sorted, the search intervals shrink. The total is $O(m \log(n/m))$. Space is $O(1)$ beyond the two list pointers.
   </ExerciseSolution>

9. A streaming system receives 1 billion integers per hour. You must report the running median after every 1000 values. Design a data structure that maintains the median under insertions only, with $O(\log N)$ per insertion and $O(1)$ median retrieval. Does the comparison lower bound prevent a faster solution?

   <ExerciseSolution hint="Two heaps: a max-heap for the lower half and a min-heap for the upper half.">

   Maintain two heaps: a max-heap for the lower half of values and a min-heap for the upper half. On each insertion, add to the appropriate heap and rebalance so sizes differ by at most 1. Median is the max of the lower heap (odd count) or the average of both roots (even count). This is optimal under the comparison model because finding the exact median requires at least $n/2$ comparisons in the worst case, and heap operations achieve this asymptotically.
   </ExerciseSolution>

10. You are building a sort for a database that stores 64-bit integer keys. Determine whether you should use a comparison sort (quicksort variants), radix sort, or a hybrid, considering: (a) worst-case guarantees, (b) memory overhead, (c) stability requirements, (d) data distributions seen in practice. Write a paragraph justifying the trade-off.

    <ExerciseSolution>
    Radix sort on 64-bit keys with base $2^{16}$ requires four passes and $O(n + 2^{16})$ space per pass. This is fast when data fits in RAM and the distribution is dense, but sparse high-bit regions waste memory. A comparison sort (introsort) uses $O(1)$ extra space and guarantees $O(n\log n)$ worst case. A common hybrid: use radix for the lower 32 bits of the key (or for the highest byte when keys cluster) and fall back to introsort when the radix distribution is too sparse.

## Deliberately Omitted

<Quiz client:load
  question="Why is TimSort excluded from this lesson's scope?"
  options={["It is a run-detection optimization, not a fundamental sorting-structure concept", "It is not used in any language", "It is slower than quicksort", "It is patented"]

  correctIndex={0}

The following topics are relevant to sorting but intentionally excluded from this lesson to maintain focus on representation-level sorting structures:

- **TimSort and adaptive merge sorts**: Detection of natural runs in real data combined with stable merge. TimSort is the default sort in Python and Java, but its galloping-mode analysis and run-length heuristics are engineering refinements rather than fundamental sorting-structure concepts.
- **Parallel sorting algorithms**: Bitonic sort, sample sort, and GPU-based radix sort. These introduce communication and synchronization costs that change sorting to a parallel decomposition task.
- **Sorting networks**: Comparator networks with fixed wiring that sort any input. These are hardware-oriented and rely on the zero-one principle rather than data-structure invariants.
- **External sort with multiple disks and fault tolerance**: Sorting in a distributed file system with node failures. The core divide-and-conquer strategy extends external mergesort, but fault-tolerance and replication concerns belong in a distributed-systems course.
- **Lower bounds for non-comparison models**: The decision-tree argument covers only comparison-based algorithms. Algebraic decision trees and the $A{\cal C}^0$ circuit lower bounds for sorting are active research areas beyond the scope of an undergraduate data structures course.
