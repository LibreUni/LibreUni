# Structure Implementation Laboratory

A hash map passes random tests but times out on repeated colliding keys. A balanced tree returns sorted keys, but `rank()` returns wrong positions after a rotation. A persistent map exposes old versions to mutation through a shared node. Each of these failures has a common root cause: the implementation lacked a precise specification, a checkable representation invariant, or a proof that operations preserve correctness.

This laboratory asks you to build one mutable and one versioned data structure behind explicit ADT interfaces. The assessed artifact is not code that passes a few examples — it is a reproducible argument that the representation is valid, operations implement the abstraction, complexity claims match the implementation, and failure and aliasing behavior is deliberate.

Choose from a dynamic array, linked deque, hash map, balanced tree, heap, trie, graph, union-find, or persistent map. The report must expose the same reasoning for the chosen structure. Treat the implementation as an engineering artifact with explicit specification, proof, tests, and benchmarks.

<MathStatement id="lab-requirement" kind="definition" title="Implementation correctness argument">
A structure implementation is correct when, for every public operation, (1) the precondition implies the abstract postcondition, (2) the representation invariant holds before and after the operation, (3) the abstraction function maps the post-state to the specified abstract value, and (4) the complexity bound is satisfied for all inputs meeting the precondition.
</MathStatement>

Write the abstract value domain and every public operation. For each operation state preconditions, postconditions, return value, error behavior, mutation, ownership, and iterator invalidation. Define equality and ordering, duplicate keys, empty/full behavior, and whether clients may retain references. If the structure is persistent, define version legality and old-version visibility. A complexity table without these assumptions is not an analysis.

<MathStatement id="spec-example" kind="example" title="Specification fragment: ordered set ADT">
The abstract value is a set of keys $S \subseteq \mathcal{K}$ with a total order $<$. Operations:
- $\text{insert}(k)$: precondition $k \in \mathcal{K}$, postcondition $S' = S \cup \{k\}$.
- $\text{contains}(k)$: postcondition returns $k \in S$, state unchanged.
- $\text{delete}(k)$: postcondition $S' = S \setminus \{k\}$.
- $\text{min}()$: precondition $S \neq \varnothing$, postcondition returns $\min S$, state unchanged.
Failure policy: missing-key delete is a no-op. Duplicate insert is a no-op.
</MathStatement>

## Part B: representation and invariants

<StructureDiagram kind="versions" title="Mutable versus versioned state" caption="The lab distinguishes destructive updates from published roots that preserve earlier observations." />

Draw or tabulate the concrete fields. State the RI in clauses that a validator can check. Include shape, ordering, metadata, ownership, and bounds. Define an abstraction function mapping each valid state to an abstract value. Identify which fields are authoritative and which are caches; cached fields require a consistency equation and update proof.

<MathStatement id="ri-def" kind="definition" title="Representation invariant (RI)">
A predicate $I(C)$ over concrete states $C$ such that:
1. Every constructor produces a state satisfying $I$.
2. Every mutator, given a pre-state satisfying $I$, produces a post-state satisfying $I$.
3. $I$ is strong enough to guarantee that the abstraction function $AF(C)$ yields a well-defined abstract value.
</MathStatement>

<MathStatement id="af-def" kind="definition" title="Abstraction function (AF)">
A (possibly partial) function $AF: C \to V$ from concrete states satisfying $I$ to abstract values in the ADT's value domain. $AF$ must be surjective onto the reachable abstract values and must respect the ADT's equality: $AF(C_1) = AF(C_2)$ whenever two concrete states represent the same abstract value.
</MathStatement>

Example dynamic-array clauses: capacity is nonnegative and representable; $0 \le size \le capacity$; logical elements occupy the first `size` cells; backing storage is owned; no operation exposes mutable storage. Example tree clauses: inherited key bounds, no illegal duplicate, child links and parent links reciprocal, root parent null, height or color valid, and subtree sizes correct. Avoid a sentence such as "the structure is well formed" without an executable interpretation.

## Part C: implementation proof

<MathStatement id="preservation" kind="theorem" title="Invariant preservation under mutation">
Let $C$ be a concrete state satisfying $I$, and let $op$ be a public operation with precondition $P_{op}$. If $op$ executes on $C$ and terminates normally, producing $C'$, then $I(C')$ holds and $AF(C')$ equals the abstract postcondition specified by $op$ for $AF(C)$.
</MathStatement>

<MathProof id="preservation-proof" for="preservation" collapsible>
Proof proceeds by cases over each operation. For each operation:
1. Identify the smallest region of the representation that can change.
2. Show that the RI clauses for unchanged regions remain satisfied (they were satisfied before and are not touched).
3. Show that the changed region satisfies every RI clause after the operation, using the precondition and the operation's internal logic.
4. Show that $AF(C')$ computes the specified abstract value.
For recursive structures, use induction on the structure size. For bulk operations, use a correspondence between old and new entries.
</MathProof>

For each mutation, identify the smallest region that can change and prove unaffected regions remain valid. For a rotation, show inorder preservation and recompute metadata. For a hash resize, show every live key is reinserted exactly once and duplicate semantics are preserved. For a persistent path copy, show old nodes are never written. For a queue, show modular indices represent FIFO order.

Include boundary cases in the proof: empty, singleton, full, root, first/last index, duplicate, absent delete, collision, resize, and failure after allocation. If exceptions are possible, decide strong, basic, or no exception safety. A strong guarantee restores the old abstract value on failure; implementing it may require allocate-and-commit ordering or a rollback log.

## Part E: complexity and measurement

Provide a worst-case table, an amortized proof if applicable, and expected bounds only with a probability model. Count key comparisons separately from pointer/array movement when key length is variable. For memory-aware structures, measure bytes per element, cache behavior, and allocations. Benchmark operation-level distributions and complete sequences, not only favorable averages.

## Counterexamples and failure modes

Every structure implementation is vulnerable to a characteristic set of defects. The following failure modes arise from the gap between specification and implementation.

<StructureDiagram kind="invalid-tree" title="AVL balance violation after rotation" caption="A stale height after a left rotation causes the ancestor balance check to miss a violation." variant="degenerate" />

<MathStatement id="failure-mode" kind="remark" title="Common failure pattern: stale cached metadata">
A cached field (height, size, color, subtree sum) is a redundant representation of information derivable from authoritative fields. If a mutation updates the authoritative structure but not the cache, the RI becomes inconsistent: the cache contradicts the authoritative state. The AF may still produce the correct abstract value, but the RI fails, and subsequent operations that depend on the cache will compute wrong results.
</MathStatement>

**Failure mode 1: stale metadata.** A rotation updates child pointers but forgets to recompute height. The tree remains structurally valid, but the height values are wrong. A subsequent insertion uses stale heights to decide balance, possibly violating the AVL invariant. The invariant validator catches this immediately because the height clause fails.

**Failure mode 2: missing parent-link update.** A rotation reassigns child pointers but does not update the parent pointers of the rotated nodes. The BST order remains correct, but parent links become inconsistent. Any operation that traverses upward (successor, predecessor, delete) will follow a stale parent pointer and reach the wrong node.

**Failure mode 3: shared mutable state in persistent structures.** Path copying copies nodes along the root-to-leaf path but occasionally mutates an existing node in place. Old versions that share that node now observe the mutation. The abstract behavior of the old version changes, violating the persistence guarantee.

**Failure mode 4: off-by-one in modular indexing.** A circular buffer or deque computes the next write position as `(head + count) % capacity`. If `count` is updated after the write instead of before, the first write overwrites the current element. If `count` reaches `capacity` without a separate full flag, the buffer cannot distinguish full from empty.

**Failure mode 5: hash resize without rehash.** A hash table doubles its bucket array on load-factor threshold but copies old entries by their old hash modulo the old capacity. Entries that collided before remain collided after resize, defeating the purpose of rehashing.

## Worked examples

### Example 1: AVL tree with invariant checks

Consider an AVL tree implementing an ordered set with keys of type `K` and a comparator. The ADT supports `insert(k)`, `contains(k)`, `delete(k)`, `min()`, `max()`, and `rank(k)` through subtree-size augmentation.

<StructureDiagram kind="tree" title="AVL tree representation" caption="Each node stores key, height, size, left, right, and parent pointers." />

**Representation.** Each node stores `key`, `height`, `size`, `left`, `right`, and `parent` (optional). `height` is the length of the longest root-to-leaf path from this node; `size` is the number of nodes in the subtree.

<MathStatement id="avl-ri" kind="definition" title="AVL tree representation invariant">
For every node $x$:
1. BST order: all keys in $x.\text{left}$ are strictly less than $x.\text{key}$; all keys in $x.\text{right}$ are strictly greater.
2. Balance: $|h(\text{left}) - h(\text{right})| \le 1$, where $h(\text{null}) = 0$.
3. Height: $h(x) = 1 + \max(h(\text{left}), h(\text{right}))$.
4. Size: $s(x) = 1 + s(\text{left}) + s(\text{right})$.
5. Parent: if $x$ has a parent $p$, then $p.\text{left} = x$ or $p.\text{right} = x$; $\text{root.parent} = \text{null}$.
6. No duplicate keys.
</MathStatement>

**AF.** The ordered set of keys obtained by an inorder traversal, respecting the BST ordering.

**Insertion proof sketch.** `insert(k)` follows the BST search path. If `k` exists, return (no duplicates). Otherwise insert a leaf with $h=1$, $s=1$. Climbing back toward the root:

- Update height and size for each ancestor.
- Check balance factor. If violated (bf $= \pm 2$), apply the appropriate rotation case (LL, RR, LR, RL). After rotation, recompute height and size for the rotated nodes.
- Stop climbing when no height change occurred or root is reached.

<MathProof id="avl-rot-proof" for="avl-ri" title="Rotation preserves inorder and metadata" collapsible>
Consider a right rotation at node $y$ with left child $x$ and right subtree $C$; $x$ has left subtree $A$ and right subtree $B$. Before rotation: inorder is $A, x, B, y, C$. After rotation: $x$ becomes root with left $A$ and right $y$; $y$ has left $B$ and right $C$. Inorder is $A, x, B, y, C$ — identical. Height recomputation: $h'(y) = 1 + \max(h(B), h(C))$; $h'(x) = 1 + \max(h(A), h'(y))$. Size recomputation: $s'(y) = 1 + s(B) + s(C)$; $s'(x) = 1 + s(A) + s'(y)$. The BST order, parent-link, and no-duplicate clauses are preserved by construction.
</MathProof>

**Validator.** After every mutation, run `check_invariants(root)`:

```python
def check_invariants(node):
    if node is None:
        return (0, 0, -inf, +inf)  # height, size, min, max
    h_l, s_l, min_l, max_l = check_invariants(node.left)
    h_r, s_r, min_r, max_r = check_invariants(node.right)
    assert max_l < node.key < min_r  # BST order
    assert abs(h_l - h_r) <= 1       # AVL balance
    assert node.height == 1 + max(h_l, h_r)
    assert node.size == 1 + s_l + s_r
    return (node.height, node.size,
            min_l if node.left else node.key,
            max_r if node.right else node.key)
```

**Test cases.** (1) Insert ascending keys $1,2,\ldots,1000$ — each triggers rotation; validate height $h \le \lfloor 1.44 \log_2 n \rfloor$. (2) Insert then delete the root repeatedly. (3) Insert random permutation; verify `rank(key)` returns the correct 1-indexed position after each operation. (4) Delete a leaf, a one-child node, and a two-child node; rebalancing may propagate. (5) Bulk-insert 100 keys, then search for all 100 plus 100 absent keys.

**Common bugs found by validator.** (a) Forgetting to update `height` after rotation — `check_invariants` catches stale heights immediately. (b) Swapping the direction of a double rotation (LR vs RL) — the BST order assertion catches misrouted subtrees. (c) Failing to update `parent` after rotation — caught by parent-link assertion. (d) Off-by-one in `size` update — caught by size assertion. (e) Deleting a two-child node without replacing the successor correctly — caught by BST order assertion.

### Example 2: indexed min-heap

<StructureDiagram kind="heap" title="Min-heap with index map" caption="A binary heap array plus a bidirectional map from item ID to array position." />

Suppose the chosen structure is an indexed min-heap with an array of records and a map from item ID to array index. Its RI has four parts: the live array prefix is a complete-tree layout; every parent priority is no greater than its children; every ID is unique; and the index map is a bijection between live IDs and positions. Insert appends a record, records its index, then sifts up while updating the map after every swap. Delete-min moves the last record to the root, deletes the old minimum's map entry, updates the moved record's index, then sifts down.

The hardest proof obligation is not the heap order alone; it is synchronized movement of the array and map. A swap helper must update both records' map entries before returning. If a comparison or callback can fail during a swap, use nonthrowing internal comparisons or specify the exception guarantee. A validator scans the prefix, checks parent order, verifies each map entry points to a record with the same ID, and checks the reverse relation. This example illustrates why cached indexes belong in the RI.

### Transfer and synthesis

5. A hash map uses open addressing with linear probing and a load factor threshold of 0.7. The specification claims that `get(k)` is $O(1)$ expected. The implementation passes all random tests, but in production with a specific key distribution, `get(k)` degrades to $O(n)$. Identify the missing assumption in the specification and design a validator that would have caught the problem during testing.

   <ExerciseSolution>
   The missing assumption is that the hash function distributes keys uniformly. Production keys may be adversarial (e.g., sequential integers modulo a power of two, or strings with common prefixes). A validator should measure the maximum probe chain length after each insertion and warn if it exceeds, say, $3 \times$ the expected chain length. Alternatively, the test harness should include adversarial keys (e.g., all keys hashing to the same bucket) and verify that the complexity bound is expressed as worst-case $O(n)$ or that the implementation uses a collision-resistant hash.
   </ExerciseSolution>

6. You are given a persistent binary tree that passes all functional tests: every old root returns the correct abstract value after updates. However, a memory profiler shows that the number of live nodes grows linearly with the number of versions rather than logarithmically. Analyze what implementation defect could cause this, and state which RI clause would expose it.

   <ExerciseSolution>
   The defect is that the implementation copies the entire tree on every update instead of path-copying only the root-to-leaf path. The RI for persistence should include a node-sharing invariant: the number of new nodes created by an update is $O(h)$ where $h$ is the tree height. A validator can count nodes reachable from the new root that are not reachable from the old root. If this count exceeds $O(h)$, the implementation is copying too much. The deeper issue: the abstraction function for persistence requires that each version's graph is exactly the specified set of nodes, and the RI should bound the structural sharing.
   </ExerciseSolution>

7. Design a metamorphic property that distinguishes a correct queue from one that occasionally reorders elements under concurrent access, assuming the queue is used sequentially. Explain why the property is not sufficient for concurrent correctness.

   <ExerciseSolution>
   Property: for any sequence of enqueue and dequeue operations, the dequeued sequence equals the enqueued sequence in FIFO order. Expressed as a state machine: enqueue $x$ then dequeue must return $x$ if the queue was nonempty before enqueue. For sequential use, this property is sufficient to establish FIFO order. For concurrent use, a queue could satisfy this property under sequential testing but violate linearizability under interleaving — a thread could observe a dequeue returning a value that was enqueued after a different enqueue whose result is not yet visible. Linearizability requires a total order of operations consistent with real-time precedence, which metamorphic testing alone cannot verify.
   </ExerciseSolution>

8. A benchmark reports that a B-tree implementation runs 2.3$\times$ faster than a hash map for point lookups on a dataset of 10 million keys. The B-tree uses 4 KB pages and the hash map uses separate chaining. The reported workload is 50% inserts and 50% lookups. Identify at least three confounding factors that could invalidate the benchmark conclusion, and design a controlled experiment that isolates the structure's cost from measurement artifacts.

   <ExerciseSolution>
   Confounding factors: (1) The hash map may be using a poor hash function or low load factor, causing excessive collisions. (2) The B-tree may benefit from CPU cache locality because the in-memory pages fit in L2 cache, while the hash map's linked-list chaining causes random pointer chasing. (3) The benchmark may not warm up the cache — the B-tree's page structure benefits from repeated access patterns. (4) The benchmark may not measure allocation time: the hash map may resize and rehash during inserts, while the B-tree allocates pages upfront. (5) Keys may be sequential integers, giving the B-tree optimal page utilization while the hash map's hash function randomizes them.

   Controlled experiment: pre-generate inputs, fix the random seed, use the same key distribution for both structures, measure cache misses (perf stat), separate warm-up from measurement, vary the load factor for the hash map, vary the page size for the B-tree, and report the full distribution of latencies (p50, p99, p99.9) rather than mean throughput.
   </ExerciseSolution>

## Deliberately Omitted

The following topics are relevant to structure implementation but are outside the scope of this laboratory. Each is listed with a brief rationale.

- **Formal verification with proof assistants (Coq, Dafny, Lean).** Encoding representation invariants as dependent types or refinement types and mechanically verifying operations is the gold standard for correctness. This lab uses proof sketches and runtime invariant checking instead because the tooling overhead is substantial and orthogonal to the conceptual methodology.

- **Property-based testing with stateful model checking (Hypothesis stateful, QuickCheck monadic).** Automated generation of operation sequences with stateful specifications can find deeper bugs than differential testing. The lab limits coverage to manual reference-model comparison and metamorphic properties.

- **Fuzz testing and sanitizer integration.** AFL/libFuzzer with address sanitizer and undefined-behavior sanitizer finds memory errors that proofs might miss. This is a complementary technique whose conceptual framework differs from the specification-driven approach emphasized here.

- **Statistical benchmark methodology.** Hypothesis testing, warm-up/cool-down analysis, and microarchitecture performance-counter interpretation are deferred to a dedicated workload-benchmarking lesson.

- **Wait-free and lock-free concurrent structures.** The laboratory restricts to sequential clients. Linearizability proofs and memory-model reasoning are a separate topic.
