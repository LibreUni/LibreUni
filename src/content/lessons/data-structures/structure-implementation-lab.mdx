# Structure Implementation Laboratory

This laboratory requires implementing one mutable and one versioned data structure behind explicit ADT interfaces. The assessed artifact is not just code that passes a few examples. It is a reproducible argument that the representation is valid, operations implement the abstraction, complexity claims match the implementation, and failure/aliasing behavior is deliberate. You may choose a dynamic array, linked deque, hash map, balanced tree, heap, trie, graph, union-find, or persistent map, but the report must expose the same reasoning.

## Part A: specification

Write the abstract value domain and every public operation. For each operation state preconditions, postconditions, return value, error behavior, mutation, ownership, and iterator invalidation. Define equality and ordering, duplicate keys, empty/full behavior, and whether clients may retain references. If the structure is persistent, define version legality and old-version visibility. If it is concurrent, define linearizability or explicitly restrict the lab to sequential clients.

Choose representative workloads: operation mix, input sizes, key distribution, update ratio, and resource limits. State expected and worst-case complexity separately. For a hash map, name the collision assumption and load policy. For a balanced tree, name the height invariant. For a heap, specify tie behavior and handle validity. A complexity table without these assumptions is not an analysis.

## Part B: representation and invariants

Draw or tabulate the concrete fields. State the RI in clauses that a validator can check. Include shape, ordering, metadata, ownership, and bounds. Define an abstraction function mapping each valid state to an abstract value. Identify which fields are authoritative and which are caches; cached fields require a consistency equation and update proof.

Example dynamic-array clauses: capacity is nonnegative and representable; 0u2264sizeu2264capacity; logical elements occupy the first size cells; backing storage is owned; no operation exposes mutable storage. Example tree clauses: inherited key bounds, no illegal duplicate, child links and parent links reciprocal, root parent null, height or color valid, and subtree sizes correct. Avoid a sentence such as “the structure is well formed” without an executable interpretation.

## Part C: implementation proof

For each mutation, identify the smallest region that can change and prove unaffected regions remain valid. For a rotation, show inorder preservation and recompute metadata. For a hash resize, show every live key is reinserted exactly once and duplicate semantics are preserved. For a persistent path copy, show old nodes are never written. For a queue, show modular indices represent FIFO order.

Include boundary cases in the proof: empty, singleton, full, root, first/last index, duplicate, absent delete, collision, resize, and failure after allocation. If exceptions are possible, decide strong, basic, or no exception safety. A strong guarantee restores the old abstract value on failure; implementing it may require allocate-and-commit ordering or a rollback log.

## Part D: test harness

Implement a slow reference model using a standard sequence/map/set and execute the same generated operation stream against both. Compare outputs after every operation. Add an invariant validator that runs after each mutation in debug mode. Record random seeds and shrink failing traces. Include deterministic tests for boundary cases and adversarial tests for complexity assumptions.

Use metamorphic properties: insert then remove a fresh key restores the model; reversing a sequence twice restores order; a heap’s repeated delete-min is nondecreasing; a graph transpose twice restores directed edges; a persistent old root returns old values after updates to a descendant. Properties should respect the ADT’s duplicate and ordering rules.

## Part E: complexity and measurement

Provide a worst-case table, an amortized proof if applicable, and expected bounds only with a probability model. Count key comparisons separately from pointer/array movement when key length is variable. For memory-aware structures, measure bytes per element, cache behavior, and allocations. Benchmark operation-level distributions and complete sequences, not only favorable averages.

Compare at least two representations on a workload that reveals their tradeoff. Explain where the benchmark agrees or disagrees with asymptotic predictions. Record hardware, compiler/runtime, seed, operation counts, warm-up, and correctness checksum. Do not claim a benchmark proves a theorem or that a theorem predicts a particular constant.

## Review scenarios

Scenario 1: a hash map passes random tests but times out on repeated colliding keys. Identify the missing assumption and propose keyed hashing, bucket trees, or limits. Scenario 2: a balanced tree returns sorted keys but rank is wrong after rotation. Check subtree-size recomputation. Scenario 3: a persistent map changes an old version after an update. Find the shared node mutation and enforce immutability. Scenario 4: a queue occasionally returns a recent item before an old one. Inspect count/full distinction and modular tail updates.

## Deliverables and rubric

Submit interface/specification (20%), implementation and RI validator (25%), correctness proofs or proof sketches (25%), adversarial/reference tests (15%), complexity and benchmark report (10%), and reproducibility/clarity (5%). A failing test is not automatically a low score if the report identifies the violated invariant and the repair plan, but an untested claim is weak evidence.

## Example review: indexed min-heap

Suppose the chosen structure is an indexed min-heap with an array of records and a map from item ID to array index. Its RI has four parts: the live array prefix is a complete-tree layout; every parent priority is no greater than its children; every ID is unique; and the index map is a bijection between live IDs and positions. Insert appends a record, records its index, then sifts up while updating the map after every swap. Delete-min moves the last record to the root, deletes the old minimum’s map entry, updates the moved record’s index, then sifts down.

The hardest proof obligation is not the heap order alone; it is synchronized movement of the array and map. A swap helper must update both records’ map entries before returning. If a comparison or callback can fail during a swap, use nonthrowing internal comparisons or specify the exception guarantee. A validator scans the prefix, checks parent order, verifies each map entry points to a record with the same ID, and checks the reverse relation. This example illustrates why cached indexes belong in the RI.

## Reproducible handoff

Include a command that runs the deterministic tests, a command that runs randomized tests with a supplied seed, and a command that produces benchmark output. Pin dependency versions or state the environment. Store generated traces for failures and make the report distinguish measured evidence from proof. A future reader should be able to change the workload and understand which complexity assumption changed rather than treating the benchmark as a fixed answer.

## Oral defense prompts

Be prepared to explain why the representation was chosen, which operation is the bottleneck, which invariant is most fragile, how a failure would be localized, and what happens when memory allocation or a comparator fails. Also explain one case where a simpler structure would be preferable. Depth means understanding tradeoffs and limits, not merely presenting the most sophisticated implementation.

## Exercises

1. Select a structure and write its complete RI and AF.
2. Design five boundary tests and five adversarial tests.
3. Give a preservation proof for its hardest mutation.
4. Build a reference-model differential harness.
5. State one complexity claim that is worst-case and one conditional.
6. Design a benchmark that avoids dead-code elimination and input-generation confounding.
7. Analyze one plausible bug trace from the scenarios above.

### Diagnostic solutions


1. A complete answer names every field and a predicate over valid states, then maps the logical traversal/content to the ADT. Include ownership and cached metadata; otherwise alias or stale-cache bugs remain outside the proof.

2. Test empty, singleton, boundary index/endpoint, full-capacity or resize, duplicate/absent operation; adversarial tests include sorted tree keys, colliding hashes, long trie prefixes, alternating resize thresholds, and stale version branches. Match tests to claimed assumptions.

3. Identify the local change, use the pre-operation RI, prove the postcondition for each branch, and show unaffected components remain valid. For recursive structures use induction; for bulk operations use a correspondence between old and new entries.

4. Generate operations, execute implementation and reference model, compare return values and abstract contents after each step, and call the validator. Record seed and shrink sequences. The model may be slower because correctness, not production performance, is its purpose.

5. Example worst-case: balanced-tree search O(log n). Conditional: hash lookup expected O(1) under bounded load and a stated hash distribution. An amortized array append is neither worst-case O(1) nor a probabilistic expectation.

6. Pre-generate inputs, consume results through a checksum, separate warm-up, repeat runs, vary n and workload, record environment, and collect operation latency and counters. Validate outputs outside or alongside a controlled sample.

7. Reproduce the shortest trace, run the validator after each operation, classify the first invariant failure, and connect it to a missing pointer/metadata update or incorrect precondition. Repair code and update the proof/test that should have caught it.
