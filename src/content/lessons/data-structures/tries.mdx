# Tries and String Indexing

A binary search tree searching for "encyclopedia" in a set of 1 billion URLs compares the full 12-character string at each of ~30 nodes — roughly 360 character comparisons. A trie (from "retrieval," Edward Fredkin, 1960 [Fredkin60]) splits on individual characters, so the same search needs exactly 12 character steps, independent of $n$.

Structure: a rooted tree where each edge is labeled with a single symbol. A path from root to node spells a prefix; a node marked *terminal* indicates that the prefix is a stored key. All keys sharing a prefix share the corresponding path. Lookup, insertion, and deletion cost $O(\ell)$ where $\ell$ is the key length, regardless of how many keys are stored. The trade-off: a basic trie uses $O(\ell \cdot n)$ nodes in the worst case, which can be memory-heavy for long strings with few shared prefixes.

The set ADT supports insert(s), contains(s), remove(s), and prefix queries such as all keys beginning with $p$ or the count of keys with prefix $p$. Define whether duplicate insertion changes a value, whether the alphabet is fixed, and whether strings are byte sequences or Unicode scalar values. Normalization and case folding are application policy, not automatic properties of a trie.

<MathStatement id="trie-invariant" kind="definition" title="Trie representation invariant">
A trie $T$ over alphabet $\Sigma$ is a rooted tree where each node $v$ has:
- a collection of outgoing edges, each labeled with a distinct symbol $\sigma \in \Sigma$ leading to child $v.\text{child}[\sigma]$;
- a boolean $v.\text{terminal}$ indicating whether the path from root to $v$ spells a stored key.

The root represents the empty string $\epsilon$; it may be terminal if $\epsilon$ is a stored key. For any two distinct edges from a node, their labels differ. A key $s$ is stored iff there exists a path from root to some node $v$ with $v.\text{terminal} = \text{true}$ and the concatenation of edge labels along the path equals $s$.
</MathStatement>

**Worked example 1: string insertion.** Insert the keys "to", "tea", "ten", and "in" into an empty trie over $\Sigma = \{\text{a},\dots,\text{z}\}$.

1. Insert "to": create root, create child `t`, create child `o`, mark terminal.
2. Insert "tea": walk root→`t`, create child `e`, create child `a`, mark terminal.
3. Insert "ten": walk root→`t`→`e`, create child `n`, mark terminal.
4. Insert "in": create root child `i`, create child `n`, mark terminal.

Prefix query "te" reaches node `te` and traverses its subtree, returning `tea` and `ten`. A dictionary comparison structure would need separate ordered searches; the trie exposes the prefix as a physical path.

**Worked example 2: binary trie for IP routing.** A binary trie stores IPv4 addresses as 32-bit strings. Insert prefix `10.0.0.0/8` (binary `00001010********...`), `10.1.0.0/16` (`0000101000000001********`), and `192.168.0.0/16` (`1100000010101000********`). The trie branches on one bit per level. Longest-prefix match for destination `10.1.5.3` traverses `0000101000000001` and returns the `/16` route rather than the `/8` route. The decision per bit is a single array lookup; no full-address comparison or range search is needed.

## Operations and correctness proof

<MathStatement id="lookup-correctness" kind="theorem" title="Trie lookup correctness" proofId="lookup-proof">
For a trie $T$ satisfying the representation invariant, the operation $\text{contains}(s)$ returns $\text{true}$ iff $s$ was previously inserted and not subsequently deleted. The operation examines at most $|s|$ edges.
</MathStatement>

<MathProof id="lookup-proof" for="lookup-correctness" collapsible>
By the invariant, exactly one path from root spells $s$: follow the unique child labeled $s[0]$, then $s[1]$, and so on. If at any step the required child does not exist, no key with prefix $s[0..i]$ is stored, so $s$ cannot be stored. If the path exists to a node $v$, then $v.\text{terminal}$ records whether $s$ is a stored key. Insertion sets $v.\text{terminal} \leftarrow \text{true}$; removal sets it to $\text{false}$. No other operation modifies $v.\text{terminal}$, so the result is correct. Each step consumes one symbol, so at most $|s|$ edges are traversed. ∎
</MathProof>

Removal unmarks the terminal node, then prunes nodes that are nonterminal and have no children, walking upward until a node is needed by another key. Pruning changes only representation, not the set of stored strings. A common bug is deleting a path shared by another key; the terminal/child checks in the invariant are the proof that pruning is safe.

If children are stored in a fixed array of size $|\Sigma|$, child lookup is $O(1)$ but each node costs $O(|\Sigma|)$ slots. A sorted vector costs $O(\log d)$ lookup with $d$ children and compact storage; a hash map gives expected $O(1)$ lookup but adds hashing and unordered iteration.

    def contains(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:

            node = node.children[ch]
        return node.terminal

    def prefix_keys(self, prefix):
        node = self.root
        for ch in prefix:
            if ch not in node.children:

            node = node.children[ch]

## Prefix queries, counts, and longest-prefix match

To enumerate keys with prefix $p$, follow $p$ then perform DFS or BFS over descendants, appending edge symbols to a mutable buffer. The traversal visits the prefix-path nodes plus the output subtree; time is $O(|p| + \text{total output symbols})$ with compact child iteration. Lexicographic order requires visiting children in sorted symbol order and using DFS; a hash-map child collection cannot guarantee order without sorting.

Counting keys under a prefix stores `subtreeCount` at each node. Inserting a new key increments counts along its path, removal decrements them, and duplicate policy determines whether an update is applied. The invariant is `subtreeCount(node) = number of terminal descendants`, including the node if terminal.

Longest-prefix matching asks for the longest stored key that is a prefix of an input string. During a traversal, remember the deepest terminal node seen. If traversal fails, return the remembered key. This supports routing-table lookup and lexical token matching. It differs from longest stored string sharing a prefix; the direction of the query matters.

## Compressed tries and radix trees

<StructureDiagram kind="btree" title="Compressed radix path" caption="A compressed trie stores a whole unary edge label while preserving prefix navigation." variant="radix" />

A trie with one-child chains wastes nodes. A compressed trie (Patricia trie / radix tree [Morrison68]) labels an edge with a nonempty string and stores only branching points or terminals. Lookup compares chunks, tracking the first mismatch. Insertion splits an edge at the longest common prefix, creates a branching node, and preserves the old and new suffixes. The invariant requires edge labels on a node to have disjoint first symbols or an equivalent unambiguous partition.

Compression reduces node count and pointer overhead but makes updates more complex: a split must preserve all descendants, terminal status, and parent links if present. The path comparison cost remains $O(\ell)$ character inspections. A radix tree is especially useful for long keys with shared prefixes and for memory-resident indexes where pointer count matters.

## Binary tries and bitwise keys

For fixed-width integers or bit strings, a binary trie branches on one bit per level. A compressed binary trie skips unary paths and stores a discriminating bit index. Longest-prefix match becomes a natural operation: follow bits while retaining the deepest matching prefix. Patricia tries are used in routing and IP lookup because prefix length and bitwise comparison align with the query [Morrison68].

The height of an uncompressed fixed-width trie is bounded by word width $W$, giving $O(W)$ operations even when $n$ is large. For variable-length strings, height depends on key length. Compressed structures can reduce the number of visited nodes but still inspect enough key bits to distinguish candidates.

## Counterexample and failure mode

<MathStatement id="failure-mode" kind="remark" title="Failure mode: unbounded node count">
A basic trie storing $n$ distinct strings of average length $\ell$ uses up to $O(n \cdot \ell)$ nodes. When keys share few prefixes (e.g., random UUIDs or hashed values), the trie degenerates to a linked list of nodes per key, negating the space advantage and making the node count approach $n \cdot \ell$.
</MathStatement>

**Example of failure.** Insert 10,000 random 32-character hex strings (e.g., UUIDs). The expected shared prefix length between any two strings is less than 2 characters. Each key creates roughly 30 new nodes, yielding ~300,000 nodes — each storing a child array or hash map. A hash table of the same 10,000 entries uses constant $O(n)$ space and offers $O(1)$ expected lookup, outperforming the trie in both space and time for this workload.

**Counterexample to naive deletion.** Consider a trie containing "test" and "testing". Deleting "test" must *not* remove the path "test" because "testing" requires it. The deletion algorithm must unmark the terminal flag and only prune nodes that are both nonterminal and childless. An implementation that deletes the "t" node on removing "test" would corrupt the "testing" path.

**Misconception: tries guarantee $O(\ell)$ time.** The bound assumes $O(1)$ child lookup. If children are stored in a sorted vector, lookup costs $O(\log d)$ per symbol, giving $O(\ell \log d)$ total — worse than $O(\ell)$ for large $d$. The bound further assumes the key length $\ell$ is known and bounded; for variable-length keys with no upper bound, $\ell$ itself is a function of the input.

## Memory and implementation

Node allocation dominates many trie implementations. An arena can improve locality, while packed edge arrays reduce per-node overhead. Store terminal flags as bit fields only if the representation handles alignment and atomic updates correctly. A child-index array into a contiguous node pool avoids raw pointers and permits serialization, but pool growth can invalidate indexes if they are addresses rather than stable offsets.

Unicode requires a defined unit. Indexing UTF-8 bytes gives a valid trie over bytes but prefix queries may cut through code-point encodings; indexing decoded scalar values changes length and normalization behavior. Grapheme clusters require even more linguistic processing. The data structure cannot decide which notion of string equality the application intends.

## Ordering and persistence

Trie nodes can be made immutable and shared between versions using the same path-copying method as persistent trees. Inserting a string copies the nodes along its prefix path and reuses unrelated subtries; deleting a key copies the path while pruning only in the new version. Subtree counts, terminal flags, and child maps must all be copied or treated immutably. This makes autocomplete snapshots consistent, but old roots retain every reachable subtrie.

Lexicographic iteration is defined by the alphabet order, not by memory address. A compressed edge label must be compared symbol by symbol, and a terminal at an internal node must be emitted before longer descendants when the shorter key sorts first. If the application wants locale-specific collation, a code-point trie's byte order is not enough; store or compute the collation key explicitly.

## Robust testing

Generate strings with empty keys, one-symbol prefixes, duplicate keys, a key that is a prefix of another, long unary chains, and maximal alphabet symbols. Compare `contains` and prefix output against a reference set, and after deletion verify no unrelated key disappeared. For radix trees, test splits at the first symbol, at the final symbol, and when one key ends exactly at the split. These cases exercise the boundaries hidden by ordinary random strings.

1. **Recall.** State the trie representation invariant and prove exact lookup correctness.

   <ExerciseSolution hint="The invariant must guarantee that every stored key corresponds to exactly one path, and every terminal path corresponds to a stored key. Formalize the uniqueness of outgoing-edge symbols.">

   Each node has a unique root path, outgoing labels are unique, and terminal marks exactly stored keys. Lookup follows the only path matching each input symbol; it succeeds exactly when that path exists and its final node is terminal.
   </ExerciseSolution>

2. **Recall.** Trace insertion and pruning for keys with shared prefixes.

   <ExerciseSolution hint="Follow the path for each key symbol-by-symbol. Pruning is only safe when a node is both nonterminal and childless—walk upward from the removed node.">

   Insert the shared prefix once and mark only complete keys. On removal, unmark the terminal; prune a node only if it is nonterminal and childless. A shared prefix remains because it has another child or terminal status.
   </ExerciseSolution>

3. **Application.** Design prefix enumeration in lexicographic order.

   <ExerciseSolution>
   Follow the prefix, then DFS with children visited in sorted symbol order. Emit the buffer whenever terminal is true. The traversal returns each descendant key once; sorting child order yields lexicographic order by the standard prefix comparison.
   </ExerciseSolution>

4. **Application.** Prove `subtreeCount` maintenance for insert and remove.

   <ExerciseSolution>
   For a new key, increment count at every node on its path exactly once; the set of terminal descendants gains one. For removing an existing key, decrement the same path; if duplicate insertion is idempotent, do not increment twice. Induction over the path proves the count formula.
   </ExerciseSolution>

5. **Application.** Compare fixed-array, sorted-vector, and hash-map child storage.

   <ExerciseSolution>
   Fixed arrays provide direct lookup but cost $\Theta(|\Sigma|)$ per node; sorted vectors use compact degree-proportional space and $O(\log d)$ lookup; hash maps use expected $O(1)$ lookup with overhead and no inherent order. Choose based on alphabet, degree distribution, and memory budget.
   </ExerciseSolution>

6. **Application.** Describe a radix-edge split and list all invariants to preserve.

   <ExerciseSolution>
   Find the longest common prefix of old edge label and new suffix; split into a parent edge prefix and two suffix edges, attach both subtrees, and preserve terminal flags if a key ends at the split. Recompute counts and parent links, and ensure outgoing first symbols remain distinct.
   </ExerciseSolution>

7. **Application.** Explain byte-oriented versus Unicode-scalar tries.

   <ExerciseSolution>
   A byte trie treats encoded bytes as symbols and can answer byte-prefix queries; a scalar trie decodes code points and answers scalar-prefix queries. Normalization, case, invalid encoding, and grapheme semantics must be specified; otherwise equal user-visible strings may map to different paths.
   </ExerciseSolution>

8. **Transfer: Autocomplete with weighted suggestions.** Design a trie extension where each stored key has an associated weight (frequency), and a prefix query returns the top-$k$ completions ordered by weight. Describe the per-node data structure, the insertion/update cost, and the query algorithm. Under what conditions does a heap-based approach at each node dominate a simple subtree traversal?

   <ExerciseSolution hint="Store a bounded priority queue at each node, or defer ranking to a post-traversal step.">

   Store a min-heap of size $k$ at each node containing the top-$k$ terminal descendants by weight. On insertion, bubble the new key's weight down the path, updating heaps. Query traverses the prefix, then returns the heap contents. Space cost is $O(k \cdot \text{nodes})$. For large $k$ or sparse weights, a post-traversal ranking (collect all candidates, then sort) may be cheaper.
   </ExerciseSolution>

9. **Transfer: Concurrent trie.** Design a trie that supports concurrent insert and lookup without global locks. Describe how you would use reader-writer locks per node, compare-and-swap on child pointers, or hazard pointers. What correctness invariant must hold for a reader that observes a partially constructed path?

   <ExerciseSolution hint="A reader must never observe a path whose terminal flag is set before the full path is constructed. Consider the ordering of child pointer publication and terminal flag writes.">

   Use per-node locks or CAS on child pointers. A writer must publish the child pointer before setting the terminal flag, and readers must read the terminal flag after confirming the child pointer exists. Memory ordering (e.g., acquire/release semantics) ensures the reader sees a consistent state. A reader that finds a missing child simply returns false; the trie is correct even if the writer has not yet finished.
   </ExerciseSolution>

10. **Synthesis: Prefix-free code from a trie.** Given a set of strings $S$, prove that the set is prefix-free (no string is a prefix of another) iff in the trie of $S$, every terminal node is a leaf. Show that the Huffman algorithm produces a tree whose trie representation satisfies this property. What is the minimal trie size for a prefix-free code over $n$ symbols?

   <ExerciseSolution hint="A terminal internal node would mean that a shorter key is a prefix of a longer one. Huffman's merging procedure ensures all codewords are leaves.">

   If a terminal node has a child, then the key at that node is a prefix of the key at the child, violating prefix-freeness. Huffman's algorithm repeatedly merges the two lowest-frequency nodes, creating a binary tree where every symbol is a leaf. The minimal trie for $n$ symbols has exactly $n$ terminal nodes; the number of internal nodes depends on the tree shape but is at most $n-1$.
   </ExerciseSolution>

## Deliberately Omitted

**Suffix trees and suffix arrays.** String data structures that index all suffixes of a string for substring search. Suffix trees are compact tries over suffixes; their construction (Ukkonen's algorithm, DC3) and the different query types (substring, LCP, pattern matching) warrant a separate lesson.

**Huffman coding and prefix-free codes.** Constructing optimal prefix codes using a binary-trie structure where leaf frequency determines code length. Huffman coding uses tries as a computational tool rather than as a query structure, and the optimality proof is a separate information-theoretic argument.

**Bloom filters and approximate set membership.** Probabilistic data structures that use hash functions and bit arrays to test set membership with false positives but no false negatives. Bloom filters are complementary to exact tries but use hashing rather than prefix storage.

**Aho–Corasick automaton for multiple pattern matching.** Building a trie with failure links for simultaneous matching of multiple patterns in a text stream. The automaton extends the trie with linear-time matching but its construction algorithm and failure-link invariant are a specialized topic.
