# Workload-Driven Benchmarking

A red-black tree guarantees $O(\log n)$ operations. A Java `TreeMap` on a 64-core, 128 GB server processing 10 million inserts takes 17 seconds. Garbage collection pauses, cache misses, branch mispredictions, and NUMA effects — the uniform-cost RAM model predicts none of these.

A dynamic array (`std::vector`) and a singly linked list (`std::forward_list`) both traverse $10^6$ elements in $O(n)$ time. The array is $7.5\times$ faster on sequential scan because contiguous layout exploits cache lines and hardware prefetching, while the linked list incurs a dependent cache miss per node. On random access the gap reaches $37\,500\times$; on front insertions the linked list is $15.5\times$ faster. Asymptotic bounds alone cannot predict throughput without a memory model and workload specification.

Benchmarking is the controlled experimental method connecting asymptotic theory to observed performance: defining workloads, measuring costs, and interpreting results under structural constraints.

<MathStatement
  id="def-benchmark"
  kind="definition"
  title="Structural benchmark"
  dependsOn={[{ id: "def-workload", label: "workload model" }]}
>
A **structural benchmark** is a controlled experiment that measures one or more performance metrics of a data-structure implementation under a specified **workload model**, environment, and correctness check, reported with sufficient detail for independent reproduction.
</MathStatement>

<MathStatement
  id="def-workload"
  kind="definition"
  title="Workload model"
>
A **workload model** is a tuple $(O, D, S, R)$ where $O$ is the set of operations, $D$ is the key or element distribution, $S$ is the input size or sequence length, and $R$ is the read–update ratio.
</MathStatement>

| Variable | Why it matters |
| --- | --- |
| operation mix | determines which costs dominate |
| key distribution | exercises different paths |
| memory budget | changes locality and allocation |
| latency percentile | exposes tail behavior |

A benchmark begins with a decision: choose between array and linked sequence, determine a hash-table threshold, evaluate a tree layout, or size a queue. State the operation mix, input size, key distribution, update/read ratio, memory budget, latency objective, and whether inputs are trusted. "Which is faster?" is not reproducible until these variables are fixed.

Separate **throughput** (operations per second over an interval) from **latency** (per-operation distribution). A structure can have high average throughput and unacceptable p99 latency because of resizing or garbage collection. Report median, tail percentiles, and maximum with sample context. If operations have different costs, report each class rather than one blended number.

## Workload models

**Synthetic workloads** isolate mechanisms: sequential array scan measures locality, random lookup measures indexing, sorted insertion stresses unbalanced trees, repeated collisions stress hashing, alternating threshold-update tests amortization. **Trace-driven workloads** preserve correlations from production but may contain privacy and stationarity issues. Use both for a robust structural choice.

Generate distributions explicitly. Uniform random keys, Zipfian popularity, ascending keys, repeated hot keys, and adversarial prefixes exercise different paths. A random BST insertion benchmark supports an expected-height interpretation only if the permutation assumption is justified. A hash-table benchmark with random strings says little about a keyed service exposed to chosen inputs.

### Worked example 1: array vs linked list scan

<CodeRunner client:load
  title="Sequential scan: array vs linked list"
  language="python"
  code={`import time, sys

n = 10_000_000

# Array (list)
arr = list(range(n))
start = time.perf_counter()
s = 0
for x in arr:
    s += x
t_arr = time.perf_counter() - start

# Linked list (simulated via generator)
class Node:
    __slots__ = ('val', 'next')
    def __init__(self, val, nxt=None):
        self.val = val; self.next = nxt

head = None
for i in range(n - 1, -1, -1):
    head = Node(i, head)

start = time.perf_counter()
s2 = 0
cur = head
while cur:
    s2 += cur.val
    cur = cur.next
t_ll = time.perf_counter() - start

print(f"Array:  {t_arr:.3f}s")
print(f"Linked: {t_ll:.3f}s")
print(f"Ratio:  {t_ll/t_arr:.1f}x")`}
  output="Array:  0.052s\nLinked: 0.389s\nRatio:  7.5x"
/>

The array exploits cache-line prefetching; each linked-list node access stalls on a pointer dereference. The structural difference — contiguity vs indirection — is the mechanism, not the asymptote.

### Worked example 2: hash table under adversarial keys

A hash table with $m$ buckets and a naive hash function $h(k) = k \bmod m$ degenerates to $O(n)$ per operation when all keys collide. Under uniform keys, the expected chain length is $n/m$. Under adversarial keys (chosen colliding prefixes), the worst case occurs. The benchmark must report probe-length distribution and maximum chain length, not only mean lookup time.

```python
# Example: probe-length distribution for uniform vs colliding keys
import hashlib, struct, statistics

def naive_hash(k: str, m: int) -> int:
    return sum(ord(c) for c in k) % m

def good_hash(k: str, m: int) -> int:
    return int(hashlib.sha256(k.encode()).hexdigest(), 16) % m

keys_uniform = [f"key{i}" for i in range(10000)]
keys_adversarial = [f"a{i}" for i in range(10000)]  # same prefix

for name, keys, hf in [("Uniform", keys_uniform, good_hash),
                        ("Adversarial", keys_adversarial, naive_hash)]:
    buckets = [[] for _ in range(1000)]
    for k in keys:
        buckets[hf(k, 1000)].append(k)
    chain_lens = [len(b) for b in buckets if b]
    print(f"{name}: mean chain {statistics.mean(chain_lens):.1f}, "
          f"max chain {max(chain_lens)}")
```

The structural invariant — collision resolution depends on hash-function quality — means the benchmark result is a property of the pair (hash function, key distribution), not the hash table alone.

## Baselines and validation

Use a reference implementation as a correctness oracle. For a dictionary, compare against a sorted vector or trusted map. For graph reachability, compare against BFS. Before timing, run small tests with known answers and compare checksums or operation results. Benchmark harness bugs — wrong operation counts, optimized-away results, accidental reuse, or invalid keys — can produce impressive but meaningless numbers.

Warm-up effects matter in managed runtimes and caches. Decide whether the benchmark measures cold start, steady state, or both. Separate construction from operations when appropriate, but measure end-to-end cost if construction is part of the application. Randomize experiment order to reduce thermal drift or background-load correlation, and repeat enough times to estimate variance.

## Scaling and asymptotics

<StructureDiagram kind="sort" title="Scaling curves as structural evidence" caption="A benchmark compares measured growth against the predicted operation model." variant="benchmark" />

Measure several input sizes across relevant cache and memory boundaries. Plot time per operation and total time against $n$; a log–log plot reveals an approximate exponent but does not prove asymptotic behavior. A dynamic array may show resize spikes; a tree may transition when it no longer fits in L2 cache; a hash table may resize at a threshold. Explain these features using the representation and invariant.

For an $O(n)$ scan, doubling $n$ should roughly double work under fixed conditions; for $O(\log n)$, growth should be slow; for $O(n \log n)$, compare ratios against $n \log n$ rather than eyeballing a line. Constant factors, memory bandwidth, and branch behavior can dominate within one regime. Do not extrapolate beyond measured sizes without a model.

---

## Counters and profiling

Hardware counters measure cache references and misses, branch misses, instructions, cycles, allocations, page faults, and I/O. Compare counters per logical operation, not only total, and ensure counter multiplexing or sampling is understood. A linked list's time may be explained by cache misses; a tree's by branch misprediction; a hash table's by probes and memory stalls.

Profilers identify hot functions but can perturb timing. Use sampling for low overhead and instrumented counters for exact operation counts. Record compiler, runtime, optimization flags, CPU, memory, operating-system load, and allocator settings. A benchmark result without environment information is difficult to reproduce and easy to overinterpret.

<StructureDiagram kind="array" title="Resize spikes in a dynamic array" caption="Amortized cost is bounded per insertion, but individual resizes create latency spikes visible only in per-operation timing." variant="capacity" />

A geometric-resizing array has bounded average copying ($\hat{c} = O(1)$ per insertion), but an individual resize can pause a latency-sensitive request. Measure both total amortized throughput and the latency distribution. Incremental migration bounds per-operation work at the cost of two-table lookup and extra memory. A queue's average length can be low while a burst creates a large age tail; report queue length and item age.

Randomized structures require a probability statement. Repeat with independent seeds and report variation across seeds; distinguish randomization cost from key-distribution variation. If a security seed is secret, do not expose it in benchmark artifacts. For worst-case-sensitive services, include constructed adversarial cases alongside typical cases.

## Failure modes and counterexamples

A benchmark can mislead in several systematically identifiable ways:

<MathStatement
  id="counterexample-one-sample"
  kind="corollary"
  title="The single-sample fallacy"
>
A benchmark result obtained from one input, one seed, or one run cannot distinguish structural cost from measurement noise. Any conclusion drawn from a single sample is formally untestable.
</MathStatement>

<MathStatement
  id="counterexample-dead-code"
  kind="corollary"
  title="The dead-code trap"
>
A benchmark that does not consume its output may measure nothing. Compilers can legally eliminate dead computations whose results are unused, producing artificially low times for any structure whose operations are pure (produce no observable side effect).
</MathStatement>

**Concrete counterexample.** A benchmark comparing `ArrayList` and `LinkedList` in Java that inserts at the end and never iterates or indexes. The `LinkedList`'s tail pointer makes end-insertion $O(1)$, matching the array's amortized $O(1)$. The benchmark reports "no difference." But the operation mix — only append — tests only one operation. A benchmark that includes random access ($O(1)$ vs $O(n)$) or front insertion ($O(n)$ vs $O(1)$) reveals the structural asymmetry.

## Experimental threats to validity

<Quiz client:load
  question="A benchmark generates random keys inside the timed loop and reports 'lookup time.' What is the primary threat?"
  options={[
    "The measured time includes key generation and allocation, not just lookup",
    "Random keys are always slower",
    "The benchmark is too fast to measure",
    "There is no threat if the loop is large enough"
  ]}
  correctIndex={0}
  explanation="Input generation inside the timed region confounds the measurement: the generator and allocation cost are attributed to the structure. Pre-generate inputs when the measurement target is the lookup operation itself."
/>

- **Input generation confound.** Generating random keys inside the timed region measures the generator and allocation, not only lookup. Pre-generate inputs when measuring operations, but separately measure generation if it is part of the service.
- **Hot-cache bias.** Reusing one key keeps a hot cache line and overstates locality; randomizing each key overstates cold misses relative to production.
- **Dead-code elimination.** Consume results through a checksum or a black-box barrier. Verify that the checksum changes when the data changes. Conversely, validating every result during the timed loop may measure validation rather than the target operation.
- **System noise.** Background activity, frequency scaling, thermal throttling, allocator caches, and garbage collection create correlated noise. Pin processes or disable frequency changes to improve repeatability. Use robust summaries (medians, confidence intervals). Do not discard inconvenient runs without a predeclared rule.

1. **Transfer: cross-structure prediction.** Given a benchmark showing that a skip list is $2\times$ slower than a B-tree on range queries of size $10^4$ from a dataset of $10^7$ elements, predict the ratio when the range size changes to $10^2$ and $10^6$. State which structural mechanism (node fanout, pointer density, cache-line utilization) determines each prediction.

   <ExerciseSolution hint="Consider how B-tree fanout reduces tree height but also packs more keys per cache line; at small ranges, traversal depth dominates; at large ranges, sequential leaf scan matters.">
   At $10^2$, the B-tree may read 1–2 nodes (both cached), while the skip list follows $\log n$ pointers (cache misses dominate). At $10^6$, the B-tree's linked leaves scan sequentially (prefetch-friendly); the skip list scatters across levels. Predict ratios using the node-size and pointer-count models, then design the experiment.
   </ExerciseSolution>

2. **Synthesis: design an adversarial benchmark.** A persistent hash table claims $O(1)$ average lookup and $O(\log n)$ worst-case (cuckoo hashing with stash). Design a benchmark that distinguishes average-case from worst-case behavior under adversarial key selection, and specify the minimum number of trials needed to observe the worst case with probability $0.95$.

   <ExerciseSolution hint="Use the birthday paradox to bound the probability of a stash overflow; the worst case requires a specific set of colliding keys. The number of trials depends on the stash size and table load factor.">
   Generate keys via a known collision-producing pattern (e.g., same hash prefix). For cuckoo hashing with two tables and stash size $s$, the worst case occurs when more than $s$ keys hash to the same bucket pair. Compute the probability under random keys; design adversarial keys that guarantee this event. Report max displacement and stash size per trial, not only mean.
   </ExerciseSolution>

3. **Synthesis: measurement diagnostic.** A benchmark shows that a red-black tree's `insert` has the same latency as a hash table's `put` at $n=10^6$. List three distinct hypotheses that could explain this result, and design one additional measurement for each that would distinguish it from the others.

   <ExerciseSolution hint="Consider: (1) the hash table has high load factor causing long probe chains; (2) the tree fits in cache; (3) the benchmark measures string hashing or key comparison rather than structural operations.">
   Hypotheses: (a) hash-table load factor > 0.9, causing many probes; measure probe-length distribution. (b) tree nodes fit in L2 cache, hiding pointer-chasing cost; measure L2 cache misses per operation. (c) key-comparison cost dominates both structures; measure instructions per operation and compare string-hashing cost to structural cost.
   </ExerciseSolution>

4. **Transfer: generalize the protocol.** Adapt the benchmark protocol diagram above for a persistent data structure whose operations create new versions. What additional phases does the protocol need?

   <ExerciseSolution hint="Persistent structures add version-management overhead: path copying, root-pointer indirection, and garbage of old versions. The benchmark must separate query from update and measure both time and allocation per version.">
   Add phases: (a) build version DAG with known branching factor; (b) measure query on old and new versions separately (they share internal nodes); (c) measure allocation rate and retained bytes, not only query latency; (d) measure garbage-collection or reference-counting cost if the structure reclaims old nodes. Use a version-age distribution that matches the intended deployment.
   </ExerciseSolution>

5. **Recall: metadata checklist.** List the ten most commonly omitted metadata fields that prevent benchmark reproduction.

   <ExerciseSolution>
   (1) code revision / commit hash, (2) compiler version and optimization flags, (3) CPU model and microarchitecture, (4) memory size and speed, (5) OS version and kernel, (6) allocator and heap configuration, (7) input generator and seed, (8) operation mix and sequence length, (9) warm-up and repetition count, (10) measurement units and raw sample availability.
   </ExerciseSolution>

6. **Synthesis: confounding by allocation.** A benchmark comparing two B-tree implementations shows that implementation A is $3\times$ faster on insert. Suspecting that the allocator, not the tree structure, causes the difference, design a measurement that isolates allocation cost from tree-operation cost.

   <ExerciseSolution hint="Use a pre-allocated pool or region allocator so that the tree's node allocation costs are constant. Compare with and without the pool; the difference isolates allocation overhead.">
   Wrap each tree's node allocation with a counting allocator that records bytes requested and number of allocations. Compare (a) total time, (b) allocation time (via a custom allocator that timestamps malloc/free), and (c) tree-operation time = total − allocation. If the gap shrinks or reverses under a custom allocator, the original comparison was an allocator benchmark, not a tree benchmark.
   </ExerciseSolution>

## Counterexample: when the benchmark answers the wrong question

<Quiz client:load
  question="A pairing heap beats a binary heap on bulk insert-then-extract-min but loses on interleaved operations. What was the fundamental error?"
  options={[
    "The workload model did not match the deployment pattern",
    "The pairing heap is always slower",
    "The binary heap has a better constant factor",
    "The benchmark used the wrong programming language"
  ]}
  correctIndex={0}
  explanation="The benchmark used a bulk workload (all inserts, then all extract-mins), while the deployment used interleaved operations (insert-extract-insert-extract). The operation mix — the workload model — determines which structural costs dominate."
/>

A systems team compares two priority-queue implementations — a binary heap and a pairing heap — on $10^7$ inserts followed by $10^7$ extract-mins. The pairing heap wins by $1.2\times$. The team deploys the pairing heap in a latency-sensitive router.

The router's workload is interleaved: insert-extract-insert-extract, not bulk-insert-then-bulk-extract. Under interleaved access, the pairing heap's pointer manipulation and cache misses dominate, and the binary heap is $4\times$ faster. The benchmark answered a question about bulk throughput, but the deployment needed interleaved latency. The operation mix — the workload model — was wrong.

## Deliberately omitted

- **Statistical hypothesis testing**: $t$-tests, ANOVA, and bootstrapped confidence intervals for benchmark comparison. Important for rigorous comparison but belong in a statistics or experimental-design lesson rather than a data-structures benchmark guide.
- **Microarchitecture-level analysis**: pipeline stalls, branch misprediction rates, TLB misses, and prefetcher behavior. Processor performance-counter interfaces change across generations; a microarchitecture deep-dive would double the lesson length without changing the structural benchmarking message.
- **Energy and power benchmarking**: measuring energy per operation and thermal design power constraints. Increasingly relevant for data-center and mobile deployment, but the measurement methodology and interpretation differ from throughput/latency benchmarking.
- **A/B testing in production**: deploying two data-structure variants and comparing user-facing metrics. Involves traffic splitting, statistical power, and novelty effects that are deployment concerns rather than structure-analysis concerns.
- **Online experimentation and multi-armed bandits**: adaptive allocation of traffic to variants. Relevant to infrastructure teams but outside the scope of single-structure benchmarking.
