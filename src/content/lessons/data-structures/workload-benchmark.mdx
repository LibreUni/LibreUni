# Workload-Driven Benchmarking

Complexity analysis predicts scaling under a model; benchmarking measures an implementation on hardware and workload. Neither replaces the other. A benchmark can reveal allocator, cache, branch, and key-comparison costs, but it cannot prove an invariant or a worst-case bound. A theorem can prove O(log n) while the chosen layout loses to a linear scan at the target size. This lesson develops a disciplined bridge between formal guarantees and empirical decisions.

## Define the question

Start with a decision: choose between array and linked sequence, determine a hash-table threshold, evaluate a tree layout, or size a queue. State the operation mix, input size, key distribution, update/read ratio, memory budget, latency objective, and whether inputs are trusted. “Which is faster?” is not a reproducible question until these variables are fixed.

Separate throughput from latency. Throughput is completed operations per second over an interval; latency is a per-operation distribution. A structure can have high average throughput and unacceptable p99 latency because of resizing or garbage collection. Report median, tail percentiles, maximum with sample context, and failure or timeout counts. If operations have different costs, report each class rather than one blended number.

## Workload models

Synthetic workloads isolate mechanisms: sequential array scan measures locality; random lookup measures indexing; sorted insertion stresses unbalanced trees; repeated collisions stress hashing; alternating threshold updates test amortization. Trace-driven workloads preserve correlations from production but may contain privacy and stationarity issues. Use both when the goal is a robust choice rather than a single demonstration.

Generate distributions explicitly. Uniform random keys, Zipfian popularity, ascending keys, repeated hot keys, and adversarial prefixes exercise different paths. A random BST insertion benchmark supports an expected-height interpretation only if the permutation assumption is justified. A hash-table benchmark with random strings says little about a keyed service exposed to chosen inputs.

## Baselines and validation

Use a simple reference implementation to verify outputs and a baseline to contextualize time. For a dictionary, a sorted vector or trusted map can serve as a correctness oracle. For graph reachability, compare against BFS. Before timing, run small tests with known answers and compare checksums or operation results. Benchmark harness bugs—wrong operation counts, optimized-away results, accidental reuse, or invalid keys—can produce impressive but meaningless numbers.

Warm-up effects matter in managed runtimes and caches. Decide whether the benchmark measures cold start, steady state, or both. Separate construction from operations when appropriate, but also measure end-to-end cost if construction is part of the application. Randomize experiment order to reduce thermal drift or background-load correlation, and repeat enough times to estimate variance.

## Scaling and asymptotics

Measure several input sizes across relevant cache and memory boundaries. Plot time per operation and total time against n; a log-log plot can reveal an approximate exponent but does not prove asymptotic behavior. A dynamic array may show resize spikes; a tree may transition when it no longer fits in cache; a hash table may resize at a threshold. Explain these features using the representation and invariant.

For an O(n) scan, doubling n should roughly double work under fixed conditions; for O(log n), growth should be slow; for O(n log n), compare ratios against n log n rather than eyeballing a line. Constant factors, memory bandwidth, and branch behavior can dominate within one regime. Do not extrapolate beyond measured sizes without a model.

## Counters and profiling

Hardware counters can measure cache references/misses, branch misses, instructions, cycles, allocations, page faults, and I/O. Compare counters per logical operation, not only total, and ensure counter multiplexing or sampling is understood. A linked list’s time may be explained by cache misses; a tree’s by branch misprediction; a hash table’s by probes and memory stalls.

Profilers identify hot functions but can perturb timing. Use sampling for low overhead and instrumented counters for exact operation counts. Record compiler, runtime, optimization flags, CPU, memory, operating-system load, and allocator settings. A benchmark result without environment information is difficult to reproduce and easy to overinterpret.

## Amortized and tail behavior

A geometric array has bounded average copying but an individual resize can pause a latency-sensitive request. Measure both total amortized throughput and the latency distribution. Incremental migration can bound per-operation work at the cost of two-table lookup and extra memory. A queue’s average length can be low while a burst creates a large age tail; report queue length and item age.

Randomized structures require a probability statement. Repeat with independent seeds and report variation across seeds; distinguish randomization cost from key-distribution variation. If a security seed is secret, do not expose it in benchmark artifacts. For worst-case-sensitive services, include constructed adversarial cases alongside typical cases.

## Experimental threats to validity

A benchmark may confound the structure with input generation. Generating random keys inside the timed region measures the generator and allocation, not only lookup. Pre-generate inputs when measuring operations, but separately measure generation if it is part of the service. Reusing one key can keep a hot cache line and overstate locality; randomizing each key can overstate cold misses relative to production.

Dead-code elimination can remove a computation whose result is unused. Consume results through a checksum or a black-box barrier, and verify that the checksum changes when the data changes. Conversely, a benchmark that validates every result during the timed loop may measure validation rather than the target operation. Validate a sample or separate correctness phase when appropriate, then explain the choice.

Background activity, frequency scaling, thermal throttling, allocator caches, and garbage collection create correlated noise. Pinning a process or disabling frequency changes can improve repeatability but may reduce representativeness. Report whether the system is isolated, and use robust summaries such as medians and confidence intervals. Do not discard inconvenient runs without a predeclared rule.

## Benchmarking amortized structures

For a dynamic array, record both operation index and duration so resize spikes can be identified. Compare total time over a complete sequence with per-operation tail. For a hash table, mark resize and rehash events and report probe-length distribution. For a persistent structure, count allocated nodes and retained bytes, not only query time; old roots can create memory pressure invisible in a short lookup benchmark.

An incremental implementation may have lower maximum latency but higher steady-state overhead. Plot cumulative work and p99 latency together. If a service has a deadline, a small number of deadline misses may matter more than a modest throughput increase. The benchmark should therefore encode the decision criterion rather than presenting a leaderboard without context.

## Interpreting negative results

If two representations perform similarly, that is evidence only over the measured regime. It may mean a dominant cost hides their difference, the input is too small, or the implementation has a bottleneck unrelated to the structure. Change one factor—size, locality, key length, update ratio, or memory pressure—and test whether the predicted mechanism appears. A negative result can refine the model rather than invalidate the asymptotic analysis.

## Exercises

1. Design a benchmark comparing arrays and linked lists for three operation mixes.
2. Explain why a median alone is inadequate for a resizable structure.
3. Choose workloads for a hash table under trusted and adversarial keys.
4. Interpret a doubling input-size experiment for O(n), O(log n), and O(n log n).
5. List metadata required for reproducibility.
6. Design a correctness oracle for a persistent map benchmark.
7. Explain how cache counters could distinguish pointer chasing from comparison cost.

### Diagnostic solutions


1. Use append/scan, random get, and insert-after-known-position mixes; keep logical elements and operation counts equal; compare dynamic array, linked list, and perhaps chunked sequence. Validate output and report throughput, latency, allocation, and cache behavior.

2. A resize or garbage-collection pause may be rare but dominate p99 or p999. Report percentiles, maximum with sample size, and number/timing of expensive events; average or median can hide service-level violations.

3. Trusted: uniform, Zipfian, repeated, and realistic keys. Adversarial: chosen colliding prefixes or keys found by probing, under fixed and randomized seeds. Bound memory and observe chain/probe tails, not only mean lookup.

4. Compare measured ratios to model ratios: linear doubles, logarithmic increases by an additive constant, and n log n grows slightly more than double. Use multiple sizes and state cache/regime changes; one ratio is not proof.

5. Record code revision, compiler/runtime, flags, hardware, OS load, allocator, input generator and seed, operation mix, sizes, warm-up, repetitions, units, and correctness checks. Preserve raw samples and analysis scripts.

6. Execute the same version history against a model that copies a map for every update. Query old and new versions after random branching; compare key/value membership and iteration according to the specified order. Retain seeds and shrink failing DAGs.

7. If pointer chasing dominates, cache misses and stalled cycles rise per logical step while comparisons remain low; if comparison cost dominates, instructions and branch/comparison counts rise with fewer misses. Compare layouts with identical comparisons to isolate locality.
