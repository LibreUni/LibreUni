# Database Assessment

This is an integrated examination of database design. Treat the case below as a specification, not as an invitation to produce a collection of unrelated SQL snippets. Every decision must connect a logical invariant to a physical mechanism and an operational consequence. A high-scoring answer can be implemented, tested, and defended under failure.

## Case study and contract

Build the data model for a multi-tenant marketplace. A tenant owns stores; a store sells products; customers place orders; each order contains one or more lines; inventory is held per warehouse; payments and shipments progress through states. Product prices at order time must not change when the catalog price changes. An order belongs to exactly one tenant and must never expose data from another tenant. A SKU may be stocked in several warehouses, but total available stock must never be negative. A payment provider may send the same webhook repeatedly and in a different order from the request that created a payment.

State assumptions: identifiers are generated by the database, timestamps are recorded in UTC, money uses integer minor units or an exact numeric type, and application retries can repeat a complete transaction. Specify whether product deletion is forbidden, soft, or cascading; “the database handles it” is not a specification. Specify the consistency target for inventory, order status, search indexing, analytics, and payment display. These targets need not all be identical, but each relaxation requires a repair path.

## Submission and evidence

Submit five parts. Part A is a normalized schema with keys, foreign keys, checks, uniqueness rules, nullability, and explicit tenant ownership. Part B is a query portfolio with relational-algebra reasoning, SQL, expected result grain, NULL behavior, and tests for empty, duplicate, and cross-tenant cases. Part C is a physical design: access paths, cardinality assumptions, representative `EXPLAIN` plans, write costs, and a benchmark method. Part D is a transaction/recovery dossier containing histories, isolation levels, lock or version behavior, WAL/backup assumptions, and retry semantics. Part E is security and operations: authorization, row filtering, secrets, audit events, migration rollback, replication, and an incident runbook.

Evidence must be reproducible. Include DDL and seed data small enough to inspect by hand, an independently calculated oracle for at least three query results, plan output with database version and relevant settings, and a test matrix. A successful query is not proof of tenant isolation; include a negative test using a tenant identifier supplied by an attacker. A benchmark without scale, warm-up, concurrency, and distribution assumptions is not a performance result. Label claims as proved by constraints, argued from a model, or measured experimentally.

## Part A: logical design

Start with functional dependencies. For an order line, `(order_id, line_no) - greater than  product_id, quantity, price_minor`; `product_id - greater than  current catalog attributes`, but not the historical order price. Explain why the latter must be copied into the line or represented by an immutable price version. Identify candidate keys and show that a surrogate key does not eliminate natural uniqueness requirements such as `(tenant_id, sku)`.

Normalize the core transactional relations to at least 3NF and discuss BCNF exceptions. A denormalized `order_total` is acceptable only if you state whether it is authoritative, how it is recomputed, and how updates remain atomic. Prove lossless decomposition for one nontrivial split using the chase or a dependency argument. Show dependency preservation where practical, and name a dependency that is enforced procedurally if it cannot be expressed as a row-local constraint.

Write constraints for positive quantities, legal status transitions, unique webhook event identifiers, and tenant-consistent references. A foreign key from `order_line(order_id)` to `order(tenant_id, order_id)` is stronger than separate foreign keys that can be mixed across tenants. Explain deferrable versus immediate constraint checking for a case where two rows are swapped. Explain why a `CHECK` constraint cannot safely contain a query over arbitrary sibling rows in ordinary relational systems.

## Part B: query portfolio

Provide SQL and reasoning for these queries: (1) a tenant-scoped order detail with lines and historical prices; (2) daily revenue including days with zero orders; (3) the top three products per store by rolling 30-day revenue; (4) warehouses whose available stock is below reorder point; (5) customers with no completed order; (6) payment webhook events that are duplicates or out of order; and (7) an audit report that must show an empty result when the caller has no permitted tenant.

For every query, state the grain before writing SQL. The revenue query has one row per tenant and calendar day, so generate the day relation and left join aggregates; placing a date predicate in the `WHERE` clause after a left join can accidentally remove zero rows. For the top-three query, use a window function after aggregating at product grain; explain the tie policy for `rank`, `dense_rank`, and `row_number`. For “no completed order,” use `NOT EXISTS` rather than an unexamined `NOT IN`, because a nullable subquery value can change three-valued logic.

Give one relational-algebra expression for the order-detail query and identify selections that can be pushed below joins. Explain why a selection on a nullable status may not be equivalent to a careless rewrite using negation. Include duplicate semantics: SQL bags preserve multiplicity unless `DISTINCT` is requested, and a join can multiply a customer row by order lines. Test duplicate line input, missing payments, NULL shipment dates, and two tenants with the same SKU.

## Part C: physical design

For each workload, choose indexes by leading equality and range predicates, not by column popularity. A likely order access path begins with `(tenant_id, created_at)` for tenant history, while line lookup needs `(order_id)` or a composite key. Inventory may need `(tenant_id, warehouse_id, sku)` and a uniqueness rule. Explain the extra work on insert, update, vacuum, and replication for each index. Do not recommend an index that is never used without showing its measured or estimated benefit.

Read a plan bottom-up. Estimate rows after each filter, identify whether a sequential scan, index scan, bitmap scan, hash join, merge join, or nested loop is selected, and compare estimated to actual rows. A nested loop can be excellent for a highly selective outer relation and terrible when the outer estimate is wrong. Explain stale statistics, skew, correlation, and parameter-sensitive plans. Compare a covering index with a heap lookup and discuss whether the reduced I/O is worth maintenance cost.

Describe one physical design for a large append-heavy order table: partitioning by time or tenant, retention policy, indexes per partition, and the failure mode of a partition-pruning assumption. Describe how you would verify a proposed partition key. Partitioning is not automatically sharding; state what remains on one database node and how cross-partition joins or uniqueness are handled.

## Part D: transactions and recovery

Write a transaction for placing an order. It must validate tenant and product ownership, reserve inventory without lost updates, insert immutable price snapshots, and create an outbox event. State the isolation level and the exact predicate or row protected. Show the outcome when two buyers compete for the last unit, including affected-row checks or a serialization retry. Do not hold database locks while calling the payment provider.

Analyse a history with two concurrent order placements and one inventory adjustment. Draw the precedence graph, identify whether the history is conflict-serializable, and draw the wait-for graph if locks are used. Give a write-skew counterexample involving a per-tenant spending limit and repair it with a summary row, serializable validation, or a different invariant representation. Define which client errors are safe to retry and which external operations require idempotency keys.

For recovery, distinguish WAL, a full backup, an incremental backup, a streaming replica, and a logical event log. Give an RPO/RTO target and show which combination can meet it. Explain point-in-time recovery after a bad migration: preserve the base backup and WAL, identify the target time, restore to an isolated environment, verify invariants, and decide whether to export corrected rows or promote a recovered copy. A replica is not a backup if corruption or an accidental delete is replicated immediately.

## Part E: security and operations

Threat-model a tenant API, an analyst role, an administrator, a compromised application credential, and a malicious webhook sender. Apply least privilege, parameterized queries, TLS, secret rotation, audit logging, and authorization at a layer that cannot be bypassed by an alternate query path. Row-level security can provide defense in depth, but explain how session identity is established and why a privileged owner or maintenance role may bypass a policy.

Design a migration adding a non-null column to a large table. Separate expand, backfill, validate, and contract phases; make the application tolerate both schemas during rollout; bound lock duration; and define rollback. Include a drill for failed replication, a disk-full incident, a long-running transaction blocking vacuum, and a leaked credential. For each incident state detection signal, immediate containment, data-integrity check, recovery action, and post-incident test.

## Diagnostic rubric and exercises

An excellent answer earns credit for invariant precision (25%), relational correctness and edge-case tests (20%), physical reasoning and measured plans (20%), concurrency and recovery analysis (20%), and security/operations (15%). A diagram without constraints, SQL without a stated grain, an index without a workload, or a backup policy without restore evidence is incomplete. A plausible answer that cannot explain a counterexample should lose more credit than a modest design whose limits are explicit.

1. Show the smallest counterexample to a design that stores only `order_total` and mutable `product.price`.
2. Write a query that returns the top three products per store and explain ties and duplicate joins.
3. Give a transaction history that violates the spending-limit invariant under snapshot isolation.
4. Choose indexes for tenant order history and warehouse stock; justify leading columns and write cost.
5. Classify a constraint violation, deadlock, lost update, corrupted page, accidental delete, and leaked password by the primary prevention and recovery mechanism.
6. Design an idempotent webhook consumer for an event that may arrive twice and out of order.
7. Propose a restore test that demonstrates the stated RPO and RTO rather than merely checking that a backup file exists.

### Diagnostic solutions


For 1, changing the catalog price retroactively changes historical reporting, and independently updating total can disagree with its lines. Store an immutable line price and derive or atomically maintain the total. For 2, aggregate first at `(store, product)`, then rank within store; join product names only after the grain is stable. Use `row_number` for exactly three rows or `rank` if ties should all appear. For 3, two transactions each read a limit with the other uncommitted, then spend; a summary row lock or serializable retry creates a conflict. For 4, lead with tenant and the common equality/range path; verify selectivity and plan use rather than indexing every filter.

For 5, constraints prevent invalid states, locking/serializable transactions prevent concurrency anomalies, WAL and backups recover durable data, and credential rotation plus authorization limits a secret leak. These mechanisms overlap but do not substitute for one another. For 6, store the provider event ID under a unique constraint, apply only monotonic state transitions, and acknowledge duplicates after proving the desired state is already recorded. For 7, restore to a clean environment, replay workload and WAL to a timestamp, measure elapsed recovery, compare checksums and business invariants, and record the maximum accepted data loss.


**Exercise.** Mark which failure is repaired by a constraint, transaction, log, backup, replica, or access policy.
