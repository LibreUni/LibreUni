# Functional Dependencies

A functional dependency $Xto Y$ says equal values on attributes $X$ imply equal values on $Y$. Attribute closure computes what a determinant implies under a dependency set. A candidate key is a minimal determinant of all attributes.

If `course_id → instructor` and a table key is `(student_id, course_id)`, repeating instructor per enrollment creates update anomalies.

## Formal definitions

For relation schema R and attribute sets X,Y subset R, a functional dependency X - greater than  Y holds on an instance r if for every pair of tuples t and u in r, t[X]=u[X] implies t[Y]=u[Y]. It is a semantic constraint about all valid instances, not a pattern observed in one sample. A dependency can hold because of the application domain even when the current table contains no repeated determinant values. Conversely, a coincidental uniqueness in a small sample does not prove a dependency.

Trivial dependencies are those where Y is a subset of X; they hold in every relation. A nontrivial dependency has at least one attribute in Y outside X. A dependency set F implies another dependency f if every instance satisfying F also satisfies f. Reasoning about implication is central to keys, normalization, and schema review.

## Armstrong’s axioms

Armstrong’s axioms are sound and complete for functional-dependency implication. Reflexivity says if Y is a subset of X, then X - greater than  Y. Augmentation says X - greater than  Y implies XZ - greater than  YZ. Transitivity says X - greater than  Y and Y - greater than  Z imply X - greater than  Z. Derived rules include union, decomposition, and pseudotransitivity. Soundness means every derived dependency holds whenever the premises hold; completeness means every implied dependency can be derived.

Decomposition is useful when a dependency has multiple attributes on the right: X - greater than  YZ implies X - greater than  Y and X - greater than  Z. Union combines X - greater than  Y and X - greater than  Z into X - greater than  YZ. Pseudotransitivity combines X - greater than  Y and WY - greater than  Z to infer WX - greater than  Z. Do not use converse rules without proof; Y - greater than  X does not follow from X - greater than  Y, and a dependency’s determinant is not interchangeable with its dependent set.

## Attribute closure

The closure X+ under F is the set of attributes functionally determined by X. Compute it by starting with X and repeatedly adding every right-hand attribute of a dependency whose left side is already contained in the current set. Termination follows because the finite attribute universe can only grow. The result is independent of dependency application order because the algorithm reaches the least fixed point.

To test whether X is a superkey for R, compute X+ and check whether it contains all attributes of R. To test whether X - greater than  Y follows from F, check whether Y is contained in X+. To test candidate-key minimality, remove each attribute of X and recompute closure; if any removal still determines R, X was not minimal. A key can contain attributes never appearing on dependency right sides, because those attributes cannot be derived from other attributes and may be necessary.

For the exercise dependencies A- greater than B, B- greater than C, and CD- greater than E, the closure of ()A() is ()A,B,C(). It does not contain D, so A is not a determinant of E. Adding D gives ()A,D,B,C(), then CD- greater than E applies and the closure contains E. Thus AD determines A,B,C,D,E under the given universe, and no smaller subset of AD works if D cannot be derived from A.

## Minimal covers

A minimal or canonical cover is an equivalent dependency set with one attribute on each right-hand side, no extraneous attribute on any left-hand side, and no redundant dependency. First decompose right sides. To test whether an attribute a in X is extraneous in X - greater than  A, compute (X-()a())+ under the current dependency set; if A is included, remove a. To test whether a dependency is redundant, compute its right side from its left side using all other dependencies.

Minimal covers are not necessarily unique, although they represent the same implication closure. The order of removals can affect the syntactic result, so record the equivalence argument rather than claiming one canonical spelling. Minimal covers support the 3NF synthesis algorithm and make dependency preservation calculations manageable.

## Keys and schema design

Given F and relation R, attributes that never appear on a right side must occur in every candidate key unless they are already in the determinant used. Start with their set, compute closure, and add attributes until all R is reached. Then minimize. Multiple candidate keys create multiple prime attributes and affect 3NF and BCNF tests. A surrogate key does not erase dependencies involving natural attributes; it may hide a business uniqueness constraint if the schema omits it.

Consider Registration(student, course, term, instructor, room). Dependencies may include (student,course,term)- greater than instructor, (course,term)- greater than instructor, and (course,term)- greater than room. The enrollment key is student,course,term, but course,term determines instructor and room, creating repeated facts across students. If instructors change, many rows require synchronized updates. The dependency analysis exposes the proper decomposition into CourseOffering(course,term,instructor,room) and Registration(student,course,term).

## Anomalies and dependency evidence

Update anomaly occurs when one fact is repeated and only some copies change. Insertion anomaly occurs when a fact cannot be stored without an unrelated fact, such as a course offering requiring at least one enrollment. Deletion anomaly occurs when removing the last enrollment also removes course or instructor information. These are symptoms of a relation whose attributes mix multiple entity or relationship facts.

Dependencies should be justified by domain rules, constraints, or authoritative data—not inferred only from current rows. If an instructor can vary by section, course_id - greater than  instructor is false and the determinant must include section or term. If email uniqueness is conditional by tenant, the dependency includes tenant. If history matters, time or version belongs in the key and dependency statement.

### Diagnostic solutions


1. Start with the determinant, repeatedly apply dependencies whose left sides are contained, and stop at a fixed point. A superkey closure contains all attributes; test each proper subset for minimality.
2. From A- greater than B, augment with C to get AC- greater than BC; from BC- greater than D, use the B component with appropriate decomposition and transitivity only when the left side matches. Every step must name the axiom or derived rule.
3. For a dependency XA- greater than B, compute X+ under F. If B is already present, A is extraneous; otherwise retain it. Recompute after other removals because extraneousness can change.
4. Split right sides, remove left-side attributes whose deletion preserves implication, then remove dependencies implied by the others. Verify the resulting closure matches the original cover’s implication for each original dependency.
5. A generated ID may uniquely identify rows while two rows still share the same email, course code, or external identifier. Declare the natural uniqueness constraint if the domain requires it.
6. Repeated instructor data may require many updates; a course cannot be inserted without an enrollment; deleting the last enrollment may erase the only course information. Decompose facts with their determinants.



## Dependency-analysis workflow

When given a schema and dependencies, write the attribute universe first and make clear whether dependencies are stated for a current snapshot or for every legal instance. Normalize right sides, compute a minimal cover, find all candidate keys, and then test each dependency against the chosen normal form. Keep a record of closures rather than relying on intuition: a determinant that looks like an identifier may fail to determine an attribute because a time, tenant, or version dimension is missing.

Use counterinstances to challenge a claimed dependency. To show X - greater than  Y does not follow from F, construct two tuples agreeing on X but differing on Y while satisfying the dependencies, or prove the target attribute is absent from X+. To show a candidate decomposition is motivated, state which repeated fact each determinant represents and which anomalies disappear. Functional dependencies are a design reasoning tool, not merely a mechanical normalization exercise.

## Exam practice

For every closure, show each added attribute and the dependency that justified it. For every candidate key, show all attributes are determined and every single-attribute deletion fails. For every minimal cover, verify equivalence in both directions or at least that each original dependency is implied and each reduced dependency follows from the original. This evidence makes a solution auditable.

Test the resulting schema with duplicate, update, and deletion fixtures, including NULL and tenant or time boundaries where the dependency’s scope changes.

Record those fixtures as regression tests.

Review them after schema changes.

Keep the proof with the migration.

And rerun it.
