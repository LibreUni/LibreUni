# Lifetime Annotations

Every reference in Rust has a **lifetime**: the scope for which that reference is valid. Lifetimes ensure references never outlive the data they point to. Most lifetimes are inferred; annotations clarify relationships when the compiler cannot deduce them.

{/* Source tracking: Lifetimes synthesized from "The Rust Programming Language" by Klabnik & Nichols, Ch. 10.3 (MIT/Apache-2.0) */}

## Why Lifetimes Exist

```rust
let r;                  // ---------+-- 'a
{                       //          |
    let x = 5;          // -+-- 'b  |
    r = &x;             //  |       |
}                       // -+       |
println!("{}", r);     //          |
```

Here, `r` references `x` but `x` goes out of scope while `r` is still alive. The borrow checker rejects this with: "borrowed value does not live long enough."

## Lifetime Syntax

Lifetime annotations use an apostrophe: `'a`, `'b`, etc. They don't change how long a value lives—they describe relationships between lifetimes.

```rust
// &i32        — a reference
// &'a i32     — a reference with an explicit lifetime
// &'a mut i32 — a mutable reference with an explicit lifetime

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

This says: the returned reference lives as long as the *shorter* of the two inputs.

## Lifetime Elision

In many cases, lifetimes follow predictable patterns. The compiler applies **elision rules** to avoid redundant annotations:

```rust
// Elided (inferred):
fn first_word(s: &str) -> &str {}

// Desugared:
fn first_word<'a>(s: &'a str) -> &'a str {}
```

The three elision rules:
1. Each input reference gets its own lifetime.
2. If there's exactly one input lifetime, it's assigned to all output references.
3. If `&self` or `&mut self` is among the inputs, its lifetime is assigned to all output references.

## Lifetimes in Structs

Structs can hold references, but every reference field needs a lifetime annotation:

```rust
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().expect("Could not find a '.'");
    let excerpt = Excerpt { part: first_sentence };
    // excerpt cannot outlive novel
}
```

## The Static Lifetime

`'static` denotes a lifetime that lives for the entire program duration. String literals have type `&'static str`:

```rust
let s: &'static str = "I live forever";
```

Be cautious with `'static` bounds—they constrain generic parameters to require values that live forever, which is often overly restrictive.
